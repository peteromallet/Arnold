# Human blockers — M3 (drivers + realized-graph + state-evolution) & M4 (services + policy spine)

Goal: drive human blockers to zero. Every design-time decision point and every runtime human-gate
below is pre-made as a DEFAULT, a MACHINE-GATE, or an AUTO-ESCALATION policy. `must_ask_peter=false`
everywhere; no genuinely irreversible taste call surfaced in these two briefs.

Source: `briefs/epic-pipeline-unification/m3-drivers-state.md`, `m4-services.md`.

## M3 — DESIGN/BUILD-TIME blockers

### M3-D1 — Snapshot granularity (whole-blob vs diff)
- **Location:** m3 Open questions L138-141.
- **Kind:** design-time.
- **Decision (default):** whole-blob copy of `state.json`. Brief already marks it *non-blocking — pick whole-blob*.
- **Rationale:** matches LWW blob surgery in `write_plan_state`; honest first cut; diff is a later optimization, reversible.

### M3-D2 — Mid-run re-realization + live resume cursor (biggest design unknown)
- **Location:** m3 Open questions L142-145; Outcome #2; Done #2.
- **Kind:** design-time.
- **Decision (machine-gate):** Done-criterion #2 IS the gate — `build_topology` rebuilt after a simulated
  `set-robustness` must yield a graph whose `predecessors(stage)` reproduces `_BLOCKED_RECOVERY_STATES` /
  `_RESUME_ACTIVE_STATES` exactly, with the resume cursor still resolving to a valid (non-deleted) node.
  Fold is deterministic (ordered cumulative over `_ROBUSTNESS_WORKFLOW_LEVELS`); cursor-validity is an
  assertion in the test. On red the milestone halts (test fails), never waits on a person.
- **Rationale:** determinism + the predecessor-equality test convert the "is the cursor still valid?" worry
  into an automated pass/fail; no human adjudication needed.

### M3-D3 — Where the reversible backend persists snapshots
- **Location:** m3 Open questions L146-148.
- **Kind:** design-time.
- **Decision (default):** sidecar dir under `plan_dir` at `.state-versions/<id>.json`, written inside the
  per-plan `flock`. A name-collision check against the executor forensic-backup naming
  (`executor.py:90-133`) is a unit assertion.
- **Rationale:** stays under the existing lock (a2 concurrency safe); collision avoidance is mechanizable as
  a test, not a judgment call.

### M3-D4 — Budget across fan-out shards
- **Location:** m3 Open questions L149-152.
- **Kind:** design-time.
- **Decision (default):** ship budget single-tenant / per-run, in-process only. Cross-shard folding is
  explicitly deferred to M4/M-fanout. Brief says *Flag, don't solve*.
- **Rationale:** the brief pre-scopes this; M4's budget authority (#5b) is where folding lands. Deferral is
  reversible and already sequenced.

### M3-D5 — Event-sourced Store: "one Store" (EPIC L33) vs "irreconcilable" (M4:39)
- **Location:** m3 Locked decisions L126-127; Scope L97-100.
- **Kind:** design-time.
- **Decision (default):** resolved toward interfaces-with-backends — event-sourced is *declared on the axis
  enum and scaffolded behind the interface, NO real backend in M3*. Only forward-only + reversible get real
  backends.
- **Rationale:** already locked in the brief; removes the contradiction without shipping the irreconcilable
  contract.

### M3-D6 — Cloud `_phase_command` shim placement (M3 vs M5/M6 seam)
- **Location:** m3 Scope L101-105; Done #7.
- **Kind:** design-time.
- **Decision (default + machine-gate):** land the guarded shim in M3 when the process driver is born; wire a
  cloud smoke oracle into the chain. Done-criterion #7 (cloud resume resolves a phase command through the
  shim + smoke run passes) is the green/red gate.
- **Rationale:** brief assigns it to M3 explicitly; the smoke oracle auto-verifies, no operator sign-off.

### M3-D7 — Acceptance toy choice (which non-planning package)
- **Location:** m3 Scope L106-110; Done #6.
- **Kind:** design-time.
- **Decision (default):** a tiny backtracking constraint-solver (brief's own example), with zero planning
  imports and all inter-step data crossing a declared Port. Green = Done #6.
- **Rationale:** any toy exercising loop+restore+budget satisfies the criterion; the solver is named in-brief
  and is the cheapest qualifying choice.

## M3 — RUNTIME blockers (baked into the product)

### M3-R1 — `set-robustness` override mid-run (operator-triggered re-realization)
- **Location:** m3 Scope L56-57; override.py 9 actions; Touchpoints L222-223.
- **Kind:** runtime.
- **Decision (machine-gate):** treat `set-robustness` as a state-mutating override that *rebuilds the graph
  and re-validates the resume cursor automatically*; if the rebuilt cursor would point at a deleted node,
  the override is rejected with a typed error and the live graph is left untouched (auto-halt-safe, no
  partial mutation). No human confirmation of the rebuild.
- **Rationale:** the cursor-validity invariant (M3-D2 test) is enforced at runtime too; the override either
  applies cleanly or is rejected — never parks waiting for approval.

### M3-R2 — `restorable_boundary` composed with process / fan-out (fail-LOUD)
- **Location:** m3 Outcome #3; Constraints L164-169; Done #5.
- **Kind:** runtime.
- **Decision (machine-gate):** declared invariant — composing `reversible` with `subprocess_isolated` or
  fan-out **raises LOUD at compose time**, auto-halting the run with a typed error. This is the desired
  behaviour, not a human gate.
- **Rationale:** whole-blob restore rolls back the RECORD not the WORLD; the loud failure is the safe default
  and is fully automated.

### M3-R3 — Loop-node teardown / max-iterations / budget-exhaustion exit
- **Location:** m3 Scope L84-90; Done #4.
- **Kind:** runtime.
- **Decision (default + auto-escalation):** mandatory `max_iterations` cap fires automatically when the
  predicate never satisfies; `finally`/teardown runs on EVERY exit path (normal/cap/exception/budget). No
  human decides when to stop the loop — the cap and budget do.
- **Rationale:** the cap is a hard, declared bound; teardown-on-all-paths is a tested invariant. The loop can
  never hang waiting on a person.

### M3-R4 — Crash under `in_process` substrate (run-down vs containment)
- **Location:** m3 Constraints L156-163; Done #3.
- **Kind:** runtime.
- **Decision (default, documented trade):** under `in_process`, a crashing step takes the run down — asserted
  as the documented trade, NOT a regression. Under `subprocess_isolated`, the step is killed via
  `kill_group` (SIGTERM→SIGKILL) on idle/wall timeout and surfaced as a contained per-step failure; parent
  survives, group reaped.
- **Rationale:** substrate choice is the decision; both behaviours are pre-specified and tested. The
  containment path auto-recovers; the in-process path is an accepted, documented property.

### M3-R5 — Reverse-recovery query (recover-blocked returns `[]`)
- **Location:** m3 Scope L69-74; a3 §3.
- **Kind:** runtime.
- **Decision (default):** recovery handlers query `predecessors(stage)` on the single realized graph; the
  graph returns `[]` for a blocked stage (the documented contract). Handlers proceed on the queried map with
  no human reverse-map maintenance.
- **Rationale:** single-source derivation removes the three hand-maintained copies; the `[]`-for-blocked
  contract is pinned by Done #2.

## M4 — DESIGN/BUILD-TIME blockers

### M4-D1 — `RecoveryPolicy` granularity vs the realized graph
- **Location:** m4 Open questions #1 (L51); Anti-scope L94.
- **Kind:** design-time.
- **Decision (default):** `classify` consumes only `ExitKind` + error-layer and emits a **target-agnostic**
  `halt(kind)` / `escalate`. It MAY query M3's `predecessors(stage)` to *name* a recover target, but must
  NOT inherit planning's `STATE_*` (that vocabulary is M5c).
- **Rationale:** keeping `halt(kind)` target-agnostic is the locked Theme-D boundary; the optional graph
  query is additive and reversible.

### M4-D2 — Budget authority home (`runtime/` vs dispatch services bag)
- **Location:** m4 Open questions #2 (L52).
- **Kind:** design-time.
- **Decision (default):** home the single budget/cost authority in `runtime/` next to `key_pool.py`, backed
  by an `fcntl.flock`'d on-disk ledger (`~/.megaplan/keypool.lock`), since rate + spend are the same
  shared-depletable-resource problem (m4 #5b is explicit about the broker home).
- **Rationale:** co-locating with the existing key/rate broker is what #5b already prescribes; "services bag"
  would re-split the resource the brief is unifying.

### M4-D3 — Global budget quota coverage (which dispatch paths)
- **Location:** m4 Open questions #2 (L52); Anti-scope L96.
- **Kind:** design-time.
- **Decision (default):** the global budget covers the Hermes/OpenRouter `acquire_key` path; Codex/Shannon
  subscription quotas are explicitly OUT (separate shared semaphore, not scoped here, per Anti-scope).
- **Rationale:** matches the a2 residual; scoping to the pooled-key path keeps the authority's contract clean
  and is what the brief already carves out.

### M4-D4 — Spend-folding source across fan-out shards (journal vs live accumulator)
- **Location:** m4 Open questions #2 (L52), #3 (L53).
- **Kind:** design-time.
- **Decision (default):** fold via a **live in-broker accumulator** (so the cap can actually stop a runaway
  composition), with the post-hoc attribution journal (#1) reconciling after. Done #4 requires a *live*
  shared spend budget that stops at cap.
- **Rationale:** the post-hoc journal cannot stop live spend (Open Q #3 says the authority must be live);
  Done #4 forces the live accumulator, so the journal is reconciliation-only.

### M4-D5 — `CostTracker` cap reads the authority (not just `state.meta`)
- **Location:** m4 Open questions #3 (L53); Touchpoints `_pipeline/runtime.py:58`.
- **Kind:** design-time.
- **Decision (machine-gate):** `CostTracker` cap MUST read the live authority (covering plan + non-plan +
  fan-out), verified by Done #4's two-tenant shared-budget-stops-at-cap test. On red, milestone halts.
- **Rationale:** Open Q #3 frames this as a confirmation; the two-tenant test mechanizes the confirmation.

### M4-D6 — Cost rollup `--dispatch` flag: opt-in vs default
- **Location:** m4 Open questions #3 (L53); d1 §3.
- **Kind:** design-time.
- **Decision (default):** opt-in via the `--dispatch` flag (d1 §3 default), so the existing rollup output is
  unchanged for current callers (back-compat constraint).
- **Rationale:** opt-in preserves byte-identical single-process behaviour and current CLI output; flipping to
  default later is non-breaking.

### M4-D7 — Liveness sink for a non-plan tenant (callback vs plan_dir-shaped scratch)
- **Location:** m4 Open questions #4 (L54); Scope #1 (L25); a4 option b.
- **Kind:** design-time.
- **Decision (default):** the injected-callback form (a4 option b) is sufficient as the primary sink; a
  `plan_dir`-shaped scratch dir is supported as an alternate when a supervisor polls `state.json`. The
  service owns one `set_active_step` entry point so plan, loop, and non-plan callers all pass one place.
- **Rationale:** callback covers the non-plan tenant without forcing a fake plan_dir; the scratch-dir option
  remains for poll-based supervisors — both are wired, no either/or human call. Resolve-before-locking is
  satisfied by supporting both.

### M4-D8 — Oracle's first real non-planning branching user
- **Location:** m4 Open questions #5 (L55); Done #7; Acceptance L18.
- **Kind:** design-time.
- **Decision (default):** ship the git-bisect / red-blue style `run(cmd)`-branching consumer from acceptance
  test #1 as the oracle's first user (any one of the three sketches qualifies; pick git-bisect as the
  concrete default). Done #7 requires "one real branching user" — non-zero, asserted.
- **Rationale:** the brief forbids shipping oracle with zero users; git-bisect is the cleanest single
  branching consumer and is already sketched.

### M4-D9 — Per-tenant budget partition field (reserve now vs build now)
- **Location:** m4 #5b L36; Anti-scope L97.
- **Kind:** design-time.
- **Decision (default):** RESERVE the per-tenant sub-budget field in the ledger schema now; do NOT build the
  partitioning logic (that is M6 trust-boundary work).
- **Rationale:** reserving avoids a later migration; building it now is out of scope per Anti-scope.

### M4-D10 — Event envelope schema version + payload validation strictness
- **Location:** m4 Scope #2 (L28-29); Done #5; Constraints L60.
- **Kind:** design-time.
- **Decision (default):** add `schema_version: 1`; extract envelope + per-kind payload Pydantic models;

- **Rationale:** report-only matches the cross-cutting constraint; readers' call sites stay unchanged (Done

## M4 — RUNTIME blockers (baked into the product)

### M4-R1 — `RecoveryPolicy.classify` → {retry_fresh | retry_transient | escalate | halt(kind)}
- **Location:** m4 #5a (L35); Done #3.
- **Kind:** runtime. **This is THE core auto-escalation policy.**
- **Decision (auto-escalation):** every error is classified automatically: transient → `retry_transient`
  (bounded by per-class budget), context-exhaustion → `retry_fresh` (bounded by `max_context_retries`),
  unrecoverable-but-routable → `escalate` (to a stronger model / recover target), terminal → `halt(kind)`
  (target-agnostic, typed). Per-class budgets cap retries; on exhaustion it escalates then halts — never
  parks on a human. auto.py's three loops become bindings over this ONE policy.
- **Rationale:** this is the literal retry → escalate → halt ladder the conversion principle asks for; the
  per-class budgets guarantee termination without human intervention.

### M4-R2 — `halt(kind)` semantics — does halt wait for a human?
- **Location:** m4 #5a; Anti-scope L94 (target-agnostic); Open Q #1.
- **Kind:** runtime.
- **Decision (default):** `halt(kind)` is a *terminal, recorded run outcome* — it stops the run and emits the
  reason via the observability contract; it does NOT block awaiting a person. The `awaiting_human` /
  control-target vocabulary is explicitly M5c, NOT M4 — so M4 ships no human-wait state.
- **Rationale:** Anti-scope forbids the run-outcome/control-target vocabulary here; halt in M4 is a clean
  terminal signal an automated supervisor consumes. Any human-facing gate is deferred to M5c, where it too
  must be defaulted/gated.

### M4-R3 — Liveness / stall detection on dispatch (hermes has no watchdog)
- **Location:** m4 Scope #1 (L25); Done #2; Constraints L61 (highest-severity slice).
- **Kind:** runtime.
- **Decision (machine-gate):** the service MUST guarantee `active_step` + liveness sink for hermes at
  dispatch entry; liveness is written on token progress and NOT on silence. Done #2 is the regression guard
  (current `run_loop_worker` fails it today). A supervisor polls the sink and auto-kills/auto-retries a
  stalled step via the watchdog bounds (`idle_timeout`, `pre_first_byte_timeout`) — no human watches.
- **Rationale:** this is the a4 highest-severity slice; the sink + watchdog convert "is the step stuck?" into
  an automated kill/retry, never a human watch.

### M4-R4 — Two-tenant shared key/budget contention (double-issue + shared backoff)
- **Location:** m4 #5b; Done #4; a2.
- **Kind:** runtime.
- **Decision (machine-gate):** the `fcntl.flock`'d on-disk ledger guarantees two concurrent tenants do NOT
  double-issue a cooled key and observe a shared backoff + shared spend cap that stops the composition. Done
  #4 is the gate. Falls back to in-process pool (byte-identical) when no shared ledger configured.
- **Rationale:** the cross-process flock + wall-clock cooldown is the mechanism; contention is resolved by
  the lock, not by an operator arbitrating tenants.

### M4-R5 — Attestation (notary) verdict — does a failed attestation need human review?
- **Location:** m4 Scope #3 (L31); Done #7.
- **Kind:** runtime.
- **Decision (default):** attestation is a NOTARY — it *asserts a claim holds and never branches flow*. A
  failed attestation is surfaced as a typed result the planning binding consumes (its existing judging
  logic), not a stop-and-ask-human gate. The SDK piece is the injected `git`/`audit`/`none` strategy seam.
- **Rationale:** by definition attestation does not branch control flow, so it cannot introduce a human wait;
  any downstream judgment is the existing planning gate (separately defaulted in the planning milestones).

### M4-R6 — Oracle branch on `run(cmd)` result
- **Location:** m4 Scope #3 (L32); Done #7.
- **Kind:** runtime.
- **Decision (default):** the oracle `run(cmd) -> {exit, stdout, stderr}` result deterministically branches
  control flow at a graph node (the git-bisect/red-blue shape). The branch is data-driven (exit code /
  output predicate), not human-adjudicated.
- **Rationale:** oracle is a measuring instrument feeding a deterministic branch; no person interprets the
  result.

### M4-R7 — Composition-observability commands without a `plan_dir`
- **Location:** m4 #5c (L37); Done #6.
- **Kind:** runtime.
- **Decision (machine-gate):** `introspect`/`doctor`/`trace`/`cost` are re-homed onto the backend-agnostic
  composition contract; Done #6 requires a non-plan composition be observable by all four WITHOUT a
  `plan_dir` (re-home *proven*, not asserted). On red the milestone halts.
- **Rationale:** the proof-not-assert wording makes this a test gate; no human verifies observability by eye.

## Cross-cutting (both milestones)

### X1 — Parity gate (M3 THE gate; M4 keeps it green)
- **Location:** m3 Outcome #4 / Done #1; m4 Constraints L60.
- **Kind:** machine-gate (the canonical one).
- **Decision (machine-gate):** the `{5 robustness} × {with_prep, with_feedback} × {states} × {verdicts}`
  cross-product parity test must be GREEN — auto-proceeds on green, auto-halts on red. Honestly labelled as
  control-flow/artifact parity on the happy path, not "drift provably zero" (m3 Done #8).
- **Rationale:** this is the load-bearing oracle for the whole epic; it converts "is the projection
  faithful?" into pass/fail with no human in the loop. Note the known TIEBREAKER→ITERATE silent-downgrade
  failure mode (m3 Constraints L172-174; MEMORY) — the gate's cross-product must include the
  `gate_proceed`/`gate_proceed_blocked`/`gate_proceed_agent_availability_blocked` distinctions so a faithless
  projection is caught, not silently downgraded.

### X2 — Back-compat / characterization replay
- **Location:** m3 Constraints L180-183; m4 Constraints L59; Done #3 (byte-stable replay).
- **Kind:** machine-gate.
- **Decision (machine-gate):** all existing callers (`iterate_until`, `SubloopStep`, `write_plan_state`
  modes, `workflow_next` signature, the 14 dispatch callers, auto.py's three retry loops) keep working via
  `extra="ignore"` + name aliases + characterization-replay against recorded runs. Replay byte-stability is
  the gate.
- **Rationale:** characterization replay mechanizes "did we regress behaviour?"; additive-only changes plus
  replay leave nothing for a human to sign off.

### X3 — Pinned engine / no editable-install dogfood
- **Location:** m3 Constraints L179; m4 Constraints L60; MEMORY.
- **Kind:** design-time (operational guardrail).
- **Decision (default):** run the epic on a PINNED engine, never an editable install; any engine-side fix is
  released + re-pinned before the worktree consumes it.
- **Rationale:** the dogfood-engine-shadow bug (MEMORY) is exactly this; pinning is the standing default so no
  one decides per-milestone.
