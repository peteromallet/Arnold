v0.18.0 capability-matching — what it loses: an adversarial read of the
"one concept replaces three subsystems" pitch

## Verdict

Capability matching is the right direction and a real improvement over both
the v0.17.0 three-part fix and the current v0.16.x "fail in review, burn a
container run, shrug" status quo — but the claim that it replaces three
subsystems with one elegant concept is partially rhetorical. It shifts
complexity rather than eliminating it: the capability vocabulary becomes a
new artifact that requires governance, the plan-time hard failure replaces
a late review failure with an earlier authoring failure (which is usually
better but not always), and the `pending-human queue` is the critique's
silent-accumulator worry rebadged. The proposal genuinely wins on one axis
the other two proposals cannot touch — *solo-dev plan-time rejection of
stuck specs* — and that single win is large enough to carry the release.
But shipping it exactly as drafted leaves at least four concrete loss
modes that the v0.17.0 proposal or the critique's audit-only path would
not create. The recommendation at the end of this file is: ship capability
matching, but with four specific modifications, not as drafted.

## Concrete losses, ranked

**1. The capability vocabulary is an unowned artifact with no schema, no
linter, and no authority.** The proposal lists five human capabilities
(`drive_browser`, `inspect_runtime_ui`, `subjective_judgment`,
`verify_render_counts`, `verify_physics`) and five container capabilities
as examples. Two of those human capabilities (`verify_render_counts`,
`verify_physics`) are not capabilities in the structural sense — they are
*criteria-shaped*, one per kind of milestone. If the next milestone is an
audio mixer the author invents `verify_audio_levels`; if it is a 3D scene
`verify_mesh_topology`. Every plan gains at least one new capability
string. The `requires: [verify_render_counts]` for M0's lightbox
criterion will have exactly one matching consumer (the human reviewer who
hand-edits `workers.codex_container.verifies` to include it, or doesn't).
Compare to the v0.17.0 proposal's `verified_by: human` enum — closed set,
three values. The capability vocabulary as drafted has no enum
constraint, no `capabilities.yml` registry, no "where is this defined"
grep anchor. The file `megaplan/workers.py` or equivalent would need to
own a registry and reject unknown strings at plan time; otherwise typos
silently route criteria to the pending-human queue (`verify_rendercounts`
≠ `verify_render_counts` and neither worker declares the typo, so the
criterion passes plan assert → goes to review → reviewer can't verify →
queue). *Recovery*: add a `megaplan/capabilities.py` with a closed enum
(or a YAML registry plus a lint pass) and fail plan time on unknown
capability strings. Also: forbid criterion-shaped capabilities by style
guide; `verify_render_counts` should decompose into
`[drive_browser, inspect_react_devtools]` or similar mechanism-shaped
primitives. Without this, capability matching becomes the tagging
treadmill the critique accused v0.17 of being, with tags living in two
files instead of one.

**2. Plan-time hard failure deletes a currently-useful "inspect the
code anyway" mode.** Under v0.16.x, a spec with unverifiable criteria
still runs: the container writes code, opens a PR, and a human can look
at the diff to decide if the work was good even though review blocked.
Under capability matching, if `human_reviewer` is not configured and a
`must` criterion requires `drive_browser`, the plan is rejected at
authoring time — no code gets written. For M0 specifically this is a
win: the author learns at draft time. For a different class of specs —
exploratory milestones, "let's see what this looks like" spikes,
prototype branches where the author *wants* code to exist before
deciding whether manual verification is worth the human's time — it is
a regression. The v0.17.0 proposal preserves this mode via its
`approved_with_deferrals` label (code ships, criteria defer). The
critique's audit-only path preserves it via advisory flags. Capability
matching as drafted does not. *Recovery*: add a
`megaplan init --allow-unverifiable` flag (or the moral equivalent —
a per-plan `allow_pending_human: true` opt-in) that downgrades the
plan-time assert to a critique-level flag and routes the criterion to
the pending queue *with code still getting written*. The flag should be
loud in `status` output and require a rationale string, so exploratory
use is distinguishable from accidental use. Without it, capability
matching is strictly less flexible than today for one real workflow.

**3. The `pending-human queue` is the critique's silent-accumulator
concern renamed, with all its weaknesses.** The critique's objection to
`deferred_human` was that in a solo-operator context nobody sends the
reminder, `megaplan verify-human` never gets run, and the queue grows
unbounded. The capability-matching proposal calls the same structure a
"pending-human queue" and requires `megaplan verify-human --plan X` to
clear it before `done`. The proposal claims this is different because
`done` is *blocked* — good, that addresses critique point 1 (silent
accumulation past `done`) — but it does not address the subtler failure:
the plan parks in `awaiting_human_verify` and the human never actually
runs the CLI. In autonomous loop mode, the loop driver sees a
non-`done` state and moves on; the plan accumulates in a
`plans/awaiting/` directory with no decay, no notification, no
dashboard. *Recovery*: three small things. (a) `megaplan status
--pending-human` must be a first-class CLI that lists every such plan
across the workspace with age; (b) the loop driver should emit a
logged warning (not just skip) when it encounters a pending-human plan
older than N days; (c) `megaplan verify-human` should be the ONLY way
to clear the queue — no path via re-running review, no path via
editing the plan file, no path via a `--force` flag. The proposal as
drafted does not specify (a), (b), or (c), and without them the queue
will behave exactly the way the critique feared, just one state earlier
in the lifecycle.

**4. Grain-size mismatch between capabilities and criteria collapses
partial capability into binary.** `drive_browser` is a coarse
capability; "ImageLightbox opens with ≤5 renders" requires
`drive_browser` PLUS the React DevTools profiler PLUS knowledge of
where `useRenderBudget` writes its counts PLUS the baseline doc for
comparison. A worker can have `drive_browser` and still be unable to
verify the criterion (e.g. Playwright-capable CI without the profiler
extension, or a human reviewer who can open Chrome but does not know
how to read the overlay). The proposal's `requires ⊆ union(verifies)`
assertion treats capability possession as binary: either the worker has
it or it doesn't. This is the exact failure mode the critique flagged
in v0.17.0 for natural-language audits — the capability label is
present, the actual verification is not. *Recovery*: accept that
capability matching can only catch gross mismatches (no browser at
all vs. browser required), and document explicitly that it does NOT
guarantee a criterion will actually be verifiable by a worker whose
capability set is a superset of requires. Then pair it with the
critique's pre-critique binding audit (from the "scoped v0.17.0" path)
which can flag criteria whose `requires` list is suspiciously short
for what the prose describes. Without this, capability matching will
produce the failure mode "plan passes assert, worker accepts the
criterion, worker can't actually verify it, criterion waives silently"
— which is a new failure mode not present in the critique's simpler
audit-only path.

**5. Composite requirements cannot be expressed as flat sets.** The
proposal uses `requires: [cap1, cap2]` which is the intersection of
capabilities a verifier must hold. It has no way to express "capability
A for step 1, capability B for step 2, and the steps must be done by
the *same* verifier" vs. "capability A by any verifier AND capability B
by any verifier (possibly different)." For the M0 case: verifying
"baseline doc captures 5×3 interactions from non-HMR prod server"
requires (a) running a production build, (b) running a dev server
against it, (c) observing render counts through the overlay, (d)
writing them to a doc, and (e) a reviewer comparing the doc to a
re-observation. Steps (a)-(d) are executor responsibility; (e) is
reviewer responsibility. Capability matching treats "verify" as atomic
and assigns it to one worker. This is fine if the executor did (a)-(d)
correctly, but in the M0 case the executor did (a)-(d) *incorrectly*
(used HMR because Vite strips `import.meta.env.DEV` in non-HMR) and no
capability check catches that — the executor has
`[read_repo, run_shell, parse_diff]`, all genuinely present, and still
produced wrong output. *Recovery*: acknowledge that capability matching
is a *necessary* check not a *sufficient* one, and preserve the
pre-critique verifiability audit (critique's preferred path) as a
second layer that can reason about criterion prose. Capability matching
+ audit-prose-check is stronger than either alone. Deleting the audit
in favor of capability matching (as the "one concept replaces three
subsystems" pitch implies) is the regression.

**6. Worker discovery is unspecified and the solo-dev guarantee depends
on it.** The proposal says "if `human_reviewer` isn't configured,
human-required criteria are rejected at plan time." Configured where?
`workers.yml`? Environment variable? Detected at runtime by pinging a
Slack webhook? The difference matters: if `human_reviewer` is a static
config file entry, the solo dev just adds it on day one and every plan
thereafter silently sends criteria to the pending queue (failure mode
#3). If it is dynamic ("only counts as configured if a human has
responded to `megaplan verify-human` within the last 7 days"), the
solo-dev guarantee is real but adds lifecycle complexity. The proposal
ducks this. *Recovery*: specify a concrete file
(`megaplan/config/workers.yml`) with a concrete schema
(`human_reviewer: {enabled: bool, contact: str, last_seen: iso8601}`)
and a liveness policy (`enabled && last_seen < 30d`). Then the
solo-dev rejection path is deterministic.

**7. Gaming moved, not solved.** The proposal claims no gaming surface
because capabilities live in worker config. A motivated author still
controls the `requires:` list on each criterion. An author who writes
`requires: []` on "ImageLightbox opens with ≤5 renders" passes the
plan-time assert trivially (empty set is subset of anything) and routes
the criterion to whichever worker picks it up — typically the container,
which cannot verify render counts, which means either a false pass or
a `waived_unverifiable`-like fallback. The gaming path is now "omit
`requires`" instead of v0.17.0's "set `verified_by: human`", and is
arguably easier because omission is the default. *Recovery*: make
`requires` mandatory on `must` criteria. Missing `requires` on a `must`
is a plan-time flag. Allow empty list only with an explicit
`requires: []  # rationale: purely structural check, no runtime needed`
comment that the critique prompt enforces. Without this, gaming is
strictly easier under capability matching than under v0.17.0's tag.

**8. Migration default `requires: []` gives existing plans zero
benefit.** The v0.17.0 proposal's "default to container" at least
applied the new review rules to existing plans (their criteria got
routed through `waived_unverifiable`/`pass`/`fail` correctly). Under
capability matching, `requires: []` means every existing criterion
trivially matches every worker — so nothing flags, nothing routes to
pending-human, and the first ten plans on v0.18.0 behave exactly like
v0.16.0. The fix only kicks in for newly-authored specs that use
`requires` deliberately. The critique's audit-only path has the same
weakness; the v0.17.0 tag-with-default does not. *Recovery*: add a
one-shot retrofit CLI (`megaplan retrofit-requires --plan X`) that
runs a capability-inference prompt over each existing criterion and
writes a proposed `requires` list the author can accept. This is the
same prompt as the pre-critique audit from the critique's path, reused.
Without it, the v0.18.0 release lands and nothing changes for a month
until specs naturally churn.

## What capability matching solves that the others don't

The honest case. Capability matching has two distinctive wins:

First, **plan-time solo-dev rejection**. Neither v0.17.0 (which ships
code then defers) nor the critique's audit (which surfaces flags but
still allows unverifiable plans to run) can say at plan time "you
literally cannot author a stuck spec." That guarantee is structurally
new and operationally valuable: in the M0 case the author would have
been told *before writing any docs or code* that the spec requires a
browser and no browser-capable worker is configured. The author then
either configures one or rewrites the criterion. This is genuinely
upstream of everything v0.17.0 and the critique do.

Second, **the capability vocabulary is a shared artifact between the
author and the executor.** Under v0.17.0, the author tags
`verified_by: human` and the executor does not know what "human" means
beyond "skip this." Under capability matching, `requires: [drive_browser,
inspect_runtime_ui]` is a structured handoff: the pending-human queue
entry literally tells the human *what capabilities* to exercise.
Reviewer prompts can be parameterized by capability. Dashboards can
aggregate. This is the "ground-truth capability check" the critique
asked for as option (f), delivered. The critique's audit-only path does
not deliver this — it produces prose rationales, not structured
capability annotations.

These two wins, properly exploited, justify the release. But they do
not justify deleting the pre-critique audit; see loss #4 and #5.

## Recommendation

Ship capability matching as v0.18.0, with these four modifications:

1. **Add a closed capability registry.** `megaplan/capabilities.py`
   with a frozen enum (or `capabilities.yml` loaded at init). Plan-time
   assert rejects unknown capability strings. Style-guide forbid
   criterion-shaped capabilities; require mechanism-shaped primitives.
   (Loss #1.)

2. **Keep the pre-critique verifiability audit** from the critique's
   preferred scoped v0.17.0. Capability matching is necessary but not
   sufficient; the audit's prose-reasoning catches criteria whose
   `requires` list is technically-satisfied but semantically wrong
   (the M0 baseline-doc case). Do NOT claim "one concept replaces three
   subsystems" — honestly state that capability matching *plus* a
   lightweight audit *plus* the pending-human queue is the shipped
   surface. Three subsystems, better integrated than v0.17.0's three,
   but still three. (Losses #4, #5.)

3. **Specify worker discovery and pending-queue lifecycle concretely.**
   `config/workers.yml` with an explicit schema including
   `human_reviewer.last_seen`. `megaplan status --pending-human` CLI.
   Loop-driver warning on stale pending plans. `verify-human` as the
   only clearing path. (Losses #3, #6.)

4. **Require `requires:` on `must` criteria.** Empty list allowed only
   with explicit rationale comment that the critique prompt enforces.
   Add `megaplan retrofit-requires` for migrating existing plans.
   (Losses #7, #8.)

Do NOT ship the critique's audit-only path instead. The critique was
right that v0.17.0 as drafted was overbuilt, and right that capability
profiles are the real fix, but wrong that v0.17.0 should be the audit
alone — the audit alone catches *authorship* errors and does nothing
for the executor-capability mismatch that is the actual M0 failure.
Capability matching closes that gap in a way the audit cannot.

Do NOT ship capability matching as drafted. The elegance claim elides
four real losses (1, 3, 7, 8) where capability matching is *worse* than
v0.17.0 or the audit-only path. Each is recoverable with a small
addition; none is load-bearing against the core concept. But "ship as
drafted" would re-create the silent-accumulator failure mode the
critique was right to worry about, just with an extra state name.

The two-release sequence the critique recommended (scoped v0.17.0
audit, then v0.18.0 capability profile) is still close to right; the
modification is that v0.17.0 should also include the pending-human
queue + CLI scaffolding, so v0.18.0 is just "add the capability
registry and the requires field." Splitting the risk across two
releases is better than landing all of capability matching in one
drop. If calendar pressure forces a single release, land v0.18.0 with
modifications 1-4 above and accept that the first month of plans will
need the retrofit tool.
