# v2.7.0 — Docs, Quality Metrics, Release

**Released:** 2026-05-22
**Semver:** MINOR (backward-compatible new functionality)
**Tag:** `v2.7.0`

---

## Overview

v2.7.0 is a documentation, quality, and developer-experience release.  It adds a
newcomer-oriented cookbook, a hand-crafted gold template exemplar, a migration
guide for v2.5 → v2.7 users, an agentic usability harness, and improved header
guidance in generated templates.

No public API breakage is expected for existing v2.5 workflows, but template
authors should review the migration guide for new best practices.

---

## Breaking Changes

*None.*  This release is backward-compatible with v2.5-era workflows and public
APIs.  Template authors may optionally adopt new patterns (e.g., `copy-to-recipe`
for hand-editing) but are not required to.

---

## Deprecations (since v2.5)

| Deprecated | Replacement | Timeline |
|---|---|---|
| Old generated header text `"# vibecomfy: generated - converted by tools/convert_ready_templates.py"` | New header: `"# vibecomfy: generated"` + `"# For hand-editing, run: python -m vibecomfy.cli copy-to-recipe <id>"` | Old header remains parseable; no forced regeneration |
| `tools/convert_ready_templates.py` as the hand-editing escape hatch | `python -m vibecomfy.cli copy-to-recipe <id>` | Old tool path still referenced in test compat code; new users should use `copy-to-recipe` |
| `vibecomfy.templates.ref('name')` in **new** templates | Bind `InputSpec.node` to node objects inside `build()` | Existing `ref()` usage still works; PendingDeprecationWarning emitted |

## Runtime Schema Degradation

Runtime schema degradation is loud by default. If `object_info` schema validation is unavailable, VibeComfy logs an ERROR listing skipped class types and records them in run metadata as `schema_validation_skipped`.

Temporary off-ramp: pass `--quiet-schema-degradation` on `vibecomfy run` or set `VIBECOMFY_SCHEMA_WARN_ONLY=1` to downgrade the log to WARNING while migrating existing monitors.

## New Features

### 1. CLI Commands

New and enhanced CLI subcommands:

| Command | Description |
|---|---|
| `vibecomfy runtime eval-node` | Evaluate a node in the active runtime (local or RunPod) |
| `vibecomfy port validate-call` | Validate arguments before calling `port convert` |
| `vibecomfy nodes compatible-with` | List custom-node packs compatible with a given template |
| `vibecomfy port doctor-all` | Run all port diagnostics (check + validate + inventory) in one pass |
| `vibecomfy port export` | Export a porting artifact (scratchpad, ready template, or report) |
| `vibecomfy.cli copy-to-recipe <id>` | Copy a generated ready template into the user recipe directory for hand-editing (new recommended path) |

### 2. API Additions

| API | Description |
|---|---|
| `VibeWorkflow.lookup_id(name)` | Reverse-lookup a node ID from its human-readable name |
| `VibeWorkflow.export_to_json(path)` | Serialize the workflow graph to a JSON file |
| `VibeWorkflow.strict_types` | Enable strict type-checking mode for node inputs |
| `attempt.json` / drift / `next_action` | Structured error reporting with suggested remediation steps |

### 3. Template Authoring (v2.7 style)

- **ContextVar template authoring:** `new_workflow(METADATA)` activates a
  thread-local `ContextVar`.  Every `node(...)` or generated wrapper call inside
  the `with` block automatically binds to that workflow — no need to pass `wf`
  around.
- **Tuple-unpacked multi-output wrappers:** Nodes with multiple outputs (e.g.,
  `LoadImage` → `(image, mask)`, `WanImageToVideo` → `(positive, negative, latent)`)
  support Python tuple unpacking for cleaner emission.
- **Materialized subgraphs as Python functions:** Subgraph templates can be
  defined as plain Python functions, called inside a `new_workflow()` context
  to compose larger graphs.
- **Blank-line / label-preferred emission style:** Generated templates now
  group nodes into labeled sections with blank-line separators for readability.
- **`widget_N` cleanup:** The `translate_widget_for_key` helper maps legacy
  `widget_N` input names to their semantic equivalents.

### 4. Documentation

| Document | Description |
|---|---|
| `docs/cookbook/` (7 tutorials) | Newcomer-oriented, flat, import-safe tutorials: first workflow, tweak prompt/seed/steps, JSON-to-template, image-to-video chaining, `port doctor-all` debugging, model registry entries, Python↔UI roundtrip |
| `docs/templates/examples/gold_template_wan_i2v.py` | Hand-crafted, annotated exemplar of the Wan 2.1 I2V template.  Build-only (no GPU).  Includes explanatory comments for every section |

### 5. Agentic Success-Rate Harness

`scripts/agentic_success_rate.py` tests framework usability across 4 concrete tasks:

1. **Tweak-and-run:** Change prompt/seed/steps on a ready template and compile
2. **JSON-to-template:** Create a workflow from a ComfyUI API JSON file
3. **Port doctor-all:** Debug a failing template with diagnostics
4. **Node-splice:** Insert a custom node-pack class into a workflow

Features:
- `--dry-run` mode for CI-safe offline validation (no LLM API calls)
- Default model: `deepseek:deepseek-v4-pro` (OpenRouter)
- `$5.00` max budget per category (configurable via `--max-budget`)
- JSON output (`--json`) for machine consumption
- Cost tracking per category

## Known Limitations

- **Bidirectional roundtrip:** Python → UI JSON → Python roundtrip is not
  lossless for all template features.  `copy-to-recipe` is the recommended
  hand-editing path.
- **Ready templates retain old header:** 48 existing ready_template files
  still carry the v2.5-vintage header.  This is expected — they remain valid
  and do not need regeneration.
- **Agentic harness real execution:** `scripts/agentic_success_rate.py`
  requires a `DEEPSEEK_API_KEY` or `OPENROUTER_API_KEY` for live LLM execution.
  The dry-run path is CI-safe.
- **Package version:** `pyproject.toml` declares `2.7.0`. Tooling should treat
  this release note as the canonical v2.7.0 record.

## Migration Checklist

See `docs/migration/v25_to_v27.md` for full per-fix guidance.  Quick-start:

```bash
# 1. Check port health
vibecomfy port check ready_templates/sources/official/video/wan_i2v.json

# 2. Validate a conversion call
vibecomfy port validate-call ready_templates/sources/official/video/wan_i2v.json

# 3. Run all diagnostics
vibecomfy port doctor-all ready_templates/sources/official/video/wan_i2v.json --json

# 4. Check custom-node compatibility
vibecomfy nodes compatible-with ready_templates/video/wan_i2v.py

# 5. Export a porting artifact
vibecomfy port export ready_templates/sources/official/video/wan_i2v.json --format ready
```
