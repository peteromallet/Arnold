### 1. `layout_store.py`: Root-level placement evaluation

**Current location:** `vibecomfy/porting/layout_store.py` (root of `porting/`)

**Importers (5 files):**
| File | Import |
|---|---|
| `vibecomfy/commands/port/_export.py` | `from vibecomfy.porting.layout_store import read_store, store_from_ui_json, write_store` |
| `vibecomfy/commands/port/_shared.py` | `from vibecomfy.porting.layout_store import read_store, store_from_ui_json, write_layout, write_store` |
| `vibecomfy/commands/port/_convert.py` | `from vibecomfy.porting.layout_store import write_layout` |
| `vibecomfy/comfy_nodes/agent/edit.py` (√ó2) | `from vibecomfy.porting.layout_store import store_from_ui_json, write_store` |
| `vibecomfy/comfy_nodes/stages/emit.py` | `from vibecomfy.porting.layout_store import store_from_ui_json, write_store` |

**Assessment:** `layout_store.py` is a sidecar I/O module (read/write `.layout.json` files, migrate v1‚Üív2, gc). It does NOT import from `vibecomfy.porting.layout.*` ‚Äî it depends on `canonical_coords`, `identity.scope`, and `identity.uid`. Conversely, the `layout/` package modules (`engine.py`, `reconcile.py`, etc.) do NOT import `layout_store.py` either. There is **no circular dependency** risk either way.

**Verdict:** `layout_store.py` **could** move under `layout/` (as `vibecomfy/porting/layout/store.py`) but it **earns its root-level placement** because:
- It is the public I/O boundary ‚Äî callers are CLI commands and agent edit code, not internal layout engine code.
- It has zero coupling to the layout engine internals (layering, lanes, placement, sizing, groups, etc.).
- Moving it would churn 6 import sites + the test file (`test_layout_store.py`) for purely aesthetic reasons.

**Recommendation:** **Keep at root level.** If you want to tighten, alias it in `layout/__init__.py`:

```python
from vibecomfy.porting.layout_store import read_store, write_store  # noqa: F401
```

But this is optional ‚Äî the current arrangement is clean.
