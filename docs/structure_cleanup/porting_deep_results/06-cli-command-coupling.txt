### 2. Are there wrappers/barrels that should exist?

**Strong candidate: `commands/_workflow_loader.py`**

Four command files use the identical pattern:
```python
loaded = load_port_source(args.workflow, schema_provider=get_schema_provider("auto"))
```
— `analyze.py:98`, `inspect.py:29`, `nodes.py:426`, `port/_widgets.py:18`

The existing `commands/_workflow_path.py` (30 lines) only does path/index resolution. A barrel that exports a `load_cli_workflow(workflow_id)` which bundles `load_port_source` + `get_schema_provider("auto")` + try/except → `print(f"Failed to load workflow: ...")` would:
- Eliminate the direct `from vibecomfy.porting.workbench import load_port_source` in 4 files
- Reduce copy-paste error handling (each file has nearly identical error print)
- Hide porting internals from non-port command modules

**`port/__init__.py` barrel re-exports are correct but have one dead entry:**

```python
from vibecomfy.porting.emit.ui import default_output_path, emit_ui_json
from vibecomfy.porting.layout import evaluate_felt_delta
from vibecomfy.registry import load_workflow_reference
```

These let submodules do `from vibecomfy.commands import port as _port; _port.evaluate_felt_delta(...)` and let tests do `monkeypatch.setattr("vibecomfy.commands.port.evaluate_felt_delta", ...)`. Three are actively used:

- `evaluate_felt_delta` — used in `_export.py:510` and patched in `test_cli_port.py:1763`
- `emit_ui_json` — used in `_export.py:480,488`
- `load_workflow_reference` — used in `_export.py:404,430` and `_doctor_all.py:126`

**However, `default_output_path`** is re-exported at `port/__init__.py:35` but is **never accessed via the barrel** — `_export.py` imports it directly (`from vibecomfy.porting.emit.ui import default_output_path`, line 12) and no code or test references `vibecomfy.commands.port.default_output_path`. This re-export is dead and can be removed.
