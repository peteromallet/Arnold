#### 2. Is that intentional or should some move under `tests/fixtures/`?

**Per-item analysis:**

**`recipes/example_tested_recipe.py.snapshot.json`** — **NOT intentional for production.** This is a snapshot baseline used by `tests/test_testing_snapshot.py` and `scripts/regenerate_snapshots.py`. The `recipes/` README defines recipes as "runnable compositions built from ready templates." A `.snapshot.json` is not a runnable composition — it's a serialized workflow used for round-trip comparison testing. **Should move to `tests/fixtures/recipes/`.**

**`workflow_corpus/input/` assets** — **Intentional, but debatable.** The `FIXTURES.md` explicitly acknowledges these are "smoke fixtures" for the corpus matrix. They're co-located with the workflow JSON they support because `workflow_corpus/official/**/*.json` references them by path. This is a pragmatic design: corpus workflows need deterministic input assets, and keeping them alongside the JSON avoids path indirection. However, this means user-facing corpus data contains test-only assets. The established pattern in this repo is `tests/fixtures/`, so the question is: do you want corpus & fixtures separated or bundled?

**`docs/templates/examples/gold_template_wan_i2v.py`** — **Hybrid.** The file serves dual purpose:
1. Pedagogical documentation (hence in `docs/`)
2. Gold reference for `tests/test_gold_template_alignment.py` (hence a test fixture)
The existing structure cleanup docs already flagged this. The test imports it from `docs/templates/examples/`. If you treat it as documentation-only, it stays. If you treat it as a test fixture, it should move to `tests/fixtures/goldens/` and the test updated.

**`vibecomfy/porting/{object_info,cache}/*@runpod-snapshot.json`** — **Runtime cache, not test fixtures.** These are serialized ComfyUI node object-info snapshots captured from RunPod. They're loaded by `vibecomfy/porting/object_info/consume.py` and `serialize.py` at runtime to do offline node resolution. They're not test-only. They could arguably live in a flat `cache/` or `data/` directory, but they're not misplaced in a fixture-sense.

#### 3. What moves are safe now?

### SAFE TO MOVE NOW ✅

| Move | Risk | Details |
|------|------|---------|
| `recipes/example_tested_recipe.py.snapshot.json` → `tests/fixtures/recipes/example_tested_recipe.py.snapshot.json` | **None** | Only referenced by `tests/test_testing_snapshot.py` and `scripts/regenerate_snapshots.py`. The regenerator script uses `REPO_ROOT / "recipes"` — that path needs updating. Low blast radius. |

### CONDITIONALLY SAFE 🟡

| Move | Risk | Details |
|------|------|---------|
| `docs/templates/examples/gold_template_wan_i2v.py` → `tests/fixtures/goldens/gold_template_wan_i2v.py` | **Medium** | Breaks `tests/test_gold_template_alignment.py` (hard-coded `GOLD_PATH`), `release_notes/v2.7.0.md` (documentation references), and `docs/templates/examples/` README. Also changes the pedagogical purpose. Recommend **leave in place** and just add a `tests/fixtures/goldens/` symlink or conftest-level resolver. |

### NOT SAFE TO MOVE ❌

| Move | Risk | Details |
|------|------|---------|
| `workflow_corpus/input/*` | **High** | Workflow JSON files reference these by relative path. Moving them breaks every corpus smoke run and every template that loads input media. The `vibecomfy/fixtures.py` module handles `copy --target input` from here. If you want to move, you need to update every JSON reference, the fixture copy logic, and all templates. **Don't do this yet** — create a separate ticket. |
| `vibecomfy/porting/{object_info,cache}/*@runpod-snapshot.json` | **High** | These are runtime-loaded by `vibecomfy/porting/object_info/consume.py` and `serialize.py`. Moving them requires updating multiple import paths in production code. They are cache files, not test fixtures. **Leave in place.** |
