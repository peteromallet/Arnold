## Audit: Navigation docs for template/corpus/recipe users

---

### Q1: Can a new user find templates, recipes, and corpus workflows from the root README?

| Artifact | Reachable from root README? | How |
|---|---|---|
| **Templates** (`ready_templates/`) | ✅ **Yes** | Repository Layout table (line 226), dedicated "Templates And Porting" section (line 169), install commands (`list --ready`, `inspect`), and the Architecture glossary table (line 161). |
| **Recipes** (`recipes/`) | ✅ **Yes** | Repository Layout table (line 227), copy-to-recipe example in install section (line 79), Architecture glossary (line 162), and a code snippet showing `load_workflow_any` usage (lines 86-99). |
| **Workflow Corpus** (`workflow_corpus/`) | ⚠️ **Barely** | Repository Layout table (line 228): one line — *"Source ComfyUI workflows used for indexing, conversion, and coverage."* No dedicated section, no link to `docs/workflow-coverage/corpus.md`, and no mention in the "Templates And Porting" section where a reader would naturally look for it. A user who scans the layout table will find it; a user reading the prose sections may miss it entirely. |

**Verdict:** Templates and recipes are well-served. `workflow_corpus/` is discoverable only if the user reads the Repository Layout table — it's invisible from the narrative sections.

---

### Q2: Is `workflow_corpus/` documented anywhere at the directory itself?

**No.** The directory has zero README files:

```
workflow_corpus/          ← no README.md
├── manifests/
│   ├── coverage.json
│   └── ready_regeneration.json
├── official/
│   ├── audio/
│   ├── edit/
│   ├── image/
│   └── video/
└── custom_nodes/
    ├── flux2/
    └── ltxvideo/ ...
```

A user who `cd`s into `workflow_corpus/` sees subdirectories and JSON files with no explanation of:
- What the directory is for
- How its structure maps to ready templates
- What `official/` vs `custom_nodes/` means
- What the manifests contain
- Where the detailed documentation lives (i.e., `docs/workflow-coverage/corpus.md`)

The *best* documentation lives at **`docs/workflow-coverage/corpus.md`**, which thoroughly explains the corpus purpose, manifest schema, status tiers, and validation commands. But there is no pointer to it from either the root README's corpus row or from the directory itself.

### Q3: What small README additions would improve navigation?

Below are the exact proposed edits.

---

#### Edit A: Add `workflow_corpus/README.md`

This is the most impactful single change. Create a new file:

```markdown
# Workflow Corpus

Source ComfyUI JSON workflows used for template generation, coverage validation, and the RunPod runtime matrix.

## Structure

```
workflow_corpus/
├── official/          # Official ComfyUI template sources (image, video, edit, audio)
├── custom_nodes/      # Community custom-node workflow sources (ltxvideo, flux2, …)
└── manifests/         # Coverage manifests
    ├── coverage.json          # Required corpus manifest with status tiers
    └── ready_regeneration.json  # Template regeneration tracking
```

- `official/` workflows are the primary sources for ready templates in `ready_templates/`.
- `custom_nodes/` workflows extend coverage to model families that require non-standard custom nodes (e.g., LTX, WanVideoWrapper).
- `manifests/coverage.json` defines the required corpus with status tiers: `raw_json` → `converted` → `ready_template` → `runtime_ready` → `runtime_green`.

## Related docs

- [docs/workflow-coverage/corpus.md](../docs/workflow-coverage/corpus.md) — full corpus schema, status language, and RunPod validation workflow.
- [docs/workflow-coverage/](../docs/workflow-coverage/) — per-family coverage maps.
- [ready_templates/README.md](../ready_templates/README.md) — how workflows graduate to Python ready templates.
```
