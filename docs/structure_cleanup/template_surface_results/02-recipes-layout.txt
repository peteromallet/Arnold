### Q1: Are recipe files organized and named clearly?

**Partially yes, with two naming mismatches.**

✅ Individual names are descriptive: `dual_pass_t2i`, `wan_i2v_lowres`, `wan_t2v_long`, `example_tested_recipe`.

❌ **Inconsistent prefix convention.** Two recipes use `wan_` (model-family prefix), one uses no prefix (`dual_pass_t2i`), one uses `example_tested_` (purpose prefix). A user scanning the directory can't infer a naming rule.

❌ **`dual_pass_t2i.py` uses `t2i`** (text-to-image shorthand), but the project's template IDs use `image/` namespacing. A reader looking for "dual pass on an image template" won't find it by template-ID grep.

❌ **`example_tested_recipe.py`** is a test-infrastructure fixture, not a user-facing "here's how to use the library" example. Its name sounds like a template for users to copy, but its content is synthetic (no `load_workflow_any`, no patches, no blocks) — it exists solely so `vibecomfy test verify recipes/` returns 0 in CI. That's an internal concern leaking into the user-facing namespace.

**Thin-recipes concern** (already tracked as `deferred` in the pristine-cleanup audit — `docs/megaplan_chains/pristine_cleanup/audit/04-layer2-architecture.md:30-31`):
- `wan_i2v_lowres.py` and `wan_t2v_long.py` are each ~3 lines of logic: `load → resolution(...).apply(wf) → return wf`. They demonstrate patch application, not composition. Not a blocking issue, but they don't showcase the library's composition value.

Its presence in `recipes/` is correct (the snapshot system needs it there), but a user cloning the repo might assume it's a recipe they should study — and be confused by its lack of template usage.

### Q3: Should recipes split into subdirs, or would that break references?

**Not recommended now — low reward, measurable risk.**

**Files that would need updating if subdirs were introduced:**

| Reference | Location | Impact |
|-----------|----------|--------|
| `python recipes/wan_t2v_long.py` | `recipes/README.md:10` | Path changes |
| `python -m vibecomfy.cli copy-to-recipe ... --out recipes/my_z_image.py` | `README.md:79` | Guidance changes |
| `recipes/my_dual_pass.py` (generic example) | `docs/testing/user_code.md:21,35-42` | Path changes in example |
| `vibecomfy test verify recipes/` | `docs/testing/user_code.md:68`, `docs/megaplan-briefs/B-user-testing.md:85` | Would still work if recursive |
| `recipes/example_tested_recipe.py` | `docs/testing/user_code.md:124` | Relative `../` path in doc link |
| `recipes/wan_i2v_lowres.py:8-11` (as code evidence) | `docs/megaplan_chains/pristine_cleanup/audit/04-layer2-architecture.md:31` | Line references (stable) |
| `dual_pass_t2i.py` (as code evidence) | `docs/architecture/python_on_the_graph.md:77` | File name reference |
| `copy-to-recipe` command | `vibecomfy/commands/copy_to_recipe.py:145` | Default output dirs in help text |
| `recipes/` directory reference | `docs/security/capability_taxonomy.md:158,160` | Directory exists check |

**Risks:**
1. **No actual user benefit** — there are only 4 recipe `.py` files. Flat is fine until 15+.
2. **Snapshot sibling convention gets murky** — subdirs would force choosing between `recipes/image/dual_pass.py.snapshot.json` (sibling) or `recipes/.snapshots/image/dual_pass.json` (centralized). The docs currently commit to sibling placement.
3. **`vibecomfy test verify` would need verification** that it walks recursively. Currently untested at the integration level for subdirectories.
4. **`copy-to-recipe` CLI help text** would need updating to reflect subdir targets.

**Verdict:** Keep flat until there are ≥8 recipe files. If/when splitting, use template-kind subdirs (`image/`, `video/`, `examples/`) — but that's a future decision.
