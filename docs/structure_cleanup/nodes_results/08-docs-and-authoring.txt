## 1. Active docs presenting `vibecomfy.nodes.<pack>` as the authoring API

Three active docs teach this import path:

- **`README.md:21`** — `from vibecomfy.nodes.core import SaveImage` in the canonical abridged example. This is the front-page guidance.
- **`docs/authoring.md:54`** — `from vibecomfy.nodes.core import CLIPTextEncode, SaveImage` in the v2.6 Ready Template Surface section. Same pattern, same authority.
- **`docs/templates/examples/gold_template_wan_i2v.py:39,55`** — Pedagogical annotated example using `from vibecomfy.nodes.core import ...` with the comment "We use the generated thin wrappers from vibecomfy.nodes.core."

**`CLAUDE.md`** does not directly show `vibecomfy.nodes.*` imports but depends on them implicitly — its "Adding a new workflow" step 7 calls `port convert`, which emits `from vibecomfy.nodes.<pack> import ...` lines.

**`docs/historical/`** is archival only; no active guidance.

## 2. Should docs present `vibecomfy.nodes.<pack>` or `_generated` imports?

**Keep presenting `vibecomfy.nodes.<pack>`.** The `nodes/<pack>.py` shim files are the intentional public API surface (as confirmed by `docs/plans/revised_plan.md`: "Shims stay — `from vibecomfy.nodes.core import SaveImage` paths survive"). Each shim re-exports from `vibecomfy.nodes._generated.<pack>`, but `_generated` is an internal implementation detail. Authoring docs should never teach users to import from `_generated` directly.

## 3. Stale phrases

- **"thin wrappers"** in `docs/templates/examples/gold_template_wan_i2v.py:39` — "generated thin wrappers from vibecomfy.nodes.core" is mildly implementation-chatter. Since these are the public API, just "generated wrappers" or "node wrappers" is cleaner.
- Planning docs (`handoff-m2b.md`, `revised_plan.md`) use **"thin shim"** and **"thin-wrapper"** to describe the same files — irrelevant for user-facing docs, but be aware the phrasing exists in internal audit trail.
