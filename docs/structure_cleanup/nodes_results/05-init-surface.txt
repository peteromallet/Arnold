### 1. Should `__init__.py` become dynamic over `_generated.MODULES`?

**Yes.** Replace the 19 hardcoded wildcard imports with a loop:

```python
from importlib import import_module
from vibecomfy.nodes._generated import MODULES

__all__: list[str] = []
for _mod_name in MODULES:
    _mod = import_module(f"vibecomfy.nodes.{_mod_name}")
    __all__.extend(_mod.__all__)
```

This auto-tracks pack additions/removals and eliminates the brittle 33KB boilerplate. The wildcard still flows through each shim (`nodes/core.py` → `nodes/_generated/core.py`) which provides runtime shim logic plus `__all__ = list(_generated.__all__)`.

### 2. Should `__all__` remain static or be computed?

**Computed.** Each shim (e.g., `core.py`) already exposes `__all__ = list(_generated.__all__)`. Accumulating those lists after import gives the correct total without a manual enumeration that inevitably drifts.

### 3. What should happen to `.pyi`?

**Mirror the same dynamic pattern.** Replace the 19 hardcoded wildcard stubs with:

```python
from __future__ import annotations
from vibecomfy.nodes._generated import MODULES as _MODULES
__all__: list[str]
```

Type checkers (mypy, pyright) don't evaluate runtime code in `.pyi` files, but they do understand `from X import *` + `__all__: list[str]` as a merged namespace. Alternatively, compute `__all__` by aggregating all `_generated/*.pyi` `__all__` declarations, but the simpler stub above is sufficient since the resolver follows the re-export chain.
