## Answers to the Three Questions

### 1. What should the root README row say for `.megaplan`?

The root README repository-layout table (lines 211–219) omits `.megaplan`, `.agents`, `.claude`, `agents`, and `agentic`. It should add a row for `.megaplan` alongside entries for the other hidden agent directories. Recommended row:

> | `.megaplan/` | Local megaplan tool state: plan states, chain logs, cloud-runner output. Gitignored; see `docs/megaplan_chains/` for authored planning artifacts. |

### 2. Should `.megaplan/README.md` exist?

**No.** This is a gitignored directory that holds ephemeral tool state. A README inside an ignored directory serves no audience:

- It won't appear on GitHub.
- It won't be cloned or distributed.
- Its guidance would be invisible to anyone who hasn't run megaplan locally.
- The relationship between `.megaplan/` (ignored runtime state) and `docs/megaplan_chains/` (tracked planning docs) is better documented in the root README table and `docs/README.md`, which are visible to all consumers.

If someone needs to understand what `.megaplan/` contains, the root README row + `docs/README.md` cross-reference is the right place — not a file that won't exist on a fresh clone.

### 3. How should this repo distinguish authored planning specs from local megaplan runtime output?

The boundary is already correctly drawn; it just needs to be documented as policy:

| Layer | Authored (tracked) | Runtime state (ignored) |
|---|---|---|
| **Chain specs** | `docs/megaplan_chains/<epic>/chain.yaml` | `.megaplan/plans/` — resolved plan-state files |
| **Milestone briefs** | `docs/megaplan_chains/<epic>/m1-*.md`, `handoff-m1.md`, etc. | `.megaplan/logs/` — execution logs |
| **Strategy/design** | `docs/megaplan_chains/<epic>/STRATEGY.md`, `README.md` | `.megaplan/plans/.chains/` — chain runner metadata |
| **Cloud state** | (none — ephemeral) | `.megaplan/cloud-*.log`, `.megaplan/recovery-prompts/` |
| **Nested runtime** | (exception: chain docs live at `docs/megaplan_chains/<epic>/`) | `.megaplan/` at any level (e.g., `docs/megaplan_chains/<epic>/.megaplan/`) |

**Policy rule:** All megaplan **planning content** — chain specs, milestone briefs, strategy docs, operator guides, handoff artifacts — lives as tracked `.md` files under `docs/megaplan_chains/<epic>/`. All megaplan **runtime state** — plan state JSON, chain runner metadata, execution logs, cloud-runner output, recovery prompts — lives in `.megaplan/` directories (at any level, always gitignored). Never commit runtime state; never put planning specs inside `.megaplan/`.

## Summary Policy (concise)

1. **`.megaplan/` — runtime state only.** All directories named `.megaplan/` (at any level) are gitignored and hold ephemeral megaplan tool state. No authored content goes here. No README needed.

2. **`docs/megaplan_chains/` — authored planning only.** All chain specs, milestone briefs, strategy docs, operator guides, and handoff artifacts are tracked `.md` files under `docs/megaplan_chains/<epic>/`. These are durable, reviewed planning documents — they are project documentation.

3. **`.agents/`, `.claude/` — local agent config.** Gitignored per-user skill/config files. No README needed. The tracked `agents/` directory documents the public agent interface that these skills implement.

4. **`agentic/` — tracked harness.** The Sisypy agentic-test harness. Run output goes to `out/agentic/reports/<tag>/`.

5. **Never commit `.megaplan/` runtime state** into the repo. The `.gitignore` pattern already enforces this; the policy is to keep it that way and not nest authored content inside `.megaplan/` directories.
