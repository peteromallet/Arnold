## Q3: README/policy vs. moving files?

**Keep in place + add a README/policy** is the recommended path.

Rationale:

1. **They're not in the way.** Both epics are complete; the files are inert (no active chain runner references them). There's no maintenance burden.

2. **Moving would break the one doc link** in `docs/architecture/python_on_the_graph.md:50` and require updating internal `idea:` relative paths — risk for zero gain.

3. **They're the canonical location** for their content. Moving them to `docs/megaplan_chains/frontier-hardening/chain.yaml` and `docs/megaplan_chains/scratchpad-emitter/chain.yaml` would create two new directories there but serve no functional purpose (the epics are done).

4. **The real problem is ambiguity** about the relationship between `.megaplan/chains/` and `docs/megaplan_chains/` — which a README solves directly.

## Recommendation

**Option A (recommended): Keep in place. Add a README at `.megaplan/chains/README.md` documenting the policy:**

> - `.megaplan/chains/*.yaml` — canonical executable specs for **completed** epics (frontier-hardening, scratchpad-emitter)
> - `docs/megaplan_chains/*/chain.yaml` — executable specs for **active** epics (excellence, node_resolution, pristine_cleanup, readable_ready_templates)
> - New chain specs should be placed under `docs/megaplan_chains/<epic-name>/chain.yaml`
> - Completed epics may stay in `.megaplan/chains/` or be migrated to `docs/megaplan_chains/<epic-name>/` at the team's discretion

**Option B (if you want strict consistency):** Move both YAMLs to `docs/megaplan_chains/frontier-hardening/chain.yaml` and `docs/megaplan_chains/scratchpad-emitter/chain.yaml`, then:
- Update `docs/architecture/python_on_the_graph.md:50`
- Update internal `idea:` paths from relative (`.megaplan/ideas/...`) to absolute or `docs/megaplan_chains/.../ideas/...` (these idea dirs live under `.megaplan/ideas/` though, so this gets messy)
- Mark `.megaplan/chains/` as deprecated in a README

**I recommend Option A.** The two files are stable, complete, and referenced from only one doc. The cost-to-benefit ratio of moving them is poor. A README clarifies the convention without touching established content.
