## Recommendation

### What belongs in `docs/megaplan_chains/`

**Move all authored/design material** currently under `.megaplan/` to `docs/megaplan_chains/`. Rationale:

1. **The gitignore is correct for runtime state** — `.megaplan/` contains massive runtime artifacts (telemetry NDJSON up to 2.5 MB, plan-run JSON blobs, PID files). Keeping authored design docs in the same tree forces force-tracking with `git add -f`, which is fragile and confusing for collaborators.

2. **`docs/megaplan_chains/` already exists** — the project already references it:
   - `pyproject.toml` lines 98-100 cite `docs/megaplan_chains/node_resolution_epic/testing/testing.md`
   - `tests/test_porting_provenance.py` line 17 imports from `docs/megaplan_chains/node_resolution_epic/`

3. **Separation of concerns** — `docs/` is git-tracked by default; `.megaplan/` is gitignored by default. Putting authored design material under `docs/` eliminates the need for force-tracking entirely.

4. **Discoverability** — engineers look for design docs under `docs/`, not inside a dot-directory that `.gitignore` hides.

### What must stay gitignored under `.megaplan/`

All runtime-generated state:
- `plans/`, `epics/`, `telemetry/`, `wakeup/`, `logs/` directories
- All `*.log` files at the root of `.megaplan/`
- Nested `.megaplan/` directories inside briefs/chains (these are accidental artifacts of worktree execution inside a brief that had its own `.megaplan` seed)

### What's borderline

- **`debt.json`** (3663 lines, 235 KB) — mixed authored/auto content. Could move to `docs/megaplan_chains/debt.json` but its size suggests it may be auto-updated. If auto-updated, it should stay gitignored under `.megaplan/`. Recommend moving only the human-curated ticket markdown files and leaving `debt.json` where it is.
- **`schemas/`** (17 JSON Schema files) — These are authored metadata protocols. They are referenced by the megaplan engine at runtime, but they are design artifacts, not runtime state. **Move to `docs/megaplan_chains/schemas/`**.

## Least-Risk First Batch

**Batch 1** (move briefs + chains + ideas + tickets + audit doc to `docs/megaplan_chains/`) is the safest starting point because:

- These are plain Markdown and YAML files with no hardcoded path dependencies inside the runtime engine (chain configs reference each other via relative paths *within the moved tree*, which stay valid if the subtree is kept intact).
- The moved files will no longer be force-tracked; they will naturally be tracked via `docs/`.
- No code changes are needed — the megaplan engine reads brief paths from chain YAML `idea:` fields, which reference `.megaplan/briefs/...`. After the move, the engine would need a config base-path override or a symlink `.megaplan/briefs → docs/megaplan_chains/briefs/`. A symlink is the lowest-risk mechanism:
  ```bash
  ln -s docs/megaplan_chains/briefs .megaplan/briefs
  ln -s docs/megaplan_chains/ideas .megaplan/ideas
  ln -s docs/megaplan_chains/chains .megaplan/chains
  ```
  This way the engine sees the same paths, but the canonical copy lives in `docs/`.

**Do NOT start with Batch 2 (schemas)** — the engine reads schemas from `.megaplan/schemas/` at runtime with no configurable base path. Moving them requires an engine patch or a symlink, which should be coordinated with the megaplan-harness package owner.
