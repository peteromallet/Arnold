A local `.megaplan/README.md` can serve as developer orientation without conflating local tool state with project docs.

**Suggested content for `.megaplan/README.md`:**

```markdown
# .megaplan/

This directory is git-ignored. It holds local agent-run planning artifacts
used by the megaplan/arnold harness.

## Why ignored

The `.megaplan/` tree combines authored design briefs with runtime execution
state (plan sessions, chain traces, locks, logs, telemetry). Keeping it
entirely local avoids committing transient LLM outputs, eval verdicts,
API keys, and session locks.

## What is here

| Path | Purpose |
|---|---|
| `briefs/` | Authored design briefs and chain specs — most valuable content. Still git-ignored because they are consumed locally by the harness. |
| `schemas/` | JSON plan-session schemas (plan, execution, critique, gate, etc.) |
| `chains/` | YAML chain orchestration definitions |
| `ideas/` | Forward-looking design RFCs |
| `debt.json` | Curated tech debt registry |
| `plans/` | Per-session execution artifacts (transient) |
| `epics/` | Timeline event logs (transient) |
| `.state-locks/` | Runtime lock files (transient) |
| `logs/` | Driver session logs (transient) |

## Committed equivalents

Authored planning material that has been promoted to committed docs lives in:
- `docs/megaplan_chains/` — promoted chain specs (excellence_epic, readable_ready_templates, pristine_cleanup, node_resolution_epic)
- `docs/agent-edit/` — agent-edit contracts and architecture docs

## Related

- `docs/structure_cleanup/megaplan_briefs/` — analysis of what to track vs ignore
```
