### 2. Hypothesis Usage

**Two Hypothesis users found — but they're structurally dissociated from the `property/` directory:**

1. **`tests/test_codemod_hypothesis.py`** (374 lines) — Full Hypothesis `@given`-based property tests for `port_convert_workflow`. Uses 7 property tests with handwritten composite strategies. **Located in the root `tests/` directory, NOT in `tests/property/`.** This is a discoverability issue — a developer looking for Hypothesis tests in `tests/property/` will miss this file entirely.

2. **`tests/property/test_emitter_fuzz.py`** (443 lines) — Uses **seeded Python `random.Random`**, not Hypothesis. Despite being the only file in `tests/property/`, it's a classic fuzz test with deterministic seeding rather than a Hypothesis property test.

**Hypothesis dependency is declared** in `pyproject.toml` under `[project.optional-dependencies] dev`: `"hypothesis>=6.100"`.

**Gap:** No `pyproject.toml` marker for Hypothesis tests, so they can't be selected via `pytest -m hypothesis`. The Hypothesis tests run as part of the default test suite (no marker filter). With `max_examples=40` per test × 7 tests = 280 Hypothesis examples per run, this adds meaningful time to default runs.

### 8. Immediate Cleanup Actions (Safe to Do Now)

1. **Fix `tests/edgecases/test_json_formats.py::test_boolean_and_null_like_inputs`** — The test name says "boolean and null-like" but the test only checks numeric values (42, 20, 7.5). Rename to `test_numeric_inputs_preserved` or add actual boolean/null test cases.

2. **Fix misspelling of `edgecases/` → `edge_cases/`** — This is a directory rename that would affect the existing `known_failures.txt` entry. **Recommend deferring** due to the complexity of tracking down every import reference. Document the naming instead.

3. **Add `__init__.py` docstrings** to `tests/security/__init__.py` and `tests/edgecases/__init__.py` and `tests/parity/__init__.py` and `tests/intent/__init__.py` — currently empty or non-descriptive. Characterization has a good model docstring that should be replicated.

4. **Register a `hypothesis` pytest marker** in `pyproject.toml` so Hypothesis tests can be filtered (`pytest -m hypothesis`). Currently they run as part of the default suite.

5. **Reset gate context var in root conftest** — The root `conftest.py` resets `_CURRENT_WORKFLOW` and `comfy_backend` cache, but does NOT reset `_gate_context_var` from `vibecomfy.security.gate`. This could cause test-pollution across security tests.

### 9. Implementation Recommendations (Separated from Structure)

- **Move `tests/test_codemod_hypothesis.py` into `tests/property/`** — It's a true Hypothesis-based property test and belongs with the fuzz test. This requires updating imports and ensuring the module path doesn't conflict.
- **Consolidate `tests/edgecases/` into the existing `tests/property/` or merge into root** — The 10 edge case files test mixed surfaces; some overlap with property tests, some are pure unit tests. A cleaner split would be: property/fuzz in `property/`, boundary/error conditions in the root or inline.
- **Add `security` pytest marker** — Security tests can't be selected independently. A `pytest -m security` filter would be useful for CI gate profiling.
- **Add `edgecase` or `edge` pytest marker** — So edge case tests can be toggled separately from unit tests.
- **Remove deterministic redundancy** — `tests/edgecases/test_determinism.py` is redundant with `tests/test_codemod_hypothesis.py` Property 2 (deterministic Python) and Property 3 (importable). Drop the edge case file and its standalone tests.
- **Move `tests/edgecases/test_runtime_failures.py` into `tests/test_errors.py`** — The three tests (missing node, empty workflow, invalid ready_id) are a better fit for the existing error tests file.
