## Audit Review: `docs/` Organization

### 1. Natural Subfolders

| Proposed Folder | Files | Rationale |
|---|---|---|
| **`architecture/`** | `vibeworkflow.md`, `contextvars_pattern.md`, `adding_templates_models.md` | Core IR, binding pattern, lifecycle — stable reference |
| **`workflow/`** (or **`authoring/`**) | `authoring.md`, `template_porting_workbench.md`, `workflow_onboarding_parity_tooling.md`, `workflow_corpus.md`, `custom_nodes.md` | How to use the system day-to-day |
| **`coverage/`** | `ltx_workflow_coverage.md`, `wanvideo_wrapper_coverage.md`, `z_flux_workflow_coverage.md` | Model-family coverage maps — natural group, cross-reference each other |
| **`cleanup/`** (or **`audits/`**) | `corpus_parity_allowlist.md`, `readable_ready_template_cleanup_plan.md`, `template_cleanup_followups.md`, `decorator_template_emitter_completion.md`, `subgraph_inner_id_recoverability.md`, `custom_node_wrapper_codegen.md` | Temporal audit/plan docs — lower ongoing relevance after M1 close |
| **`examples/`** | `gold_template_wan_i2v.py` | Anomalous `.py` file among `.md` — belongs in an examples folder |

### 2. Highest-Confidence Moves

- **Coverage triple → `coverage/`**: All three docs (LTX, WanVideoWrapper, Flux/Z) share the same structure (source streams → ready templates → runtime plan) and already link to each other.
- **Audit/cleanup docs → `cleanup/`**: Six docs tied to specific sweeps/plans (corpus allowlist, emitter completion, subgraph investigation, codegen pipeline). They are valuable artifacts but not day-to-day authoring references.
- **`gold_template_wan_i2v.py` → `examples/`**: It's a `.py` executable in a markdown directory. Rename to `.md` or move to `examples/` to avoid toolchain confusion.
- **`contextvars_pattern.md` → `architecture/`**: Only 49 lines, purely architectural — buried among operational docs otherwise.

### 3. Files That Should Remain Prominent

Keep these at the top level or in a well-indexed `workflow/` subfolder:

- **`authoring.md`** — primary authoring surface doc; referenced by multiple other docs
- **`adding_templates_models.md`** — the lifecycle entry point (363 lines, cross-referenced heavily)
- **`workflow_corpus.md`** — defines corpus manifest structure; referenced by coverage docs
- **`template_porting_workbench.md`** — operational CLI reference for port/check/convert/doctor
- **`custom_nodes.md`** — node resolution policy and lockfile reference

Consider a top-level `docs/README.md` index file referencing the subfolders.

### 4. Risks & Reference Checks

- **Cross-ref breakage**: `authoring.md` links to `docs/python_composition_dsl_plan.md` (not in candidate list — verify it exists or was retired). `workflow_corpus.md` links to `docs/structural_issues.md` (ditto). Subfolder moves require updating all relative `[text](file.md)` links.
- **Staleness risk**: `corpus_parity_allowlist.md` (316 lines, dated 2026-05-26) enumerates 80 entries — high maintenance burden if coverage evolves. `subgraph_inner_id_recoverability.md` is a one-off M1 investigation with zero cross-references; could be archived after M1 close.
- **`.py` in `docs/`**: `gold_template_wan_i2v.py` is a valid Python file that `import` / linters / formatters will pick up. Tools that glob `docs/**/*.md` will miss it. Either add a `docs/examples/` folder or rename to `.md` with a code block.
- **Cross-folder visibility**: Coverage docs reference `ready_templates/video/...` paths — these are code paths (not doc paths), safe from the move, but verify no doc-to-doc relative links between coverage docs and cleanup/ or architecture/ docs exist.
