## Audit: Environment / Secrets Handling

### 1. Is `this.env` safe/intentional to track at root?

**Not safe; semi-intentional (developer convenience file).**  

`.gitignore` line 36 lists `this.env`, so it is **protected from accidental `git add`**. It is **not currently tracked by git**. However:

- The file sits at the repo root with a nonstandard name (`this.env` instead of `.env` or `.env.example`).
- No source code loads `this.env`. All env-file loading points use `.env` (dot env):
  - `vibecomfy/commands/runpod.py` line 42: `load_dotenv(root / ".env")`  
  - `scripts/run_local_agent_comfy.sh` line 61: reads `~/.hermes/.env`  
  - `vibecomfy/cli/_debug.py` line 415: mentions `~/.hermes/.env`
- The only reference to `this.env` in code/docs is in planning/audit markdown files that flag it as a problem.

### 2. Does it contain secrets or local paths?

**Yes ŌĆö live API credentials in plaintext.**

```
OPENAI_API_KEY=***          ŌåÉ redacted (safe)
ANTHROPIC_API_KEY=***       ŌåÉ redacted (safe)
GEMINI_API_KEY=***          ŌåÉ redacted (safe)
FAL_KEY=873bb4e7-7ee0-4fc9-bf5d-65617511dbcb:549aee9a0d3da96240a01433bfdbc40e  ŌåÉ LIVE, NOT REDACTED
FAL_API_KEY="45f9bb...5e22"  ŌåÉ partial credential
```

**`FAL_KEY` is a real, non-redacted credential.** A single `git add -f this.env` or a pre-gitignore history rewrite would publish it. No local paths (no `/Users/...` strings) are present.

### 3. Should it be renamed, moved, ignored, or kept?

**Replace with a tracked `.env.example` template, then delete `this.env` locally.**  

Four reasons:

| Issue | Evidence |
|---|---|
| **Secrets leak risk** | `FAL_KEY` is live in plaintext; gitignore is the only protection |
| **Name inconsistency** | Code loads `.env` (dot env), not `this.env`; no documentation tells contributors where to put API keys |
| **`.env` is NOT gitignored** | `.gitignore` lists `this.env` (line 36) but not `*.env` or `.env`. If someone copies to the conventional `.env` name, it will be tracked. |
| **No onboarding documentation** | README.md (250 lines) doesn't mention API key setup at all. A new contributor has no way to know where to put keys. |

### 4. What exact safe change should happen now?

**Two-file change, no disruption to developer workflow:**

**Step A ŌĆö Create `.env.example` (tracked) at repo root:**

```
# ~/.config/vibecomfy/env ŌĆö API keys for VibeComfy tools and agentic backends.
# Copy this file to ~/.config/vibecomfy/env (or export the vars in your shell profile).
# VibeComfy's runpod command also looks for a .env file next to the runpod-lifecycle checkout.
#
# Required for agentic workflows:
OPENAI_API_KEY=***
ANTHROPIC_API_KEY=***
# Required for image/video generation via Fal.ai:
FAL_KEY=***
FAL_API_KEY=***
# Required for image generation via Gemini:
GEMINI_API_KEY=***
```

**Step B ŌĆö Add `.env` to `.gitignore` (alongside the existing `this.env` entry):**

In `.gitignore`, add `.env` on a new line near line 36 (the `this.env` line), so the conventional filename is also protected:

```
this.env
.env
```

**Step C ŌĆö Delete `this.env` locally** after verifying the `.env.example` is in place.

**Step D ŌĆö (Optional but recommended)** Add a short "API Keys" section to `README.md`:

> ### API Keys
> Copy `docs/env.example` to `~/.config/vibecomfy/env` (or export the vars in your shell profile) with your API keys. Required vars: `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `FAL_KEY`, `FAL_API_KEY`, `GEMINI_API_KEY`.

### Firm Recommendation

| Action | Priority | Rationale |
|---|---|---|
| **Create `.env.example`** | **Immediate** | Gives contributors a documented, safe template. One live key (`FAL_KEY`) is on disk right now. |
| **Add `.env` to `.gitignore`** | **Immediate** | Closes the gap where the conventional dot-env name is unprotected. |
| **Delete `this.env`** | **Immediate** | Once `.env.example` exists, `this.env` is redundant and carries continued leak risk. |
| **Document API keys in README** | This PR or next | Removes the onboarding friction for new contributors. |

The existing gitignore entry on line 36 already prevents accidental commits of `this.env`, but the plaintext `FAL_KEY` on line 6 is a credential that should never have been stored in the repo tree, even ignored. The `.env.example` pattern eliminates the risk entirely: credentials never live in the repo directory, and contributors get a documented setup path.
