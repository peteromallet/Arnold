## `.github/` Audit — Findings

### Inventory

```
.github/workflows/
  ci.yml               # Fast CI suite — push + PR
  strict-ready.yml     # Strict-ready template gates — PR + push (main, megaplan/*)
  parity.yml           # Canonical parity check — push + PR
  schema_freshness.yml # Schema freshness hash verification — workflow_dispatch only
  e2e_matrix.yml       # Weekly E2E RunPod matrix — schedule + workflow_dispatch
  nightly-runpod.yml   # Daily RunPod smoke — schedule + workflow_dispatch
```

No other files exist. No `.github/README.md`, no stale artifacts, no generated output.

### Q1: Are workflow files logically named and grouped?

**Yes.** All six files live in the standard `workflows/` subdirectory. Each filename clearly maps to its purpose:

| File | Trigger | Purpose |
|---|---|---|
| `ci.yml` | push + PR | Primary fast CI gate (tests, coverage, snapshot check, oracle gate) |
| `strict-ready.yml` | PR + selective push | Template static index + strict-ready gates |
| `parity.yml` | push + PR | Canonical ready-template parity check |
| `schema_freshness.yml` | workflow_dispatch | On-demand per-pack schema hash drift detection |
| `e2e_matrix.yml` | weekly + manual | Multi-template RunPod E2E matrix |
| `nightly-runpod.yml` | daily + manual | Single-pod RunPod smoke test |

One minor naming inconsistency: `e2e_matrix.yml`'s workflow name says `"e2e-matrix"` (name: in YAML) but a more descriptive name like `"runpod-e2e-matrix"` would signal the RunPod dependency without reading the file. Not worth changing.

### Q2: Is any stale or generated output committed under `.github/`?

**No.** Every file is a hand-authored CI workflow definition. No build artifacts, no stale generated outputs, no binary files, no temp files.

### Q3: Should `.github/README.md` exist?

**No.** There is no adoption barrier that a `README.md` would solve:

- Every workflow file has extensive YAML comment headers documenting its purpose, triggers, and steps (especially `ci.yml`, `schema_freshness.yml`, `e2e_matrix.yml`, `nightly-runpod.yml`).
- Cross-references to CI already exist in `docs/testing/overview.md` and `docs/comfy_version_support.md`.
- The `docs/structure_cleanup/` analysis already classifies `.github/` as "permanent VCS/CI infrastructure" in its deletion-risk review.
- Adding one would be documentation churn with no measurable benefit.

### Q4: What safe cleanup is available without changing CI behavior?

| # | Item | Action | Risk |
|---|---|---|---|
| 1 | **`strict-ready.yml` uses `actions/setup-python@v5` + pip** instead of `uv` like all other workflows | **Leave alone.** Standardizing to `uv` is a deliberate CI change (different resolver/venv), not safe cleanup. | Cosmetic inconsistency only |
| 2 | **`schema_freshness.yml` has ~130 lines of embedded Python** (lines 92–220) doing per-pack hash verification | **Extract to `tools/check_schema_freshness.py`** and call `python -m tools.check_schema_freshness` from the workflow. Behavior is identical; the Python becomes unit-testable and lintable. | Zero — pure refactor |
| 3 | **`ci.yml` line 29 says `name: ci`** — the shortest name in the set | Naming is fine. No change needed. | — |
| 4 | **No stale files to delete** | Nothing to prune. | — |

### Exact Recommendations

1. **Keep all 6 workflow files in place as-is.** No deletions, no renames, no trigger changes.

2. **Optionally extract the inline Python from `schema_freshness.yml`** into `tools/check_schema_freshness.py` and replace the embedded script with a single `run: python -m tools.check_schema_freshness` call. This is the only safe refactor available — CI behavior is identical, and the logic becomes maintainable/testable.

3. **Do not create `.github/README.md`.** The workflows are self-documenting, and the team already references CI from `docs/testing/`.

4. **No other cleanup available.** `strict-ready.yml`'s pip-vs-uv inconsistency is cosmetic; standardizing would be a CI change, not cleanup.

**Bottom line:** The `.github/` tree is clean, properly named, and contains no stale output. The only actionable item is extracting `schema_freshness.yml`'s embedded Python to a tool module — and even that is optional.
