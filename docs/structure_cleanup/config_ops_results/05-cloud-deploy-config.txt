### Question 1: Does `cloud.yaml` earn root placement?

**No.** Move it to `deploy/cloud.yaml` or `.cloud/cloud.yaml`.

**Rationale:**

- `cloud.yaml` is a **Railway PaaS deployment config** specific to one use case (the megaplan cloud chain runner). It is not Python packaging config, not CI config, not project-wide infrastructure.
- A root-scanning new contributor sees `cloud.yaml` and cannot tell if it's Kubernetes, Railway, Terraform, or something else without opening it.
- The `structure_cleanup` audit already flagged this: `docs/structure_cleanup/results/08-user-navigation.txt` (line 29) recommends moving to `deploy/cloud.yaml`.
- **No Python code, no CI config, no tooling references `cloud.yaml` by hardcoded path.** Zero hits in `.py`, `.toml`, `.cfg`, `Dockerfile*`, or `.github/` files. All references are `--cloud-yaml cloud.yaml` CLI flags — fully overridable.

### Question 3: Should there be a `deploy/` or `ops/` folder?

- **`deploy/` — yes, safe to create.**
  - `cloud.yaml` is only referenced via `--cloud-yaml <path>` CLI flags (6 spots in `docs/megaplan_chains/readable_ready_templates/README.md`).
  - None of the shell scripts (`megaplan_cloud_operator_loop.sh`, `megaplan_cloud_recovery_loop.sh`) hardcode `cloud.yaml` — they reference chain specs only.
  - No Python imports, no CI config, no tooling depends on the bare name.
  - **Breakage if moved:** ~6 CLI arg strings in one doc file. Trivial find-and-replace.

- **`ops/` — not needed.**
  - `scripts/` already holds all operational scripts (8 RunPod + 3 megaplan cloud) and is documented in `scripts/README.md`.
  - Creating `ops/` would divide operational scripts between two directories without clear boundary, confusing users and breaking the `scripts/README.md` contract.
