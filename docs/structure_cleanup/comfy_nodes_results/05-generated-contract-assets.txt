### Should Both Be Checked In?

**Current status: The generated file is not consumed by anything.** Zero production imports:

```
$ grep -r "agent_edit_response_contract_generated" --include="*.js" --include="*.mjs"
# Only references: the generator script, the drift-guard test, and docs.
```

The generated file exists as a **standalone reference artifact** — a drift-guarded snapshot to ensure Python→JS contract alignment. Per the existing cleanup doc (`docs/structure_cleanup/vibe_root_results/05-generated-cache-artifacts.txt`), the current decision is **Keep, intentionally checked in**.

**My assessment:** Defensible but currently redundant. The generated file adds maintenance burden (two copies of every constant must stay in sync) without any consumer benefit. The ideal future state would be to have the hand-maintained file **import from the generated file** (for constants + snake_case recovery functions), eliminating the duplication. But as designed today, if you check in the generated file, you must keep the duplication alive.

### Cleanup Recommendation

**Defer — but document as a known debt item.** The safe path:

1. **Do not delete** the generated file now — it's the drift-guard target. Deleting it would break the test.
2. **Do not remove** the duplication from `contract.js` — the hand-maintained file doesn't import the generated one.
3. **Cleanup candidate** for a future refactor: Make `agent_edit_response_contract.js` import the constants and snake_case recovery functions from `_generated.js`, then delete the duplicated inline copies.

### Verification Command

```bash
python -m pytest tests/test_agent_contract_codegen.py -v
```

This drift-guard test runs the generator against a temp file and does a byte-for-byte comparison with the committed generated file. It will fail if:
- `contracts.py` constants changed without regeneration
- The generator script was modified
- The generated file was hand-edited
