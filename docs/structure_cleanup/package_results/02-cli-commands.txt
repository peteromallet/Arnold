### 1. Is command registration explicit and discoverable?

**Yes — excellent design.** Registration follows a clean three-tier contract:

1. **`COMMANDS` tuple** (`commands/__init__.py`, lines 16–42) — an explicit manifest of every top-level CLI verb and its module path.
2. **`register_commands()`** iterates the tuple, calls `import_module()`, then calls `module.register(proxy)` — enforced by a `callable` check in `load_command()`.
3. Each command module must expose a `register(subparsers)` function that calls `subparsers.add_parser(...)` and sets `set_defaults(func=...)`.

Every command is discoverable by reading the 24-line `COMMANDS` tuple or by running `vibecomfy --help`.

The `_SubparsersWithParents` wrapper (lines 69–86) transparently injects `--yes`/`--non-interactive` flags into every subparser — a clean cross-cutting concern without individual command modules needing to opt in.

**Entrypoint contract** (what each command module must export):
```python
def register(subparsers: argparse._SubParsersAction | _SubparsersWithParents) -> None:
    p = subparsers.add_parser("name", ...)
    p.add_argument(...)
    p.set_defaults(func=_handler)
```

### 2. Are command modules grouped coherently?

**Generally yes.** Breakdown:

| Group | Location | Subcommands |
|---|---|---|
| **Flat top-level** | `vibecomfy/commands/*.py` (non-underscore) | sources, workflows, nodes, analyze, search, inspect, contract, validate, doctor, fetch, models, run, runtime, session, logs, debug, runpod, watchdog, schemas, check, agentic, copy-to-recipe, test, config |
| **Port sub-tree** | `vibecomfy/commands/port/` (package, 13 submodules) | check, convert, widgets, export, validate-call, doctor-all, inventory, repair, rules, lint, simulate |

The port subpackage is well-factored: `_register.py` handles all argparse DSL, individual `_cmd_*` modules are separated by concern, and `__init__.py` re-exports the monkeypatch-sensitive symbols so tests can `setattr(...)` on the package.

The utility modules (`_output.py`, `_index_files.py`, `_workflow_path.py`, `_model_entries.py`) are shared internally by multiple commands — appropriate use of the private convention.

### 3. Stale / dead files under CLI/commands

**Two confirmed dead files:**

#### ❌ `vibecomfy/commands/port.py` — 1970-line zombie god-file

- Original monolithic `port` module, **superseded by `vibecomfy/commands/port/` package**.
- When Python imports `vibecomfy.commands.port`, the directory package (`port/__init__.py`) wins — `port.py` is never loaded.
- All 9 imports in tests resolve to the package, not this file.
- Zero references in `vibecomfy/` source code (only mentioned in `port/__init__.py` docstring: *"was split out of a single port.py god-file"*).
- **~1970 lines of dead code** on disk.

#### ❌ `vibecomfy/commands/_analyze_names.py` — 295-line unreferenced duplicate

- Near-identical copy of `analyze_names.py` (same `_CLASS_ALIASES` dict, same `analyze_names()` function at line 96).
- Difference: imports from `vibecomfy.porting.emit.emitter` vs `vibecomfy.porting.emitter` (the public version imports from the shorter path).
- **Zero imports** across the entire repo.
- The public `analyze_names.py` is the one actually used (imported by `analyze.py` line 19).

#### ⚠️ `vibecomfy/commands/analyze_names.py` (public) vs `_analyze_names.py` (private)

These are clearly the same module duplicated during a refactor. The public one won the dependency race. The private one is a leftover.
