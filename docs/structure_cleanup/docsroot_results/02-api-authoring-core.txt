### Q1: Should any move?

**No — all 5 belong as root entry guides.**

Rationale per file:

| File | Candidate Subdir | Verdict | Why |
|---|---|---|---|
| `docs/authoring.md` | `docs/api/` | ❌ Stay at root | It's the **primary authoring reference** — covers blocks, patches, ready templates, recipes, router, plugin discovery, artfacts/ops. It's far broader than "API docs." The existing `docs/api/README.md` has a narrow scope: the stable public import surface. `authoring.md` is a how-to-write-code guide, not an API reference. |
| `docs/vibeworkflow.md` | `docs/architecture/` | ❌ Stay at root | It's the **foundational IR contract** — 14 lines that define the data model. Root-level visibility ensures every reader (agent or human) finds it immediately. |
| `docs/api_stability.md` | `docs/api/` | ❌ Stay at root | `docs/api/` exists, and this 7-line policy doc *could* move there. But it's referenced by name in the structure cleanup briefs as a root brief. Moving a 7-line doc would require updating at least 1 reference for negligible discoverability gain. |
| `docs/errors_and_doctor.md` | `docs/runtime/` | ❌ Stay at root | `docs/runtime/` doesn't exist yet. Creating it for one doc is premature. The doc is about both authoring (porting) and runtime — it straddles categories. |
| `docs/comfy_version_support.md` | `docs/architecture/` | ❌ Stay at root | It's a policy/infrastructure doc about CI version targeting, not architecture. Root-level makes it findable for developers setting up CI. |

**Strongest precedent**: The project's own prior cleanup pass already documented this decision. `docs/structure_cleanup/docsroot_briefs/02-api-authoring-core.md` groups exactly these 5 files under the category **"docs root briefs"** — meaning the project has already analyzed this and concluded they belong at root.

### Q3: Safe Recommendation for This Pass

**Do NOT move these files. Leave them at `docs/` root.**

The evidence chain is:

1. **Precedent exists.** The `docs/structure_cleanup/docsroot_briefs/02-api-authoring-core.md` — a prior cleanup analysis — explicitly treats these 5 as "docs root briefs." Moving them contradicts the project's own prior organization decision.

2. **No subdirectory fits better than root.**
   - `docs/api/` exists but is scoped to the **stable import surface** (an API reference). `authoring.md` is a guide, not a reference.
   - `docs/runtime/` doesn't exist.
   - `docs/architecture/` is a mismatch for operational/user-facing docs.

3. **~23 links would need updating** for no behavioral or discoverability gain. That's pure churn.

4. **`authoring.md` internally references** `architecture/`, `templates/`, and `../AGENTS.md` via relative paths — any move would require rewriting those paths too.

**Exact paths to leave untouched (all five stay as root entry guides):**

```
docs/authoring.md
docs/vibeworkflow.md
docs/api_stability.md
docs/errors_and_doctor.md
docs/comfy_version_support.md
```

**If a future pass wants to reduce root-doc sprawl**, the right order is:
1. Create `docs/runtime/` first (if the runtime module grows its own doc set)
2. Then consider moving `errors_and_doctor.md` into it
3. Only then reassess whether `api_stability.md` belongs in `docs/api/` (only when that directory has enough mass to justify the link churn)
