## Analysis & Recommendation

### Answers

#### 1. Should these stay at docs root or move?

**Keep all three at `docs/` root.** They are cross-cutting reference docs read by agents and developers, not runtime lifecycle docs. Moving them would cause churn without payoff because:

- **`docs/node_pack_reconciliation.md`** has ~20 hardcoded string references in production code (`vibecomfy/schema/validate.py:22-55`). These are display-only strings, not broken links, but changing 20+ lines for a cosmetic move is not worthwhile.
- **`docs/custom_nodes.md`** is linked from `docs/README.md`, `CLAUDE.md`, `AGENTS.md`, and via relative paths from subdirs (e.g., `../../custom_nodes.md` from `docs/megaplan_chains/node_resolution_epic/README.md`).
- **`docs/comfy_version_support.md`** is referenced in `tests/test_comfy_version_compat.py` and from megaplan docs.
- None of these three fit cleanly into `docs/runtime/` (runpod/execution lifecycle), `docs/templates/` (template authoring guides), or `docs/migration/` (version migration). `comfy_version_support.md` is the closest fit for `docs/migration/`, but still crosses the cost/benefit threshold negatively.

#### 3. What is the least-risk organization improvement?

**Do nothing to these three files.** The existing `docs/structure_cleanup/` analysis already flagged `docs/` as **HIGHEST RISK** due to the density of cross-links (see `docs/structure_cleanup/docs_results/08-link-risk.txt`). These three files are not clutter — they are actively referenced core docs.

If you *must* reduce docs-root density, the only low-risk move is:

> **`docs/comfy_version_support.md` → `docs/migration/comfy_version_support.md`**

This file is about version-migration strategy, `docs/migration/` already exists with `docs/migration/v25_to_v27.md`, and `docs/roadmap_agentic_comfyui.md:267` already groups them together conceptually: *"already has `docs/comfy_version_support.md` + `docs/migration/v25_to_v27.md`"*. But even this single move breaks **6 references**.
