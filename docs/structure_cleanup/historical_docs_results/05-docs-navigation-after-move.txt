## Decision on `docs/plans/` Directory

**Recommendation: Keep `docs/plans/` as a README-only directory (do not delete, do not remove from navigation).**

Rationale:

1. **`docs/README.md` links to `[Plans](plans/)`** — removing the directory would create a 404 for anyone clicking that nav link, and the directory still serves a valid purpose as the designated location for *future* active/in-flight plans.

2. **The constraint says** *"Do not recommend deleting docs/plans/ if external references still need a stable index"* — while the two production-code references point to specific files (not the directory), the `docs/README.md` nav link is itself a stable index entry. More importantly, keeping the directory signals "this is where new plans land" rather than having plans drift back to `docs/` root or get scattered.

3. **Semantic clarity**: The cleanup draws a boundary — `docs/historical/` for completed/archived plans, `docs/plans/` for active/in-flight plans. Removing `docs/plans/` would lose that boundary and force any future plan to either live at `docs/` root or get incorrectly dumped into `docs/historical/`.
