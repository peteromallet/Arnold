## 1. .gitignore Additions Recommended

**Add these patterns:**
```
# Scratch/migration scripts
_fix_t6.py
_regen_templates.py

# Generated build artifacts
asset_manifest.json
finalize.json

# Tool caches (currently rely on fragile per-dir .gitignore stubs)
.ruff_cache/
.import_linter_cache/
```

`custom_nodes.lock` is a generated lockfile (header: `# Generated by vibecomfy nodes lock`). Like `uv.lock`, it should remain **tracked** — it pins node dependency versions for reproducibility. Do not ignore it.

`version_matrix.json` is a durability contract pinning the pinned ComfyUI commit. Keep tracked; do not ignore.

## 3. Tracked Scratch-Looking Files Needing Migration

These are **tracked by git** and look like scratch or misplaced planning docs:

- **`plan_v2.md`, `revised_plan.md`** — Root-level planning documents that should live under `docs/planning/` or be archived if stale.
- **`CLAUDE.md`** — Agent context file. Belongs in `docs/` if durable; fine at root if actively used by tooling, but check whether the project prefers it at root vs. documented elsewhere.
- **`agentic/briefs/` (20+ briefs) + `agentic/actors_m4/`, `agentic/actors_m5/`** — Agent-run artifacts: brief markdown files and generated test harness `.py` files. The markdown briefs are essentially task prompts; the actor `.py` files are synthetic test fixtures. If they're durable eval infrastructure, move under `tests/agentic/`. If ephemeral, **git-remove** them and add to `.gitignore` under `agentic/`.

## 4. Risks

- **`custom_nodes.lock` is generated but should be tracked** — Do _not_ add to `.gitignore`. It serves the same role as `uv.lock` for node dependency pinning. Ignoring it would break reproducible installs.
- **`agentic/` migration is medium risk** — The actor `.py` files import from `vibecomfy` and may be used by CI/eval pipeline. Verify references before moving or removing.
- **`plan_v2.md` / `revised_plan.md`** — Check `git log` before deleting; they may contain historical decision records worth keeping in `docs/archive/`.
- **Broad `_*.py` glob is tempting but dangerous** — Could catch a legitimate module if one is ever named this way at root. Stick to explicit filenames.
