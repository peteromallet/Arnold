**Substantive agreement.** The written answer correctly captures the amendment-protocol intent and the plan reflects it faithfully for M2–M4. Two gaps exist.

### Where the plan reflects the answer

- **Amendment protocol**: M1 defines `docs/arnold/workflow-manifest-amendments.md` with a version-bump and review protocol.
- **M2**: Scope says compiler-discovered missing invariants must update M1 contract tests and the amendment doc.
- **M3**: Done Criterion 13 requires updating M1 contract tests, M2 compiler tests, and the amendment doc before M4 starts.
- **M4**: Done Criterion 15 requires the same for content-type/state/inventory discoveries.
- **Blocking**: Sequential `depends_on` in `chain.yaml` plus done criteria ensure no milestone completes while contract tests are stale; downstream milestones cannot start until upstream ones finish.
- **Post-merge safety net**: `chain.yaml`'s `post_merge_conformance` and M6 Done Criterion 31 recheck the integrated result so merge order cannot resurrect stale contracts or deleted surfaces.

### Gap 1: M5 and M6 omit the amendment clause

The written answer says "**Any** downstream discovery" must go through the amendment protocol. Yet M5 and M6 contain no `If X discovers manifest/kernel/runtime contract changes, update M1/M2/M3 contract tests..."` clause. M5 anti-scope says "Do not reopen DSL/runtime architecture," and M6 anti-scope says "No new architecture features," so the omission is defensible — but the written answer's "any" is broader than the plan's explicit coverage.

### Gap 2: "Block downstream milestones" is implicit, not gated

The plan doesn't model an explicit "if amendment occurred, re-run upstream gates before proceeding" mechanism. It relies on (a) sequential `depends_on` preventing downstream start until upstream completion, and (b) `post_merge_conformance` as the catch-all. If M4 amends M1 contracts during its branch, M2 and M3 are already "done" — there's no forced re-validation of M2/M3 against the amended M1 until post-merge. This works but is a softer guarantee than an explicit re-gate.

### Recommended plan edits

1. **M5 scope, add**: "If shipped-pipeline, CLI, doc, or scaffold work discovers a manifest/kernel/runtime contract gap, update M1/M2/M3 contract tests and `workflow-manifest-amendments.md`; do not silently widen contracts."
2. **M6 scope, add**: "If deletion/conformance work discovers a contract gap that was masked by deleted surfaces, update M1/M2/M3 contract tests and `workflow-manifest-amendments.md`."
3. **chain.yaml `post_merge_conformance`**: add a sentence — "If any downstream milestone amended M1/M2/M3 contracts, re-run the full upstream milestone gate suite (not only the amended tests) before declaring merge-result conformance green."
