## Position on Expectation 3 — Completeness vs. Overreach

**Verdict**: M1's scope is ~80% appropriate but carries execution-coupled design risk on retry, loops, and fanout/fanin. These three should be stub-only in M1, with concrete shapes deferred to M3. One schema field is missing.

---

### Why (the core tension)

M1 is a schema-design milestone. It must define enough that M2 (compiler) can target the manifest and M3 (runner) has a stable contract. But fields whose shape depends on *how* the runner operates — backoff policies, loop evaluation order, fanin reduce semantics — cannot be correctly designed before the runner exists. M1's own brief asks: *"What minimal manifest schema is sufficient to unblock M2 without over-designing runner internals?"* That question answers itself for the three execution-heavy fields.

### Field categorization

| Category | Fields | Why |
|---|---|---|
| **M1-proper** | nodes, refs, edges, source spans, topology hashes, manifest hashes | Structural/identity concerns. No overreach. |
| **M1-schema + M3-verify** | policies, budgets, reentry IDs, subpipelines, human gates | Need schema slots so M2 can compile references. Shapes are provisional; carry TODOs for M3 validation. |
| **Defer-to-M3 (stub only)** | retry policy shape, loop control, fanout/fanin reduce | Tightly coupled to execution engine. M1 should define only node-type discriminators (`RetryNode`, `LoopNode`, `FanoutNode`) with opaque `config: dict` bags. Concrete backoff/bound/reduce shapes go in M3. |
| **Missing** | artifact binding schema | No field exists for a node to declare expected input/output artifacts with hash types. M3's content-addressed store needs this anchor. Add `artifact_specs: list[ArtifactSpec]` to M1. |

### Milestone corrections

1. **M1**: Narrow retry/loop/fanout to discriminators + opaque config bags. Add `ArtifactSpec` (name, content_type, hash_algo, optional binding). Keep everything else as-is.

2. **M2**: No change — the compiler only needs discriminators to lower pattern nodes; it doesn't need concrete retry/loop/reduce shapes.

3. **M3**: Owns concrete retry (`RetryPolicy` with max_attempts, backoff, retry_on), loop (`LoopControl` with until, max_iterations, break_on), and fanout/fanin (`FanoutSpec`, `ReduceStrategy`). Must backfill M1 stubs and update manifest hash stability tests.

### Proof gates

- **M1 exit**: Schema module exists with all 14 categories represented (3 as stubs). Artifact binding field present. Tests prove deterministic serialization.
- **M3 exit**: Stub shapes replaced with concrete dataclasses. Manifest hash changes documented. Existing M2 compile tests still pass against updated schema.
- **Cross-check**: M2 canonical Megaplan compile test does not silently depend on stub shapes being concrete.
