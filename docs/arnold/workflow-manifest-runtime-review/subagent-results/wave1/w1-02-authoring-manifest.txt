## Verdict: **Right — builders/patterns survive as compile-time authoring helpers; manifest alone is the executable contract.**

### Why the expectation is correct

The line the briefs draw — runtime executes manifests, not live Python object graphs, generator frames, or compatibility wrappers — is non-negotiable for three reasons that survive scrutiny:

1. **Deterministic resume/replay.** Manifest-coordinate resume (M3) requires `manifest_hash`, `reentry_id`, `scope_stack`, and `event_sequence` — all impossible if the runtime trusts live object identity, generator frame locals, or GC-dependent graphs.

2. **Cross-boundary execution.** A manifest serializes to JSON/YAML and can be shipped to a subprocess, container, or remote worker. Live `Stage`/`Edge`/`PipelineBuilder` graphs cannot without pickling the whole interpreter.

3. **Topology verification.** Deterministic topology hashing and compile-time validation (M2 done criteria) depend on inspecting *data*, not object graphs that may carry hidden closures, bound methods, or mutable state.

### The nuance: builders/patterns DO survive — as manifest-compiling constructors

The briefs do NOT ban builder surfaces. M2 explicitly creates `arnold.patterns` — constructors for `agent`, `branch`, `critique`, `fanout`, `human_gate`, `loop`, `merge`, `panel`, `retry`, `review`, `revise`, `subpipeline`, `tournament`, `external_call`. These ARE builders. The difference is *what they produce*:

| Dies in M6 | Survives as canonical |
|---|---|
| `PipelineBuilder`, `Stage`, `Edge`, `ParallelStage` — live object graphs passed to executor | `arnold.workflow.Pipeline(steps=[...])` — pure data that compiles to `WorkflowManifest` |
| `run_pipeline()` accepting live Python callables | `arnold.patterns` constructors returning serializable step values |

M2 Locked Decisions allow this explicitly: *"Helpers may standardize defaults but must return pure Arnold step values and may not hide topology, side effects, budgets, or dynamic routes."*

### What needs tightening in the plan

The M1 "Locked Decisions" say runtime executes *"not package-private graph executors"* — but M2 `arnold.patterns` is effectively a graph constructor. The plan should clarify the boundary explicitly: **Constructor surfaces compile → manifest. Manifest → runtime. No constructor object shall cross the compile boundary into execution.** Without this clarification, someone could argue `arnold.patterns` is just a renamed `PipelineBuilder`.

### Corrections by milestone

- **M1**: Add one sentence to Locked Decisions: *"Authoring helpers (including `arnold.patterns`) produce pure data that compiles to manifest; they shall not be importable from execution modules."*
- **M2**: Add a conformance test that `arnold.patterns` constructors produce only JSON-serializable, manifest-compatible data — no bound methods, closures, or live instances.
- **M3**: Add an import-boundary test that `arnold.execution` does not import `arnold.patterns`.
- **M6**: Delete `PipelineBuilder`/`Stage`/`Edge` but retain `arnold.patterns` as the blessed authoring surface. Update M6 done criterion #3 to explicitly list what survives.

### Proof gates

1. Compile a patterns-built pipeline → serialize manifest → deserialize in a separate Python process → execute → identical event journal.
2. AST/import scan confirming `arnold.execution.*` imports zero symbols from `arnold.patterns` or `arnold.workflow` (only `arnold.workflow.manifests` for schema types).
3. Any `arnold.patterns` constructor called with a closure, lambda, or bound method raises at construction time, not at runtime.
