## Verdict

**Right, with one missing nuance.**

## Why

The prohibition is correct in spirit — a neutral workflow runtime that imports product code isn't neutral, and the entire architecture collapses into a Megaplan-specific engine. The plan consistently reinforces this: M1 done criterion #5, M3's constraint against importing `arnold_pipelines.megaplan`, M4's locked decision that "Neutral Arnold packages must not depend on Megaplan product code," and sequencing point #9 ("keep policy vocabulary product-owned; generic runtime should route string decisions/capabilities without knowing Megaplan-specific gate recommendations").

However, the plan under-specifies **where the shared vocabulary boundary lives**. Neutral runtime must dispatch on capability strings, effect contracts, and event schemas that both sides agree on. If those types/enums live in product code, the runtime unavoidably imports product code to do basic dispatch. If they live in `arnold.kernel`, that's fine — but no milestone brief explicitly declares that ownership. The current "product-owned policy vocabulary" principle (sequencing anchor #9) only covers Megaplan-specific gate recommendations, not the shared contract surface.

There's also an ambiguity: absolute "must not import" could accidentally forbid harmless `TYPE_CHECKING`-guarded imports. The plan should be explicit.

## Plan Corrections

1. **M1** should add a brief to its scope: define shared contract vocabulary (capability string constants, effect contract base types, event family identifiers) in `arnold.kernel.*`, not in product. This makes the kernel the single source of truth for the wire protocol.

2. **M3** should add an explicit constraint: the runner resolves all capabilities and effects via string-keyed registry/dispatch — it must never import product-specific policy classes, prompt templates, or gate logic. This closes the loophole where "I didn't import the module, I just used a string that happens to match" becomes a dodge.

3. **M1** and **M4** should clarify whether `TYPE_CHECKING` imports across the boundary are permitted (for type narrowing in tests/tooling) or prohibited. If prohibited, say so. If permitted, state the exception explicitly so scanners don't false-positive.

## Tests/Gates

- **AST import scanner** (M1/M6): walks `arnold/workflow/`, `arnold/patterns/`, `arnold/execution/`, `arnold/kernel/` and fails on any `import arnold_pipelines` or `from arnold_pipelines` statement. Should run in CI pre-merge.

- **Installed-wheel smoke gate** (M6): imports `arnold.workflow`, `arnold.kernel`, `arnold.execution` in a clean venv and asserts `arnold_pipelines` is absent from `sys.modules`. Proves no transitive import leakage.

- **Capability dispatch test** (M3): confirms the runner can route a novel capability string (not in any product registry) without importing product code — the runtime surface accepts the string and either routes or rejects cleanly, never resolving it through a Megaplan import.
