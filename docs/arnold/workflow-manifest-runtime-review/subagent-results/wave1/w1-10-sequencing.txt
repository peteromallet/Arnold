## Verdict: Sequencing Is Fundamentally Correct but Brittle

The linear M1→M2→M3→M4→M5→M6 chain matches End-state expectation 10 and is architecturally coherent. However, three structural risks and two overloaded milestones threaten delivery.

---

### Sequencing Risks

**1. Manifest freeze without compiler feedback (M1↔M2 gap).** M1 locks `WorkflowManifest` v1 before M2 builds the compiler. If M2 discovers the manifest is missing fields for branch/loop/fanout/subpipeline lowering, the contract must be reopened — but no defined protocol exists for doing so. M5's escape hatch ("unless a migration bug exposes a missing invariant") comes too late.

**2. No integration smoke between runner and product (M3↔M4 gap).** M4's Megaplan migration is the first real consumer of M3's fake-backend runner. If M3's execution semantics don't cover a Megaplan edge case (human-gate resume, tiebreaker subpipeline retry), M4 stalls with no fallback. A thin integration gate between M3 and M4 — compile and fake-run the canonical Megaplan shape before migrating product code — would catch this early.

**3. M5 is four workstreams in one milestone.** Shipped-pipeline migration, CLI commands, docs/scaffolds, and inventory scanning are bundled. The inventory (which feeds M6's deletion list) shouldn't gate on completing all pipeline migrations — it should begin as soon as M4 lands, in parallel with M5a pipeline work.

---

### Missing Seams

- **No shadow-run period.** M4 proves parity through goldens, not live side-by-side execution. A temporary dual-run seam (old pipeline writes reference, new pipeline compares) would de-risk behavior regressions that goldens miss.
- **No contract amendment process.** If M2 or M3 needs manifest changes, there's no lightweight amendment path short of reopening M1. A `MANIFEST_AMENDMENTS.md` living document with version-bump protocol would suffice.
- **No incremental purge.** M6 is all-or-nothing deletion after M5 completes. An incremental deprecation-visible-in-M4 (emit warnings, tag deprecated imports) would surface external consumers before the final cut.

---

### Corrections by Milestone

| Milestone | Action |
|-----------|--------|
| **M1** | Split: M1a (guardrails/goldens, can run parallel with M1b) and M1b (manifest contract). Add amendment protocol doc. |
| **M3** | Add done criterion: "Canonical Megaplan shape from M2 compiles and fake-runs end-to-end." This becomes the M3→M4 gate. |
| **M4** | Add dual-run seam: optional shadow execution producing drift reports. Emit deprecation warnings on legacy import paths. |
| **M5** | Split into M5a (pipeline migration + CLI) and M5b (docs/scaffolds + inventory). Start inventory in M5a as soon as M4 lands. |
| **M6** | Add incremental gate: require M4 deprecation warnings to have been visible for one milestone before deletion. No pure-binary cutover. |

---

### Proof Gates

1. **M1b→M2 gate:** M1 manifest compiles a minimal branch+loop+fanout+subpipeline shape (hand-authored JSON) before M2 starts.
2. **M3→M4 gate:** M2's canonical Megaplan test manifest fake-runs through M3 runner with all node types exercised.
3. **M4→M5a gate:** Golden parity passes AND dual-run drift report shows zero un-explained diffs across 10 planning runs.
4. **M5b→M6 gate:** Inventory scanner returns zero unresolved shipped-pipeline blockers AND deprecation warnings have fired since M4.

**Bottom line:** The sequence is right; the seams between milestones are the risk. Add integration gates, split M1 and M5, and mandate a deprecation visibility period before M6 deletion.
