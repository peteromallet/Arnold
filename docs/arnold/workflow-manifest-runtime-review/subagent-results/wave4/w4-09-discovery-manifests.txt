Here is the inventory and disposition for the workflow-manifest-runtime plan artifacts, based on my inspection of the six milestone briefs, the review synthesis, the review subagent results (waves 1–4), and the existing codebase.

---

## Inventory and Disposition

**1. Discovery Manifests (old `arnold.pipeline.discovery.manifest`)**

- **Role:** AST-based extraction of pipeline metadata without importing modules. `Manifest` dataclass holds name, description, profile, modes, driver, entrypoint, API version, capabilities, and a content hash.
- **Final Disposition:** Survives as package metadata only. The M1 contract states explicitly: "Package discovery manifests are not runtime manifests." The neutral Arnold `arnold.pipeline.discovery.manifest` reader (`ARNOLD_IDENTITY_SCHEMA = "arnold.pipeline-manifest.v1"`) is used at discovery time for the old pipeline layout. The new `WorkflowManifest` v1 (to be built in M1/M2) is a separate, runtime-oriented contract. The old discovery manifest becomes a parallel metadata artifact scoped to package identity and trust grading, not to runtime execution.
- **Plan Hook:** M1 defines `WorkflowManifest` v1 dataclasses/serialization; M1 must "state whether [discovery manifests] survive as package metadata and how trust gates map to workflow manifests." M5 produces pipeline ID registry JSON artifacts alongside discovery metadata for every surviving shipped pipeline. W4 review adds subagent briefs specifically for discovery manifests (w4-09).
- **Gaps:** The mapping between old discovery manifest trust gates (`AUTO_EXEC`, `QUARANTINED`, `BLESSED`) and the new workflow manifest regime is not specified in detail. The `judge_manifest.py` discovery sidecar and `trust.py` `derive_tenant_id` have no explicit final disposition. Which fields from the old `Manifest` dataclass flow into `WorkflowManifest` is unresolved.

---

**2. Package Disposition**

- **Role:** `docs/arnold/package-disposition.md` and associated validation govern where pipeline packages live and what is public/provisional/internal.
- **Final Disposition:** Retained and strengthened. M4 produces a `legacy-surface-inventory.md` mapping every old surface to its M6 fate. M5 produces a "closed migration inventory" per shipped pipeline (migrate/delete/whitelist). M6 enforces package disposition validation against the final file tree with no rows pointing to deleted paths. Disposition becomes a conformance gate, not a planning hint.
- **Plan Hook:** M5 expands generated-artifact and per-pipeline disposition tables. M5 also produces `package-disposition-validation` and `generated-artifact-manifest` covering `_codex_skills`, `_composed`, pipeline SKILL.md, projection docs, registry JSON, docs generators, and scaffold templates.
- **Gaps:** The mechanism for auto-discovering and validating package disposition against the file tree is mentioned but not specified. Whether `package-disposition.md` is regenerated or hand-maintained is ambiguous.

---

**3. Pipeline IDs**

- **Role:** `arnold/pipeline/pipeline_id_registry.py` validates JSON registry files with `PipelineIdRegistry` containing `name`, `stable_id`, `typed_contract_capable`, `seam_id`, `previous_stable_id`, and other fields. Enforces uniqueness, standardizes naming conventions, and checks conformance to naming rules.
- **Final Disposition:** Survives in a migrated form. M5 must produce pipeline ID registry JSON artifacts for every shipped pipeline that survives. M5 adds inventory scanners that forbid old public authoring APIs. Discovery trust classification tests in M6 must pass with no hardcoded path fragments pointing to deleted directories.
- **Plan Hook:** M5 work: "pipeline ID registry JSON artifacts and package-discovery metadata for every shipped pipeline that survives." The W4 review wave includes w4-09 specifically for discovery manifests and pipeline registry artifacts.
- **Gaps:** The relationship between old `PipelineIdRegistry` (name/stable_id/contract-capable) and new `WorkflowManifest` pipeline identity is not defined. Whether the old registry JSON format is migrated 1:1 or replaced by a manifest-derived artifact is open.

---

**4. Registry Files (general)**

- **Role:** `arnold/pipeline/registry.py` (pipeline registry), `arnold/pipeline/schema_registry.py`, discovery registry, and the neutral content-type registry (new in M1) compose the registration infrastructure.
- **Final Disposition:** Split. The old pipeline registry is replaced by manifest-backed discovery. A new neutral content-type registry is created in M1 under `arnold.kernel.content_types` with `type_id`, `schema_version`, `schema_hash`, `retention_policy`, and provenance fields. Megaplan registers content types (plan, receipt, capsule, delta, gate signal, etc.) in M4 as product-owned registrations over the neutral kernel registry.
- **Plan Hook:** M1 defines neutral kernel registries. M4 registers Megaplan content types "without adding product schema knowledge to the neutral kernel." M3 implements content-type registry enforcement and artifact retention pins.
- **Gaps:** The old `schema_registry.py` disposition is unstated. Whether the old pipeline registry survives as a transitional query surface or is deleted in M5/M6 is open.

---

**5. Discovery Trust Gates**

- **Role:** `arnold/pipeline/discovery/trust.py` computes `TrustGrade` (`AUTO_EXEC`, `QUARANTINED`, `BLESSED`) from module filesystem origin, never from user-supplied metadata. Uses `in_tree_path_fragment` parameter and `BLESSED_ALLOWLIST`.
- **Final Disposition:** Survives as a discovery-time concept but likely transitions. The plan states "package discovery manifests are not runtime manifests" and M1 must state "how trust gates map to workflow manifests." The trust concept may be re-chartered under the new manifest regime or remain as a package-discovery-only check.
- **Plan Hook:** M6 requires "discovery trust classification tests pass with no hardcoded path fragments pointing to deleted directories." The W4 review includes w4-09 on discovery manifests. The review synthesis notes: "Define old discovery manifest vs. new WorkflowManifest coexistence."
- **Gaps:** The exact mapping between `TrustGrade` and `WorkflowManifest` execution gates is deferred to M1. Whether trust grades constrain runtime manifest execution (or remain a discovery-only concern) is unresolved.

---

**6. Docs Generation**

- **Role:** Regenerated agent-facing docs: `_codex_skills`, projection docs (`docs/reference/arnold-projections.md`), package authoring docs, package disposition data, scaffolds, and SKILL.md files. Some are hand-maintained, some generated.
- **Final Disposition:** All retained but updated to teach only the new explicit-node authoring surface. M5 must regenerate every generated artifact so no public/operator artifact references deleted surfaces. M6 enforces generated-artifact freshness (clean diff after regeneration). Registry JSON artifacts from pipeline IDs become generated outputs.
- **Plan Hook:** M5 "update docs for workflow authoring, manifest, runtime, patterns, package authoring, examples, and projections." M5 produces a generated-artifact manifest with owner, generator/check command, final location, and M6 disposition. W4 review addresses generated artifacts explicitly (w4-03).
- **Gaps:** The generator tooling for docs is not specified (existing `_gen_corpus.py`, `_gen_golden_traces.py`, `sync-skills.sh`). The review mentions `scripts/generate_arnold_docs.py` as a touchpoint, but it does not exist yet in the repo.

---

**7. Package Authoring Contracts**

- **Role:** `docs/arnold/package-authoring-contract.md` defines the contract for pipeline packages, including `build_pipeline()`, SKILL.md expectations, and metadata requirements.
- **Final Disposition:** Updated for the new explicit-node regime. M2 explicitly defines "the explicit-node package authoring contract: package-level `build_pipeline()`, required metadata, pipeline ID registration, SKILL.md expectations, discovery metadata, and how package manifests relate to WorkflowManifest output." M5 regenerates all package authoring docs.
- **Plan Hook:** M2 touchpoints include `docs/arnold/package-authoring-contract.md`. The package contract is broadened in M5 to cover all shipped pipelines.
- **Gaps:** The precise `build_pipeline()` signature and return type (Pipeline vs WorkflowManifest) is stated but not locked beyond "returns `workflow.Pipeline(...)`." Whether the old `arnold pipeline` entrypoints and the package contract coexist during migration is not detailed.

---

**8. Old vs New Manifest Systems**

- **Role:** **Old:** `arnold.pipeline.discovery.manifest` (AST extraction, `Manifest` dataclass, `ARNOLD_IDENTITY_SCHEMA`). `arnold.pipeline.discovery.judge_manifest` (sidecar JSON, `JUDGE_MANIFEST_SCHEMA`, evaluand records). **New:** `WorkflowManifest` v1 (to be built) with nodes, refs, edges, policies, source spans, capabilities, budgets, retry/loop/fanout/subpipeline, reentry IDs, topology hash, manifest hash, artifact binding, content-type registration, retention pins, provenance chains, generic suspension.
- **Final Disposition:** The old manifest system stays as package metadata (discovery-time only). The new `WorkflowManifest` becomes the runtime source of truth. They coexist with distinct purposes: old for package identity/discovery, new for execution. The old system may be pruned in M6 if its fields are fully subsumed by the new registry JSON artifacts.
- **Plan Hook:** M1 defines the new manifest and kernel contracts. M1 explicitly scopes: "Define how existing discovery/package manifests coexist with or are superseded by WorkflowManifest v1." The old `arnold.runtime` disposition is defined in M3 (refactor reusable pieces into `arnold.kernel`/`arnold.execution`, mark rest for deletion).
- **Gaps:** The old `judge_manifest` sidecar system has no stated disposition. Whether old `Manifest` discovery fields are migrated into `WorkflowManifest` or kept separate is open. The review synthesis flags this: "Define old discovery manifest vs. new WorkflowManifest coexistence." No existing behavioral golden covers discovery → runtime manifest transition.
