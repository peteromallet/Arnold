Now I have a comprehensive view of all plan artifacts, the wave1 review results, and what's still open. Here's my analysis:

## Verdict

The plan correctly owns branch, loop, fanout, retry, subpipeline, human-gate, and suspension as generic runtime primitives — and correctly defers concrete retry/loop/reduce *shapes* from M1 to M3. But three runtime semantics are **missing entirely**, two have **product-policy leakage**, and one critical deferral remains implicit.

---

### Missing Runtime Semantics

**1. Cancellation.** No milestone mentions cancelling an in-flight workflow or node. The runtime needs a `CANCELLED` terminal state (alongside `COMPLETED/FAILED/SUSPENDED`), a cancellation event, and a rule for propagation (cancel children of a cancelled fanout parent? abandon siblings?). Without it, runaway workflows have no generic kill path.

**2. Node-level timeout.** The plan has suspension `deadline_seconds` (human-gate adjacent), but no generic *execution* timeout: "this node must complete within N seconds or transition to TIMED_OUT/FAILED." Distinct from suspension — a node can time out *while* executing, not just while suspended.

**3. Pipeline-level deadline/TTL.** No mechanism for a manifest-level deadline after which the entire run terminates. M3's fake-backend tests can't exercise deadline-driven abort without it.

---

### Product-Policy Leakage

**1. Reducer ownership is blurred.** M2 lists `merge` as a pattern constructor; M3 says fanout/merge *mechanism* is generic but `majority_vote` vs `weighted_vote` is product policy. Yet no milestone draws the line: the runtime provides `fanout_completed(N)` → dispatch reducer → emit `fanout_reduced` event. The reducer *function* is product-owned and resolved via `module:qualname`. This must be explicit.

**2. Override/fallback edges carry policy assumptions.** M3 says "preserve generic override edges, fallback edges, dynamic topology overlays… as policy-blind primitives." But what constitutes an override vs. a fallback is inherently semantic — the runtime must treat them as opaque edge types with product-registered dispatch, not hardcode routing decisions.

---

### Deferrals That Must Become Explicit

**1. Escalation.** `chain.yaml` has `on_escalate: abort: stop_chain` at the chain level, and M3 mentions "supervisor event routing." But there's no node-level escalation primitive: when a node fails and the retry policy is exhausted, what event fires? Where does escalation route? This is a generic runtime concern that M4 (Megaplan) will consume, not define. M3 must add it.

**2. Compensating actions.** The event journal enables replay/audit, but no milestone addresses what happens when a node fails *after* a preceding node committed a side effect (`git_push` succeeded, then a later node fails). The runtime doesn't need to *perform* compensation, but it must provide an event hook (`node_failed_with_prior_effects`) so product packages can register compensations.

---

## Milestone Edits

| MS | Edit |
|----|------|
| **M1** | Add `CANCELLED` and `TIMED_OUT` to node terminal state enum. Add `node.timeout_seconds: Optional[float]` and `manifest.deadline_seconds: Optional[float]` as schema slots (opaque to M1, concrete semantics in M3). Add `escalation_target: Optional[NodeRef]` to node schema. |
| **M3** | Add explicit sections: **(a) Cancellation** — `cancel(manifest_hash, scope?)` emits `run_cancelled`, propagates to active children, terminal. **(b) Timeout** — node-level `timeout_seconds` enforced by runner; pipeline-level `deadline_seconds` checked before each node dispatch. **(c) Escalation** — `node_failed` + `retries_exhausted` → route to `escalation_target` if present, else `run_failed`. **(d) Reducer dispatch** — runtime invokes `module:qualname` reducer when all fanout children complete; reducer returns merged artifact ref. Runtime knows only the dispatch contract. **(e) Compensation hook** — `on_prior_effects` event emitted when a node fails after upstream effects committed; product registers handlers. |
| **M4** | Add note: Megaplan's tiebreaker subpipeline consumes M3's generic escalation target. Megaplan's override/fallback edges register as product-policy routing over M3's opaque edge types. |

## Proof Gates

1. **M1 gate:** Schema module defines `CANCELLED`, `TIMED_OUT` in `NodeTerminalState`; `node.timeout_seconds`, `manifest.deadline_seconds`, and `escalation_target` exist as optional fields. Tests prove round-trip serialization.

2. **M3 cancellation gate:** Fake-backend test starts 3-node pipeline, cancels at node 2, asserts `run_cancelled` event emitted, node 3 never starts, resume from event journal confirms cancelled terminal state.

3. **M3 timeout gate:** Node with `timeout_seconds=0.1` that sleeps forever transitions to `TIMED_OUT` within tolerance. Pipeline with `deadline_seconds=1` that would run 10s terminates after deadline.

4. **M3 escalation gate:** Node with `retry(max_attempts=2)` fails all attempts, `escalation_target` routes to a handler node, handler executes, pipeline completes via escalation path — not `run_failed`.

5. **M3 reducer gate:** Fanout with 3 children and `reducer="mypkg.reducers:concat"` completes all children, invokes reducer with child artifact refs, reducer output becomes parent node artifact. Reducer module is not imported by `arnold.execution`.

6. **Product-policy boundary gate:** `grep -r "escalat(e|ion).*megaplan\|override.*megaplan\|fallback.*megaplan" arnold/execution/ arnold/kernel/` returns zero matches.
