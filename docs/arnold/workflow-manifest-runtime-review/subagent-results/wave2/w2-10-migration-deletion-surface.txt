## Verdict

**The migration/deletion surface is the plan's weakest edge — the w2-10 subagent never ran, leaving it unreviewed by adversarial analysis. Three material scope gaps, two compatibility-creep vectors, and five unsafe deletion assumptions remain uncorrected.** Fixing them turns M6 from a risky all-or-nothing cut into a verifiable, deprecation-gated purge.

---

## Missing Scope

1. **No committed bridge-caller inventory.** `_BRIDGE_CALLERS.md` is referenced informally in w1-08 but absent from M4/M5/M6 touchpoints. The ~19 `run_pipeline_with_policy` callers, subloop execution paths, resume paths, and demo entries have no tracked repointing status. M4 must produce this as a committed artifact.

2. **On-disk state migration unaddressed.** `.megaplan/plans/*/state.json`, `plan_v*.md`, receipt JSON, capsules, and gate signals must survive the runner transition. No milestone defines a one-shot migration tool to convert old state into content-addressed blobs. Without it, M6 orphanes historical plan state.

3. **Old `arnold/runtime/` disposition missing.** M3 builds `arnold.execution` and `arnold.kernel` as replacements, but the existing `arnold/runtime/` package (envelope, resume, event_journal, dry_run) is never declared for deletion, merge, or refactor in any milestone.

4. **External consumer discovery absent.** M6's Open Questions ask whether external consumers exist, but no milestone defines a mechanism (deprecation warnings, shadow-run drift reports, PyPI pre-release) to surface them before deletion.

---

## Compatibility Creep

1. **Bridge scaffolding has no removal deadline.** w1-08 correctly identifies that `_bridge.py`, `_compatibility.py`, `_forward_m2_m3.py` must survive M4-M5 for repointing callers. But M6 brief says "No shims is the target" without acknowledging these temporary shims exist. The contradiction invites either premature deletion or permanent leakage.

2. **M2 pattern constructors creep into product territory.** 16 constructors are listed including `panel`, `tournament`, and `merge`. These are Megaplan product compositions, not neutral Arnold primitives. Baking them into `arnold.patterns` creates a new compatibility surface that M6 would then need to delete.

---

## Unsafe Deletion Assumptions

1. **M6 lists `arnold.pipelines.megaplan` for deletion** in Scope IN, but Locked Decisions says it "survives as the permanent product package." This internal contradiction could delete the product home.

2. **"Delete top-level `megaplan` package"** — but it doesn't exist on `origin/main`. Only the console entrypoint exists. The gate cannot pass or fail meaningfully.

3. **All-or-nothing deletion.** M6 deletes everything after M5 without incremental deprecation. No milestone mandates deprecation warnings be visible for one full milestone before deletion. A pure-binary cutover leaves no safety margin.

4. **No merge-result contamination gate.** M6's Done Criteria test only the feature branch. A merge from a stale branch (e.g., `native-python-pipelines`, old worktrees) could reintroduce deleted surfaces. No cross-branch quarantine protocol exists.

5. **Golden parity without live smoke.** M6 authorizes deletion based on golden fixtures alone. At minimum, one live-backed workflow run must match golden behavior before deletion.
