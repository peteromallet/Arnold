## Verdict: Missing Control/Observability Scope, Policy Leakage, and Duplicate State Risks

### 1. Control Interface Absent from Plan (MISSING SCOPE)

The neutral `arnold.control.interface.ControlBinding` protocol — with `valid_targets`, `recover_targets`, `apply_transition`, `synthesize_artifacts` — is a live, implemented control-plane surface with a full product implementation (`PlanningControlBinding`, ~570 lines). **No milestone (M1–M6) names it.** It is not in M1's kernel contracts, M3's runtime primitives, or M4's migration scope. It imports `RunOutcome` from `arnold.runtime.outcome`, a module that may not yet exist in the M1–M3 timeline. If this protocol is to survive the clean break, it needs a home in M1's kernel vocabulary and M3's runtime event/control flow. If it is legacy, it needs an explicit deletion gate in M6.

**Milestone edit**: Add to M1 "Locked Decisions" or kernel touchpoints: `arnold.kernel.control` carrying `ControlBinding`, `ControlTarget`, `ControlTransition`, `ControlProjection`, `ControlTransitionResult`. Add import-boundary test that neutral control carriers import no product code.

### 2. Dual State Authority — Manifest Events vs. `state.json` (DUPLICATE STATE RISK)

M3 mandates: *"Derive state from the event journal and artifact store; avoid mutable overwrite-only state as authority."* M4 adds: *"Status, trace, and resume should project from manifest events/artifacts rather than product-owned parallel state."*

But megaplan's `_core/state.py` is a 1000+ line state machine with `write_plan_state` in 8+ modes (`replace`, `executor-key-merge`, `patch-key`, `patch-many`, `active-step-heartbeat`, `merge-meta-list`, `legacy-migration`, `reversible`), CAS versioning under `_state_meta.versions`, snapshot/restore, heartbeat dedup, and driver-lock PID tracking. The `StateStoreBackend` protocol reserves `event_sourced` but raises `NotImplementedError`. After M3, two sources of truth will coexist: the manifest event journal AND megaplan's `state.json`. The plan offers no reconciliation strategy.

**Proof gate**: Before M4 starts, require a committed `docs/arnold/state-authority-migration.md` specifying: (a) which megaplan `write_plan_state` modes survive as event-journal projections, (b) which are deleted, and (c) the replay/reconstruct path from journal→state for resume.

### 3. Cost Tracking Fragmentation (POLICY LEAKAGE)

Three cost surfaces exist: (i) megaplan `append_history` → `state["meta"]["total_cost_usd"]` with `journal_then_execute` at-most-once wrapping; (ii) `arnold.pipeline.cost_types.py` (neutral `CostResult`); (iii) `arnold.pipelines.megaplan/observability/cost.py` (product cost module). M3 mentions "budget caps" and "capability checks" but gives no migration path for how the manifest runner enforces budgets versus megaplan's cost accumulation in `state["meta"]`. The effect ledger (`journal_then_execute`) is a product-owned idempotency layer not represented in any kernel contract.

**Milestone edit**: M3 "Capability, budget, and idempotency enforcement fail closed" criterion should add: *"Neutral budget enforcement reads from manifest events, not from product-owned `state['meta']['total_cost_usd']`."*

### 4. Governor/Supervisor Abstraction Missing from Kernel Contracts (MISSING SCOPE)

Megaplan has a full Governor abstraction (`arnold.pipelines.megaplan.runtime.governor`) with `BudgetExceeded` and `charge(envelope)`. The `PlanningControlBinding` has a `supervisor` projection surface. The plan mentions "auto-supervisor transitions, stall/idle/blocked/orphan recovery" as M4 migration items but the Governor is not defined in any M1–M3 kernel contract. The supervisor event routing promised in M3 ("Preserve... supervisor event routing as policy-blind primitives") has no kernel module target.

**Milestone edit**: Add `arnold/kernel/governor.py` to M1 touchpoints with `Governor` protocol and `BudgetExceeded` carrier. Add to M3 done criterion: *"Supervisor event routing dispatches through policy-blind kernel primitives, not product-owned transition tables."*

### 5. Override Path Duplication (POLICY LEAKAGE)

Two override mechanisms operate in parallel: (a) `ControlBinding.apply_transition` — protocol-based, produces `StateDelta` sequences with CAS versioning; (b) `write_plan_state(mode="merge-meta-list")` — state-level merge of `notes`, `overrides`, `user_action_resolutions`, `quality_gate_resolutions` with content-hash dedup. These are not reconciled. The `ControlBinding` path produces `StateDelta` objects expecting CAS enforcement; the merge-meta path bypasses CAS entirely using hash-based dedup. If both survive M4, concurrent override application can produce silent divergence.

**Milestone edit**: M4 must add a done criterion: *"All operator overrides flow through a single `ControlBinding.apply_transition` path; `merge-meta-list` mode is retired or proven to be a read-only projection of control transition events."*

### 6. Trace/Event Journal Triangulation (DUPLICATE OBSERVABILITY)

Three event/trace surfaces exist: `NativeTraceHooks` (writes `state.json`, `events.ndjson`, `stages.json`, `artifacts.json`, `checkpoint.json` per run), `megaplan/observability/events.py` (plan-scoped NDJSON with monotonic sequence counter and `_envelope_ctx`), and the planned M3 append-only event journal. The plan does not specify which survives, whether they merge, or how the M3 journal relates to the existing NDJSON format.

**Proof gate**: Before M3 implementation, require a committed `docs/arnold/event-journal-spec.md` defining the canonical event schema, journal file layout, and migration of existing `NativeTraceHooks` and megaplan event emitters into the new journal.

### 7. Specific Milestone Edits

| Milestone | Edit |
|-----------|------|
| **M1** | Add `arnold/kernel/control.py` (ControlBinding protocol), `arnold/kernel/governor.py` (Governor protocol), `arnold/kernel/effect_ledger.py` (idempotency model) to touchpoints. |
| **M2** | Add constraint: *"Compiler must not widen manifest to accommodate product-owned control target types."* |
| **M3** | Add done criterion: *"Event journal is the single source of truth for state, cost, and resume; megaplan `state.json` is a disposable projection."* |
| **M4** | Add constraint: *"`_core/state.py` `write_plan_state` mode surface must shrink to ≤ 3 modes (replace, event-project, reversible) before M5."* |
| **M6** | Add deletion target: *"`NativeTraceHooks` trace directory emission is deleted; all trace output flows through the manifest event journal."* |

### Summary

The plan is architecturally coherent at the manifest→compile→execute level but has a **control/observability blind spot**: the existing `ControlBinding` protocol, Governor, effect ledger, dual state authority (`state.json` vs. event journal), and three parallel trace systems are unaddressed. Without the edits above, M4's migration will hit a wall where the manifest runtime produces clean events but megaplan's deeply embedded `write_plan_state` machine continues operating on `state.json` as authority — creating a permanently split control plane. The proof gates (state-authority-migration.md, event-journal-spec.md) are the minimum blocking artifacts before M3 implementation begins.
