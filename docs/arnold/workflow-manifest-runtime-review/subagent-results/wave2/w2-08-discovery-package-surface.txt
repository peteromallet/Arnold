## Verdict: Missing Package/Discovery Work and Deletion Hazards

### Critical Findings

**1. M6 Self-Contradiction in Locked Decision.** M6 states: "`arnold_pipelines.megaplan` survives as the permanent product package. `arnold.pipelines.megaplan` does not." This is a literal text garbling â€” the same identifier appears in both survive and delete sentences. The intent from M4 is that `arnold_pipelines.megaplan` (new namespace-package home) survives while `arnold.pipelines.megaplan` (old location) is deleted. But the M6 brief renders them identically. **Fix:** Correct the second occurrence to `arnold.pipelines.megaplan` (old location).

**2. Two Manifest Systems â€” No Coexistence or Migration Plan.** The current discovery pipeline (`arnold.pipeline.discovery.manifest`) uses a native-first manifest contract with `driver` requirements, while the workflow-manifest-runtime M1 defines a new `WorkflowManifest` v1 contract. The plan has no milestone addressing how these two manifest systems relate, whether old discovery manifests become deprecated, or how pipeline packages declare themselves under the new explicit-node regime versus the old native-first regime during the transition. This is a **discovery trust gate gap**.

**3. No Explicit-Node Package Authoring Contract Defined.** M5 lists `docs/arnold/package-authoring-contract.md` and `docs/arnold/package-contract.md` as touchpoints, but these currently document the native-first `@pipeline` decorator/`NativeProgram` model. The plan has no milestone that defines the explicit-node package authoring contract (what a shipped pipeline package looks like under `arnold.workflow`/`arnold.patterns`). M2 defines the DSL shape but doesn't define the package-level contract that M5 must teach and M6 must audit.

**4. Pipeline ID Registry JSON â€” Code Exists, Artifact Missing.** `arnold.pipeline.pipeline_id_registry.py` implements loading and validating pipeline ID registry JSON files, but no actual `pipeline_id_registry.json` file exists anywhere in the repo. M5's inventory must produce these, but the task isn't explicitly called out beyond the generic inventory requirement.

**5. No Namespace-Package Wheel Configuration.** The plan calls `arnold_pipelines` a namespace package (M4), but `pyproject.toml` only packages `arnold` under `[tool.hatch.build.targets.wheel]`. There's no hatch config for the `arnold_pipelines` namespace. This is a **wheel packaging gap** that blocks M4 and makes M6's installed-wheel gate untestable.

**6. Deletion Hazard â€” py.typed Audit Not Milestoned.** `py.typed` markers exist at `arnold/py.typed`, `arnold/pipeline/py.typed`, `arnold/pipelines/megaplan/py.typed`, etc. M6 says "public type-surface checks prove no stale re-exports... leak deleted APIs," but no milestone actually audits py.typed markers end-to-end. After deleting public surface from `arnold.pipeline.__init__` (Stage, Edge, PipelineBuilder, etc.), the py.typed promises will be silently wrong unless regenerated or audited. The type-surface gate is named but not implemented anywhere in the plan.

**7. Package Disposition Manifest â€” 275 Rows, No Update Milestone.** `docs/arnold/package-disposition.yaml` has 275 disposition rows covering the current package layout. After M4-M6 migration, many rows will point to deleted or moved paths. No milestone explicitly requires updating or re-validating this manifest. M5 mentions "package disposition data" in its scope, but as a regeneration target, not as a disposition manifest update task.

**8. Discovery Trust Gate â€” `megaplan/pipelines` Path Fragment Will Break.** `arnold/pipelines/megaplan/discovery/trust.py` hardcodes `megaplan/pipelines` as a legacy in-tree path fragment. After M6 deletes old paths, this fragment may resolve to nothing or to a non-existent directory. The trust gate silently returns QUARANTINED, but no test validates post-migration trust classification.
