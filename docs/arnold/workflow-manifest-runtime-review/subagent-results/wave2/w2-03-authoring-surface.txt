## Verdict: **Plan is structurally sound but the authoring surface has critical under-specification — three missing surfaces, one over-scoped risk, and five public/private ambiguities that will cause contract churn if unresolved before M2.**

---

### Missing Authoring Surfaces

**1. Authoring DX and error ergonomics.** The plan covers compile-time *rejection* (closures, lambdas, bound methods) but never addresses the *quality* of authoring errors. What does an author see when they pass a closure to `arnold.patterns.agent()`? If error messages are inscrutable stack traces, the authoring surface is hostile regardless of architecture correctness. M2 needs an explicit "authoring error message" conformance requirement — every rejection path must produce a human-readable message naming the offending node, the violation, and the fix.

**2. Authoring composability and reuse.** The plan has `subpipeline` as a pattern but no discussion of pipeline-as-library authoring — importing pipelines from other packages, versioning pipeline definitions, composing sub-pipelines with parameter overrides, or sharing pattern configurations. Without this, every non-trivial pipeline author hits a wall. Add to M2 or M5: pipeline composition/import/reuse semantics and tests.

**3. Prompt-as-code authoring surface.** M4 mentions "dynamic prompt builders via importable `module:qualname` callables" but there's no dedicated authoring construct for prompt templates, prompt composability, or prompt versioning. This is a first-class authoring concern that deserves its own pattern constructors or at minimum explicit shape documentation in M2, not an ad-hoc M4 discovery.

---

### Over-Scoped Authoring

**M2's 16 pattern constructors are too many too early.** `panel`, `tournament`, and `merge` appear to be Megaplan-specific compositions, not universal workflow primitives. Defining them in M2 before M4 validates Megaplan's actual needs risks baking in wrong abstractions. **Edit:** Move `panel`, `tournament`, and `merge` to M4 as product-owned compositions built from the core patterns proven in M2. M2 should ship only the primitives needed to build the canonical Megaplan shape: `agent`, `branch`, `critique`, `fanout`, `human_gate`, `loop`, `retry`, `review`, `revise`, `subpipeline`, `external_call`.
