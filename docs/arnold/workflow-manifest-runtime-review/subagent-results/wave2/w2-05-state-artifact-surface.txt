## Verdict

**The plan has strong architectural boundaries but two material gaps in the state/artifacts surface:**

1. **No concrete state-migration path.** The M1-M3 milestones define the *target* artifact model (content-addressed storage, append-only events, artifact roots), but no milestone addresses how existing Megaplan on-disk state — plans, receipts, state.json, capsules, deltas — survives the transition. M4 bundles "move or re-author" without distinguishing new implementation from migration of historical data. A plan run today must be resumable tomorrow.

2. **Receipt/content-type ownership is blurry.** The plan correctly says "shared kernel vocabulary lives in neutral Arnold packages," and the review synthesis affirms "product packages populate policy/capability registries." But the milestone briefs don't apply this to the artifact surface: where do content-type schemas, receipt canonicalization, and taint/provenance *rules* live vs. *enforcement*? The kernel should own content-type *registration*, hash binding, and provenance *chains*. Product owns the specific schemas (receipt shape, plan capsule shape, delta format) and registers them.

## Recommended Milestone Edits

**M1:** Replace `arnold/kernel/human.py` touchpoint with `arnold/kernel/suspension.py`. Add `arnold/kernel/content_types.py` (registry interface only, no product schemas). Add artifact retention contract.

**M3:** Add provenance-chain verification to done criteria. Add `arnold/kernel/content_types.py` implementation.

**M4:** Split the migration scope into (a) register Megaplan content types (plan, receipt, capsule, delta, gate-signal) against kernel registry, (b) implement product artifact adapters, (c) migrate old on-disk state with a one-shot tool, (d) rebuild plan-repository-as-artifact-root.

**M6:** Add gate: no stale-format state files remain under `.megaplan/`. All historical state must be reachable via content-addressed artifact roots.
