## Wave 7 Assessment: Semantic Parity for Agent/Tool/Model/Effect Dispatch

**Confidence: 72/100**

The plan correctly separates **what** from **who** — neutral kernel defines contracts, product code registers handlers. Four review waves have tightened this boundary substantially. However, several concrete gaps could erode the neutral/product seam and weaken guarantees in execution.

---

### Top Capability/Effect Parity Risks

**1. `module:qualname` as boundary bypass (HIGH).** M2 resolves prompt builders, reducers, and policies to `module:qualname` strings at compile time. If M3's runtime later calls `importlib.import_module` on those strings to execute them, the neutral runtime is de facto importing product code through string indirection. The M3 constraint says the runner "must not import product policy classes" but is silent on dynamic resolution. This is the single largest semantic leak.

**2. Effect idempotency as a boolean, not a contract (HIGH).** The plan requires external effects to "declare idempotency policy" but never defines the contract. If the kernel only checks `idempotent=True` without an idempotency-key derivation scheme, replay safety is illusory — the kernel can't know whether two effect invocations are semantically identical. The key derivation protocol must be neutral; the derivation function is product-registered.

**3. Budget enforcement is post-hoc, not pre-execution (MEDIUM).** The plan states budget enforcement "reads from manifest events and effect ledgers." This models budget as an audit trail, not a gate. Real enforcement must reserve budget before effect execution (pre-allocate on capability check, settle on completion). Without pre-reservation, a burst of concurrent effects can collectively exceed budget before any event hits the journal.

**4. `arnold.agent` classification ambiguity (MEDIUM).** M3's open questions acknowledge that `arnold.agent` contains dynamic import shims whose status as "neutral adapter" vs. "compatibility alias" is unresolved. If concrete Megaplan-forwarding adapters live in `arnold.agent`, they form a permanent product-to-neutral bridge that M6 deletion gates may miss because `arnold.agent` is classified as neutral.

**5. Callback recovery shifts from push to poll without notification primitive (LOW-MEDIUM).** Current recovery (auto-supervisor, stall/orphan detection) likely uses direct callbacks. The event-journal model replaces this with polling. The plan provides "policy-blind primitives" for override/fallback/supervisor routing but doesn't specify whether `arnold.execution` emits a notification/event-bus that product code can subscribe to for time-sensitive recovery. Polling the journal introduces latency.

---

### Exact Plan Edits Needed

**M3 Scope (capability dispatch):** Replace "neutral string-keyed registries" with:
> "Protocol-typed capability registries. Capability identifiers carry a `dispatch_protocol` discriminator (`tool_call`, `model_call`, `external_effect`). The kernel validates handler protocol compatibility at registration time. `arnold.execution` calls handlers through abstract protocol interfaces. It never resolves `module:qualname` strings at runtime — all callable resolution happens at compile time in M2 or through product startup registration."

**M3 Constraints (add):**
> "The runtime must not call `importlib.import_module` on any string originating from a manifest field. Product callables are registered through the capability/effect registry at process start. The registry returns opaque callables; the runtime has no knowledge of their module origin."

**M1 Kernel Contracts (effect idempotency):** Replace boolean requirement with:
> "External effect contracts include an `idempotency_key_schema` field (neutral schema definition) and `derive_idempotency_key` protocol (product-registered derivation function). The kernel computes and records the idempotency key in the effect event before execution. Replay skips effects whose keys match previously committed events in the same manifest/scope/run. The key schema is neutral; derivation logic is product-owned."

**M3 Runtime (budget):** Add to budget enforcement:
> "The governor reserves budget on capability-check events before effect execution begins. It releases unspent budget on effect completion or failure. Both `budget_reserved` and `budget_settled` events are journaled. Execution is blocked when reservation would exceed remaining budget."

**M3 Runtime (add notification primitive):**
> "`arnold.execution` provides a neutral lifecycle listener protocol. Product packages register listeners for node/effect lifecycle transitions. Notifications fire before journal commit when policy requires pre-journal action. The listener interface is neutral; product code supplies concrete subscribers."

**M1 Locked Decisions (add):**
> "`arnold.agent` is either: (a) a pure neutral protocol surface defining `ToolAdapter`/`ModelAdapter` abstract interfaces only, OR (b) explicitly classified as a mixed surface. If mixed, every concrete-to-product adapter is inventoried in M1 and becomes an M6 deletion gate. Dynamic import shims in `arnold.agent` that forward to product code are M6 deletion targets unless re-chartered with explicit neutral rationale."

### Contract-Tested vs. Product-Tested

**Contract-tested** (neutral `arnold.kernel`/`arnold.execution` suites):
- Capability protocol typing: does registration reject mismatched handler protocols?
- Effect idempotency key derivation + replay skip: does the kernel correctly deduplicate?
- Budget reservation/settlement ordering: does pre-reservation block over-budget execution?
- Import boundary: zero product imports via static AST + `importlib` call scan
- Notification bus ordering: do lifecycle events fire before journal commit?
- Manifest hash integrity, suspension/resume without human semantics, deterministic event ordering

**Product-tested** (`arnold_pipelines.megaplan` suites):
- Golden behavior parity: identical planning outcomes with manifest execution vs. legacy
- Prompt builder equivalence: same prompt output through registry vs. direct call
- Callback recovery latency: auto-supervisor transitions within acceptable windows
- Budget policy: `total_cost_usd` tracking parity between old and new systems
- Human-gate-on-generic-suspension semantics, override/fallback routing parity
- Installed-wheel dispatch: tool/model calls reach live backends through neutral contracts
