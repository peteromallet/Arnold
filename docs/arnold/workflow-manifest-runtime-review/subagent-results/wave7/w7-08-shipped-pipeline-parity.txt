The plan is structurally sound — it sequences manifest/runtime work (M1-M4) before touching shipped pipelines (M5), then purges (M6). But M5 is where the semantic-parity rubber hits the road, and the plan has a real unresolved tension.

## Top Shipped-Pipeline Parity Risks

**1. Every shipped pipeline currently uses old native semantics — none use explicit-node.**

Every `build_pipeline()` on disk (creative, doc, jokes, epic-blitz, evidence_pack, folder_audit, deliberation, writing_panel_strict, select_tournament, live_supervisor, _template) returns `arnold.pipeline.Pipeline`, imports `Stage`/`Edge`/`ParallelStage`, and attaches a `native_program` compiled from `@phase`/`@pipeline` generators. The plan orders this correctly (M5 after M4), but M5's scope is enormous — ~15 pipeline roots, ~20 test roots, all CLI, all docs, all generated assets, all scripts, all package metadata, plus producing a closed inventory. A single milestone trying to rewrite a dozen pipelines into explicit-node DSL while preserving behavior is the highest failure-density point in the chain. The plan acknowledges this by allowing M5a/M5b workstreams but still gates M6 on M5 completion — meaning one stuck pipeline blocks the whole purge.

**2. The "whitelist with rationale" escape hatch.**

M5 allows pipelines to be "explicitly whitelisted as internal test coverage." Combined with "Keep restricted-Python generator material private/context-only unless needed as explicitly whitelisted internal coverage," this creates a path where pipelines like `_deliberation_example`, `briefs`, `_template`, or even `t19`/`t20` stubs could pass inventory while retaining old native semantics — as long as someone writes a rationale. The plan's locked decision ("Every shipped pipeline must expose a canonical `build_pipeline()` returning `workflow.Pipeline` or `WorkflowManifest`") would then conflict with the whitelist provision. The plan doesn't resolve which wins when a pipeline is hard to migrate but not clearly dead.

**3. No final package location for non-Megaplan shipped pipelines.**

M4 creates `arnold_pipelines/megaplan/`. M5 says old `arnold/pipelines/...` paths are "source-to-migrate/delete unless explicitly re-chartered." But evidence_pack, folder_audit, deliberation, and _template currently live under `arnold/pipelines/` with no stated target home. If they migrate to explicit-node, do they become `arnold_pipelines/evidence_pack/`? Stay under `arnold/pipelines/` re-chartered? Move into `arnold_pipelines/megaplan/pipelines/`? The plan's "declare final package location" requirement is correct but the locations themselves are unspecified — that's a decision M5 must make without reopening architecture, and wrong choices create M6 deletion headaches.

**4. Native decorator → explicit-node gap is not just syntactic.**

The current pipelines use native generators (`@pipeline`, `@phase`) that execute live Python — suspending mid-function for human gates (evidence_pack), dynamic fanout from generator state (doc's `OutlineArtifactReader`), and topic-seeded adapter classes (jokes' `_JokesNativeAdapter`). Rewriting these as explicit-node static data (`workflow.Pipeline(steps=[...])`) requires expressing all that control flow through manifest primitives (branch, loop, fanout, suspension). The M2 canonical Megaplan fixture plus M3 runner must be expressive enough to capture these patterns. If they aren't, M5 either widens M1-M3 contracts (violating the "no reopening" constraint) or whitelists pipelines as exceptions.

**5. Test migration is a hidden multiplier.**

Tests under `tests/pipelines/` (test_creative_pipeline.py, test_doc_pipeline.py, test_jokes_pipeline.py, etc.) and `tests/arnold/pipeline/` (test_builder.py, test_executor.py, test_executor_parallel.py, etc.) all import old types. M5 adds these to the inventory but the plan doesn't specify whether tests must be rewritten or can be whitelisted as internal coverage. If whitelisted, they become keepalive roots that M6 cannot delete — defeating the clean break.

Whitelist must name an owner and expiry. No shipped pipeline with a public `build_pipeline()` may be whitelisted."

2. **M5 scope**: Add one sentence naming the final package location for each non-Megaplan shipped pipeline before M5 starts. E.g.: "evidence_pack → `arnold_pipelines/evidence_pack/`; folder_audit → `arnold_pipelines/folder_audit/`; deliberation → `arnold_pipelines/deliberation/`; _template → `arnold_pipelines/_template/` or delete."

3. **M5 open questions**: Add "Which native-generator patterns (dynamic fanout from generator state, topic-seeded adapters, conditional suspension) have no M2/M3 manifest equivalent and must be added to M2/M3 contracts before M5 can close them?"

4. **M5 done criteria #1**: Change to "Every shipped pipeline is migrated to `workflow.Pipeline`/`WorkflowManifest` with compile/dry-run/fake-run/docs tests, or deleted. Whitelist entries are permitted only for pipelines with zero public surface and must include owner/expiry."

5. **M5→M6 gate**: Add "M6 cannot delete any path that has an M5 whitelist entry unless the whitelist is burned down first."

## Minimum Evidence for Migrate/Delete/Whitelist Decisions

- **Migrate**: Pipeline has a public `build_pipeline()`, is importable by other packages, has CLI/docs/tests referencing it, or is listed in any SKILL.md / generated asset. Must produce compile + dry-run + fake-run green.
- **Delete**: Pipeline has zero callers (AST + dynamic import scan), zero docs/scaffold/generated-skill references, zero CLI entrypoints, and zero test coverage outside its own directory. Confirm with `rg` across the whole repo.
- **Whitelist**: Pipeline fails the delete bar but cannot be migrated within M5 cycle. Requires: named owner, expiry date (before M6 or M6 is blocked), explicit rationale, and a concrete plan to migrate or delete by expiry. Must be tracked in the conformance allowlist that M6 burns down.
