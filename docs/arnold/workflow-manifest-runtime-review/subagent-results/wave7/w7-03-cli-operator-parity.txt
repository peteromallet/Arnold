## Top CLI/Operator Semantic Risks

### 1. Status still reads mutable `state.json`, not event-journal projections
`status_view.py` and `feedback.py` import `_core.state.write_plan_state` and read `state.json` fields like `current_state`, `meta`, `latest_failure`. M4 says `_core/state.py` authority must "shrink toward event projection," but there is **no committed field-level mapping** from old `state.json` to manifest-event projections, and no test that both produce identical status output for the same run. A "projection" can silently diverge.

### 2. Trace reads old `events.ndjson`, not the M3 event journal
`trace.py` calls `read_events()` which reads the per-plan `events.ndjson` file backfilled by `events_projection.py` from the old `arnold.runtime.event_journal`. The M3 plan defines a **new** canonical event journal under `arnold.kernel`/`arnold.execution` â€” there is no bridge between old and new journal formats, no parity test, and no commitment that `arnold workflow trace` will produce identical output to the old `megaplan trace`.

### 3. Resume cursor is not manifest-coordinate
Current resume reads `state.json` `resume_cursor` dicts with stage names and program counters. M3 defines manifest-coordinate resume via `manifest_hash` + `reentry_id` + `scope_stack`. M3's open question â€” "How should legacy resume aliases be represented?" â€” remains **unanswered in any milestone**. Every pre-migration suspended run has no defined resume path.

### 4. Override applies to `state.json`, not control-transition events
The 11-routed-action override system in `arnold.py` dispatches through `override_catalog_for("megaplan")` â†’ product registry, and `native_hooks.py` injects overrides into `state["meta"]["overrides"]` via `merge_state()`. The plan says "All operator overrides should flow through one control transition path" (M4 constraints) and M3 defines `ControlBinding`/`ControlTransition` â€” but there is **no mapping** from old override catalog actions to new control-transition events, and no gate requiring it.

### 5. Hidden compatibility dispatch via `arnold/cli/__init__.py`
The thin dispatcher lazy-imports `arnold.pipelines.megaplan.cli.arnold` â€” an M6 deletion target. M5's touchpoints mention it but there's no committed re-pointing to `arnold_pipelines.megaplan.cli.arnold`, no gate testing it, and no installed-wheel proof.

### 6. No installed-wheel CLI contract exists
Root `pyproject.toml` has **no `[project.scripts]`** stanza. M1 done-criterion #10 calls for wheel scaffolding, M5 mandates installed-wheel tests, but no milestone specifies a concrete `arnold = "arnold.cli:cli_entry"` entrypoint or proves it works.

### 7. `arnold workflow inspect` is undefined at runtime
M2 defines compile-time inspect/dry-run. M5 mentions `inspect` for operators but only as "where supported." There is no runtime `arnold workflow inspect <run-id>` that reads manifest events/artifacts to show node state, ref values, or capability outcomes.
