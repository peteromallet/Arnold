# Semantic Parity Assessment: Workflow-Manifest-Runtime Plan

## Confidence Score: **62/100**

The plan is strong on code-side migration and contract-level rigor, but it systematically under-instruments the **regeneration pipeline** — the generator code, verification gates, and provenance metadata that would guarantee generated artifacts teach the *same* behavior the runtime enforces.

---

## Top Generated-Artifact Risks (ranked)

### 1. **Agent-facing skills drift into stale surfaces** (CRITICAL)
`_codex_skills/` and `_composed/` are the primary teaching interface for downstream agents. M5 requires regeneration so they "reference only surviving public surfaces," but the plan provides no semantic validation test. A skill file can pass `ast` checks (no deleted imports) while teaching `PipelineBuilder`-era topology patterns, old `run_pipeline()` invocation, or pre-manifest resume/replay semantics. Agents would then emit code that compiles but breaks at runtime.

### 2. **Generator code uses old imports to produce "new" output** (HIGH)
`scripts/generate_arnold_docs.py`, skill syncers, and projection generators are themselves code that must be migrated. If a generator still imports `arnold.pipelines.megaplan` (the old path) or walks old package trees, it will produce docs/skills/dispositions that are *syntactically* clean diff but *semantically* reference stale package layouts, old CLI entrypoints, or pre-manifest pipeline IDs. The plan has no ordering constraint that generators be migrated *before* their output is regenerated.

### 3. **Projection docs describe old runtime state shapes** (HIGH)
`docs/reference/arnold-projections.md` and runtime docs must describe event-journal projections, artifact roots, and resume cursors. M3 defines the new event journal as single source of truth; M4 maps old `.megaplan` state. But M5's requirement to "update docs" doesn't include a conformance test proving that documented state shapes match actual `arnold.execution` event schemas. Projection docs could describe old `state.json` fields long after the runtime has switched to event sourcing.

### 4. **Pipeline ID registries and `WorkflowManifest` identity diverge** (MEDIUM)
M5 produces "pipeline ID registry JSON artifacts" but the relationship between registry IDs, `WorkflowManifest.id`, discovery manifest fields, and `manifest_hash` is never concretely defined. M1 defers it as an open question; M2 touches discovery manifests but doesn't resolve the identity model. Generated registries could use old `judge_manifest.py`-derived IDs while the runtime uses new manifest hashes for resume integrity.

### 5. **CLI help text and `--help` output lag behind command mapping** (MEDIUM)
M5 requires a command mapping table but doesn't require regenerating or validating CLI `--help` text, error messages, or shell completion scripts. These are operator-facing artifacts that embed command names, argument syntax, and behavioral descriptions. A clean command mapping table doesn't guarantee `arnold workflow run --help` describes the manifest-backed runner rather than the old `run_pipeline()` path.

### 6. **Package disposition data references deleted package paths** (MEDIUM)
Package disposition data shipped with `arnold_pipelines.megaplan` must teach downstream consumers which packages exist, their public status, and their entrypoints. If generated before M6 deletion or against old paths, it could list `arnold.pipelines.megaplan` (old) or `arnold.pipeline` modules that M6 deletes.

## Exact Plan Edits Needed

### M1: Add semantic artifact provenance contract
Add to Scope/IN:
> Define `GeneratedArtifactProvenance` fields: `generator_module`, `generator_source_hash`, `manifest_contract_version`, `generation_timestamp`, `input_surfaces_hashed`. Every regenerated artifact must embed or be accompanied by provenance metadata.

### M2: Lock package authoring contract into generated artifact requirements
Add to Done Criteria:
> The package authoring contract (`docs/arnold/package-authoring-contract.md`) must specify that `build_pipeline()`-bearing packages provide a `regenerate_artifacts()` or equivalent hook, and that generated skills/docs/dispositions must pass a semantic conformance gate (not just clean-diff).

### M5: Add four concrete gates
Replace "Regenerating docs/skills/registries/disposition/scaffolds yields a clean diff" with:

> 10. Regenerated skills pass semantic validation: every code block, import, class reference, and invocation example in generated skills/docs compiles against the installed `arnold.workflow` / `arnold.execution` public surface and produces no `AttributeError`, `ImportError`, or deprecated-warning at runtime.
> 11. Generator migration must complete before final artifact regeneration: `scripts/generate_arnold_docs.py`, skill syncers, projection builders, and registry generators must themselves import only surviving M6 surfaces, verified by AST + dynamic import scan.
> 12. Every generated artifact carries a provenance block (generator identity, source hash, contract version). A freshness check (`arnold workflow check --artifacts`) must fail on stale provenance.
> 13. A semantic conformance test suite exercises generated pipeline examples from docs, skills, and scaffolds through compile → dry-run → fake-run → resume, failing if any documented example diverges from runtime behavior.

### M6: Strengthen artifact conformance gate
Change "Enforce generated-artifact freshness" to:

> Enforce generated-artifact semantic conformance: every shipped generated skill, composed rule, projection doc, scaffold template, package disposition entry, and pipeline registry row must pass (a) provenance-freshness check, (b) import-surface scan against surviving M6 public API, (c) compile-and-fake-run of any embedded pipeline example, and (d) diff against regeneration-from-current-generators yielding zero drift.

## Freshness/Provenance Gates That Should Exist

| Gate | What It Checks | Who Enforces |
|------|---------------|--------------|
| **Provenance stamp** | Every generated artifact carries `generator_source_hash`, `manifest_contract_version`, `generation_timestamp` | M5 generation; M6 scan |
| **Regen-before-gate** | Generator code is migrated *before* its output is regenerated | M5a/M5b ordering constraint |
| **Semantic compile gate** | Every code block in skills/docs compiles against current installed public surface | M5 done criteria; M6 conformance |
| **Fake-run doc examples** | Documented pipeline examples actually run through the M3 runner | M5 done criteria |
| **Stale artifact detector** | `arnold workflow check --artifacts` fails if provenance hash ≠ current generator hash or manifest contract version ≠ current | M5 CLI; M6 gate |
| **Import surface scan on generated content** | Generated skills/docs/dispositions contain zero references to M6-deleted modules, classes, or entrypoints | M6 conformance (must use dynamic + string-aware scan, not just AST) |
| **CLI surface audit** | CLI `--help` output, error messages, and shell completions describe only surviving M6 commands and semantics | M5 CLI tests; M6 gate |

The plan's core weakness is treating generated artifacts as "disposable outputs" (M5 Locked Decisions) while simultaneously expecting them to teach correct behavior. Disposability and correctness are in tension: if they're disposable, the plan needs less rigor; if they're teaching surfaces, they need the same semantic conformance as code. The plan currently splits the difference without resolving it.
