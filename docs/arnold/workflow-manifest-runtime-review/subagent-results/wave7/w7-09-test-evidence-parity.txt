## Confidence Score: **54/100**

The plan names the right gate *categories* but under-specifies their evidentiary standards. Several gates are one-sample, editable-only, or have built-in escape hatches that a motivated implementer could walk through without proving semantic parity.

---

## Top Evidence Weaknesses

**1. Golden normalization is underspecified (M1, M4, M6).**
M1 says goldens are "normalized for volatile fields" but never defines the normalization contract. Timestamps, ordering, random seeds, topology hashes — any field can be declared "volatile" post hoc. The no-regression rule covers only "import/package-only moves," not internal refactors, helper rewrites, or "the new system emits equivalent but differently-shaped output." Without a locked normalization spec, goldens can be updated for convenience and still pass.

**2. The canonical Megaplan fixture is a single point of evidence (M2→M3→M4).**
One named fixture (`canonical_megaplan_manifest`) is the M3-to-M4 integration gate. It's described as covering all routing families, but it's still one pipeline shape compiled once. A single passing fixture proves that fixture works, not that the compiler+tuner correctly encode arbitrary Megaplan semantics. There is no combinatorial test matrix across topology variants, error paths, or edge cases.

**3. Fake-run proves execution, not semantic fidelity (M3, M4).**
Fake-run tests verify the runner can traverse a manifest. They do not verify the manifest correctly encodes intended behavior. The chain is: *author intent → DSL → compile → manifest → execute*. Fake-run only tests the last link. There is no manifest-diff, semantic-equivalence assertion, or invariant checker that compares two manifest versions and declares them semantically equivalent.

**4. Live-smoke is a single run (M4 DC6).**
"At least one live-backed smoke run succeeds after fake-backend parity." One run. No edge cases, no failure recovery, no resume-from-real-suspension, no multi-turn gate iteration with a real model. This gate proves the pipe isn't completely broken, not that it behaves correctly under load or matches legacy behavior against live APIs.

**5. Installed-wheel gates are aspirational, not mandatory until M6 (M4, M5, M6).**
M4 requires installed-wheel smoke for `arnold_pipelines.megaplan`, but the canonical fake-run gate (M3 DC10) and M4 parity tests are not required to pass from a built wheel. M5 says "where feasible." Only M6 hardens this. An editable-only test suite can pass for four milestones and then discover wheel-packaging problems at the last minute — after deletion decisions are locked.

**6. Characterization tests are mentioned but never formalized.**
M5 inventories "characterization fixtures" and "oracle/parity fixtures," but no milestone defines a characterization *methodology*: capture old-behavior traces, lock them, then prove new-behavior traces are semantically equivalent. The plan relies on goldens + parity, but both can be updated.

**7. Conformance allowlist "re-chartering" is a built-in escape hatch (M6).**
M6 says conformance allowlists must "burn down to zero legacy exceptions or re-chartered with owner, expiry, and non-legacy rationale." Re-chartering is an explicit alternative to deletion. Combined with dynamic-import detection that must catch `importlib`, `__getattr__`, and `__all__` — complex to fully audit — this gate is vulnerable to "we audited it, remaining entries are re-chartered."
