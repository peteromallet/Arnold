## Identity/Discovery Coherence Assessment

**Confidence: 38/100**

The plan names every identity surface but never declares a canonical root or derivation chain. M1's critical open question — *"How does a surviving package pipeline ID relate to `WorkflowManifest.id`, `version`, `manifest_hash`, package metadata, and generated registry artifacts?"* — goes unanswered across all six milestones. Each milestone independently touches identity artifacts without a binding answer, creating latent split-brain.

---

### Top Identity/Discovery Risks

**1. No identity root declared (M1 Open Question → never closed)**
`WorkflowManifest.id`, `WorkflowManifest.version`, `manifest_hash`, pipeline registry IDs, and discovery manifest entries are all named but no derivation order is specified. M2 says `build_pipeline()` returns either `workflow.Pipeline` or `WorkflowManifest` — but `Pipeline.id` is set by the author, `manifest_hash` is computed by the compiler, and registry IDs in M5 are generated artifacts. Without a declared root, four different identity values can coexist for the same pipeline.

**2. Package discovery manifests vs WorkflowManifest — split-brain architecture**
M1 correctly states discovery manifests "are not runtime manifests" and must "survive as package metadata." But M1 never resolves whether discovery manifest fields (pipeline identity, trust class, tenant) are *derived from* `WorkflowManifest` or independently authored. If independent, you get two identity authorities. If derived, no milestone specifies the derivation. M5 generates "package-discovery metadata" but doesn't wire it to manifest identity.

**3. Trust classes and tenant derivation orphaned after M1**
M1 assigns disposition of `judge_manifest.py`, trust classes (auto-executable/quarantined/blessed), `schema_registry.py`, and `derive_tenant_id`. M4 (product migration) never mentions reconnecting Megaplan product identity through these. M5 generates registry artifacts but omits trust/tenant fields entirely. M6 has one done criterion about trust classification tests — but by then the derivation gap is baked in.

**4. Generated metadata can silently disagree with manifest identity**
M5 regenerates `SKILL.md`, composed skills, projection docs, package disposition, and pipeline ID registries. Nothing in M5 requires these artifacts to derive identity from `WorkflowManifest.id/version/hash`. A regenerated `SKILL.md` could reference `pipeline_ids.json` IDs that don't match the compiled manifest hash. Installed-wheel discovery would then see two conflicting identities.

**5. `build_pipeline()` dual return type creates identity ambiguity**
M2 locks `build_pipeline()` to return either `workflow.Pipeline` or `WorkflowManifest`. If it returns `WorkflowManifest`, `id` is baked in at authoring time. If it returns `workflow.Pipeline`, the compiler may assign or derive `id`. Neither path is canonical, so two packages could disagree on when identity is fixed, producing different `manifest_hash` values for semantically identical pipelines.

**6. Stale registry after package relocation (M4→M5→M6)**
When M4 moves product to `arnold_pipelines.megaplan` and M5 relocates shipped pipelines, old ID registries and discovery manifests at `arnold/pipelines/megaplan/` paths become stale. The plan says old paths are "source-to-migrate/delete" but never specifies ID registry consolidation, cross-reference migration, or a single-source-of-truth registry location.

**7. Installed-wheel discovery bypass**
Installed-wheel tests are required, but no milestone specifies that discovery from a built wheel MUST resolve the same identity as from the source tree. A wheel could ship a stale discovery manifest JSON that disagrees with the compiled `WorkflowManifest` embedded in the same package.

### Exact Plan Edits Needed

**Edit 1 — M1: Close the open question**
Replace M1's open question with a locked decision: *"`WorkflowManifest.manifest_hash` (content-addressed, compiler-computed) is the canonical identity root. `WorkflowManifest.id` is a stable human-chosen alias stored in the manifest. Pipeline registry IDs, discovery manifest entries, trust classifications, tenant derivations, and generated metadata MUST derive from `manifest_hash` + `id` and MUST NOT carry independently authored identity fields."*

**Edit 2 — M1: Add identity derivation table**
Add a committed table mapping each identity surface → derivation rule:
| Surface | Derives From |
|---|---|
| `WorkflowManifest.id` | Author-chosen, stable |
| `WorkflowManifest.manifest_hash` | Compiler (content-addressed over manifest DAG) |
| Pipeline registry ID | `{id}@{manifest_hash[:12]}` |
| Discovery manifest `pipeline_id` | `registry ID` (read from registry, not independently set) |
| Trust class | Package-level `pyproject.toml` → compiled into manifest → discovery manifest copies it |
| `derive_tenant_id` | Hash of (`manifest_hash`, package namespace) |
| Generated SKILL.md header | MUST include `manifest_hash` and `id` |

**Edit 3 — M2: Lock `build_pipeline()` return type**
Change M2's locked decision: *"`build_pipeline()` returns `workflow.Pipeline` only. `WorkflowManifest` is compiler output, never hand-authored. The `Pipeline.id` field is the stable human-chosen identity; `manifest_hash` is assigned at compile time. Any `build_pipeline()` that returns a pre-built `WorkflowManifest` fails conformance."*

**Edit 4 — M4: Add trust/tenant reconnection**
Add to M4 scope: *"Reconnect Megaplan product identity to the M1 identity derivation table. Confirm that `arnold_pipelines.megaplan`'s discovery manifest, trust classification, and tenant derivation survive the package move and produce identical results to the pre-move values."*

**Edit 5 — M5: Wire generated metadata to manifest root**
Add to M5 scope: *"Every generated artifact (SKILL.md, registry JSON, projection doc, composed skill, package disposition row) MUST include `manifest_hash` and pipeline `id` in its header/metadata and MUST fail regeneration if these disagree with the compiled manifest."*

**Edit 6 — M5: Single registry location**
Add to M5: *"Pipeline ID registries consolidate to a single `arnold_pipelines/pipeline_ids.json` at the namespace-package root. Per-package `pipeline_ids.json` files are read-only projections. Discovery from installed wheel MUST validate against this root registry."*

### Required Conformance Tests

1. **Identity derivation chain test**: Given a `workflow.Pipeline` with `id="foo"`, compile → extract `manifest_hash` → verify pipeline registry ID, discovery manifest ID, and generated `SKILL.md` header all match the derivation table.
2. **Cross-surface identity agreement**: After M5 regeneration, assert `manifest_hash` in the compiled manifest == `manifest_hash` in `pipeline_ids.json` == `manifest_hash` in discovery manifest == `manifest_hash` in `SKILL.md`.
3. **Trust-class roundtrip**: Set trust class in package metadata → compile manifest → read discovery manifest → assert trust class matches and no independent trust field exists.
4. **Tenant derivation determinism**: Call `derive_tenant_id(manifest_hash, package_namespace)` from source tree, editable install, and installed wheel → assert identical output.
5. **Stale registry detection**: Create a pipeline with `id="A"`, compile, move package to `arnold_pipelines`, regenerate registries, then assert old `arnold/pipelines/megaplan/pipeline_ids.json` is absent or flagged stale.
6. **build_pipeline() return-type conformance**: Scan all `build_pipeline()` functions → assert none return pre-built `WorkflowManifest`; all return `workflow.Pipeline`.
7. **Installed-wheel identity parity**: Install wheel, run discovery, extract `manifest_hash` and registry ID → assert identical to source-tree compilation.
8. **Generated metadata freshness gate**: Regenerate all M5 artifacts → assert clean diff; if any artifact changes its identity header, fail.
9. **Discovery-manifest independence check**: Parse discovery manifest → assert no identity field exists that isn't traceable to `manifest_hash` or `id` via the derivation table.
10. **No identity bypass via dynamic import**: Scan for `importlib.import_module` or `__getattr__` that resolves pipeline identity without going through the canonical registry.
