## Confidence Score: **42/100**

The plan has strong architectural principles but critical gaps in guaranteeing that *every surviving pipeline/run* replays identically post-migration. It leans heavily on classification (migrate/archive/ignore/delete) without proving classification decisions are lossless for runs that matter.

---

## Top State/Resume Risks

### 1. Manifest-version mismatch during in-flight resume (CRITICAL)
M3 quarantines manifest-hash mismatches but never specifies what happens to a run that was *suspended under manifest v1* and must resume after the manifest has evolved. The amendment protocol (M1) covers how to *change* the manifest, not how to *revive* runs trapped under an old hash. If a real Megaplan run is suspended mid-gate during the M4 migration window, the plan offers no replay path — just quarantine semantics. The run is effectively dead.

### 2. Old `state.json` authority never fully transferred
M4 classifies old `.megaplan/plans/*/state.json` as "migrate, project read-only, archive, ignore, or delete" — but there is no committed schema mapping from old mutable-state fields to new event-journal projections. If `state.json` contained computed fields (e.g., accumulated cost, gate-pass counts, supervisor state) that have no direct event-journal equivalent, those runs cannot be reconstructed. The plan allows "archive" or "ignore" as valid classifications with "rationale," which means runs can be silently abandoned.

### 3. Stale locks have no migration-time resolution protocol
M4 says lock files need "owner/PID/TTL/stale-lock policy or explicit archival/deletion rationale." But there's no rule for what happens when the migration tool encounters an active lock from a pre-migration run. Who breaks it? Under what authority? A stale lock on a plan directory could silently exclude that run from migration.

### 4. Nested run replay is unspecified
Nested `.megaplan` run directories and `.hermes_state` conventions are inventoried (M4) but have no replay semantics. Subpipeline promotion and nested scope cursors are defined for *new* runs (M3) but the plan never maps old nested-run event trees into the new unified journal. A parent run with nested sub-runs cannot be faithfully replayed.

### 5. Replay cursor compatibility is an open question
M3 asks: "How should legacy resume aliases be represented so topology changes quarantine safely unless explicit aliases exist?" This is flagged but *never answered in any milestone*. Without explicit old→new cursor aliasing, every pre-migration run that was suspended is unreplayable under the new runtime. The classification system provides no "alias" category.

### 6. Projection drift between old mutable state and new event projections
M4 says `_core/state.py` authority must "shrink toward event projection" and M5 says status/trace/resume "should project from manifest events/artifacts." But there's no test that old projections and new projections produce identical outputs for the same run. The plan allows them to diverge as long as both are "projections."

### 7. Hash/topology evolution breaks traceability
The amendment protocol (M1) covers *contract* changes but not *run-level* hash lineage. If a pipeline's topology changes (e.g., a node is added between M2 and M4), old runs that references the old topology hash have no defined replay path. The plan quarantines mismatches but provides no tool to *adapt* old runs to new topology.

## Exact Plan Edits Needed

1. **M3**: Add an explicit "manifest-version bridging" section: when a run resumes under a newer manifest hash, the runtime must either (a) replay using the original manifest hash (stored immutably), or (b) validate and project the old event stream into the new topology with explicit compatibility gates. Quarantine alone is not a replay policy.

2. **M4**: Replace the classification table with a **lossless migration guarantee**: every `state.json` field must map to a specific event-journal projection, with a conformance test proving round-trip fidelity for all golden runs. "Archive" and "ignore" classifications must name the specific runs dropped and why.

3. **M4**: Add a lock-resolution protocol: migration tooling must detect active locks, identify owner/PID, and either (a) confirm the owning process is dead with a timeout check, or (b) refuse migration with a report. No silent lock-skipping.

4. **M4**: Add nested-run replay: map old nested `.megaplan` and `.hermes_state` directory conventions to subpipeline scope stacks and nested event journals. Prove a parent run with children replays faithfully.

5. **M3/M4**: Resolve the legacy-resume-aliases open question: define an explicit alias table mapping old reentry coordinates (stage name, program counter, checkpoint ID) to new manifest-coordinate cursors. Every old suspended run must have at least one valid alias or be explicitly quarantined with operator notification.

6. **M4**: Add a projection-parity test: for every golden run that survives migration, prove that `_core/state.py` legacy projections and new event-journal projections produce identical status/trace/resume outputs.

7. **M1**: Amend the manifest-amendment protocol to cover run-level hash lineage: every manifest version bump must include a migration path for in-flight runs, and the runtime must store the original manifest hash with every event so replay can fall back to the original contract.

## What Must Be Tested Before Deletion

1. **Resume an old-format suspended run under the new runtime** — using real pre-migration `.megaplan` state from production or a faithful golden, prove it resumes and completes identically.

2. **Replay an old-format completed run** — full event-by-event replay against new event journal, prove final state projection matches legacy golden.

3. **Nested run replay** — a parent with subpipeline children replays with correct scope nesting.

4. **Manifest-hash evolution replay** — a run started under manifest v1 resumes under manifest v2 (post-amendment) with explicit compatibility validation.

5. **Stale-lock migration** — tooling correctly identifies, reports, and handles an active lock on old `.megaplan` state during migration.

6. **Projection parity** — old `state.json` projection and new event-journal projection produce identical outputs for the same run (field-by-field).

7. **Hash lineage traceability** — every event in a migrated run can be traced back to its originating manifest hash, and the full hash chain is verifiable.

Without these, deletion (M6) will be deleting the only remaining evidence that old runs ever worked.
