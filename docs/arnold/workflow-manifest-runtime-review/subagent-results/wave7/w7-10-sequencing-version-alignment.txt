### Top Risks

**1. M3 dynamic topology overlays can widen the manifest contract without back-propagation to M1/M2 (HIGH)**

M3 scope explicitly says: *"Define dynamic topology overlays as manifest amendments or control-transition events."* M1 reserved slots for retry/loop/fanout/reducer, but did **not** reserve deterministic slots for overlay lists, amendment-event types, or control-transition payloads. If M3 introduces new manifest fields, M2's compiler and M1's contract tests become stale. Unlike M2 (which has an explicit *"update M1 contract tests and amendment doc"* gate), M3 has **no equivalent trigger**. The amendment protocol M1 creates only names "M2/M3 discoveries" — but M3 itself needs a WAIT gate requiring it to update M1/M2 before M4 begins.

**2. M4 content-type registration / state migration can discover M1 contract gaps with no amendment path (HIGH)**

M4 registers Megaplan content types into the neutral registry M1 defined. If M4 discovers that M1's `content_type` schema lacks needed fields (e.g., migration-epoch markers, dual-format compatibility windows), M1's contract must be amended. Neither M4's scope nor its done criteria mention updating M1 manifests or tests. The amendment protocol is scoped to M2/M3 — M4 has no formal path to change the contract it consumes.

**3. M4/M5 operator command mapping conflict (MEDIUM)**

M4 produces a *"Megaplan-specific operator command inventory."* M5 produces a *"repo-wide CLI command mapping."* These two inventories can produce contradictory dispositions for the same command (e.g., M4 marks a command as `migrate to arnold workflow status` while M5 marks it `delete`). M5's scope says it "broadens" M4's inventory but doesn't require **reconciliation** — it could silently overwrite M4 decisions.

**4. M5 generates assets into a package layout M4 defined — no re-inventory gate (MEDIUM)**

M5 regenerates skills, composed rules, projection docs, and pipeline-ID registries into `arnold_pipelines/megaplan/` — a package M4 created. If M5's generation creates new files or import paths that M4's legacy inventory didn't capture, M6's deletion scan (which is a *"closed subset of M5 inventories"*) will miss them. M6 requires no unresolved M5 rows, but doesn't require M5 to **re-run its own inventory** after generation.
