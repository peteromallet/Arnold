## Top Parity Risks

### 1. Gate condition fidelity — 9 transitions, not 4
The current gate stage has 9 distinct *conditionally-gated* transitions (`gate_unset`, `gate_iterate`, `gate_escalate`, `gate_tiebreaker`, `gate_proceed_blocked`, `gate_proceed_agent_availability_blocked`, `gate_proceed`, plus two override-force-proceed paths via different conditions). The plan reduces this to "branch/loop/fanout/retry" primitives in M2/M3. The `gate_proceed_agent_availability_blocked` condition specifically checks `preflight_results` for `claude_available`/`codex_available` — that's product logic the neutral runtime can't evaluate without baking in Megaplan knowledge. **Risk**: implementer collapses 9 transitions into 4 decision edges and passes tests by routing all proceeds the same way, losing blocked-agent detection.

### 2. Override edge dual-path collapse
`STATE_CRITIQUED` has *two* distinct "override force-proceed" transitions: one via `gate_escalate` (for explicit operator overrides) and one via `gate_proceed_agent_availability_blocked` (for auto-escalation when agents are unavailable). If the explicit-node DSL compiles these to a single `override_edge("force-proceed", "finalize")`, the auto-escalation pathway is silently lost. M4 must preserve both as separate manifest edges with distinct condition keys.

### 3. Tiebreaker is recursive loop-back, not a one-shot subpipeline
The current tiebreaker resolves to `"iterate"` → critique → gate, and the gate CAN trigger tiebreaker again. This is a loop-back-into-loop, not a leaf subpipeline. M2 requires a "canonical Megaplan-shaped manifest fixture" but the plan language says "tiebreaker subpipeline/retry path" — singular. The fixture must exercise at least two tiebreaker rounds to prove the recursion isn't flattened.

### 4. Resume execute-authority verification has no manifest expression
The current `_resume_execute_authority_failure` checks `finalize.json` task lists against stored cursors with 5 distinct failure modes before advancing past execute. M3's resume model (`manifest_hash`, `reentry_id`, `scope_stack`, `artifact_root`, `event_sequence`) doesn't address *authority-increasing* transitions where the runner must verify external evidence (task completion in finalize.json) before clearing the cursor. The M3-to-M4 fake-run gate must include an execute-authority failure scenario.

### 5. Robustness topology: compile-time selection vs runtime overlay ambiguity
The plan says M3 should handle "dynamic topology overlays as manifest amendments or control-transition events." But the current robustness system (`_workflow_for_robustness`) is a *pre-execution* topology selection that removes steps and rewires transitions. "Bare" robustness removes prep, critique, gate, execute, and review entirely — that's a different manifest, not a runtime overlay. The plan is ambiguous: does robustness produce different compiled manifests or does the runner dynamically skip nodes?

### 6. Feedback phase as optional topology extension unaddressed
The current `with_feedback` flag extends the workflow (review → feedback → done) by patching `_workflow_for_robustness` at init-time. M4 mentions "feedback-as-phase" but M2/M3 provide no mechanism for compile-time optional topology extensions. This risks being implemented as a runtime hack instead of a manifest variant.

### 7. State-authority migration contention
M3 says "Derive state from the event journal and artifact store" and M4 says status/trace/resume "should project from manifest events/artifacts." But the current system has `state.json` as the primary authority with 18+ state values, and resume cursors that reference it directly. The plan doesn't specify whether `state.json` survives as a projection, gets fully replaced, or coexists during migration — and the golden tests use `state.json` contents as their comparison target.
