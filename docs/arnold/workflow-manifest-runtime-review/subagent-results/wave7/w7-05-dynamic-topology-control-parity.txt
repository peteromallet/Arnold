## Confidence: 70/100

The plan names every control/topology concept you ask about and repeatedly asserts the generic-vs-product boundary. But it lacks the concrete event families and dispatch contracts that would prove the boundary holds under implementation. The vocabulary is right; the wiring diagram is missing.

## Top 5 Control/Topology Risks

1. **Dynamic topology overlay mechanism is forked, not decided.** M3 says overlays are "manifest amendments or control-transition events." These have opposite replay semantics — amendments change the canonical manifest (affecting resume-from-cursor), while events only record decisions. The plan never picks one, and the fake-run gate can pass either way, hiding a fundamental architectural divergence.

2. **Override/fallback edges are named but never shaped.** M2's canonical fixture includes "override/fallback placeholders" and M3 promises "generic override edges, fallback edges" — but no manifest field, edge type, or routing rule is specified. Without a concrete shape, the compiler can accidentally bake product policy into the edge schema (e.g., a `fallback_reason: MegaplanReason` enum leaking into the neutral manifest).

3. **Supervisor condition detection boundary is unowned.** M3 says "supervisor event routing as policy-blind primitives" but never states whether the kernel detects stall/idle/blocked/orphan or whether product emits those events through a generic signal API. If the kernel detects them, it imports product concepts. If product emits them, there's no generic event family for it to emit into.

4. **Compensation is a single-word placeholder.** "Compensation hooks" appears exactly once (M3 scope). No compensation event family, no Saga-style completion tracking, no idempotency contract for undo operations, no manifest slot for compensation targets. This is the least-developed topology primitive in the plan.

5. **Escalation and supervisor transitions may be the same mechanism but are never reconciled.** Both use "targets" and "routing." If they share a `ControlTransition` event family, that needs to be stated. If they're distinct, the manifest needs separate slots. The plan mentions both separately without clarifying.

## Exact Plan Edits Needed

### M1 (`m1-baseline-manifest-contract.md`)

**Add to "Scope / IN":**
> - Define the concrete control-transition event family: `ControlTransition` events carry `transition_type` (override, fallback, escalate, supervisor_promote, compensation_trigger), `source_node`, `target_node`, `trigger`, `payload_schema_hash`, and `idempotency_key`. Product packages register `transition_type` meanings; the kernel owns the event envelope and dispatch.
> - Define dynamic topology overlay as a control-transition event (not a manifest amendment). Overlays are recorded in the event journal and projected into topology views; they do not mutate the canonical manifest. This preserves deterministic replay from any cursor.
> - Reserve manifest slots for: `override_edges`, `fallback_edges`, `escalation_targets`, `compensation_targets`, `supervisor_promotion_routes`, each as lists of `(condition_ref, target_node_ref, policy_ref)` tuples where `condition_ref` and `policy_ref` are product-registered importable identities.
> - Add compensation contract: `CompensationScope` with `completed_steps`, `compensation_target`, `idempotency_policy`, and `reversal_ordering` (reverse-completion or explicit).

**Add to "Open Questions" (or resolve):**
> - Resolved: Dynamic topology overlays are control-transition events, not manifest amendments. Rationale: amendments break cursor-based replay; events preserve it.

## Summary

The plan's architecture *intent* is sound: neutral runtime dispatches generic control events, product registers meanings. But the concrete evidence — event family names, manifest field shapes, the amend-vs-event decision, compensation model, and supervisor detection boundary — is missing. Those six edits close the gap between "the plan says the right things" and "the plan can be implemented without accidentally baking Megaplan into the runtime."
