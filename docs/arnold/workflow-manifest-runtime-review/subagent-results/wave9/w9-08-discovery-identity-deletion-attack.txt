## Deletion-Attack Analysis: How Identity/Discovery Gates Can Falsely Pass

### Top 10 False-Pass Vectors

**1. Registry-to-compiler hash drift.** M5 produces `pipeline_ids.json` with manifest hashes at generation time. M6 deletes pipelines without recompiling survivors and cross-checking that every registry hash matches `compiler(build_pipeline(current_tree, id)).manifest_hash`. The M6 done-criteria check "no rows pointing to deleted paths" guards file paths, not hash identity. A registry row for `creative_pipeline` with hash `abc123` survives because the path exists — but the pipeline was re-authored at M5 and the registry holds the pre-rewrite hash. Identity tests pass because the `id` string is valid; no gate verifies the hash resolves to the current compiler output.

**2. Discovery manifests as parallel identity authorities.** M1 defers the coexistence/supersedence question ("Define how existing discovery/package manifests coexist with or are superseded by `WorkflowManifest` v1"). If `SKILL.md`, `judge_manifest.py` sidecars, or package `__init__.py` discovery blocks still carry independent pipeline references (trust class, entrypoint path, version strings) after M6, those references can survive deletion of the actual pipeline. M6 done criterion 8 guards "hardcoded path fragments pointing to deleted directories" — but a discovery manifest referencing `"pipeline": "epic_blitz"` and trust class `"blessed"` passes directory checks when the pipeline implementation was moved/deleted and the discovery block wasn't regenerated.

**3. Trust classification caching stale pipeline IDs.** Discovery trust classes (auto-executable, quarantined, blessed) in `schema_registry.py` or `judge_manifest.py` can embed pipeline ID strings that survive M6 directory deletion but point to quarantined or migrated-away implementations. If the trust rule says `"epic_blitz" → auto_executable` and `epic_blitz` has a surviving `build_pipeline()` at a new path but the trust entry references the old compiled manifest hash, the trust gate passes (the ID exists) but routes to the wrong compiled artifact.

**4. Tenant derivation producing orphan IDs.** `derive_tenant_id` can output IDs that syntactically match surviving pipeline registry entries without verifying those pipelines still compile from the final tree. M6 has no tenant-derivation resolution gate. A derived tenant `"tenant_megaplan_v1"` might map to a pipeline ID present in the registry but whose `build_pipeline()` was deleted — the registry wasn't pruned post-deletion.

**5. Generated-artifact freshness gap.** M6 done criterion 13 checks "freshness check passes after regenerating docs/skills/registries/disposition/scaffolds" — but freshness typically means "source hash matches generator." If the generator itself reads stale pipeline registries or hardcoded ID lists, regenerated output is "fresh" yet embeds deleted pipeline IDs and stale manifest hashes. Generated `_codex_skills` could reference pipeline routes that no longer compile.

**6. Canonical fixture matrix staleness.** M2's fixture matrix (`canonical_megaplan_manifest.*`) is checked into the repo and M6 done criterion 19 verifies M1/M2/M3 tests pass against the installed wheel. But if the fixture files reference pipeline IDs or manifest shapes from pre-M5 pipelines that were subsequently deleted or restructured, the tests pass because the fixtures still exist on disk — they're just testing stale identities. No gate verifies fixtures correspond to current `build_pipeline()` outputs.

**7. Resume cursors referencing deleted manifests.** M3 defines in-flight runs replaying with original manifest hashes or quarantine. M6 deletes old surfaces but never scans surviving resume-cursor fixtures to verify every `manifest_hash` in a cursor resolves to a compilable manifest in the final tree. A test fixture with `manifest_hash: "deadbeef"` from a pipeline deleted at M6 still replays in tests because the fixture file wasn't deleted.

**8. Whitelist-escape stale-reference path.** M5 whitelist rows (zero-public-surface internal coverage) can reference pipeline IDs, import paths, or test utilities that are themselves stale. M6 done criterion 10 burns down conformance allowlists to zero legacy exceptions — but a re-chartered non-legacy row with "owner, expiry, independent review" can still embed a pipeline ID whose canonical implementation was moved. The dynamic-import proof only checks the import resolves; it doesn't verify the imported object compiles to the expected manifest hash.

**9. Multiple pipelines sharing the same `id`.** M1 defines `workflow.Pipeline.id` as the stable alias. If M5 migrates a pipeline from old path to new path but preserves the same `id`, and the old compiled manifest was registered in `pipeline_ids.json` before the migration finalized, the registry holds the old hash. Identity tests see the `id` matches a surviving pipeline but miss that the hash references a stale compile. No M6 gate requires hash re-derivation from final-tree compilation.

**10. CLI output fixtures locked to stale data.** M6 done criterion 20 requires surviving CLI `status/trace/inspect/override` outputs match "locked semantic fixtures." If those fixtures were locked at M5 before final deletions and contain pipeline IDs, manifest hashes, or artifact roots from subsequently deleted surfaces, the CLI passes the fixture comparison while projecting stale identity data to operators.

### Exact Gates / Plan Edits Needed

**M6 must add these gates:**

| # | New Gate | What It Catches |
|---|----------|-----------------|
| A | **Pipeline-identity resolution**: For every `id` in every `pipeline_ids.json`, resolve to `build_pipeline()`, compile, assert `registry.hash == compiler_output.manifest_hash`. | Vectors 1, 6, 9 |
| B | **Discovery-to-manifest reconciliation**: Every discovery manifest row and trust classification must reference a pipeline ID whose compile-from-final-tree hash matches any embedded or referenced manifest hash. Stale trust entries with no surviving pipeline → hard fail. | Vectors 2, 3 |
| C | **Tenant-resolution gate**: `derive_tenant_id` output for all registered pipelines must map to an ID in the final registry, and that row must pass the pipeline-identity resolution gate. | Vector 4 |
| D | **Generated-artifact embedded-ID scan**: Beyond import-path scanning, search all generated skills/composed/docs/registries/scaffolds for pipeline ID strings and manifest hashes; each must resolve to a surviving pipeline with matching compiler hash. | Vector 5 |
| E | **Resume-cursor manifest resolution**: Every manifest hash in surviving resume-cursor fixtures/tests must compile from the final tree. Unresolvable hashes → quarantined or deleted. | Vector 7 |
| F | **Whitelist pipeline-identity sub-gate**: Re-chartered conformance allowlist rows that reference pipeline IDs or module paths must also pass the pipeline-identity resolution gate (A). | Vector 8 |
| G | **CLI fixture re-lock from final tree**: CLI semantic fixtures must be regenerated from the final installed wheel, not carried forward from M5 lock. | Vector 10 |

**Plan edits to M1 and M5:**

- **M1**: Resolve the open question about discovery manifest survival. Add a concrete disposition: "Discovery manifests survive only as regenerated artifacts derived from `id + manifest_hash`; old discovery manifest formats and trust classification rules that carry independent pipeline references are M6 deletion targets."
- **M5 done criterion 10**: Amend to require "pipeline ID registries regenerated as the final M5 step, with a committed cross-check that every registry `manifest_hash` equals `compiler(build_pipeline(current_tree, id)).manifest_hash`."
- **M5 done criterion 8**: Expand "clean diff" to include "no pipeline ID or manifest hash references to non-surviving or stale-compiled pipelines."
- **M6 done criterion 7**: Expand from "no rows pointing to deleted paths" to "no rows pointing to deleted paths, stale pipeline IDs, or manifest hashes that do not match current compiler output for the referenced pipeline."

### Evidence Required for Final Manifest-Identity Proof

A single cross-referenced ledger table committed before M6 closes, with columns:

| Pipeline ID | Package Path | `build_pipeline()` Location | Registry JSON | Registry Hash | Compiler Hash (final tree) | Match | Discovery Trust | Tenant ID | Generated Artifact Refs | Survives |
|---|---|---|---|---|---|---|---|---|---|---|
Every row proves `compiler(build_pipeline(final_tree, id)).manifest_hash == registry[id].manifest_hash`. This ledger is produced by an automated script that (a) enumerates all pipeline IDs from all registries, (b) resolves each to its `build_pipeline()`, (c) compiles and hashes, (d) cross-checks against every registry, discovery, trust, tenant, generated-artifact, and resume-cursor reference. The script fails M6 if any hash mismatch or orphan reference exists. This is M6 gate A above, elevated to a blocking done criterion.
