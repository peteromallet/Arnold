## Top Ways Deletion Could Falsely Pass (M6 Green = False Negative)

**1. `__pycache__` survivor imports.** Python imports `.pyc` files without `.py` source. If M6 deletes `arnold/pipelines/megaplan/__init__.py` but leaves `__pycache__/__init__.cpython-311.pyc`, `import arnold.pipelines.megaplan` still succeeds. A `.py`-only AST scan sees nothing; runtime sees the cached bytecode. Same for every deleted package.

**2. Installed-wheel staleness.** The current `pyproject.toml` builds `packages = ["arnold"]` into the wheel before deletion. If the M6 conformance suite tests against an editable install (where files are truly deleted) but never rebuilds the wheel and re-installs from it, the wheel that ships to users still contains every old module. A subprocess import test from a clean venv with the stale wheel passes locally but the published artifact is dirty.

**3. `arnold/pipeline/__init__.py` keeps `arnold.runtime` alive transitively.** The file currently does `from arnold.runtime.envelope import RuntimeEnvelope` and re-exports it in `__all__`. If M6 deletes `arnold/runtime/` but neglects to strip those re-export lines from `arnold/pipeline/__init__.py`, the import fails at the re-exporter — but if M6 wraps them in `try/except ImportError` to make the conformance import scan pass, old consumers that `from arnold.pipeline import RuntimeEnvelope` see it silently swallowed. Either way, the re-export chokepoint must be removed cleanly.

**4. `arnold/pipelines/megaplan/__init__.py` dynamic `__getattr__` and `import_module`.** The file defines `_SYMBOL_EXPORTS` mapping ~130 symbols to `arnold.pipelines.megaplan.*` dotted paths. `__getattr__` calls `import_module(...)`. A static AST scan sees no `import` statements and reports clean — but accessing any symbol at runtime triggers dynamic import of old paths. If M6 "deletes" by emptying `_SYMBOL_EXPORTS` but leaves the `__getattr__` machinery and module-level side effects (`_register_megaplan_content_types()`, `_install_model_adapter_once()`), the import gate passes but old behavior leaks.

**5. Console entrypoints baked into wheel metadata.** If `pyproject.toml` (or the eventual `arnold_pipelines` wheel config) defines `[project.scripts]` keys like `megaplan = "arnold.pipelines.megaplan.cli:main"`, those survive in `PKG-INFO`/`RECORD` after source deletion. `pip install` the wheel, run `megaplan`, and old code executes. The M6 gate must audit entrypoints from an installed wheel, not the source tree.

**6. `arnold/pipeline/__init__.py` re-exports of `PipelineBuilder`, `Stage`, `Edge`, `run_pipeline`, etc.** M6 Done Criterion #3 says `from arnold.pipeline import Stage` must fail. But `Stage` is defined in `arnold/pipeline/types.py` and re-exported. If M6 removes the re-export from `__init__.py` but doesn't delete `types.py`, `from arnold.pipeline.types import Stage` still works — and every old consumer that uses the dotted path survives. The conformance gate must test both `__init__` re-exports and full dotted imports of every public symbol.

**7. `.pyi` type-stub ghost modules.** Zero `.pyi` files exist today, but if a type-stub distribution or `py.typed`-marked package ships `.pyi` for deleted modules, type checkers resolve old imports. M6 Done Criterion #9 mentions "type stubs" but the check must be explicit: scan every `.pyi` in the built wheel and fail if any declares a deleted module.

**8. `agent/` top-level package or namespace package collision.** The repo has a top-level `agent/__init__.py`. If the wheel includes `agent/` as a separate package (or if `arnold_pipelines` is a namespace package that merges with an existing `arnold.pipelines` on `sys.path`), old import paths can be resurrected through package-path merging at install time.

**9. Test/support code imports from old paths that are excluded from scans.** M6 requires `tests/_pipeline/` deletion, but `tests/arnold/pipelines/megaplan/`, `scripts/`, `tools/`, conftest fixtures, and golden generators may still `import arnold.pipelines.megaplan` or `from _pipeline import ...`. If the import-graph scan whitelists test directories, old surfaces survive as importable from any code that runs with the repo on `sys.path`.

**10. Module-level side effects suppressed but not removed.** `arnold/pipelines/megaplan/__init__.py` calls `_register_megaplan_content_types()` and `_install_model_adapter_once()` at import time. If the deletion check catches `ImportError`, the gate is green, but these initialization steps silently don't run for the new path — behavior diverges without any test noticing because the conformance suite never exercises the old-init-then-fail path.

## Evidence That Would Make Me Confident the Old Surface Is Truly Gone

1. **Fresh wheel → clean venv → subprocess import failure.** A script that: (a) builds the wheel from the post-deletion commit, (b) creates a clean venv with `python -m venv /tmp/audit_venv`, (c) installs only that wheel, (d) runs `python -c "import megaplan"`, `python -c "import arnold.pipelines.megaplan"`, `python -c "from arnold.pipeline import Stage, Edge, PipelineBuilder"`, and `python -c "from arnold.pipeline.native import NativeProgram"` — and every one fails with `ModuleNotFoundError`.

A semantic scan of every regenerated asset finds no reference to `arnold.pipelines.megaplan`, `arnold.pipeline.Stage`, `_pipeline`, `PipelineBuilder`, `run_pipeline`, or `arnold.runtime`.

7. **Git worktree and `sys.path` isolation.** The audit runs from a clean checkout (no stale `.pth` files, no editable installs, no `PYTHONPATH` pointing at the repo root). `python -c "import sys; print([p for p in sys.path if 'megaplan' in p])"` returns `[]`.
