## Deletion Attack Analysis: Wave 9

### Top Ways Capability/Runtime Deletion Could Falsely Pass

**1. String-encoded import paths in registries and manifests.** M3 requires neutral capability dispatch through "string-keyed registries/contracts," and M2 resolves hooks to `module:qualname` strings. If a `module:qualname` string in a manifest or registry dict references a deleted surface (`arnold.pipelines.megaplan.old_module`), AST boundary scans pass because the string is data, not an import statement. The scan gap is that M6's dynamic-import scanning (criterion 4) must explicitly parse all registry dicts, manifest ref fields, and config payloads for embedded module paths—not just `importlib.import_module` call sites.

**2. Adapter internals not exercised by conformance.** M3 requires `arnold.agent` adapters satisfy neutral dispatch protocols without product imports. But an adapter class could internally `from arnold.pipelines.megaplan.schemas import MegaplanSignal` in a method body that conformance tests never call. AST scans on the adapter module pass because nothing at the module level imports deleted surfaces; the import is conditional or lazy. M6 must require *execution-coverage* of all adapter registration paths, not just static scans.

**3. `TYPE_CHECKING` exceptions as leaks.** M1 permits `TYPE_CHECKING` imports across the neutral/product boundary "if the exception is explicit and scanner-enforced." A type stub or `TYPE_CHECKING` block importing a deleted surface is technically allowed under M1's explicit exception. When that deleted surface disappears, the code still runs (type-checking imports are never executed), but the type contract is now broken. The scanner must prove zero `TYPE_CHECKING` imports reference deleted modules.

**4. Transitively tainted prompt/policy ref chains.** M4 registers Megaplan prompt-builders as `module:qualname` callables. At compile time, the ref `arnold_pipelines.megaplan.prompts.gate_review` resolves fine. But if that function internally imports from a deleted `native_hooks` or `_core` module, the compiler never discovers it—only runtime execution hits the deleted import. M6's gate must trace the full import closure of every registered callable, not just its direct module.

**5. Callback recovery pointing at deleted surfaces.** M4 preserves "execute-callback-failure recovery." If the recovery handler registration references old `native_runner.recover()` or `_bridge.fallback()`, and the registration happens at process startup inside a `try/except ImportError` block, the scan sees no import because the import is guarded. M6 must forbid `try/except` around any import that could mask a deleted dependency.

**6. Dynamic registration via startup hooks swallowing failures.** If product packages register handlers at startup (M3) and a registration function does `try: from old_module import X; registry['x'] = X; except: pass`, the startup succeeds silently until the capability is invoked. Scans must parse all registration call sites and reject any `except` that swallows an `ImportError` for a module not in the surviving allowlist.

**7. Generated artifacts with stale provenance.** M5 requires regenerated skills/docs to pass semantic checks. If the regeneration step itself produces correct output but the generator code still contains `import deleted_surface` (because the generator isn't run during conformance scans, only its output is), the scan passes. M6 criterion 13 checks regeneration diffs but must also AST-scan the generator scripts themselves.

**8. Installed-wheel leak through namespace package sidecars.** `arnold_pipelines/__init__.py` or `py.typed` markers may be remnants from before the migration. If the namespace package `__init__.py` contains a stale re-export or if `py.typed` in a deleted subpackage survives the wheel build, type checkers resolve deleted types. M6 must verify wheel contents at the file level, not just import behavior.

### Exact Gates/Plan Edits Needed

**M6 done criteria additions:**

- **22.** Every `module:qualname` string in every manifest fixture, registry dict, capability map, policy table, CLI dispatch table, and config payload is resolved against the surviving installed-wheel import graph; no string resolves to a deleted module.
- **23.** Full import closure of every registered prompt-builder, policy, reducer, and capability handler is traced; no transitive dependency lands on a deleted surface.
- **24.** Zero `try/except ImportError` (or bare `except`) in production code guards an import of any module listed in the M5 deletion inventory.
- **25.** Generator scripts, tools, sync utilities, and operator scripts pass AST import scans against the surviving surface; no generator references a deleted module even if generated output is clean.
- **26.** `arnold.agent` adapter registration is exercised end-to-end with full execution coverage; no adapter method body imports a deleted surface under any tested code path.
- **27.** `TYPE_CHECKING` imports across the neutral/product boundary are enumerated; none reference a module deleted in M6.

**M3/M4 amendments:**

- M3's "Product packages register handlers at process startup" must add: *Registration code must not use try/except to silently skip unavailable imports. Any import that may fail must be a hard gate with a scanner-detectable pattern.*
- M4's callback recovery must explicitly name the surviving registration module and commit to zero legacy recovery imports.

### Evidence Needed to Prove Runtime Dispatch Is Clean

1. **Import-closure trace file** generated by a tool that walks every `module:qualname` string in every manifest, registry, and config file, resolves each to a module, and produces the full recursive import graph—proving no path reaches a deleted module.

2. **Coverage report over adapter registration** showing every `arnold.agent` adapter method, every product handler registration, and every callback recovery path was executed in the conformance suite with import tracing enabled.

3. **Wheel-contents manifest diff** comparing the M6 built wheel against the M5 deletion inventory, proving no deleted `.py`, `.pyi`, or namespace `__init__.py` survives.

4. **Generator AST scan report** covering `scripts/`, `tools/`, and all generator modules, showing zero imports of deleted surfaces—independent of generated output checks.

5. **`try/except ImportError` audit** enumerating every such guard in production code and proving none masks a deleted dependency, or that all such guards were removed.

6. **Runtime dispatch trace** from a live-backed run exercising every capability, prompt, policy, reducer, control transition, and callback—with import hooks recording every module loaded—proving no deleted module is ever loaded at runtime.
