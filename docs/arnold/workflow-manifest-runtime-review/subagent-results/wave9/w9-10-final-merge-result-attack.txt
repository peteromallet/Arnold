## Deletion Attack: How the Final Merge Falsely Passes

### Top Ways the Merge Falsely Passes

**1. Git merge reintroduces files M6 deleted (the cardinal sin).** Each milestone branch passes alone because deletions are clean on that branch. But Git three-way merge does not propagate deletions when the file was touched (not deleted) on the other side of the merge. If M5 modified `arnold/pipelines/megaplan/__init__.py` and M6 deleted it, a naive merge resolution that picks the M5 side silently resurrects the deleted surface. No conformance gate reruns *after* the merge resolution — it ran on M6's pre-merge branch only.

**2. Generated artifacts pass freshness on M5's tree, rot across the merge boundary.** M5 regenerates skills, projection docs, pipeline registries, and scaffolds that reference "all surviving public surfaces." M5's freshness check passes because legacy surfaces were still present (or at least not yet deleted). After M6 deletions land, those same generated artifacts now embed references to deleted paths, deleted import identities, and deleted CLI entrypoints. But M6's freshness gate (criterion 13) tests regeneration against the *current* file tree — if the generated artifacts were committed on M5's branch and the merge brings them forward without re-triggering regeneration, the gate falsely passes because the stale artifacts are the baseline.

**3. Stale tests survive the merge because deleted code also survived.** If attack #1 resurrects a deleted module *and* attack #1 also resurrects its tests (e.g., `tests/_pipeline/test_old_runner.py`), the tests pass and mask the violation. The conformance scan only flags *imports* of deleted surfaces; if the deleted surface exists on disk due to merge error, the import succeeds and the test goes green. Absence is only verifiable when the thing is actually absent.

**4. Installed-wheel gate runs on M6's branch, not the merge result.** M6's done criteria require installed-wheel tests, but the chain says `merge_policy: review`. If the reviewer runs installed-wheel tests on M6's pre-merge branch (where deletions are intact), they pass. Then the merge into the integration branch happens, and no one rebuilds the wheel from the merge result. Editable-mode conformance on the merge checkout misses missing `py.typed`, missing package data, wrong entrypoints, and namespace-package gaps.

**5. Manifest version skew across merged amendments.** M3 (criterion 13) and M4 (criterion 13) can amend the M1 manifest contract and back-propagate contract tests. If M3 added a `retry_policy_v2` field to the manifest schema and M4's compiler was written against the pre-amendment schema, each passes alone because each tested against its own branch's version of the contract. After merge, the runtime expects the field but the compiler doesn't emit it, or vice versa. No merged-contract coherence gate exists.

**6. Dynamic import slipstream.** M6 requires catching `importlib.import_module`, lazy `__getattr__`, and `__all__`. But if a dynamic import string was on M5's branch as `importlib.import_module("arnold.pipelines.megaplan." + plugin_name)` and M6 deleted `arnold.pipelines.megaplan`, the string-based import survives the merge because no static AST scan can match a runtime-composed string. The import only triggers in production paths not covered by any milestone's tests.

**7. Conformance allowlist re-chartering during merge review, not during M6 implementation.** A row that was unresolved on M5's whitelist may be "re-chartered with non-legacy rationale" by the human reviewer during merge, bypassing the independent review, owner assignment, and dynamic-import proof that M6 would have required if handled during implementation.

### Exact Gates/Plan Edits Needed

**A. Merge-result rebuild gate (add to chain.yaml, after M6).** Add a mandatory `merge_result_conformance` stage that executes *after* all six branches are merged into a single integration checkout. This stage must:
- Build the wheel from the merge result, not from any feature branch
- Run the full installed-wheel conformance suite (M6 criteria 1–21) against that wheel
- Re-run M5's generated-artifact freshness and regeneration checks
- Re-run M1/M2/M3 contract/runtime tests against the merge-result wheel
- Fail if any file present in the merge checkout but absent from the wheel references a deleted surface  

**B. Merge deletion integrity gate.** The merge-result conformance must include a `git diff --diff-filter=D origin/main..MERGE_HEAD` check that enumerates every file deleted across the full chain. Then verify that none of those files exist in the merge checkout. Git merge can silently resurrect deleted files when the merge-base diverges; this explicit enumeration catches that deterministically.

**C. Generated-artifact re-generation gate at M6, not just M5.** Move M5 criterion 8 (regeneration yields clean diff) from a one-time M5 gate to a re-executed M6 gate. M6 must not only check freshness — it must delete all generated artifacts, regenerate them from the post-deletion tree, and prove the diff is empty. Stale generated artifacts brought forward from M5 commits fail this.

**D. Manifest contract coherence gate across amended versions.** Add a gate after M4 that compiles the canonical Megaplan-shaped fixture matrix to a manifest, runs it through the M3 runner, and verifies that the manifest hash produced by the merged compiler matches the hash expected by the merged runtime. If M3 and M4 amended the contract in mutually incompatible ways, the hash chain breaks.

**E. Dynamic-import runtime trap.** M6's conformance suite must instrument `importlib.import_module` and `__import__` during a full test run (including installed-wheel CLI smoke) and fail if any call references a path on the M5 deletion inventory. Static AST scans are insufficient.

**F. Allowlist freeze before M6 merge.** All M5 whitelist rows must be resolved (migrate/delete/re-charter) *before* the M6 branch is cut, not during merge review. Re-chartering during merge review must be treated as a plan amendment requiring the same independent review and owner assignment as implementation-phase re-chartering.

### Evidence to Prove the Final Merged State Is the Actual Clean-Break State

1. **Wheel diff against deletion inventory.** Build the wheel from the merge checkout. Diff the wheel contents against the committed M4 legacy-surface inventory and M5 migrate/delete outcomes. Prove every `delete`-classified path is absent from the wheel. Prove every `migrate`-classified path has a corresponding surviving path in the wheel.

2. **Full import closure from installed wheel.** Install the merge-result wheel in a clean venv. Run `python -c "import arnold_pipelines.megaplan"` and every surviving public entrypoint. Then run an import-graph tracer that proves no import chain reaches any path on the deletion inventory through any mechanism (direct, lazy, dynamic, `__getattr__`, console dispatch).

3. **Generated-artifact provenance proof.** The `GeneratedArtifactProvenance` contract from M1 must be enforced: every generated artifact in the merge-result wheel carries a `generator_source_hash` and `manifest_contract_version`. Re-run generation from the merge checkout and prove all provenance hashes match. A mismatch proves the artifact was generated from a pre-deletion tree.

4. **Full test suite against installed wheel, not editable checkout.** Every test (M1 contract, M2 compiler, M3 runtime, M4 parity, M5 pipeline/docs/CLI, M6 conformance) must pass against the installed wheel. If any test passes in editable mode but fails against the wheel, the merge is not clean-break.

5. **Canonical manifest hash stability proof.** Compile the canonical Megaplan fixture matrix to a manifest on the merge checkout, compute `manifest_hash`, and verify it matches the hash that M2's locked fixture expects. Then run, replay, and resume that manifest through the M3 runner and verify event journal hashes match. A hash mismatch proves manifest version skew across merged milestones.

6. **Conformance allowlist = zero legacy rows.** The allowlist must contain zero entries that reference any path, import identity, or entrypoint classified as `delete` in M4/M5 inventories. Any remaining row must have independent review evidence, non-legacy rationale, owner, expiry, and a dynamic-import trace proving the allowed path is not a renamed legacy surface.
