## Deletion Attack Analysis: Generated Artifacts (Wave 9)

Assume M6 gates appear green: all AST/import scans pass, freshness diffs are clean, compile/fake-run of embedded examples succeeds, and conformance allowlists are burned down. Here is how generated skills, composed rules, docs, projections, package disposition files, registries, and scaffolds could still teach deleted behavior.

---

### Top Ways Generated-Artifact Deletion Could Falsely Pass

**1. Generator source escapes scanning.** The M5/M6 scan gates (`git ls-files` + AST) cover `arnold/`, `arnold_pipelines/`, `docs/`, and `tests/`. But generators live in `scripts/generate_arnold_docs.py`, `sync-skills.sh`, and oracle/tool files. If a generator internally imports `arnold.pipeline.PipelineBuilder` to extract docstrings or introspect types, the generated output may be superficially clean while the generator itself remains a live import of deleted surfaces — and no gate catches it.

**2. Template/stub files carry hardcoded legacy references.** Scaffold templates and `SKILL.md` stubs are often Jinja2 or string-format files, not Python. They can contain literal `from arnold.pipeline import Stage`, `PipelineBuilder(...)`, or old import paths. AST scans don't parse templates. The generated artifact passes freshness because the template is the source of the stale reference and is never diffed against a migration spec.

**3. Data files encode deleted paths as inert strings.** `pipeline_ids.json` entries can carry `"module": "arnold.pipelines.megaplan.pipeline"`, and package-disposition JSON rows can point to `arnold/pipelines/evidence_pack/`. These are not Python imports — no AST or dynamic-import scanner catches them. Registry entries survive M6 intact, quietly teaching operators and discovery tools to look at paths that no longer exist.

**4. Freshness check uses cached regeneration, not actual regeneration.** If "regeneration yields a clean diff" relies on comparing committed artifacts against a stale virtualenv or cached intermediate, the diff is empty by construction. The generator may have never run against the post-deletion source tree. Similarly, if the freshness check only verifies that the generator's source hash hasn't changed (not that the generator itself is clean), committed artifacts pass with no actual regeneration.

**5. Transitive reference chains are unverified.** A generated skill imports `arnold_pipelines.megaplan.pipeline.build_pipeline`, which passes the import scan. But `build_pipeline` internally imports a product module that still `import _pipeline` — a deleted surface. The skill looks clean in isolation but delegates to contaminated code. No gate traces the transitive closure of generated-artifact imports through product code into deleted paths.

**6. Projection docs preserve deleted behavior in "historical" or "migration" sections.** `docs/reference/arnold-projections.md` or migration guides may include code blocks showing `PipelineBuilder`, `Stage`, or `run_pipeline()` under a heading like "Legacy API (removed)" or "Previous approach." These pass "no public examples of old builder/native APIs" scans because they're labeled as deprecated reference — but they still teach the deleted surface to any reader or LLM agent.

**7. Installed-wheel bundles stale copies.** The source-tree scan passes, but `pyproject.toml` `package_data` or `MANIFEST.in` includes a `data/_generated/` directory with cached skills/registries from before M6. The wheel content gate checks for "no deleted package data" by looking for `arnold/pipelines/megaplan/` directories, but stale JSON/CSV/TOML data files under legitimate package data paths pass because they don't look like deleted code modules.

**8. Semantic compile/fake-run test coverage is minimal.** The M5 criterion requires compile + fake-run of "embedded examples" in generated artifacts. If an artifact contains 20 code examples but only one trivial one is extracted and tested, a deleted reference in example #17 passes. The check is presence-based, not coverage-based.

**9. Composed rules reference deleted pipeline semantics.** A composed rule YAML may specify `pipeline: t19-external-builder` or `pattern: NativeParallel`, which resolve to old semantics. The composed rule file is clean Markdown/YAML with no Python imports — scanners pass it — but the rule's semantic target was deleted.

**10. Regeneration is non-idempotent or environment-dependent.** A generator produces different output depending on which packages are installed. If the regeneration test runs in an environment where deleted shims are still pip-installed, the output may embed references that vanish when the shim is removed — but the freshness check runs with the shim present and passes.

---

### Gates and Plan Edits Needed

**A. Add generator-source scan gate (M5/M6).** The generated-artifact manifest must include each generator's module path. M6 must AST-scan every generator module for references to deleted surfaces *before* regeneration runs. This closes attack #1.

**B. Add template-data scan gate (M5).** Expand the scanner surface to include `*.jinja`, `*.j2`, `*.mustache`, and scaffold template directories. Semantically scan these for string literals matching deleted module paths, class names, and `import` lines. This closes attack #2.

**C. Add data-file content scan gate (M6).** M6 criterion 13 (generated-artifact freshness) must be broadened to include JSON, YAML, TOML, and CSV registry/disposition/manifest files. Scan for path fragments (`arnold/pipelines/megaplan`, `arnold.pipeline`), module references as string values, and `pipeline_ids.json` entries pointing to deleted locations. This closes attacks #3 and #7.

**D. Require clean-checkout regeneration (M5 criterion 8 tightening).** Replace "diff is clean" with: regenerate in a *clean checkout* where M6 deletions are already applied (or in a simulated post-deletion import environment). Regeneration must succeed without importing any deleted surface, and the output must match the committed artifacts. This closes attacks #4 and #10.

**E. Add transitive import closure gate (M6).** For every symbol referenced by a generated artifact, verify transitively that its module's full import graph contains no deleted paths. This closes attack #5.

**F. Prohibit copy-pasteable deleted examples in docs (M5 anti-scope addition).** Add to M5: projection docs, migration guides, and historical references must not contain executable or syntactically-valid code blocks using deleted `import` lines, class names, or function calls. A semantic scan must flag ` ```python ... from arnold.pipeline import Stage ``` ` in any doc. This closes attack #6.

**G. Require full embedded-example extraction (M5 criterion 8 tightening).** Every code fence in every generated artifact must be extracted and run through compile + fake-run, not only a curated subset. This closes attack #8.

**H. Add composed-rule semantic resolution gate (M5).** Composed rule YAML must resolve their `pipeline` and `pattern` keys against the surviving pipeline ID registry. Any rule referencing a deleted or whitelist-only pipeline must fail. This closes attack #9.

### Evidence That Generated Artifacts Match the Clean-Break System

**1. Clean-checkout regeneration proof.** A single script that: (a) checks out the final merge result, (b) deletes all generated artifacts, (c) runs the full generator suite, (d) diffs the regenerated output against the committed artifacts. A pass proves generators are self-contained and produce identical output in a deletion-clean environment.

**2. Generator import-surface attestation.** Every generator file passes the same production import scan as `arnold/` code. Its AST is signed (hash) and the hash chain links generator → run → output artifact.

**3. Transitive import graph certificate.** A machine-readable graph for every generated artifact showing every referenced symbol → its defining module → that module's import closure, with proof no edge crosses into a deleted surface. Generated by a tool, checked in, re-verified by M6.

**4. Full embedded-example extraction report.** A JSON manifest listing every code fence extracted from every generated file, with a pass/fail for compile + fake-run against the installed wheel. 100% pass required, with no skipped or excluded fences.

**5. Installed-wheel roundtrip attestation.** Install the final wheel in a fresh virtualenv. Run every generator from the installed package (via console entrypoints). Diff the output against the committed source-tree artifacts. Identical output proves no environment-dependent regeneration.

**6. Composed-rule registry closure proof.** Every composed rule's `pipeline`/`pattern` key resolves to an entry in the final pipeline ID registry. No registry entry points to a deleted module path or whitelisted internal-only pipeline. The registry itself passes the data-file scan.

**7. Provenance chain closure per M1.** Each shipped artifact carries `GeneratedArtifactProvenance` with `generator_hash`, `manifest_contract_version`, and `input_surface_hashes`. M6 verifies every artifact's provenance chain is complete, the generator hash matches the clean generator source, and the manifest contract version is current.
