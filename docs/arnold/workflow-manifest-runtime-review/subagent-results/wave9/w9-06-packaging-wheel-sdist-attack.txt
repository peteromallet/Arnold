## Deletion Attack: Packaging Liability Analysis (Wave 9)

Assume M6 source-tree scans pass: `git ls-files` is clean, AST/import scans green, no production code imports deleted surfaces. Packaging can **still** ship deleted code through these vectors:

---

### Top Ways Packaging Falsely Passes

**1. Wheel/sdist inclusion divergence.** `pyproject.toml` `[tool.setuptools.packages.find]` with `include=['arnold*']` sweeps old `arnold/pipelines/megaplan/` into the wheel alongside `arnold_pipelines/megaplan/`. Same for `MANIFEST.in` globs (`recursive-include arnold *`) that predate M4's package split. Source-tree scans don't check build config exhaustiveness.

**2. Namespace-package shadowing.** `arnold_pipelines` is a namespace package (no `__init__.py`). If the old `arnold/pipelines/megaplan/__init__.py` ends up in the wheel—via residual package data, stale `find_packages`, or a leftover `arnold/pipelines/__init__.py`—it hijacks the import namespace. `import arnold.pipelines.megaplan` can resolve to the deleted path even if `arnold_pipelines` is correctly configured.

**3. `py.typed` leakage.** A `py.typed` marker in a surviving ancestor package (`arnold/pipelines/`) or a stale one bundled with deleted package data signals to type checkers that the old surface is typed. Type stubs and mypy cache then resolve deleted symbols without any source-tree import.

**4. Entrypoint ghost targets.** Console scripts in `[project.scripts]` or `[project.entry-points."console_scripts"]` that reference `arnold.pipelines.megaplan.cli:main` or `megaplan:main` survive source-tree deletion. `pip install` will fail at runtime, but the wheel metadata silently carries the dead reference.

**5. Top-level compatibility `megaplan/` package.** If `megaplan/` was ever a real package directory (not just a console entrypoint module), `find_packages()` or `MANIFEST.in` may still include it. Even an empty `megaplan/__init__.py` in the wheel satisfies `import megaplan`—violating M6 Done Criterion #1.

**6. Build isolation cache contamination.** `pip wheel` and `python -m build` may use stale build directories (`build/lib/`, `.egg-info/`) that weren't cleaned. Editable installs leave `.pth` files pointing to old paths. A clean checkout CI check is not the same as a `rm -rf build dist *.egg-info && pip wheel .` check.

**7. Dynamic `setup.py`/`setup.cfg` `packages=` drift.** If `setup.py` dynamically computes `packages` via `setuptools.find_packages(include=['arnold*', 'megaplan*'])`, and the package split isn't reflected in exclusions, old paths get bundled. M6 must prove the build config is statically auditable, not just that `git ls-files` is clean.

**8. `MANIFEST.in` vs sdist stale data.** `MANIFEST.in` `graft`, `recursive-include`, and `include` directives can reference directories that still exist on disk (build artifacts, nested checkouts, `.megaplan` working data). The sdist may ship deleted code that was never in `git ls-files` because it was generated or cached.

**9. Wheel RECORD silent inclusion.** The wheel's `RECORD` file lists every installed path. Source-tree scans won't catch files in `RECORD` that came from `MANIFEST.in`, `package_data`, or `data_files` without explicit wheel-contents inspection.

**10. Test-only packaging drift.** If `tests/` directories include `conftest.py` with `sys.path.insert` or editable-mode `.pth` files, they can make deleted imports pass during `pytest` even when the installed wheel would fail. M6's Done Criteria require installed-wheel tests, but many test suites default to editable mode.

### Wheel/Sdist Evidence Needed for Confidence

To prove deletion is real (not just source-tree-clean), M6 must produce:

1. **`dist/` diff report**: Unpack both the pre-M6 and post-M6 wheel. Diff RECORD files. Every deleted path must be absent; every surviving path must be justified by inventory traceability to M5 outcomes + M4 bridge/state inventories.

2. **Import-failure matrix from installed wheel**: A CI matrix that `pip install`s the built wheel in a clean venv and runs `python -c` for **every** deleted import path from M6 Done Criteria items 1-4, asserting `ModuleNotFoundError` or `ImportError` (not silently returning a stale namespace).

3. **`pip install --no-cache-dir` from sdist**: Prove the sdist builds and installs cleanly. The sdist is the distribution artifact that downstream packagers (PyPI, distro maintainers) consume; wheel-only checks miss sdist contamination.

4. **`find` over unpacked wheel**: `find unpacked_wheel/ -name '*.py' | sort` compared against the M5-to-M6 deletion inventory. No deleted module `.py` file may appear.

5. **Editable-install smoke after uninstall**: `pip uninstall arnold -y && pip install -e . && python -c "import megaplan"` must fail. Editable installs often bypass wheel packaging entirely and mask deletion bugs.

6. **Namespace `__path__` inspection**: `python -c "import arnold.pipelines; print(arnold.pipelines.__path__)"` from installed wheel must not list old `arnold/pipelines/megaplan/` as a discovered path.

**Summary**: Without explicit wheel/sdist unpack-then-audit gates, M6's source-tree deletion is packaging-theater. The plan currently trusts build config too much. Add gates A-H to M6, I-J to M4, K to M5, and demand the six evidence artifacts above. M6 Done Criterion #11 says "Built wheel and sdist contain... no deleted package data" but doesn't define the verification method. The method is: unpack, diff RECORD, grep paths, assert import failures.
