## How Old CLI Deletion Falsely Passes — Top 7 Vectors

**1. The `arnold/cli/__init__.py` lazy-import trap.** The ONLY production dispatch path — `arnold/__main__.py` → `arnold.cli.cli_entry()` → `main()` — does `from arnold.pipelines.megaplan.cli.arnold import main as _arnold_main` *inside a function*. A static AST scan sees zero imports of deleted modules at module load time. If M6's conformance test stops at `import arnold` without actually invoking `python -m arnold <subcommand>` from an installed wheel, deletion falsely passes — the import succeeds but the CLI crashes on first use.

**2. There are no `[project.scripts]` to audit, but `-m` entrypoints go unchecked.** `pyproject.toml` defines zero console_scripts. M6 Done Criterion #5 says "CLI entrypoints contain no references to deleted surfaces," but a naive implementation interprets "no entrypoints listed → nothing to check." Meanwhile `python -m arnold` (via `arnold/__main__.py`), `python -m arnold.pipelines.megaplan`, and `python -m arnold.pipelines.megaplan.cli` all route to the old CLI. None require a `[project.scripts]` entry.

**3. Three `__main__.py` files create parallel dispatch paths.** `arnold/pipelines/megaplan/__main__.py` and `arnold/pipelines/megaplan/cli/__main__.py` both import from `arnold.pipelines.megaplan.cli`. If M6 deletes the `cli/` directory but either `__main__.py` survives, `python -m arnold.pipelines.megaplan` still invokes the deleted surface. The deletion gate must check all three, not just `arnold/`.

**4. Parser snapshots are the fossil record of old command names.** `tests/characterization/test_cli_parser_snapshot.py` serializes the full argparse tree from `arnold.pipelines.megaplan.cli.parser.build_parser()`. The parser help text is saturated with `megaplan` subcommand names, `megaplan init`, `megaplan config profiles list`, etc. A M5/M6 migration that moves parser code to `arnold_pipelines.megaplan.cli` but doesn't regenerate the snapshot fixture — or worse, regenerates it from the *new* parser that's been updated but whose help text still embeds old names — makes deletion appear green.

**5. `scripts/generate_arnold_docs.py` is a live old-surface importer.** It directly imports `arnold.pipelines.megaplan.cli.arnold` and `arnold.pipelines.megaplan.cli.parser` (lines 41-42), plus calls `_handle_pipelines` (line 656). If M6 deletes the old CLI modules, this script breaks — but the M6 done criteria only gate *regenerated* artifacts, not the generator's own import surface. If the script isn't run (merely "checked for freshness"), the stale import survives.

**6. 30+ test files import old CLI paths directly.** Tests in `test_feedback.py`, `test_epic_cli.py`, `test_pipeline_runnable_e2e.py`, `conftest.py`, `test_override_strict_notes.py`, `test_chain_worktree_base_ref.py` and many others all import `arnold.pipelines.megaplan.cli` at module level. If these tests are updated to import from `arnold_pipelines.megaplan.cli` (new location) but the old `arnold.pipelines.megaplan.cli` modules are deleted, deletion "passes" only if every such test was either migrated or deleted — a combinatorially large conformance check that's easy to under-audit.

**7. Lazy dispatch *within* the old CLI creates nested survival.** `arnold/pipelines/megaplan/cli/arnold.py`'s `_megaplan_main()` itself does `from arnold.pipelines.megaplan.cli import main as megaplan_main`. If the outer dispatch (vector 1) is fixed but the *moved* CLI code's internal lazy imports still point to the old dotted path, subcommands break selectively. Conformance that tests only `arnold workflow --help` misses `arnold workflow run` failing 14 calls deep.

## Evidence Proving Only the Event-Backed CLI Remains

A single self-contained evidence gate, runnable by a reviewer:

1. **Build wheel, install in clean venv.** Run `pip install dist/arnold-*.whl` into a temp venv with no editable source on `PYTHONPATH`.

2. **Old entrypoints fail.**
   ```
   python -m arnold.pipelines.megaplan        # → ModuleNotFoundError
   python -m arnold.pipelines.megaplan.cli    # → ModuleNotFoundError
   ```

3. **Surviving entrypoint works end-to-end.**
   ```
   python -m arnold workflow --help            # → only arnold.workflow / arnold.execution event-backed help text
   python -m arnold workflow check             # → compiles, validates manifest
   python -m arnold workflow status <plan-id>  # → output derived ONLY from manifest event journal (prove by checking no state.json read)
   python -m arnold workflow trace <plan-id>   # → event-by-event journal projection
   ```

4. **Every surviving subcommand's `--help` output is piped through a regex deny-list:** `megaplan`, `PipelineBuilder`, `Stage`, `Edge`, `ParallelStage`, `run_pipeline`, `arnold.pipelines.megaplan`, `state.json`. Zero hits.

5. **The installed wheel's `entry_points.txt` (if any) and `RECORD` are scanned:** no paths matching `arnold/pipelines/megaplan/cli/`, no console_script named `megaplan`.

6. **`git grep 'arnold\.pipelines\.megaplan\.cli' -- '*.py' ':!docs/archive/' ':!.megaplan/'` returns zero hits** (excluding committed migration/retirement docs).

7. **Run the regenerated `generate_arnold_docs.py` against the installed wheel:** exits zero, output diff from pre-M6 is empty (modulo intentional renames).

All seven gates must pass against the merge result, not the feature branch. Any single failure = M6 blocked.
