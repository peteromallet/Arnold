# C1 Atomicity Failure Table

> Generated for `arnold.workflow.execution_attempt_ledger.v1`
> Schema reference: `arnold/workflow/execution_attempt_ledger.py`
> Diagnostic reference: `PersistenceFailureMode`, `PersistenceFailureDiagnostic`, `ReconciliationDiagnostic`
> Applies to: Megaplan phase handlers, `arnold.pipeline.native` adapters, and any runtime that produces `LedgerEvent` streams.

## Overview

Each boundary transition (prep→plan, plan→critique, …, execute batch, finalize, gate, override) consists of a sequence of durable writes that are **not** wrapped in a single transaction. The following table enumerates every atomicity gap — the crash points where a subset of writes may have succeeded while others have not — and prescribes the deterministic reconciliation that a C4+ runtime **must** apply when it encounters a partially-written boundary on restart.

The table is organized by crash point relative to the write sequence. For each crash point, the _visible state_ column describes what a runtime restart or C1 observer would see on disk. The _deterministic reconciliation_ column describes what the runtime must do to recover a consistent state.

---

## Write Sequence (Nominal)

For a typical phase handler (prep, plan, critique, gate, revise, tiebreaker, execute batch, finalize, review, override), the nominal write sequence is:

1. **Dispatch** — `set_active_step` writes invocation metadata to `state.json` (phase, invocation_id, session).
2. **Worker execution** — worker runs, produces artifacts and output.
3. **Result publication** — `_emit_phase_result` writes `phase_result.json` atomically; `_emit_boundary_receipt` writes `boundary_receipts/{boundary_id}.json` atomically and appends audit JSONL.
4. **External effects** (execute/finalize only) — `git add`, `git commit`, `git push` of output artifacts.
5. **Terminal append** — state history update in `state.json`; routing ledger append in `routing_ledger.jsonl`; event append in `events.ndjson`.

Each step is an independent durable write (or sequence of writes) with no transactional coordination across steps. A crash at any point leaves a partially-written boundary.

## Atomicity Failure Table

| # | Crash Point | Visible State | PersistenceFailureMode | Deterministic Reconciliation |
|---|-------------|---------------|------------------------|------------------------------|
| **CP1** | **Before dispatch** — after the scheduler decides to advance to the next boundary but before `set_active_step` writes `state.json`. | `state.json` still shows the previous phase and iteration. No `phase_result.json` for the new phase. No boundary receipt. No new artifacts. `events.ndjson` may contain a `step_start` event (best-effort before dispatch). | `indeterminate` (the `step_start` event may be absent or present; the write to `state.json` never happened so there is no durable record of the dispatch decision). | 1. Read `state.json` current_phase and iteration. 2. Read `events.ndjson` to find the most recent `step_start` event whose phase differs from `state.json`. If found, that event's phase is the intended dispatch target — re-dispatch from that phase. 3. If no `step_start` event exists for the gap, the scheduler's dispatch decision was not committed. Treat as a **no-op**: the previous boundary is still the active boundary; re-attempt the same boundary (idempotent). 4. Emit `reconciliation` event with `ReconciliationOutcome.RECOVERED` and `authority_disposition="re_dispatched"` or `"no_op"`. |
| **CP2** | **During result publication — after `phase_result.json` write, before boundary receipt write** — `_emit_phase_result` succeeded (atomic write of `phase_result.json`), but `_emit_boundary_receipt` (which writes `boundary_receipts/{boundary_id}.json` + audit JSONL append) did not execute. | `phase_result.json` exists with the correct phase, invocation_id, exit_kind, and artifacts_written. `boundary_receipts/{boundary_id}.json` is **absent**. Artifacts exist on disk. `state.json` may still show the old phase (history not yet updated). | `partial_write` — the phase result is durable but the boundary receipt (the declarative evidence of the boundary crossing) is missing. | 1. Detect: `phase_result.json` exists AND `boundary_receipts/{boundary_id}.json` is absent. 2. Reconstruct the boundary receipt from `phase_result.json`, the current artifacts, and `state.json`. 3. Write `boundary_receipts/{boundary_id}.json` atomically. Append audit JSONL. 4. Emit `reconciliation` event with `ReconciliationOutcome.RECOVERED`, `recovered_evidence_refs` pointing to the reconstructed receipt and `phase_result.json`, and `authority_disposition="receipt_reconstructed"`. |
| **CP3** | **During result publication — after boundary receipt write, before history update** — `_emit_phase_result` and `_emit_boundary_receipt` both succeeded, but `_finish_step` crashed before writing the history entry to `state.json`. | `phase_result.json` exists. `boundary_receipts/{boundary_id}.json` exists. Artifacts exist. `state.json` still shows the **old** phase and iteration (no history entry for the completed step). The routing ledger and `events.ndjson` may or may not have the terminal event. | `partial_write` — the boundary evidence is fully durable but the state transition is not committed. | 1. Detect: `phase_result.json` + boundary receipt exist, but `state.json` history does not contain the corresponding phase/iteration entry. 2. Apply the state delta: advance `current_phase` to the resolved next step, increment iteration, append the history entry with `result="success"` (or the exit_kind from `phase_result.json`). 3. Write `state.json` atomically. 4. Append routing ledger and events if absent (idempotent via invocation_id). 5. Emit `reconciliation` event with `ReconciliationOutcome.RECOVERED` and `authority_disposition="state_advanced"`. |
| **CP4** | **Before external effects — result and state published, but `git commit`/`git push` not yet executed** — all local writes are durable, but the external side effect (pushing artifacts to the remote repository) has not occurred. | All local state is consistent: `phase_result.json`, boundary receipt, `state.json` (current_phase advanced), artifacts, routing ledger, events. The git working tree has the artifact changes staged or unstaged, but no commit or push exists. | `unknown` (from the ledger's perspective — the crash happened outside the ledger's write domain; the ledger sees a complete, successful boundary with no persistence failure). | 1. Detect: boundary is fully complete locally, but the remote repository does not have the expected commit. 2. If the phase requires external effects (execute, finalize), re-attempt the git operations idempotently: `git add`, `git commit` (with the same message/content, which produces the same tree hash), `git push`. 3. If `git push` fails due to a concurrent push, pull-rebase and re-push (the artifact content is deterministic). 4. Emit `external_effect_outcome` event referencing the `external_effect_intent` event. 5. If the external effect is permanently unavailable (remote unreachable), emit `persistence_failed` with `PersistenceFailureMode.STORE_UNAVAILABLE` and `quarantined_authority_advance=False` (no authority advance is needed; the boundary is complete locally). |
| **CP5** | **After external effects — git push succeeded but crash before terminal event append** — the remote repository has the artifacts, but the local `events.ndjson` and/or `routing_ledger.jsonl` do not have the terminal event for this boundary. | All state files are consistent. Artifacts exist locally and remotely. `events.ndjson` may end with the last worker event but lacks the boundary-completion event. `routing_ledger.jsonl` may be missing the final routing row. | `partial_write` — the external effect outcome is durable but the terminal ledger event is missing. | 1. Detect: boundary receipt exists AND `events.ndjson` does not contain a completion event for this invocation_id. 2. Append the missing terminal event(s) to `events.ndjson` and `routing_ledger.jsonl` using the idempotency key derived from `invocation_id`. 3. If the append succeeds, emit `reconciliation` event with `ReconciliationOutcome.RECOVERED`, `recovered_evidence_refs` pointing to the existing boundary receipt, and `authority_disposition="terminal_event_appended"`. 4. If the append fails (quota, store unavailable), emit `persistence_failed` with the appropriate `PersistenceFailureMode`. The boundary is **still complete** — the missing terminal event is an observability gap, not a correctness gap. |
| **CP6** | **During terminal append — crash while appending to `routing_ledger.jsonl`** — the routing ledger append is a best-effort `record_step_routing` call that uses `fcntl` file locking and JSONL append. A crash during this append may leave a truncated or partial line. | All state files and boundary evidence are complete and consistent. `routing_ledger.jsonl` may contain a **truncated last line** (incomplete JSON object) or a **duplicate line** (if the append was retried after a partial write). `events.ndjson` may or may not contain the terminal event. | `partial_write` — the NDJSON append was interrupted, leaving an unparseable or duplicate entry. | 1. Detect: validate the last line of `routing_ledger.jsonl` (parse as JSON; if parsing fails, the line is truncated). 2. If the last line is truncated: truncate the file to the last valid newline boundary and re-append the complete row (idempotent via `invocation_id`). 3. If a duplicate row is detected (same `invocation_id` and `phase`), keep only the first occurrence. 4. Emit `reconciliation` event with `ReconciliationOutcome.RECOVERED` and `authority_disposition="routing_ledger_repaired"`. 5. **Do not** treat a missing routing ledger entry as a boundary failure — the routing ledger is best-effort observability, not a correctness dependency. |
| **CP7** | **During `events.ndjson` append** — the `events.ndjson` journal is append-only with per-event `seq` numbers and a sidecar `.events.seq` file. A crash during append may produce a truncated line, a missing seq increment, or a gap in the sequence. | All state files and boundary evidence are complete. `events.ndjson` may contain: (a) a truncated last line, (b) a complete last line but `.events.seq` was not updated, (c) a sequence gap (the seq counter was incremented but the line was never fully flushed). | `partial_write` — same NDJSON failure modes as the routing ledger, plus sequence counter desynchronization. | 1. Validate last line of `events.ndjson` (parse as JSON). If unparseable, truncate to last valid newline. 2. Read `.events.seq` and compare with the `seq` field of the last valid event. If `.events.seq` > `last_event.seq`, a gap exists — the seq counter advanced but the event was lost. 3. For a seq gap: emit `persistence_failed` event with `PersistenceFailureMode.PARTIAL_WRITE`, `target_event_sequence` = the missing seq number, and `quarantined_authority_advance=False`. Then append the missing event (reconstructed from boundary evidence) with the next available seq. **Do not** reuse the gapped seq number — seq numbers are never recycled. 4. If `.events.seq` < `last_event.seq`: update `.events.seq` to match. 5. Emit `reconciliation` event summarizing the repair. 6. **Sequence gaps** (where a seq number has no corresponding event) are `indeterminate` — consumers must not silently skip them. The gap is recorded as a `persistence_failed` event for downstream audit. |
| **CP8** | **During state.json atomic write** — `save_state_merge_meta` or equivalent writes `state.json` atomically (write to temp file, rename). A crash during this atomic write should leave the old `state.json` intact (the temp file is abandoned). However, if the filesystem does not guarantee atomic rename, a partial `state.json` may exist. | `state.json` is either: (a) the old version (atomic rename succeeded before crash — the temp file was renamed over the target), (b) a partial/temp file (`.state.json.tmp` or similar exists but rename did not complete), or (c) corrupted (the rename partially overwrote the file — unlikely on modern filesystems but possible on network filesystems). | `write_failed` if the target is corrupted; `unknown` if a temp file is present but the target is intact. | 1. If `.state.json.tmp` (or similar temp file) exists and `state.json` is valid JSON: delete the temp file. The old state is intact. 2. If `state.json` is missing or unparseable AND a temp file exists: rename the temp file to `state.json` (the atomic write completed the temp file write but not the rename). 3. If both are corrupt or missing: `PersistenceFailureMode.UNKNOWN` — require manual intervention. The boundary evidence (`phase_result.json`, boundary receipt) is the authoritative record; reconstruct `state.json` from it. 4. Emit `reconciliation` event with `ReconciliationOutcome.RECOVERED` or `REQUIRES_MANUAL_INTERVENTION`. |

## Phase-Specific Crash Points

The following crash points are specific to particular phase families because they involve operations not present in the generic write sequence.

| # | Phase Family | Crash Point | Visible State | PersistenceFailureMode | Deterministic Reconciliation |
|---|-------------|-------------|---------------|------------------------|------------------------------|
| **CP9** | **Execute (batch)** | Crash after `handle_execute_one_batch` completes a sub-batch but before the batch checkpoint is written. Some sub-batches produced artifacts and receipts; others did not. The `_emit_execute_boundary_receipt` in the execute handler emits per-sub-batch receipts. | Some `boundary_receipts/execute_*.json` files exist for completed sub-batches. The batch checkpoint (`batch_checkpoint.json` or equivalent) is absent or incomplete. `state.json` may not reflect partial progress. | `partial_write` — the execute batch is partially complete. | 1. Enumerate completed sub-batches by scanning `boundary_receipts/execute_*.json`. 2. Compare with the work plan (task list in `state.json` or `plan.md`). 3. Re-dispatch only the **uncompleted** sub-batches. Each sub-batch handler is idempotent (same inputs → same outputs). 4. After all sub-batches complete, write the batch checkpoint and the aggregate boundary receipt. 5. Emit `reconciliation` event with `ReconciliationOutcome.RECOVERED`, listing completed and re-dispatched sub-batches. |
| **CP10** | **Execute (approval)** | Crash after `evaluate_destructive_approval` emits an approval receipt but before the destructive action (e.g., `git push --force`) executes. | Approval receipt exists. The destructive action has not executed. `state.json` shows execute phase in progress. | `partial_write` — the approval decision is recorded but the side effect is absent. | 1. Detect: approval receipt exists but the expected side effect (tag, push, deployment marker) is absent. 2. Re-execute the side effect idempotently (the approval receipt authorizes it). 3. Emit `external_effect_outcome` event. |
| **CP11** | **Finalize** | Crash after artifact finalization writes (`finalize_artifacts.json`, `final_projection.json`) but before the final state transition to `STATE_FINALIZED`. | Artifacts and receipts are complete. `state.json` still shows `STATE_EXECUTED` or `STATE_BLOCKED`. | `partial_write` — the finalize evidence is durable but the state marker is stale. | 1. Detect: finalize artifacts/receipts exist but `state.json` current_state is not `STATE_FINALIZED`. 2. Set `current_state = STATE_FINALIZED` and write `state.json` atomically. 3. Emit `reconciliation` event with `ReconciliationOutcome.RECOVERED`. |
| **CP12** | **Override (authority)** | Crash after the override receipt is written but before the overridden state transition is applied. | `boundary_receipts/override_*.json` exists. `state.json` may still reflect the pre-override state. The override's target phase artifacts may or may not exist. | `partial_write` — the authority decision is recorded but the state transition is incomplete. | 1. Detect: override receipt exists but `state.json` does not reflect the override's state delta. 2. Apply the override's state transition (change `current_phase`, update `current_state`, record the override in history). 3. The override is an authority decision — it takes precedence over any other state. Apply it unconditionally. 4. Emit `reconciliation` event with `ReconciliationOutcome.RECOVERED` and `authority_disposition="override_applied"`. |

## Ledger-Level Reconciliation

Beyond individual boundary crash points, the ledger itself may exhibit gaps that require cross-event reconciliation.

| # | Condition | Visible State | Reconciliation |
|---|-----------|---------------|----------------|
| **LR1** | **Missing `started` event** — the first event in a ledger stream is not `started`. Events exist with `sequence >= 2` but no `sequence == 1`. | Ledger contains events but `first_event.event_type != STARTED`. | 1. Synthesize a `started` event from the identity and provenance of the first available event. 2. Assign `sequence = 1`, `causal_predecessor_sequence = 0`. 3. Mark `persistence_status = INDETERMINATE` (the event was reconstructed, not recovered from durable storage). 4. Emit `reconciliation` event referencing the synthesized event. |
| **LR2** | **Gap in event sequence** — events exist with sequences 1, 2, 4 (seq 3 is missing). No `persistence_failed` event records the gap. | `events` tuple has a sequence discontinuity. | 1. For each missing sequence number, emit a `persistence_failed` event with `PersistenceFailureMode.UNKNOWN` and `target_event_sequence` = the missing number. 2. The gap is recorded for audit. Consumers MUST NOT silently skip gaps. 3. If recovery evidence exists (e.g., spool files, worker transcripts), attempt to reconstruct the missing event and insert it. |
| **LR3** | **Duplicate idempotency key** — two events in the same attempt stream share the same `idempotency_key`. | Duplicate keys detected during validation. | 1. Keep the event with the lower `sequence` number (first durable write). 2. Mark the duplicate as `persistence_status = INDETERMINATE` with a `reconciliation` event referencing both. 3. Do not delete the duplicate — the ledger is append-only. |
| **LR4** | **Lifecycle precedence violation** — e.g., a `completed` event appears before any `started` event. | `events` tuple violates `_LIFECYCLE_PRECEDENCE`. | 1. The precedence violation indicates corrupted or out-of-order appends. 2. Sort events by `sequence` (authoritative order). 3. If precedence still fails after sorting, the ledger is irrecoverably corrupt — emit `persistence_failed` with `PersistenceFailureMode.CHECKSUM_MISMATCH` and `quarantined_authority_advance=True`. 4. Authority MUST NOT advance past a precedence-violating ledger. |

## Key Design Constraints

1. **No transaction coordinator exists** — each write in the sequence is independently durable. The reconciliation procedures above are the **only** correctness mechanism. Any C4+ runtime MUST implement them before claiming atomicity.

2. **Sequences are monotonic and never recycled** — when a gap is detected (LR2), the missing sequence number is recorded as a `persistence_failed` event. The reconstructed event uses a new sequence number.

3. **Idempotency keys are the retry safety mechanism** — every `LedgerEvent` carries an `idempotency_key`. Reconciliation procedures use these to detect duplicates and safely re-append events.

4. **Authority advances are quarantined on irrecoverable failures** — when `quarantined_authority_advance=True`, the runtime MUST NOT advance to the next boundary until manual intervention resolves the failure.

5. **The routing ledger and events.ndjson are best-effort observability** — missing entries in these journals do not affect boundary correctness. They are repaired for audit completeness, not functional necessity.

6. **This table is C1 specification, not C1 implementation** — C1 freezes the schema and documents the reconciliation contract. C4+ runtimes implement the procedures.
