# ADR: Skill Distribution Strategy

**Status:** Accepted  
**Date:** 2026-05-19  

## Principles

### One canonical home per skill

Every `megaplan-*` skill lives at exactly one path under `megaplan/data/`. No
skill has copies in `~/Documents/poms_skills/`, `~/.claude/skills/`, or
anywhere else — those locations hold only symlinks into the canonical home.
When reconciling drift, the package source always wins.

### Symlink everywhere

Every `_GLOBAL_TARGETS` entry in `megaplan/cli.py` uses `install: "symlink"`.
Copy mode is retired. The one exception — composed bundles (main skill with
agent-specific appendices) — is handled by pre-composing at commit time into
`megaplan/data/_composed/`. Those pre-composed files are single-source from
the installer's perspective, so `add_symlink_targets()` creates symlinks into
`_composed/` just like it does for single-file skills.

This strategy makes drift structurally impossible: editing a canonical file
updates every install target through the symlink chain, and there is no
install-time composition that could produce different content at different
install targets.

The three `_composed/` files are the main `megaplan` skill bundled with
agent-specific appendices (Claude subagent instructions, Codex subagent
instructions, or the Cursor rule header). They are generated by
`megaplan setup --regen-composed` and committed to the repo.

## Drift-guard mechanism

Install the canonical raw git hook with:

```bash
megaplan setup --install-hooks
```

Use `--force` to replace an existing `pre-commit` hook with Megaplan's
canonical drift guard. Fresh clones should run this once after checkout.

The installed `.git/hooks/pre-commit` script runs before every commit:

```bash
#!/usr/bin/env bash
set -euo pipefail
# Full source lives at megaplan/data/pre-commit-hook.sh.
...
```

**Contract:** `--regen-composed` reads `instructions.md` and the appendix
files, computes the three composed bundles, and compares each against the
existing file in `_composed/`. Exit 0 means no diff (all bundles match).
Exit 1 means one or more bundles were regenerated — the hook `git add`s them
and fails the commit. The developer re-runs `git commit` and the regenerated
files are included.

This is a commit-time guard, not a runtime guard. It catches drift before it
lands in history. The coverage hole (Claude reading a skill without invoking
the megaplan CLI) is acceptable because the symlink chain guarantees the
installed file is always the committed canonical — there is no separate copy
to drift from.

### Why not a pre-commit framework hook?

The repository does not use the `pre-commit` framework (no
`.pre-commit-config.yaml` exists). A raw `.git/hooks/pre-commit` script is
simpler and avoids adding a dependency for a single check. If the repo adopts
`pre-commit` in the future, the regen check should be migrated into the
framework config.

## Placeholder resolution

`_composed/claude_skill.md` and `_composed/codex_skill.md` contain resolved
numeric values, not literal `{max_execute_no_progress}` template strings.
`handle_regen_composed()` calls `_claude_subagent_appendix()` and
`_codex_subagent_appendix()`, which resolve `{max_execute_no_progress}` and
`{max_review_rework_cycles}` via `get_effective()` at regen time.

This means different developers may have different resolved values in their
working trees (depending on their local config). The pre-commit hook will
flag this as drift, and the developer should either align their config or
accept the resolved values in the commit.

## How to extend (adding a new skill)

1. **Create the canonical file** under `megaplan/data/` (e.g.,
   `new_skill.md`). Include proper YAML frontmatter with `name:` matching
   the skill slug.

2. **Add a `_canonical_*_skill()` helper** in `megaplan/cli.py` following
   the existing pattern:
   ```python
   def _canonical_new_skill() -> str:
       return resources.files("megaplan").joinpath("data", "new_skill.md").read_text(encoding="utf-8")
   ```

3. **Add a `bundled_global_file()` branch** that returns the canonical
   content unchanged (early return, before the inline composition block):
   ```python
   if name == "new_skill.md":
       return _canonical_new_skill()
   ```

4. **Add `_GLOBAL_TARGETS` entries** with `install: "symlink"` for each
   agent that should discover the skill:
   ```python
   {"agent": "claude", "detect": ".claude", "path": ".claude/skills/megaplan-new/SKILL.md", "data": "new_skill.md", "install": "symlink"},
   {"agent": "codex", "detect": ".codex", "path": ".codex/skills/megaplan-new/SKILL.md", "data": "new_skill.md", "install": "symlink"},
   ```

5. **If the skill is composed** (header + body + appendix), add it to the
   `--regen-composed` writer in `handle_regen_composed()` and create the
   pre-composed file under `megaplan/data/_composed/`. Update the canonical
   home table above. Most sub-skills are single-source and do not need this.

6. **Run the tests:** `pytest tests/test_setup_no_shadow_skills.py -v`.
   `test_every_target_uses_symlink` will fail if any new entry uses a
   non-symlink install mode.

## Personal vs package split

`~/Documents/poms_skills/` is the personal-toolbox home for **non-megaplan**
skills (bakeoff, subagent-launcher, hivemind, etc.). These skills are managed
separately and are not covered by this strategy. All `megaplan-*` skills live
in the megaplan package under `megaplan/data/` — `poms_skills/` should never
contain a `megaplan-*` directory.

## Subcommand naming

`megaplan setup --regen-composed` is a flag on the existing `setup` subparser
(in `megaplan/cli.py`). It runs the regen logic as a standalone operation
without triggering the full install flow. The flag does not require `--force`.
