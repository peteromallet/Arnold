vibecomfy: model-distribution v1 — extract HuggingFace URLs from workflow JSONs, populate READY_REQUIREMENTS["models"], add a small fetch CLI and doctor integration. Don't build a bundle/manifest format yet.

Source repo: /Users/peteromalley/Documents/reigh-workspace/vibecomfy

# Problem
Today vibecomfy has a half-built model-distribution story:
- Upstream Comfy-Org workflow JSONs in `workflow_corpus/official/*.json` already carry inline `"url": "https://huggingface.co/..."` fields per model-loader node (see workflow_corpus/official/video/wan_t2v.json:38, ltx2_3_i2v.json:699, flux2_klein_4b_t2i.json:1082, etc.) plus markdown notes in human-readable form.
- The materializer in `scripts/materialize_ready_templates.py` builds `ready_templates/*/*.py` from those JSONs but never extracts the URLs — every scratchpad has `READY_REQUIREMENTS = {"models": [], "custom_nodes": [...]}` with `models` empty.
- There is no downloader. `vibecomfy doctor` checks runtime readiness but doesn't check model presence.
- The de-facto convention is HuggingFace via the upstream Comfy-Org URL fields, but nothing in vibecomfy reads or enforces it.

# Goal
Close the loop in five mechanical steps:
1. Materializer extracts URLs from workflow JSON and populates `READY_REQUIREMENTS["models"]` with structured entries.
2. A small `vibecomfy.fetch` module downloads missing models from URLs to the right ComfyUI subdir.
3. `vibecomfy fetch <workflow>` CLI command resolves a workflow's model list and downloads the missing ones.
4. `vibecomfy doctor <workflow>` reports missing models with their URLs.
5. Tests for extraction, fetch (mocked httpx), and doctor.

The bundle/manifest format and SHA pinning are explicitly OUT of scope — they're a separate plan.

# Verified facts (do not re-research)
- Inline URL convention is in upstream JSONs:
  - workflow_corpus/official/video/wan_t2v.json:38 — text_encoders/umt5_xxl_fp8_e4m3fn_scaled.safetensors
  - workflow_corpus/official/video/wan_t2v.json:79 — vae/wan_2.1_vae.safetensors
  - workflow_corpus/official/video/wan_t2v.json:200 — diffusion_models/wan2.1_t2v_1.3B_fp16.safetensors
  - The URL field lives at the same JSON level as `class_type` and `inputs` for loader nodes.
- Materializer entry: `scripts/materialize_ready_templates.py` (writes `READY_REQUIREMENTS = {...}` literal into each generated scratchpad).
- Scratchpad shape: `ready_templates/{image,video,edit}/<name>.py` declares `API_WORKFLOW`, `READY_METADATA`, `READY_REQUIREMENTS`, and `build()`.
- Existing CLI registration: `vibecomfy/cli.py` + per-command modules under `vibecomfy/commands/` (e.g. `commands/doctor.py`).
- Existing dep is `httpx>=0.25` (pyproject.toml). No `huggingface_hub` dep — keep it that way for v1.

# Steps

## S1: Extract URLs during materialization
- In `scripts/materialize_ready_templates.py`, walk the source JSON and collect `(node_id, url)` entries from any node that has a top-level `"url"` field.
- For each URL, parse it to derive:
  - `name`: the basename (e.g. `wan2.1_t2v_1.3B_fp16.safetensors`)
  - `subdir`: the ComfyUI models subdir to land it in. Derive from the URL path — for HF Comfy-Org `split_files/<subdir>/<name>` URLs, use `<subdir>` (text_encoders, diffusion_models, vae, clip_vision, loras, latent_upscale_models, etc.). For other URL shapes, fall back to a heuristic based on the loader's `class_type` (CheckpointLoaderSimple → `checkpoints`, CLIPLoader → `text_encoders`, VAELoader → `vae`, LoraLoader → `loras`, ControlNetLoader → `controlnet`, UNETLoader → `diffusion_models`).
  - `url`: the URL as-is (drop `?download=true` query param if present).
- Emit `READY_REQUIREMENTS["models"]` as a list of `{"name": ..., "url": ..., "subdir": ...}` dicts in materialization order, deduplicated by `(name, subdir)`.
- Re-materialize all existing scratchpads as part of this step.

## S2: vibecomfy.fetch module
- New file `vibecomfy/fetch.py`:
  - `models_root() -> Path` — discover ComfyUI's models root via `comfy.cmd.folder_paths.folder_names_and_paths` if importable, else fall back to `os.environ["COMFY_MODELS_ROOT"]`, else `Path("ComfyUI/models")`.
  - `local_path(entry: dict) -> Path` — `models_root() / entry["subdir"] / entry["name"]`.
  - `is_present(entry: dict) -> bool` — checks the local path exists and has nonzero size.
  - `download(entry: dict, *, force: bool = False) -> Path` — uses `httpx.stream("GET", url, follow_redirects=True)` to download to a `.tmp` sibling file, then rename atomically. Raises `RuntimeError` with a clear message on 401/403 (license gating) and on 404. Skips if `is_present(entry)` and not `force`.
  - `download_many(entries: list[dict], *, force: bool = False) -> list[Path]` — sequential for v1 (parallel is a follow-up). Reports per-entry status via stdout (one line per entry: skipped/downloaded/failed).
- No `huggingface_hub` dep. Use plain HTTP + `Authorization: Bearer $HF_TOKEN` if the env var is set (for license-gated repos).

## S3: vibecomfy fetch CLI command
- New file `vibecomfy/commands/fetch.py`:
  - `vibecomfy fetch <workflow>` — resolves the workflow (same resolution as `inspect`/`doctor`), reads `READY_REQUIREMENTS["models"]`, calls `download_many`, reports counts.
  - Flags: `--force` (re-download present), `--dry-run` (print what would be fetched).
  - Register in `vibecomfy/cli.py` alongside the other commands.

## S4: doctor integration
- Extend `vibecomfy/commands/doctor.py` to:
  - Read `READY_REQUIREMENTS["models"]` from the resolved workflow.
  - For each entry, call `is_present`. Report missing entries with name + url + expected local path.
  - Exit nonzero (or surface a flag in the existing doctor output schema) when models are missing.

## S5: Tests
- `tests/test_materialize_extraction.py` — unit-test the URL/subdir derivation against a small fixture JSON (built inline, not snapshot) plus one real upstream sample.
- `tests/test_fetch.py` — mock httpx to verify: skip-when-present, download-to-tmp-then-rename, 401/404 error messages, dry-run.
- `tests/test_doctor_models.py` — verify doctor reports missing models from a fake `READY_REQUIREMENTS["models"]` and a tmp models root.
- `tests/test_fetch_cli.py` — argparse + dispatch.

# Out of scope (do NOT add)
- Bundle/manifest format (.toml manifest, vibe pack zip, install command). Separate plan.
- SHA256 verification. HF doesn't reliably expose hashes via resolve URLs; first-run hash + cache is a follow-up.
- Civitai-specific auth / API key handling.
- Parallel/multi-connection downloads.
- Auto-fetch on `vibecomfy run` (must be explicit `vibecomfy fetch`).
- A registry / hub for sharing.
- huggingface_hub dependency.

# Design constraints
- Default to HF, but the downloader is URL-driven, not provider-driven. A Civitai/R2/S3 URL works too — only the subdir derivation has an HF-aware fast path.
- Strip `?download=true` query params (Comfy-Org sometimes appends these).
- Strip `split_files/` prefix when deriving subdir from HF Comfy-Org URLs.
- License-gated repos (FLUX.2-klein-base-9b is one): download() returns a clear error pointing at the URL when 401/403, doesn't try to handle the agreement flow.
- Atomic writes (rename from .tmp). No partial files left behind on Ctrl-C.
- No new top-level deps. httpx is already there.
- Sequential downloads; this is v1 not perf-critical.

# Open questions
- Should `vibecomfy fetch` accept a `--workflow` group flag for multi-workflow batch fetching, or stick to one workflow per invocation? Lean toward one per invocation; users can `xargs` if they want.
- Should the materializer also extract custom-node references (already in `READY_REQUIREMENTS["custom_nodes"]` but as bare names)? Out of scope here — we don't have URLs for those in the JSON.
- Should `models_root()` honor a vibecomfy-specific env var like `VIBECOMFY_MODELS_ROOT` in addition to `COMFY_MODELS_ROOT`? Yes — fall through both before the hardcoded default.
