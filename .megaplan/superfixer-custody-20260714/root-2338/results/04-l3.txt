**4. Code-fix commit never deployed/retriggered**

This is **the exact detection gap**, and it was pre-identified by the 2026-07-03 subagent review (`.megaplan/audits/6hr-auditor-20way-subagent-review-20260703.md` L26: "WEAK: repair fixes committed but not installed", L40-41: "meta-repair returned FIXED but did not retrigger ordinary repair; or a patch was committed but the installed wrapper remained stale").

**Mechanics of the gap:**

- `_superfixer_cycle_evidence` computes `absent_or_stale_l2` as `not current_l2`, where `current_l2 = bool(meta_ages and min(meta_ages) <= 60 and not failed_l2)`. If the meta-repair ran, created a commit, and wrote meta evidence (records/logs), then `absent_or_stale_l2` is **False** → the true-stall gate blocks at `l2_backstop_failure_not_proven`.

- The `_l2_fix_vs_resolver_drift_finding` (six_hour_auditor.py L649) detects false-fixed-L2 only when the **brief projection claims recovery** (`shipped_fix="fixed"` AND `brief_outcome` in recovered set) but incident/resolver show active work. If the projection never updated to "recovered" (because the fix was never deployed and the chain never advanced), this drift detection never fires.

- The reason tokens `"false_fixed_l2"`, `"l2 false success"`, `"no ordinary repair retrigger"` searched by `_l2_failure_fingerprint` are **never directly generated by the gather program**. They only appear as substring matches from reconciler drift findings (e.g., `"l2_fix_vs_resolver:false_fixed_l2_result"`), which require the brief to claim recovery first.

- `verify_recovery` (escalation L726) catches this post-hoc (requires `ordinary_retriggered`), but only during `_terminal_reverification` for already-dispatched repairs — not as a proactive pre-escalation gate.

### Exact Backstop Gap

**The L3 true-stall gate has no pre-dispatch check for "meta-repair evidence exists and is fresh but the code fix was never deployed/retriggered to the running session."** The pipeline treats fresh L2 evidence as a backstop that worked, even when the commit sits un-deployed. The 2026-07-03 review recommended (L57): "Expand `meta_repair_summary` to include partial failures: … FIXED but no retrigger, no ordinary repair attempt after meta record" and (L70): "Add active-install freshness checks so 'FIXED <sha>' means the running wrapper changed." Neither recommendation is implemented in this version.

### Likely Narrow Regression

A session with `claim_count=0`, dead runner, stale nonterminal custody, and a meta-repair commit that was never deployed **would NOT be escalated by current L3**, because:

1. `_superfixer_cycle_evidence` sets `absent_or_stale_l2 = False` (fresh meta evidence exists)
2. `_l2_failure_fingerprint` returns `failed=False` (no drift contradiction without projection claiming recovery)
3. `classify_true_stall` blocks at `l2_backstop_failure_not_proven`
4. The finding stays `report_only` — the session remains stuck indefinitely

The minimal fix is adding an `installed_freshness` check in either `_superfixer_cycle_evidence` or `_l2_failure_fingerprint` that compares the deployed wrapper/runtime SHA against the meta-repair's output commit SHA, and treating a mismatch as L2 backstop failure even when meta evidence is present and fresh.
