## Megaplan ↔ AgentBox Overlap Audit: Machine Provisioning & Credential Sync

### 1. Directly Reusable

**SSH Provisioning — `providers/ssh.py`**
The `SshProvider` class already handles the transport layer AgentBox needs: `_sync_deploy_dir` (rsync/scp), `_remote_run` (SSH → Docker exec), `upload_file` (base64), `read_remote_file`, and container lifecycle (`build`/`deploy`/`docker rm -f`/`docker run -d`). For a bare-metal Hetzner target, you’d remove the Docker middleman — the `_remote_run` command just drops the `docker exec` wrapper and runs directly. The `ssh.host`, `ssh.user`, `ssh.port`, `ssh.identity_file` config fields from `cloud.yaml` map cleanly to the AgentBox machine target.

**Credential Seeding — `auth.py`**
`seed_codex_oauth()` and the `OAUTH_SEEDS` dataclass already implement the pattern: base64-encode a local file, pipe it through SSH, write to both `/workspace/.creds/<file>` (persistent) and `/root/<file>` (runtime). The entrypoint template (`entrypoint.sh.tmpl`) restores from `/workspace/.creds/` on reboot. AgentBox’s proposed `/secrets/` directory maps directly to this existing dual-location pattern. The `OAuthSeed` struct (label, local_relative, persistent_dest, root_dest) is generalizable to any secret file.

**Entrypoint Bootstrap — `entrypoint.sh.tmpl` + `template.py`**
The entrypoint already handles: directory scaffolding (`/workspace`, `/root/.ssh`, `/root/.codex`), git identity config, GitHub credential helper, repo clone-if-missing (`ENSURE_REPO_BLOCK`), OAuth restore from persistent volume, Claude refresh-token shim, agent routing config, and `arnold` install/upgrade. AgentBox’s startup sequence is a superset — add Guardian launch and Discord listener.

**Multi-Repo Clone — `render_ensure_repos_block()` in `template.py`**
Already iterates `spec.repo` + `spec.extra_repos` and calls `render_ensure_repo_command()` for each. AgentBox extends this by changing the workspace layout (sibling dirs under `/workspace/repos/`) but reuses the same clone-if-missing logic. The `extra_repos` field in `cloud.yaml` may already support this.

### 2. Needs Extraction or Generalization

| Existing Piece | Location | Change Needed |
|---|---|---|
| `SshProvider` | `providers/ssh.py` | Decouple from Docker (`docker exec` → raw SSH). Add a `bare_metal` flag or new provider subclass. |
| `seed_codex_oauth()` | `cloud/auth.py` | Generalize from OAuth-specific seeds to arbitrary secret files. The `OAUTH_SEEDS` tuple should become a config-driven list. |
| `render_ensure_repo_command()` | `cloud/template.py` | The AgentBox layout (`/workspace/repos/<name>.git` or bare repo) differs from the single-`repo.workspace` pattern. Extend to accept a `workspace_root` and repo-name-based subdirectory. |
| Entrypoint runner block | `cloud/template.py` | Current modes are `auto`/`chain`/`idle`. Add an `agentbox` mode that launches Guardian daemon + Discord listener instead of a single tmux session. |
| Worktree mechanics | `bakeoff/worktree.py` | `validate_worktree_name`, `ensure_no_inprogress_op` are reusable, but worktree-per-operation + per-runner isolation needs a new coordinator. |
| `arnold-supervise` | Referenced in entrypoint | The supervisor wrapper exists for auto/chain. Need to understand its internals to extend for Guardian-style periodic checks. |

### 3. Missing Pieces (Net New)

- **Operation registry** — the core AgentBox state (operation ID, kind, repos, worktrees, branches, tmux session, log path, status, pending approvals). Nothing like this exists in Megaplan today beyond plan-level state files.

- **Guardian daemon** — the periodic supervisor that scans operations, checks tmux/process liveness, classifies states, restarts safe runners, and files approvals. Entirely new code.

- **Discord Operator** — Discord-triggered agent with access to AgentBox state and tools. No Discord integration exists in Megaplan.

- **Safety/approval system** — explicit pending approvals for risky actions (product decisions, merge conflict resolution, PR merges, quality debt). Current Megaplan has `auto_approve` as a binary toggle.

- **Per-operation worktree isolation** — the `--in-worktree` flag exists but is scoped to a single plan. AgentBox needs a factory that creates worktrees from bare repos, assigns them to operations, and cleans them up.

- **`agentbox.db` + `config.yaml`** — the AgentBox plan references a SQLite operation registry and a manager config. These need schema design.

### 4. Risks and Gotchas

1. **Docker coupling.** The entire cloud stack assumes a Docker container. The entrypoint, health server, and `tmux` session all run inside Docker. For a bare-metal Hetzner VM, you either keep Docker as the sandbox layer (simpler, riskier isolation) or strip it out (more work, better perf for git worktrees). The AgentBox plan describes bare worktrees on the filesystem — this conflicts with the current Dockerized model.

2. **Secret rotation blind spot.** The Claude refresh-token shim handles rotation, but codex OAuth tokens have no refresh path. If a codex token expires on the remote machine, the Guardian has no way to refresh it without user intervention. AgentBox needs a credential health check and re-seed flow.

3. **Single-repo bias in entrypoint.** The entrypoint template interpolates `REPO_URL`/`REPO_BRANCH`/`WORKSPACE_PATH` as singletons. `render_ensure_repos_block` handles multi-repo clone, but `WORKSPACE_PATH` is used downstream for `cd`, agent routing, and runner launch. AgentBox needs the concept of a repo *registry*, not a single workspace.

4. **GitHub token scope.** The entrypoint configures a global git credential helper with `GITHUB_TOKEN`. In multi-repo AgentBox with mixed public/private repos, a single token may not suffice. Also, `safe.directory` is set only for `WORKSPACE_PATH` — needs to cover all repos.

5. **`arnold-supervise` internals are opaque.** The Guardian needs to understand chain/plan state. If `arnold-supervise` black-boxes this, the Guardian can only check process liveness, not semantic progress.

6. **Port conflicts.** The entrypoint binds `resources.port` (default 8080) for the health server. AgentBox adds a Discord listener — needs its own port or a reverse proxy.

### 5. Recommended First Slice

**Phase 1: Bare-metal SSH provisioning + multi-repo clone + credential sync + Guardian skeleton.**

1. Extend `SshProvider` with a `bare_metal=True` mode that runs commands directly (no Docker). Target: Hetzner CX53 via `ssh.host` + `ssh.identity_file`.

2. Generalize `auth.py` to `seed_secrets(secrets_list: list[SecretSpec])` where each spec has `local_path`, `persistent_dest`, `root_dest`. Backward-compatible with existing OAuth seeds.

3. Extend `render_ensure_repos_block()` to use `/workspace/repos/<name>` layout with bare-repo option (`git clone --bare`). Write a `cloud.yaml` extension for `repos: [{name, url, branch}]`.

4. Write a minimal Guardian that: starts as a tmux session, loads a `config.yaml` from `/workspace/manager/`, runs a loop every N minutes, and prints a heartbeat. No operation registry yet — just prove the daemon stays alive and has access to all repos.

This slice validates the provisioning and credential pipelines without touching the operation registry or Discord, and produces a machine you can SSH into with all repos cloned and credentials live. Everything after this builds on that foundation.
