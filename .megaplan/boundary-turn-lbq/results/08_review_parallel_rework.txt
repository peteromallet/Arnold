## Verdict (Question 8)

**No, not yet.** The `BoundaryTurn` design currently covers only the single-worker `promote_scratch()`-style review path (`review_output.json` capture → validate → promote). It does not model the parallel/extreme review merge as a reducer `BoundaryTurn`, nor does it address the fact that the parallel path completely bypasses `promote_scratch()` and assembles its own merged payload from scattered per-check worker outputs. The design doc's Stage Coverage table says "Wrap existing scratch promotion and route result into review/rework transition policy," but the extreme path has no scratch to wrap — it has a fan-out-reduce pattern that the design calls out in the Risks section ("over-generalizing could erase meaningful phase behavior, especially execute batch reduction, review rework") yet never resolves. All 14 preservation-checklist items listed for Review are gated on both paths surviving, but the abstraction as proposed only models the first.

## Review Behavior at Risk

1. **Parallel merge ≠ scratch promotion.** The extreme path (`run_parallel_review` in `review/parallel.py`) fans out N per-check worker units (each writing `review_check_{check_id}.json`) plus a criteria-verdict side-unit (`review_criteria_verdict.json`), then merges them in `handle_review` via `_synthesize_review_rework_items()`, flag-ID deduplication, and verdict override. The merged payload is not validated through `promote_scratch()` at all — canonical promotion to `review.json` happens inline. A `BoundaryTurn` that only wraps `promote_scratch()` would miss this entire path.

2. **Infrastructure-failure detection lives post-merge.** `_review_infrastructure_failure()` runs inside `_finalize_review_outcome()` after both single-worker and parallel payloads have been assembled. A naive BoundaryTurn that validates only at draft-capture time would lose the structured infra-failure signals (explicit `review_completion_status == "incomplete"` and `source in _REVIEW_INFRASTRUCTURE_SOURCES`) that prevent bogus executor passes.

3. **Empty-approved backfill.** `_backfill_empty_approved_review_from_execution()` runs before `_merge_review_verdicts()` and requires `finalize.json` to be available for task/sense-check enumeration. If BoundaryTurn promotes review scratch before finalize is in scope, backfill won't fire.

4. **Verdict merge into finalize projection.** `_merge_review_verdicts()` mutates a `deepcopy(finalize_data)` and returns coverage counts; those counts later drive the blocked/needs_rework/success routing in `_resolve_review_outcome()`. The BoundaryTurn design lists `finalize.json` as a canonical target for the `finalize` stage, not `review`. The review stage's merge into finalize is a cross-artifact side effect that the current `BoundarySpec` has no slot for.

5. **Rework caps.** `_resolve_review_outcome()` reads history entries to count prior rework cycles and applies `max_review_rework_cycles` / `max_robust_review_rework_cycles`. This depends on `state["history"]` which BoundaryTurn's `PromotionResult.state_delta` could carry — but only if the turn is opened with state access, which the current `BoundaryTurn.open(spec, plan_dir, state)` signature supports. The risk is that the turn's validator layer doesn't know about history, so cap-busting logic stays outside the boundary.

6. **Transition-policy denial artifacts.** `_persist_review_done_transition_decision()` writes `transition_decision_review_done.json` and, on denial, rewrites `review.json` with outcome overrides and routes back to `STATE_EXECUTED`. This is a second promotion within a single handler invocation — BoundaryTurn's single `promote()` call model doesn't account for conditional re-promotion.

7. **Parallel review receives no scratch promotion at all.** Lines 1477–1517 of `review.py` build `merged_payload` entirely from the parallel result, validate it with `_audit_review_payload_or_raise`, and write `review.json` directly. There is no `review_output.json` to capture; the "scratch" files are per-check and per-criteria, not a single draft. A BoundaryTurn that expects exactly one draft path per turn can't model this.

## Specific Design-Doc Edits and Acceptance Tests

**Design-doc edits** (to `megaplan-boundary-turn-design.md`):

1. **Stage Coverage table, Review row** — change the BoundaryTurn target from "Wrap existing scratch promotion and route result into review/rework transition policy" to:
   > "Single-worker: wrap existing `promote_scratch()` → `_finalize_review_outcome()` in a json-kinded `BoundaryTurn`. Parallel/extreme: model the fan-out merge as a child-subpipeline `BoundaryTurn` whose reducer produces the merged payload, then feed that payload into the same `_finalize_review_outcome()` post-boundary flow. The reducer turn must preserve infrastructure-failure detection, empty-approved backfill, rework caps, maker-stop, and transition-policy denial."

2. **Multi-Stage Semantics, Review rework subsection** — add:
   > "Parallel review is a subpipeline boundary: each per-check unit is a child turn scoped to the parent review turn; the criteria-verdict side-unit is an additional child; the merge/reduce step is a reducer turn whose canonical target is the review payload that feeds `_finalize_review_outcome()`. The reducer must capture verified/disputed flag-IDs from all children and synthesize `rework_items` with deterministic-check grounding before verdict normalization."

3. **Preservation Checklists, Review** — add two explicit bullets:
   > "- Parallel review reducer turn merges per-check JSON drafts without losing per-check `concerned_task_ids`, `findings.deterministic_check`, or `flag_id` provenance."
   > "- Transition-policy denial path (conditional re-promotion of `review.json`) is modeled as a second promotion within the same review turn rather than a separate turn."

**Acceptance tests** (new, `tests/megaplan/boundary_turn/test_review_boundary.py`):

| Test | What it proves |
|---|---|
| `test_single_worker_review_boundary_wraps_scratch` | Single-worker path through BoundaryTurn produces identical `review.json`, `final.md`, receipts, and history as current handler |
| `test_parallel_review_boundary_child_turns` | N per-check child turns + criteria-verdict child turn all produce valid per-check JSON; reducer merges without dropping `deterministic_check` fields |
| `test_parallel_review_reducer_preserves_infra_failure` | A child returning `review_completion_status: "incomplete"` causes reducer to produce infrastructure-failure signal, not rework items |
| `test_boundary_turn_preserves_empty_approved_backfill` | Approved review with empty task/sense verdict arrays triggers backfill from `finalize.json` before canonical promotion |
| `test_boundary_turn_transition_policy_denial_repromotes` | Policy denial rewrites `review.json` with `policy_denied` outcome and writes `transition_decision_review_done.json` within the same turn |
| `test_boundary_turn_rework_cap_escalation` | After N rework cycles, the turn produces `STATE_BLOCKED` with blocker-list in issues, not a silent force-proceed |
| `test_boundary_turn_maker_stop_in_creative_mode` | `needs_rework` verdict with `stop_signal.requested==True` routes to `STATE_DONE` with recorded stop note |
| `test_boundary_turn_flag_updates_survive` | `verified_flag_ids` / `disputed_flag_ids` from parallel reducer are propagated to `update_flags_after_review()` |

These tests collectively ensure that both single-worker and parallel/extreme review paths survive migration under BoundaryTurn without regressing any of the 14 preservation-checklist behaviors.
