## 1. Verdict

**Not yet sufficient.** The parent/child BoundaryTurn model correctly identifies the structural invariant — child turns produce evidence, only a parent reducer advances state — but the design doc hand-waves the `subpipeline` draft kind and provides no concrete reducer-turn lifecycle. The current tiebreaker implementation (researcher→challenger→synthesis→decide) is far from the BoundaryTurn API surface, and three critical gaps would cause it to fail the preservation checklist.

---

## 2. Subpipeline/Tiebreaker Behavior At Risk

**Child turns bypass the draft→capture→validate→promote lifecycle.** The orchestrator (`tiebreaker_orchestrator.py:79-116`) calls `run_step_with_worker` and then `atomic_write_json` directly to `tiebreaker_researcher*.json` and `tiebreaker_challenger*.json`. These are canonical writes, not draft fills. There is no harness-provided seed template, no unmodified-detection, no validation-before-promotion. A researcher that writes to the wrong path or produces malformed JSON is not caught by a boundary — it's caught downstream in the decide handler's `read_json` call, or worse, silently swallowed.

**The decide handler conflates reduction and promotion.** `handle_tiebreaker_decide` (`_tiebreaker_impl.py:76-155`) reads child outputs, constructs a `TiebreakerDecision`, appends to `tiebreaker_decisions.json`, records an audit, mutates the flag registry, and sets the next state — all in one monolithic block. BoundaryTurn requires that only `promote()` writes canonical artifacts, mutates state, emits receipts, and selects the next transition. The design doc (line 102) is explicit here, and the tiebreaker violates every clause.

**Gate.json input is not formalized as reducer evidence.** The decide handler re-reads `gate.json` (line 92) to extract `tiebreaker_question`, `tiebreaker_flag_ids`, and `tiebreaker_fuzzy_group_id`. Subpipeline semantics (lines 183-188) say the parent sees "decision, payload, evidence refs, and route" — but they don't specify that prior-turn canonical artifacts like `gate.json` must be passed to the reducer as input evidence. Without this, the reducer turn has no principled way to consume the context that triggered the tiebreaker.

**Flag registry mutation is not checkpointed.** Line 124-129 writes `settled_by_tiebreaker` directly into the flag registry. In the ordered-promotion model (lines 117-121), state deltas must move through resumable checkpoints alongside artifact writes. If the process dies after writing `tiebreaker_decisions.json` but before mutating the flags, the plan is in an inconsistent state with no recovery path.

**Versioned runs are not modeled.** The tiebreaker supports `_v2`, `_v3` suffixes for repeated runs (`_next_version_suffix`, line 36-45). BoundaryTurn has `iteration`/`attempt` metadata on `BoundaryDraft` but the subpipeline section never explains how child-turn versioning maps to those fields. Should each versioned researcher run be a new child turn with incremented iteration? Should the reducer aggregate across versions?

**Audits are not receipts.** `record_tiebreaker_audit` (line 122) is called directly. In the BoundaryTurn model, observability events should be emitted as `EventRef` tuples during promotion (lines 85-86), not as ad-hoc side effects inside a handler.

## 3. Specific Design-Doc Edits and Acceptance Tests

### Edits to `megaplan-boundary-turn-design.md`

**a) Replace "For subpipelines" section (lines 183-188) with a concrete specification:**

```
For subpipelines:

- Each child turn is a full BoundaryTurn with its own draft_kind
  (typically "json"), template, validator, and promoter. The child
  promoter writes child-scoped canonical artifacts only — never parent
  artifacts or parent state.
- The reducer is a separate BoundaryTurn with draft_kind "reducer".
  Its draft surface is a structured payload (JSON) containing: decision,
  child_evidence_refs (paths to child canonical artifacts), route, and
  state_delta_proposal.
- The reducer promoter validates child evidence exists under expected
  paths, constructs the final decision artifact, applies state deltas
  (including flag registry mutations), emits receipts, records audits
  as observability events, and sets the next workflow transition — all
  in ordered, fail-closed checkpoint sequence.
- Prior-turn canonical artifacts required as reducer input (e.g.,
  gate.json) are passed as read-only evidence refs in the reducer's
  BoundaryDraft.seed_context, not by re-reading inside the handler.
- Versioned child runs use BoundaryDraft.iteration. The reducer
  aggregates across all iterations for the same parent_turn_id.
```

**b) Add to the Tiebreaker preservation checklist (after line 401):**

```
- Child researcher/challenger turns capture only expected draft paths;
  direct writes to tiebreaker_researcher*.json or
  tiebreaker_challenger*.json are rejected as boundary violations.
- Reducer turn validates that child canonical artifacts exist before
  promoting tiebreaker_decisions.json.
- Flag registry mutation occurs during reducer promotion, not before.
- Reducer turn fails closed if gate.json evidence ref is missing or
  child artifacts are missing/unmodified.
```

**c) Add to the Open Questions section:**

```
- Should the reducer turn use the same draft path for reprompts
  (like gate reprompts do), or should each reducer attempt produce a
  new versioned decision entry?
```

### Acceptance tests to add

1. **Child-turn wrong-path write test:** Run tiebreaker researcher, have the model write to `tiebreaker_decisions.json` instead of the expected draft path → capture status is `"missing"` or `"invalid"` with a boundary violation diagnostic; no canonical artifact is promoted.

2. **Reducer-missing-evidence test:** Run decide with researcher output missing → reducer validation fails before `tiebreaker_decisions.json` is touched or state is advanced.

3. **Flag-registry atomicity test:** Simulate a crash between `tiebreaker_decisions.json` write and flag registry mutation → on resume, the plan either replays the full promotion (idempotent) or detects the incomplete promotion and fails closed with a clear diagnostic.

4. **Versioned-iteration test:** Run tiebreaker twice (v1, v2) → both iterations appear as distinct child turns under the same `parent_turn_id`; reducer sees evidence from both iterations and can select the latest or aggregate.

5. **Direct canonical write rejection test:** In a file-tool worker, model writes directly to `tiebreaker_researcher_v1.json` (the canonical path, not a draft) → the child turn captures it as a boundary violation diagnostic, not as valid output.
