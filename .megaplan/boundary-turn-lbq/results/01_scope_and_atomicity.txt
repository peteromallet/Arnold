## 1. Verdict: **Sound**

The BoundaryTurn design draws the right scope boundary. The code evidence strongly confirms the design's central claim: **plan-dir artifact promotion is the atomic boundary; target-repository mutations are external side effects that can only be recorded, not rolled back.**

Here is the concrete lifecycle I traced in `batch.py`:

1. **Worker mutates the target repo** (`_run_and_merge_batch`, line 1028): `run_step_with_worker` invokes Codex/Hermes, which edits actual project files. These writes are irreversible — no git stash or rollback happens.
2. **Harness captures before/after git snapshots** (lines 1003, 756–788 in `quality.py`): `_capture_git_status_snapshot` hashes every tracked file *before* the worker runs. After the worker, `_observe_git_changes` (lines 657–742) diffs before vs. after, identifies phantom claims and unclaimed changes, and records them as advisory deviations.
3. **Quality gates and auto-attribution run against the mutated repo** (lines 1098–1154): `_collect_quality_deviations`, `_auto_attribute_unclaimed_paths` — all before any plan-dir canonical artifact is promoted.
4. **Batch artifact promoted atomically** (line 1198): `atomic_write_json(batch_artifact_path(plan_dir, batch_number), payload)` — only NOW is a canonical artifact written.
5. **Aggregate `execution.json` promoted only on final batch** (line 1570): `write_plan_artifact_json(plan_dir, "execution.json", ...)`.
6. **State, history, receipts appended** (lines 1597, 1609–1671): `save_state_merge_meta`, `append_history`, `write_receipt` — all AFTER artifacts.

The structured_output handler (`structured_output.py`) already enforces the same boundary for JSON phases: only `promote_scratch()` may write canonical artifacts, and direct model writes to canonical paths are silently ignored (line 153). The design doc's `PromotionResult.external_side_effects` field (line 84) exists precisely to carry what execute already records today — git deltas, quality deviations, attribution records, scope drift, timeout evidence — without pretending they are transactional.

---

## 2. What functionality would be lost if this scope were wrong

If someone tried to **extend BoundaryTurn to guarantee rollback of target-repository mutations**, three categories of functionality would break:

**A. Resumable partial execution would be impossible.** Execute already writes `resume_cursor` (lines 1590–1595, 81–87) so a blocked/failed run can resume from the last completed batch. If the harness tried to `git stash` or `git reset --hard` after a quality-gate deviation, it would destroy the partial work needed for diagnosis and resumption. The blocked-task reset mechanism (`_reset_blocked_tasks_to_pending`, lines 1781–1811) deliberately *preserves* the repo state and only clears per-attempt metadata in `finalize.json`.

**B. Observation and evidence collection would become impossible.** The entire quality-observation pipeline — `_observe_git_changes`, `_collect_quality_deviations`, `_auto_attribute_unclaimed_paths` — depends on comparing pre-worker and post-worker git states. If the harness rolled back the repo before observation, it would observe nothing. Phantom-claim detection, unclaimed-change detection, line-count regression checks, and scope-drift gates would all silently no-op.

**C. Receipt integrity and audit trail would collapse.** Execute receipts (`build_receipt` at line 1655, `write_receipt` at line 1668) record drift, batch payloads, artifact hashes, and `execution_audit.json`. These receipts are *evidence of what happened*, not guarantees of correctness. If BoundaryTurn promised atomicity it couldn't deliver, the receipts would carry false claims about rollback state, making audit/debugging unreliable.

## 3. Specific design-doc edits and acceptance tests

### Design-doc edits

**a) Add a "Why Not Rollback" subsection** after line 121 (end of "Lifecycle" section):

> **Why this boundary is not transactional for the target repository:** The worker mutates the project directory during its run via arbitrary shell commands, file writes, and git operations. The harness cannot reverse these mutations without destroying evidence needed for resumption, quality observation, and audit. Instead, execute records before/after git snapshots, attributes changes to tasks, runs quality gates against the mutated tree, and writes all evidence into `execution_batch_N.json` and `execution_audit.json` *before* promoting the aggregate `execution.json`. The promotion boundary protects plan-dir canonical artifacts and state transitions; the target repository is observed, not controlled.

**b) Clarify `PromotionResult.external_side_effects` semantics** (after line 88): Add a code comment:

```python
# external_side_effects are recorded evidence of mutations that already
# occurred in the target repository. They are NEVER promises of rollback.
# They exist so downstream phases (review, finalize, gate) and audits
# can inspect what the worker did without re-executing.
```

**c) Update the "Non-Goals" section** (line 270–271): Strengthen:

> - Do not attempt to make target-repository mutations transactional **or reversible**. The harness observes and records side effects; it does not reverse them. Git snapshots, quality deviations, auto-attribution, and scope drift are evidence artifacts, not rollback mechanisms.

### Acceptance tests

**Test 1: Phantom-claim detection survives despite mutated repo**
- Setup: Worker claims `src/a.py` changed but only actually edits `src/b.py`.
- Assert: `_observe_git_changes` produces a phantom-claim deviation. `execution_batch_1.json` contains the deviation. The repo retains both `src/a.py` (unmodified) and `src/b.py` (modified). No rollback occurs.

**Test 2: Blocked-task resume preserves repo state**
- Setup: Batch 1 of 3 completes with a quality-gate deviation. `result=blocked`.
- Assert: `resume_cursor` is written. `state=STATE_BLOCKED`. On resume with `--batch 2`, `_reset_blocked_tasks_to_pending` clears per-attempt metadata but does NOT touch the repo. The working tree is unchanged.

**Test 3: Canonical artifact promotion is atomic within plan-dir**
- Setup: Worker succeeds but `atomic_write_json` for `execution_batch_N.json` is made to fail mid-write (inject fault).
- Assert: No partial `execution_batch_N.json` exists. State is NOT advanced. `_record_execute_blocked` or `record_step_failure` fires. The run is resumable from the prior batch.

**Test 4: Direct canonical write is ignored**
- Setup: Worker writes to `plan_dir/finalize.json` instead of the scratch file.
- Assert: `promote_scratch` (or BoundaryTurn equivalent) ignores the canonical-path write. Only the scratch file or inline payload is promoted. A boundary-violation diagnostic is recorded.

**Test 5: External side effects recorded before aggregate promotion**
- Setup: Full execute run with git changes, quality deviations, and auto-attribution.
- Assert: `execution_batch_N.json` contains `deviations`, `files_changed`, and auto-attribution records. `execution_audit.json` exists. `execution.json` aggregate is written LAST. Receipt references all batch payloads and drift.
