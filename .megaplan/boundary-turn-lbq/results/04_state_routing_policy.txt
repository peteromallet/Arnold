## 1. Verdict

**BoundaryTurn can return a validated `WorkflowTransition` proposal, but it must not _own_ next-route selection.** Routing is status-driven: the auto-driver (`auto.py`) re-derives the next step from `state.current_state` via `workflow_next()`, _ignoring_ the handler's returned `next_step` (see gate.py L459–464). The design doc's `PromotionResult.workflow_transition` is the right seam — a proposal that handlers, the transition policy, and the auto-driver can accept or override — but flattening routing into a generic `BoundaryTurn` field would erase the phase-specific policy that makes gate, review, and execute routing correct and safe.

---

## 2. Routing Behavior at Risk

Five concrete risks arise if routing is collapsed into a generic `BoundaryTurn` return:

**A. Status-driven re-derivation (gate.py).** `_apply_gate_outcome` does not just return a `next_step` — it _sets `state.current_state`_ to `STATE_GATED`, `STATE_CRITIQUED`, `STATE_BLOCKED`, or a tiebreaker state. The auto-driver then calls `workflow_next(state)` which resolves the next transition from the `WORKFLOW` dict and robustness overrides. If `BoundaryTurn` were to return, say, `"finalize"` as a plain string without simultaneously mutating state to `STATE_GATED`, the auto-driver would re-derive `"gate"` and loop forever. The doc's `WorkflowTransition` type can carry both the target step _and_ the state mutation, but the two must stay coupled.

**B. Blocking classes and severity-gated termination.** `_critique_terminate_branch` (gate.py L454–490) distinguishes `STATE_BLOCKED` (correctness/security flags → human required) from `STATE_GATED` (cosmetic-only → force-proceed to finalize). A generic `BoundaryTurn` routing string like `"blocked"` or `"proceed"` loses this distinction. The auto-driver's `on_escalate` / `on_failure` branching and `AUTOMATION_TERMINAL_STATES` handling depend on the exact state value, not a string summary.

**C. Phase result and recovery policy.** `auto.py` reads `PhaseResult` (line 2555–2563) to classify exit kinds (`timeout`, `internal_error`, `context_exhausted`, `blocked`) and applies `RecoveryPolicy`, tier escalation, and retry caps. A `BoundaryTurn` that emits a `WorkflowTransition` without the corresponding `PhaseResult` would break the auto-driver's retry/blocked/external-error classification (lines 394–433), making it unable to distinguish a retryable stream stall from a permanent config error.

**D. Transition-policy evidence gating.** `TransitionPolicy.evaluate_review_done()` (transition_policy.py L89–140) inspects `review_completion_status`, `review_verdict`, `repository_inspected`, `blocking_contradictions`, and `evidence freshness`. It can _deny_ a review→done transition even when the handler returns `"success"`. This is a separate policy surface from the handler's own routing logic. A `BoundaryTurn` that owns routing would either bypass this check or need to embed it, coupling the boundary abstraction to evidence-contract internals.

**E. Review rework routing.** `_resolve_review_outcome` (review.py L736–805) returns `("needs_rework", STATE_FINALIZED, "execute")` — routing back to execute, not done. The rework-cap enforcement (`max_review_rework_cycles`), maker-stop for creative mode, and blocker-vs-cosmetic classification are review-specific; collapsing them into a `BoundaryTurn` route string like `"rework"` would lose the cap counter, the state mutation to `STATE_FINALIZED`, and the blocking item list that the auto-driver needs.

## 3. Specific Design-Doc Edits and Acceptance Tests

### Edits to `megaplan-boundary-turn-design.md`

**Add to § "Proposed Abstraction" after line 89:**

> `workflow_transition` is a **proposal**, not a command. Handlers set it; the transition policy may deny it; the auto-driver re-derives routing from state. `BoundaryTurn` must not bypass `workflow_next()` or the evidence-contract gating in `TransitionPolicy`.

**Add to § "Non-Goals":**

> - Do not move `_apply_gate_outcome`, `_resolve_review_outcome`, or `TransitionPolicy.evaluate_review_done` into `BoundaryTurn`. These remain handler/policy-layer concerns.

**Add to § "Risks":**

> - If `PromotionResult.workflow_transition` is treated as authoritative rather than advisory, the auto-driver's status-driven re-derivation loop will diverge from the handler's intent, causing stall-detection false positives or infinite loops on gate/review boundaries.

### Acceptance Tests

1. **Gate routing survives BoundaryTurn wrapping.** Wrap `_apply_gate_outcome` behind a `BoundaryTurn` facade; assert that an ITERATE recommendation with `prior_rounds >= max_iter` and an open correctness flag sets `state.current_state = STATE_BLOCKED` and `workflow_next(state)` returns `[]` (no revise edge), not `["revise"]`.

2. **Review→rework routing preserved.** After `BoundaryTurn` promotion of a `needs_rework` review, assert `next_state == STATE_FINALIZED`, `next_step == "execute"`, and that `TransitionPolicy.evaluate_review_done()` is still evaluated (and passes for a non-done transition).

3. **Auto-driver ignores BoundaryTurn's next_step.** Mock a `BoundaryTurn` that returns `WorkflowTransition(step="done", state="done")` from a gate phase while `state.current_state` is `STATE_CRITIQUED`. Assert the auto-driver resolves `next_step` via `workflow_next(state)`, not from the mock.

4. **Evidence-contract denial overrides BoundaryTurn.** A review `BoundaryTurn` returns `workflow_transition` with `to_state=STATE_DONE`, but `TransitionPolicy` finds `repository_inspected: false`. Assert the auto-driver blocks the transition and the plan remains in review.

5. **Phase-result classification survives facade.** A gate `BoundaryTurn` promotion that fails with `ExitKind.internal_error` must still produce a `PhaseResult` that the auto-driver classifies as non-retryable (not a stream stall), so it does not incorrectly retry a permanent error.
