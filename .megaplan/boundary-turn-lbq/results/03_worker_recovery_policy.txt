## 1. Verdict

**The provisional answer is correct and necessary, but incomplete.** The three-policy split (`strict_file_fill`, `legacy_recovery`, `inline_only`) is the right taxonomy. Starting every existing phase with `legacy_recovery` and tightening per-phase only after provider-path tests is the only safe migration strategy. However, the design doc and current code also reveal two concrete gaps that will cause regressions *even under `legacy_recovery`* unless addressed:

1. **BoundaryTurn's proposed `capture()` must not consume the raw worker output stream before `model_seam._recover_payload_with_provenance` has had its turn.** Today, the hermes worker parses raw text, extracts JSON from tool-markup (DeepSeek/Kimi `<write_file>` recovery), reconstructs execute/gate payloads, *then* routes through `model_seam.capture_step_output`, which itself does file-based recovery. If `BoundaryTurn.capture()` normalizes or classifies the payload *before* that recovery cascade runs, it will break every flaky-provider path that depends on late-stage file recovery.

2. **The `FallbackPolicy.inline_when_unmodified` default in `BoundarySpec` (line 63 of the design doc) is ambiguous and dangerous.** Today's `promote_scratch` distinguishes between *missing* (no file) and *unmodified* (file exists, byte-identical to seed). An unmodified scratch file means the model was *instructed* to fill it but *didn't* — which is a different diagnostic signal than "no file tool available." Collapsing both to inline fallback under a single policy name loses that signal and silently masks non-compliance.

**Verdict:** Proceed with the three-policy split, but add two invariants before the first migration step.

## 2. Recovery Behavior That Must Remain

These recovery paths are load-bearing for flaky providers and must survive any `BoundaryTurn` wrapping:

### A. Hermes-side recovery (must fire *before* BoundaryTurn capture)

- **Tool-markup JSON extraction** (`_extract_json_from_mutating_tool_markup`, lines ~370–560 of `hermes.py`): DeepSeek and Kimi models, when told "fill the JSON template file," often emit `<write_file path=...>...</write_file>` or `<invoke name="write_file">` XML blocks containing the JSON payload inside `<parameter name="content">` tags. This recovery parses those blocks and substitutes the extracted JSON as the model response. A `strict_file_fill` BoundaryTurn that rejects anything not at the expected draft path would kill these models outright.

- **Empty-response retry** (`_run_attempt`, lines ~2173–2360): Up to 3 retries with exponential backoff (2s, 4s, 8s) on `worker_parse_error` with "0 chars." This catches context overflows and API hiccups.

- **Execute/gate payload reconstruction** (`_reconstruct_execute_payload`, `_reconstruct_gate_payload`, lines ~2397–2426): When `model_seam.capture_step_output` rejects the parsed payload, execute and gate have phase-specific reconstructors that salvage partial output from agent conversation messages. BoundaryTurn's `validate()` must preserve this second-chance path.

- **Retry template rewriting** (`_rewrite_output_template`, lines ~2137–2159): On retry (e.g., MiniMax→OpenRouter fallback), the scratch template is rewritten fresh via the registry builder. BoundaryTurn must not treat this as a new turn — it's the same attempt with a refreshed seed.

### B. Model-seam recovery (must remain callable from BoundaryTurn.validate)

- **`_recover_payload_with_provenance`** (lines ~1055–1175 of `model_seam.py`): When raw JSON parsing fails, this walks: (a) the output file path, (b) a fallback template path (`*_output.json`), (c) JSON candidate extraction from raw text. It validates each candidate against the step schema and uses scoring functions for critique completeness and plan structure. This is the *last line of defense* for providers that produce malformed inline JSON but valid file JSON.

- **Phase-specific normalizers** (lines ~368–401): Plan, review, execute, gate, critique, critique_evaluator, prep-distill, and finalize each have normalizers that coerce, strip, and alias fields. BoundaryTurn must route through these before schema validation.

- **Compatibility projection** (`_compatibility_projection`, line ~1276): Legacy-mode steps still go through compatibility projection. BoundaryTurn must honor this for any step not yet migrated to `CompatibilityMode.NATIVE`.

### C. Handler-side promotion (the current `promote_scratch` semantics)

- **Missing/unmodified scratch → inline fallback** (`promote_scratch`, lines ~198–204): This is the correct behavior for `legacy_recovery`. The worker's `.payload` (inline JSON) is accepted when the scratch file is absent or untouched.

- **Modified-invalid + file_fill_instructed → hard failure** (lines ~207–218): This is SD3 and must be preserved. When the worker was explicitly told to fill the file and produces garbage, falling back silently would mask model non-compliance.

- **Unknown-key stripping** (`_strip_unknown_keys`, lines ~66–75, 228–238): Unknown top-level keys are silently dropped before promotion. This is *not* a recovery path — it's a schema hygiene feature — but must not be lost.

- **Expected-path-only reads** (lines ~41–51, 111): The handler only reads the expected scratch path. Wrong-path writes (e.g., model writing directly to `finalize.json` instead of `finalize_output.json`) are silently ignored. This is the boundary's core invariant.

## 3. Design-Doc Edits and Acceptance Tests

### Required design-doc edits

**Edit 1 — Add a `capture()` sequencing invariant to §Lifecycle:**

After step 5 ("The turn captures only expected draft paths and records raw worker output"), insert:

> 5a. For `legacy_recovery` turns, raw worker output (both text and file paths) MUST be passed through the existing hermes/model-seam recovery cascade (`_extract_json_from_mutating_tool_markup` → `_recover_payload_with_provenance` → phase normalizers) *before* capture classification. Skipping this cascade to read only the expected draft file will regress DeepSeek/Kimi tool-markup recovery and file-based rescue of malformed inline JSON.

**Edit 2 — Replace the `FallbackPolicy.inline_when_unmodified` default:**

Change `BoundarySpec` line 63 from:
```python
fallback_policy: FallbackPolicy = FallbackPolicy.inline_when_unmodified
```
to:
```python
fallback_policy: FallbackPolicy = FallbackPolicy.legacy_recovery
```

And add to the `FallbackPolicy` enum definition:

> `legacy_recovery`: Mirrors current `promote_scratch` semantics exactly. Missing/unmodified scratch → inline worker.payload. Modified-invalid + instructed → hard fail. Modified-invalid + not instructed → inline fallback. Worker-side recovery (tool-markup extraction, payload reconstruction) runs before classification.
>
> `inline_when_missing`: Only fall back to inline payload when the draft file is genuinely absent (no template written). An unmodified draft is a diagnostic event, not a silent fallback. For phases where file tools are unavailable.
>
> `strict_file_fill`: Only the expected draft path counts. No inline fallback. No wrong-path recovery. No tool-markup extraction. Modified-invalid is always hard failure. For phases proven stable across all providers.

**Edit 3 — Add to §Non-Goals:**

> - Do not run BoundaryTurn capture/classify before the hermes worker's `parse_agent_output` and model-seam recovery cascade have completed. BoundaryTurn wraps promotion, not raw-output parsing.

### Acceptance tests to add before migrating any phase

1. **`test_boundary_turn_deepseek_write_file_markup_recovery`**: Hermes worker with a DeepSeek model that responds to file-fill instructions by emitting `<write_file path="finalize_output.json">...valid JSON...</write_file>`. Under `legacy_recovery`, `BoundaryTurn.capture()` must produce `status="filled"` with the extracted JSON. Under `strict_file_fill`, this must fail (or at minimum log a diagnostic).

2. **`test_boundary_turn_malformed_inline_but_valid_file_recovery`**: Worker produces malformed inline JSON (missing closing brace) but wrote valid JSON to the scratch file. Under `legacy_recovery`, `_recover_payload_with_provenance` must rescue the file payload and produce `status="filled"`. Under `strict_file_fill`, only the scratch file is read anyway, so this must still pass.

3. **`test_boundary_turn_unmodified_scratch_diagnostic`**: Worker is instructed to fill the scratch file but returns text-only output without touching the file. Under `legacy_recovery`, `capture()` produces `status="unmodified"` and falls back to `worker.payload`. Under `inline_when_missing`, this must produce `status="unmodified"` and *not* fall back — the handler must see the diagnostic and decide. Under `strict_file_fill`, this is a hard failure.

4. **`test_boundary_turn_modified_invalid_hard_fail`**: Worker with `file_fill_instructed=True` modifies the scratch file but writes non-JSON content. All three policies must raise `ValueError` — this is the invariant from SD3.

5. **`test_boundary_turn_empty_response_retry_passthrough`**: Worker hits a context overflow (empty response), retries 3 times internally, then succeeds on the 4th attempt. BoundaryTurn must see only the final successful capture, not error on the transient empty responses. The retry loop lives in the worker, not in BoundaryTurn.

6. **`test_boundary_turn_canonical_write_ignored`**: Model writes directly to `finalize.json` (canonical path) instead of `finalize_output.json` (scratch path). BoundaryTurn must ignore the canonical-path write — `classify_scratch` only reads the expected scratch path. This must produce `status="missing"` (if no scratch file) or `status="unmodified"` (if seed exists untouched), never `status="filled"` from the wrong path.

7. **`test_boundary_turn_legacy_recovery_preserves_all_provider_paths`**: A parametrized smoke test that runs each current JSON phase (critique, critique_evaluator, gate, finalize, review) through a `BoundaryTurn` facade with `legacy_recovery` and asserts the promoted payload is byte-identical to what the current `promote_scratch` path produces, for a representative set of real provider outputs captured from production (including DeepSeek tool-markup, Kimi malformed JSON, and empty-template edge cases).

---

The core principle is: **BoundaryTurn must be a promotion-time abstraction, not a parsing-time one.** It wraps how validated output becomes canonical artifacts — it must not intercept or short-circuit the worker/model-seam recovery cascade that turns flaky provider output into valid payloads. The three-policy split is the right mechanism; the missing piece is the explicit sequencing guarantee that `capture()` does not run *before* recovery.
