## 1. Verdict

The BoundaryTurn design is **directionally sound but insufficiently specified** to guarantee finalize preservation. The preservation checklist (design doc lines 360–368) and load-bearing question Q6 both enumerate the complete artifact footprint, so awareness exists. However, the design's `BoundaryTurn` abstraction, `canonical_targets`, and `PromotionResult` types have not been mapped to finalize's actual 8-artifact output and 2-path (success vs. revise-feedback) behavior. Three concrete gaps must be closed before finalize can be migrated — none are fatal, but all require explicit design-doc amendments.

---

## 2. Finalize Behavior at Risk

**A. Multi-artifact canonical targets.** The stage coverage table (line 200–201) only names `finalize.json` and `final.md`. The handler actually writes 6 artifacts on the success path: `finalize.json`, `final.md`, `contract.json`, `user_actions.md`, `finalize_snapshot.json`, and `baseline.json`. The `canonical_targets: tuple[str, ...]` field can carry all six, but the design never states the concrete mapping. Without it, an implementer could drop `finalize_snapshot.json` (relied on by downstream audit/resume code in `plan_repository.py` and test fixtures) or `baseline.json` (the per-plan cache that avoids re-running the full suite on retry).

**B. Heavyweight harness side-effect during promotion.** `_capture_test_baseline_for_plan` runs the full pytest suite — a 300-second idle-timeout operation with host-wide concurrency gating. The design states "Execute and other mutating phases must register external side effects separately" (lines 104–106), but baseline capture isn't an execute mutation — it's a read-only harness operation that happens *during* `_write_finalize_artifacts` before any canonical write. If `promote()` absorbs this without explicit side-effect registration, the BoundaryTurn lifecycle violates its own principle that promotion is a "fail-closed sequence" (line 119) of artifact writes, not a minutes-long subprocess.

**C. Finalize-to-revise feedback path.** When baseline test selection fails, `_route_finalize_baseline_selection_failure_to_revise` writes `gate.json`, `gate_carry.json`, and `finalize_revise_feedback.json`; mutates state to `STATE_CRITIQUED`; sets `last_gate` and `meta.finalize_revise_feedback`; calls `record_step_failure`; and returns a `StepResponse` routing to `revise`. The design's `PromotionResult.workflow_transition: WorkflowTransition | None` is the right hook, but `WorkflowTransition` is never defined. More importantly, this path is not a "promotion" at all — it's a *rejection that rewinds state and writes gate artifacts*. The BoundaryTurn lifecycle (lines 107–121) has no step for "promotion failed, route to alternate phase."

---

## 3. Design-Doc Edits and Acceptance Tests

**Edits to `megaplan-boundary-turn-design.md`:**

1. **Line 200–201 (stage coverage table):** Replace the finalize row with:
   > `finalize` — JSON scratch → `finalize.json`, `final.md`, `contract.json`, `user_actions.md`, `finalize_snapshot.json`. Harness-computed `validation`, baseline capture, and capability claims are added during promotion. `baseline.json` is a harness-owned cache artifact.

2. **After line 121 (lifecycle), insert:**
   > 0. The turn may execute harness-side effects (e.g., baseline test capture) before promotion begins. These are recorded as `ExternalEffectRef` entries on the promotion result and are not assumed to roll back.

3. **After line 88 (PromotionResult), insert:**
   > When promotion is blocked by a harness-level failure (e.g., unresolvable baseline test scope), the turn returns a `RouteToPhase` result instead of a `PromotionResult`. The handler writes gate/feedback artifacts, rewinds state, and the transition policy selects the next phase.

**Acceptance tests:**

- `test_finalize_boundary_promotes_all_six_canonical_artifacts` — after a successful turn, assert `finalize.json`, `final.md`, `contract.json`, `user_actions.md`, `finalize_snapshot.json`, and `baseline.json` exist and are valid JSON.
- `test_finalize_baseline_capture_registered_as_external_effect` — assert `PromotionResult.external_side_effects` includes an entry with `kind="baseline_capture"`, the pytest command, timing, and failure list.
- `test_finalize_baseline_failure_routes_to_revise_with_gate_artifacts` — assert `gate.json`, `gate_carry.json`, and `finalize_revise_feedback.json` are written; state is `STATE_CRITIQUED`; the `RouteToPhase` result points to `"revise"`.
- `test_finalize_validation_block_never_in_scratch_template` — assert the scratch template (`finalize_output.json`) contains no `validation` key, and the `validation` block in promoted `finalize.json` matches harness-computed programmatic coverage.
