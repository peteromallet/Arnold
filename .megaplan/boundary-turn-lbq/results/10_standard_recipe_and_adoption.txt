## Question 10: Does BoundaryTurn Provide a Standard Recipe?

### 1. Verdict

**No.** BoundaryTurn as currently designed is a Megaplan-internal abstraction, not a reusable recipe for the broader Arnold pipeline ecosystem. The design doc describes it entirely in terms of Megaplan's 17 phase identities, canonical artifact names, template registry modes, and handler-specific behaviors. A new pipeline author using the `Pipeline`/`Step`/`Route` explicit-node surface (e.g., `hello/`, `jokes/`, `creative/`) would find nothing actionable in this specification — no import path, no protocol to implement, no integration point with `WorkflowManifest` nodes, journal events, or the `ExecutionBackend` protocol.

The strongest evidence is the absence of any connection to `arnold.workflow` constructs. The design doc mentions `StepContract`, `TemplateRegistration`, `plan_dir`, and `Megaplan state` — all Megaplan-specific. Meanwhile `workflow-authoring.md`, `creating-a-new-pipeline.md`, and `workflow-runtime.md` operate in a completely separate vocabulary (`Pipeline`, `Step`, `Route`, `NodeState`, `event journal`, `ExecutionBackend`). BoundaryTurn bridges none of these.

### 2. Where the Recipe Is Too Megaplan-Specific

- **`draft_kind` enum** (`json`, `markdown`, `batch`, `subpipeline`) directly mirrors the five `RegistryMode` values in `template_registry.py`, which encode Megaplan handler dispatch semantics. A non-Megaplan pipeline has no file-fill vs. batch-assembly distinction.
- **Canonical artifact names** (`plan_vN.md`, `critique_vN.json`, `gate.json`, `finalize.json`, `execution_batch_N.json`, etc.) are hardcoded Megaplan paths. The spec never shows how to configure canonical targets without knowing these names.
- **The stage coverage table** lists 17 Megaplan phases (`prep`, `plan`, `revise`, `critique`, `gate`, `finalize`, `execute`, `review`, `feedback`, `tiebreaker`, etc.). There is no mapping from generic `Step(id=..., kind=...)` nodes to a BoundaryTurn.
- **The preservation checklists** reference Megaplan-specific files (`prep_triage.json`, `gate_signals_vN.json`, `gate_carry.json`, `finalize_revise_feedback.json`, `execution_audit.json`, `tiebreaker_decisions.json`) and Megaplan-specific policies (`STATE_BLOCKED` vs. `STATE_GATED`, approval gates, timeout recovery, rework caps).
- **The API** assumes `plan_dir` and `state` as Megaplan-specific shapes with no generic contract. `turn.render_prompt(base_prompt)` has no equivalent in the neutral runtime's node-payload execution.
- **`StepContract` and `TemplateRegistration`** are the only integration surface described. Both are Megaplan-only registries keyed by Megaplan phase identities, not discoverable from a `WorkflowManifest` or an authored `Step`.

### 3. Where the Recipe Is Too Vague

- **`render_prompt`** is underspecified. A general recipe must define what a base prompt is, how draft-path injection works (the doc says "names the exact draft paths" but never shows how), and whether the prompt format is a string, a chat-message list, or a provider-specific envelope.
- **`run_worker(prompt)`** is outside the API entirely but appears in the lifecycle pseudocode. A reusable recipe must define the worker protocol — is it a provider call, a subprocess, a Hermes invocation, or a `StepInvocation`-style dispatch?
- **`BoundaryValidator` and `BoundaryPromoter`** are declared as types (`Callable[..., BoundaryDraft]`, `BoundaryValidator`, `BoundaryPromoter`) but never defined as protocols with clear signatures, error contracts, or idempotency guarantees.
- **Promotion result fields** (`artifact_delta`, `state_delta`, `external_side_effects`, `observability_events`, `receipts`, `workflow_transition`) are declared as frozen dataclasses but never mapped to the neutral runtime's `artifact_written` events, `NodeState` transitions, or the journal-backed resume system.
- **Ordered promotion checkpoints** are described as "fail-closed sequence with resumable checkpoints" but never defined as discrete journal events. The runtime already has `node_suspended`, `node_resumed`, and `node_failed` — BoundaryTurn reinvents this vocabulary without reconciling it.
- **No integration contract** with `ExecutionBackend._execute_node_payload`, `_execute_effect`, or `_select_branch` — so a non-Megaplan backend author has no way to wire BoundaryTurn into the runtime loop.

### 4. Specific Design-Doc Edits and Acceptance Tests

**Edits to `megaplan-boundary-turn-design.md`:**

1. Add a section **"Generalization Boundary"** before the "Non-Goals" section that explicitly answers: *What must a non-Megaplan pipeline author do to adopt BoundaryTurn?* List the required imports, the protocol they must implement (e.g., define `BoundaryValidator` and `BoundaryPromoter` as `Protocol` classes with typed signatures), and the runtime integration point (e.g., "implement `_execute_node_payload` to call `BoundaryTurn.open`...").

2. Replace the Megaplan-specific `draft_kind` enum with a protocol-driven approach. Instead of `Literal["json", "markdown", "batch", "subpipeline"]`, define a `DraftKind` protocol with methods like `classify(path, seed) -> CaptureStatus` and `promote(capture) -> PromotionResult`. Megaplan's four kinds become implementations of that protocol.

3. Remove the hardcoded canonical artifact name list from the "Drafts Versus Canonical Artifacts" section. Replace it with a rule: *Canonical targets are declared in `BoundarySpec.canonical_targets` and must be disjoint from draft paths. No canonical path may appear in any draft-file template.*

4. Map the promotion-result dataclasses to the neutral runtime: `PromotionResult.artifact_delta` should produce `ArtifactSpec` outcomes; `state_delta` should map to `NodeState` transitions; `observability_events` should map to journal `artifact_written`/`control_transition` events; `workflow_transition` should map to `ManifestCursor` advancement.

5. Add a concrete non-Megaplan example — e.g., show how the `jokes/` pipeline (`draft -> tighten -> emit`) would use BoundaryTurn for the `draft` step with a Markdown scratch file, a heading validator, and a promotion that writes `joke_v1.md` as the canonical artifact.

**Acceptance tests the design doc should mandate:**

1. **Non-Megaplan adoptability**: Write a test that creates a `BoundaryTurn` for a phase not in Megaplan's 17-identity registry, using only the `arnold.workflow` explicit-node types and a protocol-conformant `BoundarySpec`. The test must assert that no `step_contracts.py` or `template_registry.py` import is required.

2. **Wrong-path writes**: The model writes output to the canonical artifact path instead of the draft path. Assert that `capture()` returns `diagnostics` recording the violation and that `promote()` does not read or promote the wrong-path file.

3. **Unmodified draft**: The worker runs but never touches the scratch file. Assert that `capture()` returns `status="unmodified"` and that `promote()` uses the configured `fallback_policy`.

4. **Invalid draft with strict policy**: Under `strict_file_fill`, a modified-but-invalid draft produces a hard failure. Assert that `promote()` raises before any canonical artifact is written.

5. **Resume after promotion failure**: Kill the process after the first canonical artifact write but before receipt emission. Resume and assert that `promote()` is idempotent (re-writes identical artifacts) and completes the remaining promotion steps without double-emitting receipts.

6. **Generic markdown validation**: Define a `BoundarySpec` with `draft_kind` that validates required headings, forbidden placeholders, and section ordering — without referencing any Megaplan canonical artifact name. Assert that the validator rejects a draft with a missing required heading.
