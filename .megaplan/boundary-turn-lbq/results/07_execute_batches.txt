## 2. Execute Behavior at Risk

Three behaviors are at risk if the migration is done incorrectly:

**a) Project mutations before artifact promotion.** `_run_and_merge_batch` edits the target repository (the worker runs commands, changes files) *before* `execution_batch_N.json` is atomically written (line 1198) and *before* `execution.json` is written (line 1570). The design doc is explicit: "BoundaryTurn can make those effects explicit, evidenced, and resumable, but it cannot make them atomic" (line 18–20). A naïve BoundaryTurn migration that assumes idempotent/reversible promotion would break this fundamental property. External side effects (git snapshots via `_capture_git_status_snapshot`, auto-attribution via `_auto_attribute_unclaimed_paths`, quality deviations via `_collect_quality_deviations`) must be recorded as `ExternalEffectRef` payloads in the `PromotionResult`, not gated behind successful promotion.

**b) Per-batch tier routing and session state.** The per-batch tier routing in `handle_execute_one_batch` (lines 1374–1424) mutates the active-step state mid-execution (`set_active_step` at line 1421, `save_state_merge_meta` at line 1424) so the liveness heartbeat matches the tier-selected model. If child `BoundaryTurn`s are implemented as isolated sub-transactions without awareness of the outer execute session, the per-batch model selection and session-key refresh would silently break, causing stale sessions or heartbeat failures.

**c) Reducer aggregation depends on per-batch `finalize.json` mutations.** Each batch's `_run_and_merge_batch` mutates `finalize.json` in-place (line 1200: `write_plan_artifact_json(plan_dir, "finalize.json", finalize_data, ...)`). The reducer `_build_aggregate_execution_payload` reads *all* batch artifacts and produces `execution.json` — but the per-batch merge results are already baked into `finalize.json`. A strict `BoundaryTurn` that forbids handler code from touching canonical artifacts between child turns would break this orchestration. The reducer turn must be allowed to read the side-effected `finalize.json` state, and the design doc accommodates this by treating `finalize.json` updates as a harness-computed side effect of child turns, not as model output.
