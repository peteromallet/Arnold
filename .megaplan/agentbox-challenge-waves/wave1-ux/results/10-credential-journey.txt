# AgentBox Credential UX Review

## Verdict: **FITS WITH CHANGES**

The credential architecture (SecretSpec, push/list/test/rotate, explicit spec over env-dump) is sound. The user journey for missing-credential recovery and credential confidence has real gaps, but they're surgically fixable — no redesign needed.

---

## Material Gaps

### 1. Missing-credential recovery breaks the Discord-first promise

**The gap:** Guardian's defined behavior is *"notify Peter with the exact missing credential and the `agentbox creds push/test` command."* Peter now leaves Discord, opens a laptop terminal, runs a CLI command, then returns to Discord to say "done" or retry. There's no Discord-guided recovery and no acknowledgment loop.

**Why it matters:** The product is Discord-first. If the #1 credential failure path forces Peter to context-switch to CLI, the experience feels fractured. Worse, there's no confirmation that Peter's push worked — Guardian will rediscover the same problem on its next tick, and Peter gets pinged again.

### 2. No proactive credential validation before launching work

**The gap:** The Operator launches a `megaplan_chain` without checking whether the credentials it will need (Codex OAuth, GitHub SSH, Claude refresh token) are present and valid. The operation spins up a worktree, starts a tmux session, runs for some time, then fails on a credential error. Guardian eventually detects it, notifies Peter, and the cycle restarts.

**Why it matters:** This is the difference between *"I can't start this — your Codex token is expired. Push a fresh one with this command…"* (seconds lost) and *"Your operation op-123 failed after 12 minutes because of a missing credential"* (time, context, and trust lost). The plan describes credential testing as a separate command but never connects it to the launch path.

### 3. Credential freshness is not monitored by Guardian

**The gap:** The Guardian's operational loop checks operation liveness, tmux sessions, logs, and chain state. It does not include a credential-health check. OAuth tokens expire silently. API keys get rotated on provider dashboards. The plan mentions `agentbox creds rotate` but gives no mechanism for Guardian to detect an upcoming or silent expiry.

**Why it matters:** A Codex OAuth token expiring at 2am means every operation queued for the morning wakes up to a credential failure Peter discovers reactively. The plan's human-experience goal is that *"failure should feel like: what failed, what it tried, what it needs"* — but silent expiry produces the opposite: the system didn't know, didn't warn, and only noticed when work broke.

### 4. Discord Operator has `creds_test` but no `creds_list` or guided push

**The gap:** The Discord Operator tool inventory includes `creds_test` but omits `creds_list` (show what's configured and what's expected) and any guided-push interaction. Peter can't ask *"what credentials does AgentBox have?"* or *"what credentials do I need to push?"* from Discord.

**Why it matters:** When Guardian reports a missing credential, Peter's natural follow-up from Discord is *"what credentials are missing?"* or *"show me what's configured."* Without `creds_list`, the Operator can't answer, and Peter must switch to CLI. The credential push itself must happen from the laptop, but the *understanding* step should stay in Discord.

### 5. The completion flow doesn't validate credentials before offering push/PR

**The gap:** The completion DM says *"Next: push/open PR/cleanup?"* but doesn't verify that GitHub credentials are working before presenting that option. Peter says "push," and only then does the system discover the SSH key or token is stale.

**Why it matters:** Completion is the handoff moment the plan calls out as deliberately sparse — *"summary, validation, branch/PR status, and suggested next action."* If the suggested next action quietly depends on a credential that doesn't work, the handoff erodes trust at exactly the moment it should feel clean.

## Recommended Plan Changes

### Add to "Guardian Finds A Blocker" section (after the missing-credential bullet):

```markdown
- Missing credential: notify Peter in Discord with:
  - the exact credential name and what operation needs it;
  - whether the credential is missing entirely vs. expired/failing validation;
  - a concrete next step Peter can take from Discord (e.g., "reply 'push github' and I'll guide you");
  - a confirmation path: once Peter pushes, the Operator retests and reports success or the next gap.
```

### Add to "Discord Operator" tool list:

```markdown
- `creds_list` — list configured, expected, and missing credentials with status (ok/expired/missing/untested).
- `creds_push_guide` — return the exact CLI command Peter should run on his laptop, with the credential name.
```

### Add a new subsection under Credential Sync or Guardian:

```markdown
### Proactive Credential Validation

The Operator should validate relevant credentials before launching work:

- On `run megaplan-chain ...`: test Codex, GitHub, and any chain-declared secrets before creating the worktree.
- On `run codex ...`: test the Codex auth bundle.
- On `run claude ...`: test the Claude refresh token or API key.

If any credential fails, reply in Discord with the failure and the push command — do not launch the operation.

Guardian should include a credential-health check in its periodic loop:

- Every N ticks (e.g., every 30 minutes), test all configured credentials.
- If a credential is within its expiry window (e.g., OAuth refresh token approaching silent-expiry), notify Discord proactively.
- If a credential fails validation without an operation needing it, file a low-priority pending-approval notice.
```

### Add to "Completion And Approval" section:

```markdown
Before offering "push/open PR/cleanup," the Operator should validate GitHub credentials. If they fail, include the failure in the completion DM alongside the push command.
```

## What The Plan Already Covers (Do Not Change)

- **SecretSpec abstraction** (local_path, persistent_dest, runtime_dest, permissions, health_check) — this is the right model. The gaps are about *when* it's used, not its shape.
- **`agentbox creds push/list/test/rotate` CLI** — the local-side tooling is well-defined.
- **Explicit credential classes** (GitHub, Codex, Claude, OpenAI, Discord, Supabase, etc.) — the inventory is realistic.
- **"Do not dump the entire local environment"** — the security posture is correct.
- **Guardian's bias toward autonomous repair for operational failures** — the credential *notification* behavior should stay; the gap is in the recovery *experience* around it.
- **Phase 4 as the dedicated credential milestone** — the phasing is reasonable; the proactive-validation behavior can be added to the Operator profile in Phase 6 without disrupting the build order.
- **T9 (credential sync technical approach)** — the generalization from `seed_codex_oauth()` to explicit specs is the right evolution.
