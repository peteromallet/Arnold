# AgentBox Operation Registry — Current-System Fit Review

## 1. VERDICT: **FIT WITH EXTRACTION**

The existing Megaplan store, cloud-run tracking, watchdog registry, and resident conversation mechanics provide **strong substrate** for an AgentBox operation registry, but they are **scattered across four separate systems** with overlapping concerns and no unifying record. The operation registry is not a greenfield build — it is an **extraction-and-unification task** that pulls fields from existing code and removes the duplication. The plan correctly identifies "the operation registry is the missing center." What it understates is how much the **existing systems will fight each other** if not explicitly consolidated.

## 2. EXISTING FUNCTIONALITY THAT SHOULD BE REUSED

### 2a. `store/base.py` — The store abstract interface already has the right shape

The abstract `Store` class (from the inventory) already exposes:

| Method | What it tracks | AgentBox relevance |
|---|---|---|
| `list_cloud_runs()` | Cloud chain runs with status, provider, session | **Direct.** This is the closest existing thing to an operation record. |
| `list_scheduled_jobs()` | Recurring jobs with claim mechanics | **Direct.** Guardian's periodic tick is a scheduled job. |
| `upsert_resident_conversation()` | Discord conversation state | **Direct.** Operation ↔ Discord ref mapping lives here. |
| `list_progress_events()` | Structured progress ticks | **Direct.** Heartbeat and phase-change events. |
| `create_ticket()` / `link_ticket_to_epic()` | Ticket and epic state | Indirect but useful for linking operations to tickets. |
| `events_for_plan()` | Per-plan event stream | Pattern to copy, not reuse directly. |

**Action:** The operation registry should be a new table/collection in this store, not a separate SQLite file or JSON store. The existing `list_cloud_runs` should become a **filtered view** over the unified operation table. Cloud runs are a `kind=megaplan_chain` operation with `provider=ssh|railway`.

### 2b. `watchdog/registry.py` — The WatchdogRegistry already does operation tracking

From the inventory and `scripts/megaplan_live_watchdog.py`:

```python
DEFAULT_REGISTRY_PATH = "~/.megaplan/watchdog/registry.ndjson"
```

The watchdog has `Observation` records and a `WatchdogRegistry` that tracks:
- Plan discovery (scanning `DEFAULT_SCAN_ROOTS`)
- Process liveness
- Repair attempts
- Retry outcomes
- Snapshots

This is a **partial operation registry** already. It tracks plan identity, process state, repair history, and retry caps — all of which the AgentBox plan wants in the operation registry. The watchdog's registry is currently file-backed ndjson, separate from the main store.

**Action:** Fold the watchdog's `Observation` fields into the unified operation record. Replace `~/.megaplan/watchdog/registry.ndjson` with queries against the store. The watchdog becomes a Guardian plugin that writes to the same registry rather than its own file.

### 2c. `cloud/supervise.py` — `cloud_supervise_tick` is Guardian v0 for chains

## 3. MISSING SEAMS AND INCOMPATIBILITIES

### 3a. **CRITICAL: Four separate registries must be consolidated — or they will diverge**

| Registry | Location | Tracks |
|---|---|---|
| Cloud runs | Store (DB or file) | Chain runs on cloud providers |
| Watchdog registry | `~/.megaplan/watchdog/registry.ndjson` | Plans, processes, repair state |
| Scheduled jobs | Store | Recurring job claims |
| Resident conversations | Store | Discord message state |

The plan proposes a **fifth** registry (AgentBox operation registry). If added alongside the existing four, you get:
- Cloud runs in the store AND operations in the store — duplicate identity for chain launches
- Watchdog scanning plans AND Guardian scanning operations — divergent state views
- Scheduled jobs for Guardian ticks AND a separate Guardian loop — two schedulers

**The missing seam is a migration path from four registries to one.** The plan says "add an operation registry" but doesn't specify whether `list_cloud_runs` continues to exist, whether the watchdog registry is deleted or merged, or whether scheduled jobs become a Guardian configuration concern.

### 3b. **MISSING: No `operation_id` field in existing cloud run records**

The inventory shows `list_cloud_runs` exists, but we don't see a consistent `operation_id` field across the system. The cloud CLI uses `session_name` (e.g., `megaplan-chain`) as the tmux identifier. The watchdog uses plan directory paths. The resident conversations use Discord message/channel IDs. There is **no existing foreign key** that ties a cloud run to a watchdog observation to a Discord conversation.

**The extraction seam is:** AgentBox needs to introduce `operation_id` as the primary key, then backfill it into existing cloud run records, and replace path/session-based identity with operation-id-based identity in the watchdog and resident layers.

### 3c. **MISSING: Worktree helpers assume one repo per operation**

`bakeoff/worktree.py` functions like `create_named_worktree` and `remove_worktree` operate on "the current repo." The AgentBox plan envisions multi-repo operations:

```yaml
repos:
  - name: megaplan
    branch: agent/op-123
    worktree: /workspace/worktrees/op-123/megaplan
  - name: reigh-app
    branch: agent/op-123
    worktree: /workspace/worktrees/op-123/reigh-app
```

**The missing seam is a `create_operation_worktrees(repos: list[RepoSpec])` function** that calls the existing single-repo helpers in a loop against a repo registry, not the CWD. This is extraction, not rewrite, but it's a new composition layer that doesn't exist yet.

### 3d. **MISSING: Guardian tick vs. resident scheduler conflict**

Two things exist:

1. `resident/scheduler.py` — schedules resident jobs (the inventory shows it exists)
2. `scripts/megaplan_live_watchdog.py` — standalone watchdog process

The AgentBox plan proposes a systemd-based Guardian daemon. But the resident runtime already has scheduling. **The missing seam is:** Will Guardian be a resident-scheduled job, a standalone systemd service, or both? If both, how do they avoid double-ticking the same operation?

The plan says "separate systemd services sharing the same operation store" — but the resident scheduler is inside the Discord runtime process. A separate Guardian process reading the store is fine for reads, but what about writes? Two processes writing `status`, `last_check`, and `pending_approvals` to the same operation record need a concurrency strategy. The existing store appears to use file-level or transaction-level consistency (from `store/base.py`'s `Transaction` references), but we need to verify it handles concurrent cross-process writes.

### 3e. **MISSING: Credential specs need extraction from provider-specific code**

The plan correctly identifies `cloud/auth.py`'s `seed_codex_oauth()` as the pattern. But credential handling is currently:
- Buried inside the cloud bootstrap flow (OAuth seeding happens during deploy)
- Provider-specific (SSH upload vs. Railway env vars vs. local files)
- Not separately testable

**The missing seam is a `SecretSpec` class and a `credential_registry` table** that decouples credential definitions from the cloud deploy path. The plan proposes this — confirmed as correct and necessary.

### 3f. **RISK: Bare repo worktree support is untested against existing helpers**

The plan recommends bare repos under `/workspace/repos/*.git`. The inventory shows `worktree_registered` and `remove_worktree` in `bakeoff/worktree.py`. The plan itself flags (Finding #5): "Existing worktree helpers assume a normal repo/main worktree in places such as registered-worktree discovery and removal."

**Code-inspection task to add to the plan:** Before committing to bare repos, audit `worktree_registered`, `remove_worktree`, and dirty-state carry helpers for bare-repo compatibility. Specifically:
- Does `worktree_registered` use `git worktree list` (bare-repo-safe) or inspect the main working tree?
- Does `remove_worktree` call `git worktree remove` (safe) or `rm -rf` + `git branch -D` (may miss bare-repo prune)?
- Do dirty-state carry helpers assume a main-worktree index?

### 3g. **RISK: `cloud_supervise_tick` may assume remote SSH transport**

From the inventory, `cloud_supervise_tick` calls `_tmux_chain_restart_command` which uses `tmux has-session` / `tmux kill-session` / `tmux new-session`. On a remote provider, these run via SSH. On the local AgentBox Hetzner host, they should run locally.

**Code-inspection task:** Verify whether `cloud_supervise_tick` and its callees can operate in a "local" mode without SSH wrapping, or whether a local adapter is needed.
