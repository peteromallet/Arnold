## 3. Missing Seam or Incompatibility That Must Be Addressed

### 3A. Critical: Phase Ordering Defect

**The plan puts Megaplan Native Integration at Phase 7, after Guardian (5) and Discord Operator (6).** This is backwards.

The evidence from the inventory:
- `cloud_supervise_tick()` (the Guardian's primary inspection call) operates on **Megaplan plan directories** with **chain state files**. Without a `megaplan_chain` operation adapter that creates worktrees and launches chains in the expected format, Guardian has nothing real to supervise.
- The Discord Operator's `chain_launch` tool depends on the chain adapter existing before it can route user messages to actual work.
- The plan's own "First Slice Recommended By The Swarm" section says: *"Add a single operation kind: megaplan_chain … launch by wrapping existing cloud chain / chain start behavior."* Then: *"Add Guardian v0 … for megaplan_chain, calls existing cloud_supervise_tick()."* These two steps are co-requisite, not sequential.

**Fix: Fold the megaplan_chain adapter into Phase 3 (Operation Registry) or make it Phase 4, before Guardian.**

### 3B. Missing: Worktree Parameterization Seam

The inventory confirms that `bakeoff/worktree.py` functions are **command-scoped and current-repo-scoped**:

> "*Current limitation: these are command-scoped and current-repo scoped. AgentBox needs them promoted into a machine-scoped operation service.*"

The code-inspection task the plan needs: audit every function in `bakeoff/worktree.py` for hardcoded references to:
- `~/Documents/.megaplan-worktrees` (the default worktree root)
- Current working directory for repo discovery
- `args.worktree` from argparse context

Each of these must be replaced with explicit `worktree_root: Path` and `repo: RepoSpec` parameters. This is extraction work, not rewrite work — the logic is sound, only the plumbing is too narrow.

### 3C. Missing: Guardian Plugin Interface

The plan says Guardian should call existing `cloud_supervise_tick()` as the first plugin, then add more later. But **no plugin dispatch interface exists in the plan's component list**. The operation registry stores `kind: megaplan_chain` — Guardian needs a dispatcher that maps `operation.kind` → `classifier_fn` → `repair_fn`. This is a one-file interface, but it isn't in the plan at all.

Specifically:
```python
# This mapping doesn't exist yet and must be built before Guardian v0
OPERATION_PLUGINS = {
    "megaplan_chain": ChainOperationPlugin(
        classify=cloud_supervise_tick,
        repair=cloud_repair_path,
        stop=tmux_kill_session,
    ),
}
```

### 3D. Missing: Bare Repo Worktree Paths

The plan recommends bare repos (`/workspace/repos/*.git`) but the existing worktree helpers assume a **normal repo checkout with a main worktree**. The inventory confirms this is an open risk:

> "*Existing worktree helpers assume a normal repo/main worktree in places such as registered-worktree discovery and removal.*"

This is a **code-inspection task to add to Phase 2**: grep `bakeoff/worktree.py` for any code that traverses `.git` as a directory (expecting a normal checkout) vs. operating on a `--git-dir`-qualified bare repo. Add a `repo_format: "bare" | "checkout"` flag to the repo registry and branch on it in worktree creation.

### 3E. Missing: Credential Preflight Before Launch

The plan's Proactive Credential Validation section is correct, but Phase 4 (Credential Sync) is ordered **after** Phase 3 (Operation Registry + Runner). The operation runner in Phase 3 will launch chain processes that need credentials. The plan should move credential validation into the operation launch path in Phase 3, or Phase 4 must precede Phase 3's "runner" sub-component.
