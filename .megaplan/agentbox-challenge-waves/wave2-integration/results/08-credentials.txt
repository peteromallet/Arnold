## 1. Verdict: **FIT WITH EXTRACTION**

The plan correctly identifies reusable credential patterns in the existing codebase — `seed_codex_oauth()`, preflight validation, `cloud.yaml` secrets, and the entrypoint OAuth-restore template. The `SecretSpec` generalization is the right abstraction direction and proactive validation is architecturally sound. ## 2. Existing Functionality That Should Be Reused

### (a) `seed_codex_oauth(spec, provider, writer=...)` — `cloud/auth.py`
This is the closest existing pattern to `SecretSpec`. It already:
- Takes a cloud `spec` + `provider` and copies **selected** local OAuth files to remote destinations
- Does **not** dump the entire local environment — it selects explicit credential files
- Has an event/status reporting path via the `writer` callback
- Is called at two points: during `cloud bootstrap` (line 244) and as a standalone seed (line 1338)

**Reuse verdict:** Extract the "copy named local files → remote destinations" core from this function. The `writer=` callable pattern is already a `health_check` reporting precursor.

### (b) Preflight dependency checks — `cloud/preflight.py`
Already validates prerequisites before expensive work:
```python
"claude": ("bun", "claude", "tmux"),
"codex": ("codex", "tmux"),
```
**Reuse verdict:** Extend with credential-validity checks. The structure (classify by profile → check dependencies → fail-early) is exactly the right shape for proactive credential validation. The tool-binary checks prove the "check before launch" discipline exists.

### (c) `cloud.yaml` `secrets:` field and entrypoint template — `cloud/templates/cloud.yaml.tmpl`, `entrypoint.sh.tmpl`
Already has:
- `secrets:` upload semantics in the cloud spec
- OAuth restore logic in the entrypoint template
- Environment-variable injection into the runtime

**Reuse verdict:** The `secrets:` field is a near-perfect model for `SecretSpec` — a declared list of named secrets with paths. The entrypoint restore logic should be adapted from "container startup" to "persistent filesystem permissions."

### (d) SSH provider remote execution — `cloud/providers/ssh.py`
Already transports files (including auth bundles) to remote machines and executes commands over SSH.
**Reuse verdict:** `agentbox creds push` should use the existing SSH transport, not build a new one.

## 3. Missing Seams and Incompatibilities

### Seam A: `seed_codex_oauth()` is deployment-scoped, not a standalone credential sync

The function lives inside the `cloud bootstrap` flow. Its signature is `seed_codex_oauth(spec, provider)`, where `spec` is a `CloudSpec` (cloud provider target, deployment config) and `provider` is an SSH/Railway transport. It's called during deploy, not as a standalone "sync credentials to the persistent machine."

**What AgentBox needs:** `agentbox creds push` must work independently of any deployment — it syncs credentials from Peter's laptop to the already-provisioned Hetzner box, with no chain/plan/deploy in flight.

**The missing seam:** A new extraction that takes the "copy selected files to remote" core but accepts a `SecretSpec` + target host, not a `CloudSpec` + deployment flow. The plan says "Reuse: Megaplan Cloud Codex OAuth seed logic" but doesn't specify where the extraction boundary is. Specifically:

- Does AgentBox call `seed_codex_oauth()` wrapped in a thin adapter?
- Or does AgentBox extract the inner file-copy logic into a shared `credential_sync` module that both `cloud/auth.py` and AgentBox import?

**Code-inspection task to add to plan:** Read `cloud/auth.py` in full and determine whether `seed_codex_oauth()` can be called with a trivially-constructed `CloudSpec` + `SSHProvider` targeting the persistent box, or whether the function body needs refactoring to separate transport from spec interpretation.

### Seam B: No credential health-check pattern exists anywhere

`cloud/preflight.py` checks whether tool binaries exist (`bun`, `claude`, `tmux`, `codex`). It does **not** check whether credentials are valid — e.g., "can I `gh auth status` successfully?" or "does this API key return 200 on a test endpoint?"

The plan proposes `SecretSpec.health_check` and proactive validation before launch, but:
- There is no existing pattern for what a health check looks like
- GitHub tokens, Codex OAuth, Claude refresh tokens, and API keys all need different validation commands
- The plan lists `agentbox creds test` as a CLI surface but gives no implementation guidance

**What's missing:** A design decision on how health checks work per credential class. For example:
- GitHub: `gh auth status` or `ssh -T git@github.com`
- Codex: check that the OAuth bundle is present and non-expired
- Claude: exchange refresh token for a short-lived token
- API keys: a lightweight API call that proves the key works

**The plan should specify:** either (a) `SecretSpec.health_check` is a `Callable[[], bool]` that each credential class provides, or (b) `agentbox creds test` dispatches by credential type to a known set of check commands.

### Seam C: Cloud secrets vs. persistent machine secrets overlap is unresolved

Two mechanisms exist or are proposed:
1. `cloud.yaml` `secrets:` — environment variables uploaded during cloud deploy, used per-deployment
2. AgentBox `/workspace/secrets` — persistent secret files on the box, used per-operation

The plan doesn't clarify whether these are:
- **The same mechanism** — AgentBox replaces `cloud.yaml` secrets when running on the persistent box
- **Separate mechanisms** — cloud deploys still use `cloud.yaml` secrets, AgentBox secrets are an additional layer for non-cloud operations
- **A migration path** — `cloud.yaml` secrets become a legacy mechanism that AgentBox wraps

**Why this matters:** If a Megaplan chain operation launches via AgentBox on the persistent box, does it get credentials from `/workspace/secrets` (AgentBox path) or from `cloud.yaml` `secrets:` (existing path)? Having both active is a bug farm — duplicate credentials, stale values, confusion about which path wins.

**Code-inspection task to add to plan:** Audit `cloud/template.py` and `cloud/templates/entrypoint.sh.tmpl` to determine whether the `secrets:` field is used at runtime (injected into the entrypoint) or only at deploy time. If runtime-only, AgentBox can intercept and supply from its own store. If deploy-time, a migration seam is needed.

### Change 2: Add `SecretSpec.health_check` design decision

**Add to the "Credential Sync" section, after credential classes list:**
```markdown
### Health Check Design

`SecretSpec.health_check` is a per-credential-class dispatch, not a generic function:

| Credential class | Health check command |
|---|---|
| GitHub (token) | `gh auth status --hostname github.com` |
| GitHub (SSH key) | `ssh -T -o ConnectTimeout=5 git@github.com` |
| Codex OAuth | Check that the OAuth bundle files exist and have non-zero size |
| Claude refresh token | Exchange for a short-lived token via `claude --auth-test` or equivalent |
| API keys (OpenAI/Anthropic/DeepSeek) | Lightweight models.list or chat completion with `max_tokens=1` |
| Discord bot token | `GET /users/@me` on Discord API |

Health checks return `{valid: bool, reason: str, stale: bool}`. `stale` means the credential works but is close to expiry.

AgentBox stores the last health-check timestamp and result per credential in the registry.
```

### Change 3: Resolve cloud-secrets vs. persistent-secrets overlap

**Add to Phase 4 deliverables:**
```markdown
### Cloud Secrets vs. AgentBox Secrets Overlap

**Decision needed:** When a `megaplan-chain` operation runs on the persistent box, does it get credentials from:

**(A) AgentBox `/workspace/secrets` only** — `cloud.yaml` `secrets:` is ignored.
Prefer this. AgentBox owns the machine and should be the single credential source.

**(B) Both, with AgentBox winning** — `/workspace/secrets` takes priority over `cloud.yaml` secrets on conflict.

**(C) `cloud.yaml` `secrets:` only** — AgentBox does not interfere with cloud deploy mechanics.

**Recommendation: Option A.** The persistent box is an AgentBox machine, not a cloud deploy target. When a chain runs locally, AgentBox provides credentials from its store. The `cloud.yaml` `secrets:` field is only used when deploying to Railway or a separate SSH target.

**Implementation:** AgentBox's `megaplan-chain` adapter injects environment variables from `/workspace/secrets` before launching the chain command, and the chain's entrypoint or wrapper reads from env, not from a cloud deploy step.
```

### Change 4: Add proactive validation task to Phase 3 (before Phase 4)

**Add to Phase 3 (Operation Registry and Runner) deliverables:**
```markdown
### Proactive Credential Gate (add before first launch)

Before creating a worktree or launching any operation, the runner must:

1. Classify the operation kind and its required credential classes.
2. Run `agentbox creds test` for each required credential class.
3. If any fail:
   - Block the launch.
   - Return the exact missing/stale credential and the command Peter should run (`agentbox creds push`).
4. If all pass:
   - Proceed with worktree creation and launch.

This gate reuses the `cloud/preflight.py` pattern (classify → check → fail-early) but extends it from tool-binary checks to credential-validity checks.
```
