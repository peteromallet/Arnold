### Root Cause

**Two independent failures at the L2 repair-system layer, neither of which is a design-intent flaw:**

#### 1. Artifact-read sandbox preflight/fallback (INTENT → fail)

The `arnold-meta-repair-loop` wrapper (line 418) dispatches the L2 investigator as:
```
codex exec --skip-git-repo-check --sandbox read-only … -
```
The `--sandbox read-only` flag uses **bubblewrap** (`bwrap`) for namespace isolation. On this host, the kernel denies unprivileged user-namespace creation. The sandbox cannot initialize — codex fails before opening any artifact reference. The investigator receipt confirms: `"observed_error": "bwrap: No permissions to create new namespace…"`.

The `build_meta_investigation_context()` function in `repair_investigation.py` (lines 439–588) builds a perfectly valid reference-only envelope with SHA-256 hashes and absolute paths for every artifact. But **the wrapper never preflights whether the read-only executor can actually open those paths**. There is no fallback to a Python-native or non-bwrap read-only reader. The preflight gap means the L2 investigator is handed perfect custody context it cannot consume.

#### 2. Blocker-scoped recursion poisoning (TRACKED → fail)

`meta_repair_policy.py` `check_meta_repair_recursion()` (lines 91–180) accepts `blocker_id` to scope recursion detection to a single blocker episode. But the wrapper (line 231) invokes it as:
```bash
"${CLOUD_WATCHDOG_REPAIR_BLOCKER_ID:-}"
```
When this env var is unset — which it is for the `needs-human` record (its own `blocker_id: ""`) — `blocker_id` is the empty string. The guard's blocker filter (`if blocker_id and …`) is a no-op, and **all historical session records count against the same budget**.

Three `commit_custody_failed` records accumulate. The retry-budget logic (`_MAX_COMMIT_CUSTODY_FAILURE_RETRIES = 2`) correctly separates commit-custody failures, but with 3 such records (`3 >= 2`), they all merge into `matching_ids`, exceeding `max_meta_repair_attempts=1`. The session is permanently poisoned — no meta-repair will ever launch again for `custody-control-plane-20260714`, regardless of future blocker episodes.
