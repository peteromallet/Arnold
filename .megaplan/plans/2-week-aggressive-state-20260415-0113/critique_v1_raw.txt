Those are not sufficient for the stated gates because the failures span three shared state domains, the dead-code deletions are explicit, and timeline/selection render isolation is part of the acceptance criteria.

Planning note: direct repository reads were blocked in this sandbox with `bwrap: Creating new namespace failed: Permission denied`, so this plan uses the engineering brief as the source of truth and treats caller enumeration as the first hard gate of execution rather than pretending the audit already happened.

Key constraints to preserve throughout:
- Zustand is fixed; React Query remains the server-data source of truth.
- Settings hook names and the public `useAutoSaveSettings` return shape stay stable.
- Old helpers/hooks/contexts are deleted in the same PR that introduces their successor.
- The five migration branches stay independently revertible.
- TypeScript strictness stays intact; selector and action typing must remain precise.

## Phase 2: Core Settings Store (PR 2: `megaplan/settings-store-core`)

### Step 4: Rewrite `useAutoSaveSettings` as a keyed store wrapper
**Scope:** Medium
1. Rebuild `[useAutoSaveSettings.ts](/workspace/reigh-app/src/shared/settings/hooks/useAutoSaveSettings.ts)` as a thin adapter over `createEntityStore`, keyed by `(toolId, entityId, scope)`.
2. Preserve the public return contract exactly: `settings`, `status`, `entityId`, `isDirty`, `error`, `hasShotSettings`, `hasPersistedData`, `updateField`, `updateFields`, `save`, `saveImmediate`, `revert`, `reset`, and `initializeFrom`.
3. Ensure store reuse is selector-driven so consumers stop receiving whole-object churn when unrelated settings fields change.
4. Keep React Query one-way: query result bootstraps the store; the store does not become a second cache of server data beyond the current editable entity state.

### Step 5: Move readiness/load/persistence responsibilities into the store and delete the old chain
**Scope:** Medium
1. Absorb `[useDebouncedSettingsSave.ts](/workspace/reigh-app/src/shared/settings/hooks/useDebouncedSettingsSave.ts)` into the store implementation, then delete the file.
2. Collapse `[autoSaveSettingsHelpers.ts](/workspace/reigh-app/src/shared/settings/hooks/autoSaveSettingsHelpers.ts)` and `[autoSaveSettingsLoaders.ts](/workspace/reigh-app/src/shared/settings/hooks/autoSaveSettingsLoaders.ts)` into the new bootstrap/load path, then delete them.
3. Remove render-time readiness mutation from `[useEditSettingsPersistence.ts](/workspace/reigh-app/src/domains/media-lightbox/hooks/persistence/useEditSettingsPersistence.ts)` and move any `hasInitializedRef` / `lastGenerationIdRef` / readiness bookkeeping into store state or derived selectors.
4. Replace the ref-backed pseudo-state in `[useLastUsedEditSettings.ts](/workspace/reigh-app/src/domains/media-lightbox/hooks/useLastUsedEditSettings.ts)` with a real store slice so lightbox consumers stop relying on mutable refs as implicit state.

## Execution Order
1. Do not start store rewrites until the Step 1 audit and Step 2 instrumentation are in place.
2. Use PR 1 to prove the generic store on `join-clips` before changing the central settings hook.
3. Land PR 2 before PR 3 so all caller migrations move onto one stable `useAutoSaveSettings` implementation.
4. Land PR 4 before PR 5 so timeline selection ownership is settled before the full timeline-store collapse.
5. Keep deletions in the same PR as replacements; do not carry temporary dual paths across branches.

## Validation Order
1. Start each PR with the cheapest targeted tests for the files it changes.
2. Run instrumentation-backed lightbox checks immediately after PRs 2 and 3.
3. Run selection-focused regression checks after PR 4 before touching the larger timeline provider changes.
4. Run timeline render-budget checks and drag/scroll smoke tests before the full suite in PR 5.
5. Finish each PR with `tsc` and the repo test baseline check; the final branch should match or improve the existing pass rate without unrelated fixes.

## Risks To Watch
- `useEditSettingsPersistence` and `useLastUsedEditSettings` are likely the highest-risk identity-sensitive callers; migrate them early in PR 3.
- `useShotSettings` and the ShotEditor video hooks may create multiple live consumers for the same settings key; store-key reuse and shared pending text behavior need explicit verification.
- Selection state currently leaks into timeline internals; PR 4 must carve out only shared selection ownership so PR 5 does not reintroduce cross-domain duplication.
- If the Step 1 audit contradicts the brief in a major way, pause after PR 1 and adjust the branch breakdown before rewriting the core settings hook.
