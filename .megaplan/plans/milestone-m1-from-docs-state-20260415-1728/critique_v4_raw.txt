   The factory creates a single Zustand store per domain (settings, selection, timeline, etc.) with entity state keyed by an `entityId`. Callers get a hook `useEntity(entityId)` that returns the shape needed by consumers.

   **Factory signature:**
   ```typescript
   interface EntityStoreConfig<T extends object> {
     toolId: string;                       // identifies the store (e.g. "join-clips")
     defaults: T;                          // initial settings shape
     load: (entityId: string) => Promise<{ db: T | null; lastUsed: T | null }>;
     save: (entityId: string, data: T) => Promise<void>;
     textFieldKeys?: readonly (keyof T)[];  // fields that save on blur/flush instead of debounce
     persistenceDebounceMs?: number;         // default 300
   }

   interface EntityStoreApi<T extends object> {
     useEntity: (entityId: string) => EntityView<T>;
     // plus: direct action access for tests
     getState: () => EntityStoreState<T>;
   }

   interface EntityView<T> {
     settings: T;
     status: 'idle' | 'loading' | 'ready' | 'saving' | 'error';
     error: Error | null;
     hasPersistedData: boolean;
     updateField: <K extends keyof T>(key: K, value: T[K]) => void;
     updateFields: (patch: Partial<T>) => void;
     updateTextField: <K extends keyof T>(key: K, value: T[K]) => void;
     flushTextFields: () => Promise<void>;
     save: () => Promise<void>;
     saveImmediate: () => Promise<void>;
     revert: () => void;
     reset: () => void;
   }

   function createEntityStore<T extends object>(config: EntityStoreConfig<T>): EntityStoreApi<T>;
   ```

   **Required behaviors:**
   - Single Zustand store per factory instantiation. State shape: `{ entities: Record<entityId, EntityState<T>> }`.
   - `useEntity(id)` subscribes the component to just that entity's slice. Different entity IDs do not cause cross-renders.
   - Use `shallow` equality (from `zustand/shallow`) for selectors that return objects or arrays; default `Object.is` for primitives.
   - Atomic bootstrap action: `bootstrapEntity({entityId, db, lastUsed})` merges in one dispatch — no multi-transition load sequence.
   - On first mount for an entity: `useEntity` triggers a lazy load via `config.load()` and dispatches bootstrap.
   - `updateField` / `updateFields`: optimistic write to settings, schedule debounced save.
   - `updateTextField`: same state write but does NOT schedule the debounce — intended for prompt/textarea fields that should save on blur or flush.
   - `flushTextFields`: force-save any pending text-field edits immediately.
   - Debounce: 300ms default, configurable via `config.persistenceDebounceMs`.
   - On React Query data arrival (for entities that source from RQ): scaffolding is responsible; the factory itself doesn't import RQ. The factory's `load` is invoked externally.

## Overview
The previous plan was targeting the wrong architecture for the Join Clips pilot. The user has now settled the root-cause question: for React Query-backed settings, `useEntity(id)` is reader-only, React Query remains authoritative through the existing `useAutoSaveSettings -> useToolSettings` boundary, `createEntityStore.load` is reserved for store-owned non-server entities, and any RQ-to-store sync must stay internal to the hook boundary rather than being driven by consumer bootstrap bridges. This is not a minor adjustment; it changes the Join Clips integration approach.

The revised plan keeps the milestone aligned with the original intent while removing the rejected store-authoritative variant. M1 still delivers the generic Zustand factory, still ports only `useJoinClipsSettings`, and still validates the store shape on a real consumer. But Join Clips now validates the factory in reader mode: the hook continues to source authoritative settings and persistence from the existing RQ-backed path, while adapting that state into an entity-scoped Zustand store internally for selector isolation, local reader semantics, and future migration groundwork. The factory remains generic enough to support true store-owned entities later, but that mode is not used as the authoritative architecture for Join Clips in M1.
