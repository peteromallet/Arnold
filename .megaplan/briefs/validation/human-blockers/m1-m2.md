# Human blockers — M1 (Foundation/contract-checker) + M2 (De-planning types + typed Port)

Goal: drive every human-decision point to zero. Each entry is converted to a pre-made **default**,
a **machine-gate**, or an **auto-escalation** policy. `must_ask_peter=true` only where a decision is
genuinely irreversible strategy/taste.

Scanned: `.megaplan/briefs/epic-pipeline-unification/m1-foundation.md`, `m2-deplanning-types.md`, and the
governing `.megaplan/briefs/epic-pipeline-unification/chain.yaml` (which is the artifact that actually runs and
bakes in the runtime human gates for the BUILD of these milestones).

Legend: kind = design-time (an agent executing the milestone stalls here) | runtime (gate baked into
the build/product as it runs).

## EPIC-WIDE RUNTIME GATES (apply to M1 + M2 build, sourced from chain.yaml)

### B0. `merge_policy: review` (chain.yaml:77)
- **kind:** runtime. The single biggest human blocker in the whole epic build: every milestone PR
  parks waiting for a human to review+merge before the next milestone starts.
- **decision/default:** Replace human review with the **machine-gate already specified by the briefs**:
  a milestone auto-merges when (a) `pytest -m "not slow"` is green (M1-W1 enables >40 files incl. the
  parity gate + chain-status contract), (b) the per-milestone oracle/parity gate is green
  (`tests/test_pipeline_parity.py`), (c) M1's `pipelines check` exits zero on every in-tree pipeline,
  and (d) M2's CI grep-gate (ZERO `GateRecommendation` in SDK `_pipeline` modules) passes. Set
  `merge_policy: auto` with these as required status checks.
- **mechanism:** machine-gate.
- **rationale:** The briefs explicitly designed these gates AS the substitute for human review (the
  parity gate is "a per-milestone oracle's job", M1 §Constraints); auto-merge-on-green / auto-halt-on-red
  is exactly the conversion principle. Reversible: a bad merge is caught by the next milestone's parity
  gate and revertible via M1-W3's `extra="ignore"`+`schema_version` (state stays loadable).

### B1. `on_failure: stop_chain` (chain.yaml:73)
- **kind:** runtime. A failing milestone halts the chain and waits for a human to intervene/resume.
- **decision/default:** Keep `stop_chain` as the *final* state only after an **auto-escalation ladder**:
  retry the milestone once (transient infra/idle per memory `project_execute_stall_codex_silence`,
  idle backstop 1800) → re-dispatch one tier stronger (profile premium→apex, depth high→max) → then
  stop_chain + emit a structured failure artifact. Encode the ladder in the driver, not a human.
- **mechanism:** auto-escalation.
- **rationale:** Conversion principle (c). stop_chain-only parks on a person; the ladder exhausts
  automated recovery first and only halts on a genuinely unrecoverable defect, which is itself flagged
  as an artifact, not a silent wait.

### B2. `on_escalate: stop_chain` (chain.yaml:75)
- **kind:** runtime. An ESCALATE verdict from a gate halts the chain for a human.
- **decision/default:** On escalate, auto-bump the milestone's depth/profile one tier and re-run the
  failing stage (the gate dispatch ladder already supports recommendation/escalate edges, M1-W2). Only
  after the stronger re-run also escalates does it stop_chain with an artifact.
- **mechanism:** auto-escalation.
- **rationale:** Mirrors the in-pipeline escalate→stronger-model design; never parks on a person on the
  first escalate.

### B3. "wait to run / do not `chain start` yet" (chain.yaml:10)
- **kind:** design-time. The chain.yaml is annotated review-only; a human must decide to start it.
- **decision/default:** This is the one legitimate **launch authorization** — see B-PETER below. The
  per-milestone build needs no further human start once authorized; the driver
  (`auto_approve: true`, chain.yaml:81) carries it. Default: once Peter says "go", remove line 10 and
  `chain start`; no per-milestone restart authorization.
- **mechanism:** default (one-time, see B-PETER).
- **rationale:** Single top-of-epic go/no-go is acceptable; per-milestone waits are not.
