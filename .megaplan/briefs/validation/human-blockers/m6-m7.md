# Human blockers — M6 (megaplan-as-module) + M7 (builder docs)

Goal: drive human blockers to zero. Each decision point below is converted to a
pre-made DEFAULT, a MACHINE-ENFORCED GATE, or an AUTO-ESCALATION policy. `must_ask_peter`
is false unless the decision is a genuinely irreversible strategy/taste call.

Sources: `.megaplan/briefs/epic-pipeline-unification/m6-megaplan-as-module.md`,
`.megaplan/briefs/epic-pipeline-unification/m7-builder-docs.md`, with EPIC + validation refs as cited.

## DESIGN/BUILD-TIME blockers

### B1 — M6 "Open question": which driver does planning's manifest name (process | graph | both)?
**Location:** m6 §108-112 (Open questions, "biggest open dependency").
**Default:** Manifest names a **(substrate, topology) pair = `(subprocess_isolated, process-per-phase)`** —
i.e. the substrate axis the current `auto.py` per-phase subprocess driver already uses, since that is the
production path real plans run through today. The brief itself says the pick is "settled by M3's outcome —
M6 must not re-litigate it." So the value is read from M3's realized-graph manifest, not chosen at M6.
**Mechanism:** default (pinned to M3 output).
**Rationale:** The brief explicitly defers this to M3; M6 consumes M3's substrate decision, so M6 records
whatever M3 emitted rather than re-deciding. Reversible: changing the manifest pair is a one-line manifest edit.

### B2 — M6 "Open question": where does `workflow_next`'s projection physically live?
**Location:** m6 §113-114.
**Default:** Place the projection in **M3's realized-graph layer** (`build_topology` neighborhood), exported
as the existing `workflow_next` symbol so override/status/doctor/introspect imports resolve unchanged.
**Mechanism:** default + machine-gate.
**Rationale:** Co-locating with the single source of truth (the realized graph) avoids a fourth drifting copy
— the explicit M6 goal; the import-surface characterization test (`tests/characterization/test_import_surface.py`)
enforces that the symbol stays importable from its old path, so the parity gate catches a wrong home.

### B3 — M6 "Open question": does the `arnold` surface replace `discover_python_pipelines` or wrap it?
**Location:** m6 §115-116.
**Default:** **Wrap** `discover_python_pipelines` as one source among drivers+packages behind the manifest-first
front door, rather than deleting it.
**Mechanism:** default.
**Rationale:** Wrapping is the lower-blast-radius reversible choice; creative/doc already discover through it,
so wrapping preserves their path while adding the manifest-first gate. Can be collapsed to a full replacement
later once all packages are manifest-first.

### B4 — M6 trust tier assignment: which tier does each package get (`in-tree` | `blessed` | `quarantined`)?
**Location:** m6 §46-49 (THE trust boundary), §27 "gated behind an operator trust tier".
**Default:** Tier is **derived, not operator-chosen**: packages under the repo tree / installed dist =
`in-tree` (exec allowed); packages under `~/.megaplan/pipelines` (the untrusted-author root, `registry.py:375`)
= `quarantined` by default (manifest read, no exec until promoted); `blessed` requires an explicit signed
allowlist entry. The "operator decides" becomes a deterministic path-based classifier with a config-file allowlist.
**Mechanism:** machine-gate (path-based classifier + allowlist file checked at discovery).
**Rationale:** Converts the brief's "operator trust tier" / "operator decides" runtime human gate into a
deterministic rule keyed on install location, defaulting to the safe tier (quarantined) for the only untrusted
entry point. Auto-proceeds on in-tree, auto-quarantines unknown — never waits for a person.

### B5 — M6: `arnold_api_version` supported range — what range does the SDK declare?
**Location:** m6 §57, §153.
**Default:** SDK declares `arnold_api_version` compatible range = `[1.0, <current+0)` (current major, exact
minor floor of the shipped surface). A package pinning outside the range is rejected at discovery with a loud,
catalogued error (not a silent vanish).
**Mechanism:** machine-gate (discovery-time semver check, fail-loud).
**Rationale:** A version-compat check is mechanical; the range is a build-time constant tied to the released
surface. Surfaces as a checker error, never parks on a human.

### B6 — M6: re-opened a5 import-seam trust analysis — what's the verdict / what mitigations ship?
**Location:** m6 §58-61, §153 ("a re-opened a5 covers the import seam").
**Default:** Pre-made verdict: the import seam IS high-risk (the brief already states a5's low-risk verdict
"is dangerously wrong" for the import seam). Therefore ship the mitigation unconditionally: **defer
`exec_module` to selected-to-run + quarantine the `~/.megaplan/pipelines` root** (B4). The "re-open a5"
deliverable is a written confirmation of this stance plus the manifest-first reader, not a fresh open question.
**Mechanism:** default (verdict pre-made) + machine-gate (the non-executing discovery is the enforcement).
**Rationale:** The brief has already adjudicated the risk; the remaining work is implementing the known
mitigation, which is gated by the manifest-first discovery test, not by a human risk sign-off.

### B7 — M6: per-package quota sub-budget size + `tenant_id` assignment rule.
**Location:** m6 §55-57 ("SDK-assigned `tenant_id` (not self-declared) + a per-package quota sub-budget
reserved in the broker ledger").
**Default:** `tenant_id` = SDK-derived deterministic hash of (package name + install path); default
sub-budget = a fixed fraction (e.g. equal-share of the parent run budget across discovered packages, floor
of one milestone's typical spend). Self-declared budgets are ignored.
**Mechanism:** default.
**Rationale:** The brief mandates SDK-assigned (not self-declared) ids — that removes the human/author from
the loop by construction; a fixed-fraction default is reversible via config and prevents any single package
from monopolizing the broker ledger.

### B8 — M6: name-alias + phase-slot preservation for pre-M6 plans on disk.
**Location:** m6 §120-126 (Constraints), §168 (`resume_plan` alias).
**Default:** Keep `planning` → relocated-package alias and preserve the `_RESUME_ACTIVE_STATES` phase
literals (prep/plan/critique/gate/revise/finalize/execute/review/feedback) verbatim. No human migration step.
**Mechanism:** machine-gate (a resume-a-pre-M6-plan characterization test must stay green).
**Rationale:** Back-compat is mechanically testable — fixture an old plan dir, resume it, assert no orphan.
Auto-halts the build on red, never asks a person whether the alias is "ok".

### B9 — M6: parity gate honest-labelling — what does the gate assert and at what bar?
**Location:** m6 §131-133 (honestly labelled, "control-flow/artifact parity on the happy path, not 'drift
provably zero'"), §155.
**Default:** Gate asserts control-flow + artifact parity on the happy path across
`{5 robustness}×{prep,feedback}×{states}×{verdicts}`; green = auto-proceed, red = auto-halt. Label is fixed
to "happy-path parity" per the brief — no human re-interpretation.
**Mechanism:** machine-gate.
**Rationale:** The bar and label are pre-specified by the brief and the M3 parity test; the gate is the
decider, not a reviewer.

### B10 — M6 acceptance test #1: WHICH fourth non-planning tool ships (select-tournament | snapshot/restore | run-oracle bisect)?
**Location:** m6 §137-142.
**Default:** Ship the **`select`-tournament** as the fourth tool (it is the one M7 also names as its
non-planning example, §26/§52, so the two milestones share one artifact and one acceptance path).
**Mechanism:** default.
**Rationale:** Picking the option M7 already depends on avoids divergence and a second build; any of the three
satisfies the "differently-shaped" honesty requirement, so taste is not load-bearing. Reversible — could add
the others later.

### B11 — M6 acceptance test #2: which stub becomes the "cheap new pipeline" (jokes | other)?
**Location:** m6 §143-144, §177 ("`jokes` upgraded to a real SDK pipeline").
**Default:** Upgrade **`jokes`** — the brief names it explicitly.
**Mechanism:** default.
**Rationale:** Named in the brief; no decision remains.

## RUNTIME blockers (baked into the product)

### B12 — RUNTIME: discovery trust tier "operator decides" whether to execute a discovered package.
**Location:** m6 §27, §46-49 ("gated on an operator trust tier"), EPIC §115-118 (discovery trust boundary).
**Default / mechanism:** machine-gate — same deterministic classifier as B4 evaluated at run time:
`in-tree`/`blessed` → auto-exec on selection; `quarantined` → refuse exec, emit a catalogued
`quarantined-package` event, and **auto-escalate** by skipping+flagging (the package is listed as discovered
but not runnable) rather than prompting. Promotion to `blessed` is a config/allowlist edit, never an
interactive prompt.
**Rationale:** This is the central runtime human gate in M6. A path-derived tier + skip-and-flag on
quarantine means a run never blocks on an operator approving an untrusted package; the safe default (don't
exec) holds automatically.

### B13 — RUNTIME: a discovered package fails to import / typo'd entrypoint / missing `SKILL.md` (the silent-vanish, Theme E).
**Location:** m6 §53-54, §97-98 ("discovery rejects loudly... no silent vanish"), §153.
**Default / mechanism:** machine-gate + auto-escalation — replace `except Exception: return None` with a
loud, catalogued discovery error that names the package and reason; the package is excluded from the runnable
set and surfaced in `pipelines doctor`/`check`. No human is consulted; the run proceeds with the remaining
valid packages.
**Rationale:** Converts a silent failure (which would invisibly require a human to notice) into a deterministic,
visible-but-non-blocking rejection. Fail-loud + skip = auto-proceed for the rest.

### B14 — RUNTIME: `SKILL.md` required-element enforcement at discovery.
**Location:** m6 §36, §97-98; m7 §22-23 (manifest field), §51-52 (acceptance).
**Default / mechanism:** machine-gate — discovery fails the package loudly if `SKILL.md` is absent
(folded into B13's catalogued errors). `pipelines check` exits non-zero.
**Rationale:** Pure mechanical presence check; the checker is the decider.

### B15 — RUNTIME: the planning `next_step` / human-recovery surface (9 override actions, status, doctor, feedback REVIEWED→DONE).
**Location:** m6 §127-130 (human-recovery surface), §63-74 (next-step collapse).
**Default / mechanism:** machine-gate — these surfaces must read the *projected* labels off M3's realized
graph unchanged; a parity test asserts the projected labels match pre-M6 labels for every
`{robustness}×{state}×{verdict}`. The recovery commands an operator sees are computed, not chosen.
**Rationale:** This is the existing human-recovery product surface; M6 must not change what a stuck operator
is told. The decision here is "preserve exactly", enforced by the projection parity test — no human judgement
at build or run time. (Note: these override actions are themselves operator-facing runtime gates inherited
from the existing product; M6's job is preservation, not their elimination — that is out of M6 scope per the
anti-scope, and they already have pre-made recovery commands.)

### B16 — RUNTIME (Theme D guard): a binding accidentally carries `STATE_*` as mechanism.
**Location:** m6 §28, §102-104, §155 ("no binding carries `STATE_*` as mechanism").
**Default / mechanism:** machine-gate — a grep/AST assertion in the gate fails the build if any planning
binding references `STATE_*` as a mechanism; the M5c control interface is the only allowed binding target.
**Rationale:** Mechanical eviction check, mirrors the `JoinFn`→`GateRecommendation` eviction test one layer
down. Auto-halts on violation.

## M7 DESIGN/BUILD-TIME blockers

### B17 — M7 "Open question": does the umbrella `~/.claude/skills/arnold*` skill replace or compose with megaplan-decision/observe/epic skills?
**Location:** m7 §42-43.
**Default:** **Compose** during the rename period — the umbrella skill adds an `arnold`-build entry point
and links to the existing decision/observe/epic skills rather than deleting them, since M6 anti-scope (§181)
says `arnold` is namespace-only and commands stay `megaplan`.
**Mechanism:** default.
**Rationale:** The brief locks `arnold` as namespace-only and keeps `megaplan` commands; replacing the skills
would imply a CLI/UX rename that is explicitly anti-scope. Composition is the reversible, consistent choice.

### B18 — M7 "Open question": where does per-module `SKILL.md` guidance end and umbrella how-to-build guidance begin?
**Location:** m7 §44.
**Default:** Per-module `SKILL.md` = *what this specific module does + how to invoke it* (surfaced live via
`read_skill_md`); umbrella guidance = *how to build any module* (the authoring guide + generated reference).
Boundary rule: anything module-specific lives in the module's SKILL.md; anything cross-module/general lives
in `docs/arnold/`.
**Mechanism:** default + machine-gate.
**Rationale:** A clear authored boundary rule removes duplication judgement; the generation rule (m7 §33-36 —
generated-if-typed) plus the CI re-emit-and-diff gate mechanically prevents the umbrella from duplicating
generated facts.

### B19 — M7: the GENERATED docs (`reference/`, manifest table, checker error catalogue, umbrella skill) — drift / regeneration.
**Location:** m7 §17-23, §33-36, §55 ("regenerates byte-identically in CI").
**Default / mechanism:** machine-gate — CI `--check` re-emit-and-diff; byte-non-identical = auto-fail. No
human reviews the generated docs for correctness; the generator + diff is the decider. On red, auto-escalation
is "regenerate and commit" (a mechanical fix), not a human edit.
**Rationale:** Generated-from-typed-surface means the source of truth is the code; the diff gate enforces it.

### B20 — M7 (THE headline runtime/acceptance blocker): "external-builder acceptance test assumes a HUMAN builder, no SDK author in the loop".
**Location:** m7 §8-11 (Outcome: "no access to SDK internals and no SDK author in the loop"), §50-56
(Done criteria: the external-builder acceptance test, performed "against the docs only").
**Default / mechanism:** **auto-escalation via an agent builder** — the "external developer" is operationalized
as a **sandboxed subagent** (per the project's subagent-launcher pattern) given ONLY `docs/arnold/` + the
scaffold and NO repo/SDK-internals access. It runs `pipelines new` → wires `select`+`reduce` → authors
`SKILL.md` → `pipelines check` (must exit 0) → `pipelines doctor` (must show `discovered ✓`) → `arnold run`
(must produce the winner artifact); a grep asserts no `GateRecommendation`/`STATE_*`/4-verdict leaked into the
module. All checks are mechanical exit-code / artifact / grep assertions. Green = acceptance passes; red =
the docs are insufficient → auto-file the gap and (escalation) retry with a stronger model before flagging.
**Rationale:** This is the one place both briefs literally bake in a human ("external developer", "human
builder"). It is converted to a machine-graded agent run: every success criterion in the brief is already an
exit code, an emitted artifact, or a grep — none require human taste. The doc-quality judgement ("could a
human follow this?") is replaced by "could an agent with only the docs succeed?", which is strictly stronger
and fully automatable. No human is needed to play the builder or to grade the result.

### B21 — M7: examples must be "extracted from real in-tree packs" and must run.
**Location:** m7 §25-27, §37, §56.
**Default / mechanism:** machine-gate — a CI test extracts each `docs/arnold/examples/*` snippet from its
source pack and executes it; non-extracted or non-running = auto-fail.
**Rationale:** "Can't rot" is enforced by extraction-and-run, not by a human spot-check.

### B22 — M7 gating dependency: M7 acceptance must run against the FINAL post-M6 surface.
**Location:** m7 §3-6, §46-48 (Constraints: "gate the acceptance on M6").
**Default / mechanism:** machine-gate — M7's acceptance suite is wired to depend on M6 completion (the
chain's milestone ordering); generators may scaffold earlier but the acceptance gate only runs after M6's gate
is green.
**Rationale:** Pure ordering constraint handled by the chain/epic harness, not a human go/no-go.

## Summary
- Design-time: B1-B11, B17-B19, B22 — all pre-made as defaults or machine-gates.
- Runtime (product): B12-B16, B20-B21 — the discovery trust tier (B12), silent-vanish (B13), and the
  external-builder acceptance (B20) are the three that most look like human gates; all three are converted to
  deterministic classifiers / fail-loud rejections / machine-graded agent runs.
- `must_ask_peter = false` for every entry. None is an irreversible strategy/taste call: the closest
  candidates (B1 driver pick, B10 fourth-tool pick, B17 skill replace-vs-compose) are each either deferred to
  M3's recorded output, reversible by a one-line edit, or locked by existing anti-scope.
