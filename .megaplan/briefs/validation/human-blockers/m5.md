# M5 (a/b/c/d) — Human-blocker inventory → pre-made decisions

GOAL: every human-decision point across the four M5 briefs, converted to a (a) pre-made DEFAULT,
(b) MACHINE-ENFORCED GATE, or (c) AUTO-ESCALATION policy. `must_ask_peter=false` unless genuinely
irreversible strategy/taste. Sources: `m5a-node-library.md`, `m5b-execute-realm.md`,
`m5c-control-plane.md`, `m5d-supervisor-tier.md`.

Two classes:
- **DESIGN/BUILD-TIME**: Open questions / "confirm" / "resolve in writing" / "lean" that an executing
  agent would stall on.
- **RUNTIME (baked into product)**: human-gates, override actions, clarify, awaiting_human, PR-merge
  wait, operator loop, merge_policy:review.

### C2 (design) Open-Q#2: typed reverse-edge variant on realized graph vs raw `predecessors()`

### C4 (build-gate) a3 §4.4 `workflow_next` parity across {5 robustness}×{prep,feedback}×{states}×{gate recs}

### C5 (build-gate) "All 9 override actions behave identically post-extraction" + strict-notes invariant

### RUNTIME human-gates baked into the product (M5c F6/F7) — convert each to autonomous behavior

### C7 (RUNTIME) F6 `clarify` node — "ask → BLOCK → resume on human answer" (`STATE_AWAITING_HUMAN`)
- **premade**: AUTO-ESCALATION → the clarify node remains as a general capability, but the PRODUCT default
  is auto-answer: on clarify, retry with a stronger model / use the prep-fanout research dossier to
  resolve the question autonomously; only persist `awaiting_user.json` + emit `awaiting_human` outcome if
  auto-resolution fails N times, and even then a configurable auto-default answer (best-guess + flag)
  un-halts. Never indefinitely parks on a person.
- **mechanism**: auto-escalation. **rationale**: aligns with autonomous-platform goal; clarify becomes
  retry→escalate→best-guess+flag. The MECHANISM (pause/resume hook) still ships for non-autonomous callers, but the default policy auto-proceeds.

### C8 (RUNTIME) F6 `verify-human` — `STATE_AWAITING_HUMAN_VERIFY → STATE_DONE` (criteria-verification gate)
- **premade**: MACHINE-GATE → replace the human verify with an automated criteria-verification oracle
  (the verifiability handler runs the criteria checks); green auto-advances to DONE, red auto-escalates
  (retry → stronger model → flag). No human verifier.
- **mechanism**: machine-gate. **rationale**: criteria verification is checkable; converts verify-human into an oracle pass/fail.

### C9 (RUNTIME) F7 `force-proceed` — operator manually synthesizes a PROCEED gate artifact to unblock
- **premade**: AUTO-ESCALATION → the gate's auto-proceed policy fires force-advance automatically when the
  gate consequence is satisfiable (e.g. agent-availability-blocked clears) or after the auto-escalation
  ladder exhausts; `synthesize_artifacts` is called by the SDK transition, not by a person.
- **mechanism**: auto-escalation. **rationale**: force-proceed is the recovery lever — automate it as the terminal rung of retry→escalate so a stuck gate self-clears.

### C10 (RUNTIME) F7 `recover-blocked` — operator drives a blocked run back onto a recovery edge
- **premade**: AUTO-ESCALATION → on `blocked` outcome, auto-run recovery: compute `recover_targets` via
  `predecessors()`, auto-route to the recovery target, retry; escalate model on repeat; abort+flag only
  after the ladder. (Note: `project_chain_blocked_retry_and_resume` — don't hardcode max_blocked_retries=1.)
- **mechanism**: auto-escalation. **rationale**: recovery target is computable from the realized graph; the action becomes automatic with a bounded (configurable, >1) retry ladder.

### C11 (RUNTIME) F7 `abort` — operator aborts a run
- **premade**: DEFAULT → keep as an automatic terminal of the auto-escalation ladder (retry → escalate →
  skip+flag → abort) when all recovery exhausted; abort emits `failed` outcome + flag, supervisor decides
  next. Never a manual-only action in the autonomous path.
- **mechanism**: auto-escalation. **rationale**: abort is the last rung, reached by policy not by a person.

### C12 (RUNTIME) F7 `replan` — operator triggers a replan
- **premade**: AUTO-ESCALATION → auto-replan as a recovery rung when a run is stuck pre-execute and
  recovery targets exist; bounded retries then escalate/flag.
- **mechanism**: auto-escalation. **rationale**: replan is computable recovery, not a taste call.

### C13 (RUNTIME) F7 `resume-clarify` — operator resumes after a prep clarification
- **premade**: AUTO-ESCALATION → folded into C7: once clarify auto-resolves (or best-guess+flag fires),
  resume-clarify is invoked automatically from `STATE_AWAITING_HUMAN → STATE_PREPPED`.
- **mechanism**: auto-escalation. **rationale**: the resume is mechanical once the answer exists; tie to C7's auto-answer.

### C14 (RUNTIME) F7 `set-robustness` / `set-profile` / `set-model` — operator reconfigures mid-run
- **premade**: DEFAULT → these stay available as control ops but are NOT in the autonomous critical path;
  the auto-escalation ladder uses `set-model` (escalate to stronger model) automatically. Manual
  reconfiguration is operator-optional, never blocking; runs proceed on the profile/robustness already set.
- **mechanism**: auto-escalation (for set-model) + default (others optional). **rationale**: model-escalation is the auto rung; profile/robustness changes are conveniences, not gates.

### C15 (RUNTIME) F7 `add-note` — operator annotation
- **premade**: DEFAULT → purely additive annotation; never blocking; agents emit notes automatically.
- **mechanism**: default. **rationale**: annotation has no halt semantics.

- **mechanism**: default. **rationale**: brief leans this and the cloud-arc memory (`project_cloud_arc`: core owns capability, cloud wraps). Reversible boundary statement, not irreversible.

### D2 (design) Open-Q#2: one supervisor contract for chain+bakeoff, or `select`-at-run-granularity distinct
- **premade**: DEFAULT → one supervisor tier, two variants; bakeoff's reduce IS M2 `select` at run
  granularity (the brief's locked decision §85-86). No distinct contract.
- **mechanism**: default. **rationale**: locked decision already says one tier/two variants; select is the reduce.

### D3 (design) Open-Q#3: PR-merge wait binds onto `awaiting_human`+F6, or stays supervisor-local cursor
- **premade**: DEFAULT → bind `STATE_AWAITING_PR_MERGE` onto M5c's `awaiting_human` outcome + F6
  pause/resume (uniform pause surface), NOT a separate supervisor-local cursor.
- **mechanism**: default. **rationale**: one pause surface is simpler and reuses F6; consistent with the run-outcome eviction.

### D4 (RUNTIME) PR-merge wait — `STATE_AWAITING_PR_MERGE`: run halts for a human to merge the PR
- **premade**: AUTO-ESCALATION → product default is auto-merge on green: when the PR's CI/parity gates are
  green, auto-merge (or fast-forward the branch) and resume; only emit `awaiting_human` if the merge gate
  is red after the retry ladder. Note `project_worktree_carry_breaks_pr_isolation`: re-run off clean main
  when a reviewable isolated diff is needed — but that's an operator convenience, not an autonomous block.
- **mechanism**: auto-escalation. **rationale**: merge is gated on green CI, which is machine-checkable; the human merge becomes auto-merge-on-green with red→escalate.

### D5 (RUNTIME) `ChainSpec.escalate_action` default `"force-proceed"` (control op invoked BY NAME)
- **premade**: DEFAULT → replace the literal `"force-proceed"` default with a GENERAL target from M5c's
  vocabulary (`force-advance`); the supervisor reads `valid_targets(run_state)` and calls
  `apply_transition(target)`. Default escalate policy = auto-ladder (retry→re-route→force-advance→abort).
- **mechanism**: machine-gate (enforced by D7 grep) + default policy. **rationale**: locked decision + done-criterion already mandate removing the literal; the default ladder is autonomous.

### D6 (RUNTIME) chain branches on `blocked` planning verdict → invokes `recover-blocked`
- **premade**: AUTO-ESCALATION → the supervisor branches on the general `blocked` run-OUTCOME (not
  `STATE_BLOCKED`) and auto-invokes `recover-from-stuck` (general op) with a bounded retry ladder
  (>1 retries per `project_chain_blocked_retry_and_resume`), then escalate/abort.
- **mechanism**: auto-escalation. **rationale**: converts the blocked-execute recovery into an automatic policy on the general outcome.

### D7 (build-gate) Acceptance: no supervisor code imports/branches on `STATE_*`; no op invoked by literal name
- **premade**: MACHINE-GATE → CI grep gate (SYNTHESIS §357 acceptance check at run granularity); the
  `chain/__init__.py:84` import gone, `:308` literal gone. Red = halt.
- **mechanism**: machine-gate. **rationale**: deterministic grep; mirrors M2/M5b/M5c gates.

### D8 (build-gate / acceptance) The canary epic — F8's ONLY honest acceptance
- **premade**: MACHINE-GATE → a throwaway multi-milestone chain of NON-planning M6-discovered packs (e.g.
  `select` tournament or `run(cmd)` oracle), ≥1 dependency edge, ≥1 induced per-run failure exercising
  escalate/recover, driven end-to-end by the new supervisor. Automated; green = accept. The real epic
  CANNOT self-host its supervisor mid-rebuild (SYNTHESIS #10) — so the canary is the substitute, no human
  judging required.
- **mechanism**: machine-gate. **rationale**: brief mandates the canary as the automated honest acceptance; replaces any human "looks good" sign-off.

### D9 (build-gate) Epic-level strangler liveness invariant (OLD engine boots + drives throwaway; NEW pieces run a planning-shaped plan)
- **premade**: MACHINE-GATE → gated EVERY milestone (EPIC §164); M5d is the most likely to break it
  because it rebuilds the strangler's own driver. Red = auto-halt the milestone.
- **mechanism**: machine-gate. **rationale**: liveness is an automated boot+drive test; the freeze-list keeps the OLD engine driving until the canary proves the NEW supervisor.

### D10 (build-gate) "auto.py loop buys" (context-retry, idle-kill, worktree isolation) preserved on M3 driver
- **premade**: MACHINE-GATE → characterization on a real run asserts no regression; idle-kill/retry are
  M3-driver config (pinned), not hardcoded.
- **mechanism**: machine-gate. **rationale**: regression is testable against recorded traces.

## must_ask_peter summary

**ZERO** entries set `must_ask_peter=true`. Every design-time Open-Q has a brief-stated lean adopted as a
DEFAULT; every build check is a deterministic MACHINE-GATE; every runtime human-gate (clarify, verify-human,
the 9 override actions, PR-merge wait, escalate-by-name) is converted to AUTO-ESCALATION (retry → stronger
model → best-guess/skip+flag → abort) or a MACHINE-GATE (auto-merge-on-green, criteria oracle). None is an
irreversible strategy/taste call: all the boundary statements (C1, D1, D3) are reversible interface
decisions, and all runtime gates retain the underlying capability (the pause/resume hook still ships) while
defaulting to autonomous progress.
