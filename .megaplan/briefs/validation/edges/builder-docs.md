# Builder documentation — what an external developer needs to build a custom Arnold module

**Status:** Edge design for `pipeline-unification-EPIC.md`. Scope: the *documentation set* (and the
docs-in-action tooling) an outside developer needs to ship a module on the Arnold SDK without reading
megaplan internals. The EPIC's whole premise is "**others build modules**" (`EPIC.md:9-15`,
acceptance #1 the "load-bearing test"). Today the plan ships SDK *pieces* and a contract *checker* but
**no builder-facing docs** — so the acceptance toy can only be built by someone who already knows the
codebase. That is the gap this brief closes.

## Design principle (mirrors the EPIC's own discipline)

The SDK already separates *general capability* (core owns it) from *domain content* (the package owns
it). Builder docs follow the same split along a second axis: **generated** (mechanically emitted from the
typed surface, can never drift) vs **authored** (prose a human writes once). The rule, stated up front so
every doc below inherits it:

> **If a fact is also expressed as a type, an enum, a manifest field, or a node signature, the doc that
> states it is GENERATED from that source — never hand-copied.** Prose explains *why* and *when*; types
> generate *what* and *which fields*.

This is the same anti-drift move the EPIC makes for the executable artifact (`chain.yaml↔EPIC↔briefs
lint`, `EPIC.md:137`) and the CI grep gate (`ZERO GateRecommendation in SDK modules`, `EPIC.md:140`).
The reference docs are a third member of that "regenerate as one triple" family.

## The documentation SET

Seven docs + the scaffold/checker tooling. Each is named, sited, given a purpose, and classed
generated/authored.

### 1. `docs/arnold/authoring-guide.md` — "Build a custom module in N steps" — **AUTHORED**

The on-ramp. The only doc a builder reads start-to-finish. A linear tutorial that ships the
select-tournament toy (worked example #2 below) as the running example, in ~6 steps:

1. **Scaffold** — `arnold pipelines new my-module --driver graph` (creates a green starter, see §6).
2. **Write the manifest** — declare `name`, `driver`, `arnold_api_version`, the capabilities you
   consume (`dispatch`, `emit`, `evidence`, `state`), entrypoint. (Spec → doc #3.)
3. **Pick a driver** — substrate (`in_process | subprocess_isolated`) × topology (`graph | loop`)
   (`EPIC.md:123-124`). The guide gives a 2×2 decision table; the per-driver contract lives in doc #2.
4. **Declare Ports** — every step's `produces` / `consumes` with typed `Port = (name, kind, schema,
   cardinality, version)` (`EPIC.md:96-101`). The guide teaches *why* build-time port resolution exists
   (it kills the `step_helpers.py:104` silent `v1.md` fallback); the *list* of port kinds is generated
   (doc #2).
5. **Compose nodes** — wire `produce → judge → gate → revise` from the node library using the
   `PipelineBuilder` fluent API (`builder.py:122-400`: `.input/.agent/.panel/.gate/.iterate/.escalate`).
6. **Write your `SKILL.md`** — the required package element (`EPIC.md:37,178`), with frontmatter +
   "when to invoke" + inputs + usage. Templates lifted from `epic-blitz/SKILL.md` and
   `writing-panel-strict/SKILL.md`.
7. **Check & run** — `arnold pipelines check` (the feedback loop, §6) → `arnold run my-module`.

The guide *ends* by pointing at the three worked examples (doc #4) and the reference (doc #2). Authored
because it is pedagogy and ordering — there is no type to emit it from. It carries an **embedded
doctest**: the exact scaffold + check + run commands are runnable in CI so the tutorial can't rot.

### 2. `docs/arnold/reference/` — Piece & node reference — **GENERATED** from the typed surface

The encyclopedia, and the doc most at risk of drift — so it is emitted, not written. One page per
surface, generated by a new `arnold pipelines docs` emitter that walks the SDK's typed surface and
introspects signatures + docstrings:

- **`reference/pieces.md`** — the four injected capabilities, each with its interface + Ports:
  - `dispatch` — interface + the two backends (`subprocess-cli` from planning, `async-api` from
    resident; `EPIC.md:31`); key-pool, cost, stall/liveness contract.
  - `state` — the **Store** (durable, transactional, leased) + the `StateDelta` write vocabulary
    (`replace | accumulate | deep-merge`, CAS not LWW; `EPIC.md:99-101`).
  - `emit` — `EventSink.emit(kind, payload, scope)` + backends (`events.ndjson`, Store `EpicEvent`;
    `EPIC.md:33`).
  - `evidence` — the injected strategy (`git | audit | none`) and the attestation-vs-oracle split
    (`run(cmd)→{exit,stdout,stderr}`; `EPIC.md:74,148`).
  - `config` — the N-layer precedence resolver (`EPIC.md:110,148`).
  Generated from each piece's Protocol + its docstring (e.g. `types.py:167` `Step` Protocol pattern).
- **`reference/nodes.md`** — the node library: `produce / judge / gate / revise / fan_out / reduce /
  select / escalate / clarify / verify` + macros (`critique_revise_gate_loop`, `panel_parallel`,
  `iterate_until`, `majority_vote`, `phase_zero_gate` — the real exports in `patterns.py:40-58`). Each
  node renders: signature, its `consumes`/`produces` Ports, docstring, and (post-M2) **whether its
  return type is app-neutral** — this page is where the "no `GateRecommendation` leak" guarantee
  becomes visible to a builder (today `JoinFn` returns `GateRecommendation`, `pattern_types.py:19`, the
  documented smoking gun `EPIC.md:63-64`; the generator's source-of-truth status makes the M2 fix
  legible here).
- **`reference/drivers.md`** — the 2-axis driver matrix (substrate × topology), each cell's contract:
  what state-evolution it allows (`forward-only | reversible` + `restorable_boundary`, `EPIC.md:128`),
  isolation guarantees, teardown semantics. Generated from the driver registry.
- **`reference/control-vocabulary.md`** — run-outcome enum (`succeeded | failed | escalated | blocked |
  awaiting_human`) + the `valid_targets/recover_targets` control trio (`EPIC.md:111-114`). Generated
  from the control interface so the "no `STATE_*` as mechanism" rule (`EPIC.md:160`) is enforced in docs
  too.

**Yes — these MUST be generated from the typed surface.** Rationale beyond anti-drift: the EPIC adds
Ports, `StateDelta`, `select`/`Reduce[T]`, the 2-axis driver, and the control vocabulary *across M2–M5c*.
Hand-written reference pages would be stale the day each milestone lands. A `--check` mode (CI) re-emits
and diffs against the committed pages — same pattern as the parser-snapshot test (`c493f629`).

### 3. `docs/arnold/package-contract.md` — manifest schema spec — **GENERATED** (schema) + **AUTHORED** (prose)

The contract every package must satisfy. The **field table is generated** from the manifest
Pydantic/dataclass model (so it can't drift from the validator the loader actually runs); the **prose
around each field is authored**. Fields, per `EPIC.md:37,116-117`:

- `name`, `driver` (the 2-axis value), `entrypoint` (the `build_pipeline()` callable, the existing
  discovery contract `registry.py:399`).
- `arnold_api_version` — the SemVer the package was built against; discovery refuses an incompatible
  major **without importing** (`EPIC.md:116`).
- declared `capabilities` — which SDK pieces the package consumes (drives the per-package quota
  sub-budget, `EPIC.md:117`).
- `SKILL.md` requirement — presence is a hard contract element, surfaced by the registry's
  `read_skill_md` (`registry.py:137-167`); a package missing it fails `check`.
- `trust_tier` — `in_tree | blessed | quarantined` (`EPIC.md:117`); gates whether discovery will
  `exec_module` (manifest-first, **non-executing** discovery; `EPIC.md:115-118`).

The "trust boundary" prose (why discovery reads the manifest before importing, what `quarantined`
means operationally) is authored — it is policy, not schema.

### 4. `docs/arnold/examples/` — Worked examples — **AUTHORED** (with runnable sources in-tree)

Three, deliberately spanning the EPIC's acceptance shapes. Each doc is a literate walkthrough whose code
blocks are *extracted from a real, CI-run package* (not retyped), so they stay correct:

- **`examples/jokes.md`** — the simple `graph` pipeline (EPIC acceptance #2, `EPIC.md:84-85`). Walks the
  `creative` pack (`pipelines/creative/__init__.py`) as ~50 lines of domain code + "I'm a `graph`
  driver, I need `dispatch`+`emit`." The "a new pipeline is cheap" proof.
- **`examples/select-tournament.md`** — **the load-bearing non-planning toy** (EPIC acceptance #1,
  `EPIC.md:80-83`). A bracket tournament built on `select(items, rule)→(winner, losers, scores)` and a
  data-valued `reduce` — the two P0 primitives planning never needed (`EPIC.md:66-68`). Chosen because
  it is forward-shaped-but-NOT-verdict-routed, so it exercises the de-planning of the types. This is the
  example the acceptance test (below) reproduces.
- **`examples/planning-as-composition.md`** — the "beautiful example" (EPIC acceptance #3,
  `EPIC.md:86-88`). Reads megaplan-planning itself as a composition: "`iterate` is just planning's
  binding of `revise_in_place`," the 4 verdicts are planning's `reduce` output mapped to gate
  consequences — **not** the SDK's only reduce output. Authored so a reader can point at planning and
  see the pieces, proving planning has no privilege (`EPIC.md:44-54`). This doc is only honest *after*
  M2+M6; it doubles as the visible proof that the de-planning landed.

### 5. `docs/arnold/skill-integration.md` — how a module plugs into the skill system — **AUTHORED**

Explains the two skill layers and what is generated vs authored in each:

- **Per-package `SKILL.md`** (authored by the builder): lives beside the package
  (`<pkg>/SKILL.md` for package modules, `<pkg-cli-name>/SKILL.md` for sibling-file modules —
  the exact layout the registry already resolves, `registry.py:144-167`). Surfaced at runtime via
  `read_pipeline_skill_md` so the host agent can read a discovered module's skill **without the SDK
  shipping a hardcoded skill for it**. This is the discovery-driven surface.
- **Umbrella `~/.claude/skills/arnold*` docs** (generated): the same `*_skill.md` → composed-bundle →
  installed-skill pipeline that exists today (`cli/skills.py:120` `_canonical_composed`,
  `:180` `handle_regen_composed`; sources in `megaplan/data/*_skill.md`, composed under
  `data/_composed/`, installed to `~/.claude/skills/megaplan*`). A new `arnold` umbrella skill is a
  source `arnold_skill.md` composed/installed the same way; per-package SKILL.md is **not** baked into
  the umbrella — it is discovered live. The doc states the boundary: *umbrella = how to drive the SDK
  and discover modules (generated/installed); per-package SKILL.md = what one module does (authored,
  discovered)*.

### 6. Diagnostic / scaffold tooling — docs-in-action

The builder's feedback loop. These already exist as EPIC items (`EPIC.md:119-121`); this brief frames
them as documentation that executes, and pins their doc home (`docs/arnold/tooling.md`, authored
overview; behavior generated into the reference where it has a schema):

- **`arnold pipelines new <name> --driver <d>`** — scaffolds a **green starter module**: manifest with a
  pinned `arnold_api_version`, a one-stage `graph` pipeline, a Ports stub, a SKILL.md template, and a
  smoke test. "Green starter" = `check` passes immediately, `run` produces an artifact. This is the
  authoring guide's step 1 made real; it is the *fastest possible* path to a working module.
- **`arnold pipelines check`** — the **contract checker as the inner loop**: statically proves the
  composition is wired — every `consumes` resolves to an upstream `produces`, every edge targets a real
  stage, every gate verdict has an edge (`EPIC.md:119-120`). Exit non-zero with a precise mis-wire
  message. The doc shows the error catalogue (which is generated from the checker's own error codes).
- **`arnold pipelines doctor`** — **discovery diagnostics**: per-path `discovered ✓ / rejected +
  traceback / skipped` (`EPIC.md:121`), killing the silent-vanish failure mode where a broken module
  just doesn't appear (today discovery swallows import errors, `registry.py:337-339`). This is how a
  builder learns *why* their module isn't listed.

## Doc home & generation summary

All builder docs live under **`docs/arnold/`** (a new tree, sibling to the existing `docs/pipelines.md`).
Per-package SKILL.md lives **in each package**. Umbrella skill source lives in **`megaplan/data/`** and
installs to **`~/.claude/skills/arnold*`**.

| Doc | Home | Class | Source of truth |
|---|---|---|---|
| authoring-guide.md | `docs/arnold/` | Authored | human (CI-doctested commands) |
| reference/pieces,nodes,drivers,control | `docs/arnold/reference/` | **Generated** | typed surface (Protocols, node sigs, driver registry, control enum) |
| package-contract.md | `docs/arnold/` | Generated table + authored prose | manifest model |
| examples/*.md | `docs/arnold/examples/` | Authored, code extracted from in-tree packs | the real packages |
| skill-integration.md | `docs/arnold/` | Authored | — |
| tooling.md | `docs/arnold/` | Authored overview; error codes generated | checker/doctor exit codes |
| per-package `SKILL.md` | each package | **Authored by the builder** | the builder |
| umbrella `arnold` skill | `data/` → `~/.claude/skills/arnold*` | **Generated/installed** | `arnold_skill.md` via `handle_regen_composed` |

The generated docs gain a **`--check` CI mode** (re-emit, diff against committed) so the reference and
contract tables can't silently diverge from the types — joining the EPIC's existing anti-drift gates.

## Which milestone owns this

**It needs its own milestone — propose `M7 — Builder documentation & onboarding`, gated on M6.**

Reasoning: the docs are only *honest* once the surface they describe is final. The reference is
generated from the typed surface that is still moving through M2 (Ports, `select`/`Reduce`), M3
(2-axis driver, state-evolution), M4 (the four pieces + control-observability), and M5c (control
vocabulary); `planning-as-composition.md` is only true after M6 drops `_BUILTIN_NAMES` and relocates
planning. Writing builder docs before M6 would document a surface that the next milestone rewrites —
exactly the "arriving later rewrites code" hazard the EPIC flags for the Port (`EPIC.md:101`).

It does **not** ride M6 as a sub-item: M6 is already the heaviest milestone (relocation + trust boundary
+ next-step collapse), and the docs' acceptance test (an *external* builder shipping a module) is a
distinct, expensive validation that deserves its own gate. The *generators* (doc emitter, `--check`
mode) can be scaffolded incrementally from M2 onward beside each type addition — but the **committed,
acceptance-tested doc set is M7**. M1 already owns `pipelines check/doctor/new` as code (`EPIC.md:137`);
M7 owns them *as documentation* and wires their behavior into the authoring guide.

## Acceptance test (build it, don't assert it — per `EPIC.md:79`)

> **An external builder, given ONLY `docs/arnold/` + the scaffold tooling (no access to megaplan
> internals, no SDK author on call), ships a working non-planning module — the select-tournament toy —
> that `arnold pipelines check` passes and `arnold run` executes to a real artifact.**

Concretely, executable in CI as the M7 gate:

1. Stand up a clean checkout exposing only `docs/arnold/` + the `arnold` CLI (no source browsing — e.g.
   a subagent or a fresh contributor handed just the docs).
2. Builder runs `arnold pipelines new tournament --driver graph`, follows the authoring guide to wire
   `select` + data-valued `reduce` into a bracket, authors a `SKILL.md`.
3. `arnold pipelines check` exits 0; `arnold pipelines doctor` lists the module `discovered ✓`.
4. `arnold run tournament` produces the winner artifact.
5. **Hard bar (the load-bearing part):** the builder used **zero** planning vocabulary — no
   `GateRecommendation`, no `STATE_*`, no 4-verdict enum — proving the docs describe a *general* SDK,
   not planning with the serial numbers filed off. A grep over the builder's module asserts this,
   mirroring the M2 CI grep gate (`EPIC.md:140`) and the "no binding carries `STATE_*`" proof
   (`EPIC.md:160`).

This is EPIC acceptance #1 (`EPIC.md:80-83`) made *operational through the docs path*: the existing
acceptance proves the SDK *can* express a fourth thing; this proves a stranger can *discover how to*
from the documentation alone.
