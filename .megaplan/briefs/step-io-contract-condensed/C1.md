# C1: Carrier Reconcile + Wire Enforcement at the Data Seam

Condenses old m0a / m0b / m1 / m2 (the data-plane half) into one sprint. Bases off
`arnold-generalized-pipeline @ 944d10cc` — the completed generalized-pipeline migration that
already BUILT the foundation generically. C1 builds only the genuine remainder.

## Outcome

The migration delivered the seam-contract foundation as TYPES and DECISION machinery but left four
load-bearing wiring gaps. C1 closes them so the step⇄step data seam is actually enforced, the disk
seam is a generic chokepoint, the two carriers form one thin shared core, and the `warn` tier stops
being dead weight.

1. **Reconcile the two carriers as TWO LAYERS, not a data merge.** `ContractResult`
   (`arnold/pipeline/types.py:732`) is the DATA seam primitive (payload + provenance/freshness/
   evidence/suspension). `StepIOEnvelope` + `StepIO{Operation,Classification,Diagnostic,Decision}`
   (`arnold/pipeline/step_io_contract.py:51/25/30/85/112`) are the POLICY/CLASSIFICATION layer that
   DECIDES/validates a handoff. The "thin shared core" is making the decision layer operate OVER
   `ContractResult` as its payload — a smaller, cleaner delta than collapsing two carriers. NOT a
   data-type merge.
2. **Wire typed `produces`/`consumes` enforcement at the executor data seam.** `arnold/pipeline/
   executor.py` references the contract only in a docstring (it "stores the contract, never inspects"
   it) — produces/consumes ports are NOT enforced. Wire enforcement at the executor when both sides
   are typed, routing through the already-built shadow/warn/enforce decision machinery.
3. **Lift the single disk-seam validation chokepoint to generic `arnold/pipeline/`.** The
   `read_artifact_json` / `write_artifact_json` chokepoint that already runs `decide_step_io_read` /
   `decide_step_io_write` lives at `arnold/pipelines/megaplan/store/plan_repository.py:189/239` —
   megaplan-coupled. Lift the generic disk-seam validation into `arnold/pipeline/` (next to the
   existing `arnold/pipeline/artifacts.py`) so the chokepoint is a platform primitive, leaving only
   megaplan-specific policy resolution behind in the megaplan adapter.
4. **Give the dead `warn` tier a distinct action.** `arnold/pipeline/step_io_policy.py:25` defines
   `CONTRACT_MODE_WARN` with a `.warns()` predicate (:60) and eligibility-gating (:123), but `warn`
   currently behaves IDENTICALLY to `shadow` (both log-only, no distinct surfacing). Wire `warn` to
   emit a distinct `StepIODiagnostic` action — the real middle tier between silent `shadow` and
   blocking `enforce`.
5. **Delete legacy dict-building handoff paths** at each step→step seam taken onto typed ports, so
   `StepResult.outputs` stops being a free-form `Mapping` where a typed `ContractResult` payload now
   carries the handoff.

This sprint changes no behavior on un-typed seams; enforcement is opt-in per seam via the existing
mode ladder and only fires when both sides are typed.

## Scope

IN:

- **Thin-core reconcile**: make the `StepIOEnvelope` decision layer carry/operate over
  `ContractResult` as the payload it classifies and validates, so there is ONE data carrier
  (`ContractResult`) and ONE decision/policy layer over it — not two redundant data structures.
- **Executor port wiring**: declare typed `produces`/`consumes` on the load-bearing pipeline stages
  using the existing `Port`/`PortRef` + `reads`/`writes`/`ReadRef`/`WriteRef` vocabulary
  (`arnold/pipeline/types.py:300/329/217-218/248-249`); make the executor enforce the step→step
  contract when BOTH producer and consumer are typed, routing through the existing
  `step_io_policy`/`step_io_contract` decision machinery. The generic executor must keep "no
  forbidden vocabulary / no megaplan imports."
- **Chokepoint lift**: move the generic disk-seam read/write validation into `arnold/pipeline/`
  (generic chokepoint) and have `plan_repository.py` consume it; the megaplan side keeps only
  `resolve_megaplan_step_io_policy` / `step_io_policy_adapter` / `schema_registry_adapter` /
  `pipeline_contracts` specifics.
- **Warn-tier diagnostic**: `warn` emits a distinct `StepIODiagnostic` (a surfaced, non-blocking
  warning) — observably different from `shadow` (silent log-only) and from `enforce` (blocking). A
  test proves the three tiers produce three distinct behaviors.
- **Legacy path deletion** at each newly-typed seam: remove the free-form dict-building handoff so a
  typed seam carries a `ContractResult` payload, not a loose `Mapping` (per-site delete, not
  big-bang; un-typed seams untouched).
- Per-seam shadow-first rollout via the existing mode ladder; promotion driven by the existing
  telemetry (`step_io_telemetry.py`).

OUT:

- The model seam (render/capture/tokenizers/audit) — that is C2.
- Suspension-aware composition and resume — that is C3.
- Authoring-API enforcement of `reads`/`writes` and the acceptance gate — that is C4.
- Any change to the `ContractResult` TYPE, the schema registry, the validator, or the mode-ladder
  semantics — all already built; C1 consumes them unchanged.
- Bulk migration of all 33 IO sites — C1 wires the data-seam MECHANISM + load-bearing stages; the
  long-tail migration delta folds into C4.

## Locked Decisions (carried from the prep dossier jury)

- **Thin-core, not data-merge** (LIVE-MIGRATION RECONCILIATION finding): the two carriers are two
  LAYERS — `ContractResult` = data, `StepIOEnvelope`+`StepIO{...}` = policy/classification. The
  decision layer operates OVER `ContractResult`. This is the key C1 reframe.
- **Single validation chokepoint** at the disk⇄engine seam; C1 makes it GENERIC (the migration left
  it megaplan-coupled in `plan_repository.py`).
- **Enforce only when BOTH producer and consumer are typed** — gradual typing; the 33 sites migrate
  incrementally, never one breaking flip. Eligibility rule already wired in `step_io_policy.py:123`.
- **Seam identity = the 5-tuple port-binding crossing** `(pipeline_id, producer_step, producer_port,
  consumer_step, consumer_port)`, keyed `pipeline_id::consumer.port<=producer.port`;
  content_type/path/edge/phase are telemetry, NOT identity; a failed `binding_map` lookup = legacy
  untyped read (seam `None`), never enforced. (Resolved decision #2.)
- **Schema = canonical JSON Schema, content-addressed, version-retained**; `schema_version =
  sha256:<hash of canonical>`; validate against the schema-in-effect-when-written. (Resolved
  decision #1 — already realized by the migration's `schema_registry.py`; C1 consumes it.)
- **Read-lenient / write-strict**, gated by the artifact's own `schema_version`, so frozen/in-flight
  plans stay readable on resume/status.
- **`warn` ≠ `shadow`**: the jury's "dead warn tier" refinement — `warn` gets a distinct
  `StepIODiagnostic` action, it is not absent and not a silent no-op (LIVE-MIGRATION RECONCILIATION).
- **Dogfood kill-switch** (pre-mortem risk 4): the global `read_lenient_escape` drop-to-shadow is
  already built; C1 must keep it intact when lifting the chokepoint, and the engine self-validates
  its OWN artifacts before any data seam is allowed to reach enforce.
- **Per-seam shadow-first rollout**, telemetry-driven promotion; no seam jumps straight to enforce.

## Constraints

- No behavior change on un-typed seams; every newly-typed seam enters shadow first.
- Frozen/in-flight plans must stay readable on resume/status (read-leniency gated by the artifact's
  own version); a regression test must resume a pre-contract plan.
- The lifted generic chokepoint must not import `megaplan` and must keep the executor's "no forbidden
  vocabulary" invariant.
- The dogfood kill-switch must survive the chokepoint lift (a validator bug must never wedge the
  self-hosted engine).
- Bases on `944d10cc`; must not modify the `ContractResult` type, the schema registry, the validator,
  or the mode-ladder semantics.

## Done Criteria

1. The `StepIOEnvelope` decision layer operates OVER `ContractResult` as its payload (one data
   carrier + one policy layer); a test shows a handoff classified/validated through the decision
   layer carries a `ContractResult`, with no second redundant data structure.
2. The executor enforces the step→step contract when both sides are typed (routing through
   `step_io_policy`/`step_io_contract`); a wrong-shaped handoff is rejected in enforce mode and a
   one-side-untyped seam passes through unchanged (gradual typing) — tests prove both.
3. The disk-seam validation chokepoint is GENERIC in `arnold/pipeline/` (no `megaplan` import);
   `plan_repository.py` consumes it; a pre-contract plan still resumes/loads (regression test).
4. `warn` emits a distinct `StepIODiagnostic` action, observably different from `shadow` (silent
   log-only) and `enforce` (blocking); a test proves three distinct tier behaviors.
5. Each newly-typed seam carries a typed `ContractResult` payload on `StepResult` instead of a
   free-form `Mapping`, and the legacy dict-building handoff path at that site is DELETED (gone, not
   bypassed); a test confirms.
6. The dogfood kill-switch (`read_lenient_escape` → drop every seam to lenient reads) still works
   after the chokepoint lift; the engine validates its OWN artifacts before any seam promotes to
   enforce — tests prove both.
7. Per-seam shadow→warn→enforce promotion is telemetry-driven via the existing `step_io_telemetry`;
   a seeded violation is attributed to the correct 5-tuple seam (test).
8. No model-seam, suspension, or authoring-API work is done; no change to the `ContractResult` type,
   schema registry, validator, or mode-ladder semantics.

## Rubric

- Profile: `premium`
- Robustness: `thorough`
- Depth: `high`

Rationale: C1 touches the import topology (lifting the chokepoint to generic `arnold/pipeline/`
without a megaplan import) and the contract boundary (wiring executor enforcement, reconciling the
two carriers into one thin core). A wrong cut here either re-couples the platform to megaplan or
bricks a live/frozen plan on the first enforce flip. The hard design is already settled upstream by
the migration, but the wiring is load-bearing across every pipeline, so it earns premium/thorough/
high.
