## 2. Evidence

The write path (`saveTimeline`, line 498) posts to the append service with `expected_version` (the `config_version`). The read path (`loadDbHead`, lines 299–308) queries `timeline_events.version`. These are distinct sequences.

3. **The sync protocol has no "table missing" state.** `AppSyncState` (lines 58–64) defines six states, none of which represent "event infrastructure missing." When both `dbHead.version === 0` and no bookmark exists, `syncTimeline` (lines 555–592) falls through all branches and returns `bookmark_incompatible` — which is the same state used when two valid bookmarks genuinely conflict.

4. **The `DataProvider` interface is unaware of `timeline_events`.** `DataProvider.ts` (lines 44–56) exposes `loadTimeline`, `saveTimeline`, `loadAssetRegistry`, etc. — none of which surface event-log health. The append service is now the sole write path, meaning every save must go through the Python service that populates `timeline_events`. A failure in that pipeline leaves the browser with no fallback write path.

6. **No corrective reload path exists.** `useEditorSync.ts` lines 97–129 maps `syncTimeline` results to UI states, but `bookmark_incompatible` only sets a warning — it never triggers a reload or recovery attempt. The user is stuck with a non-actionable state.

7. **The append-service response includes a `db_head` for bookmark update (line 531), but the sync's `loadDbHead` re-queries the table directly (line 539).** If the append service wrote the event but the direct SELECT misses it (e.g., replication lag, RLS misconfiguration), the sync sees stale data while the bookmark is up-to-date — a phantom `both_advanced` or `bookmark_incompatible`.

10. **Test coverage is narrow.** The `loadTimeline` test at line 199 verifies the fallback lets loading proceed, but no test verifies what `syncTimeline` returns when `loadDbHead` hits 42P01 during a sync operation. The sync tests always mock `timeline_events` as present.

## 3. Gap/Risk

**Product consequence:** If the `timeline_events` migration is absent, delayed, or rolled back on a production Supabase instance, the sync button silently degrades to `bookmark_incompatible` with no user-facing explanation. The editor remains usable for local editing and saves (since `saveTimeline` goes through the append service), but multi-device sync is permanently broken with no recovery path short of a full page reload or database migration. The user never sees an error that says "event log not available" — they see a generic bookmark conflict.

**Dev consequence:** The two-version-system architecture (`config_version` for CAS, `event_log.version` for sync head) means that any inconsistency between the append service's event emission and the materialized `timelines` row creates an unrecoverable sync state. A developer debugging a sync failure must understand that `dbHead.version` (from `timeline_events`) and `configVersion` (from `timelines`) are separate counters, and a mismatch between them is the root cause of many `bookmark_incompatible` states — but this relationship is undocumented outside the SQL migration comment.
