**Neither claim is overstated.** The `installedPackages` naming genuinely risks marketplace conflation (even if spec-guarded), and the boolean collapse is real, not theoretical. The M5 brief's Locked Decisions already identify the collapse as a problem to solve, confirming wave one's read.

**Minor overstatement in original claim**: wave one said the loader "collapses into `loaded=false`" as if no recovery path exists. In practice, the `diagnostics` array on `InstalledPackageState` does carry the reason â€” disabled packages have empty diagnostics, invalid ones carry errors. The collapse is UX-awkward but not information-destructive.

### 2. Add explicit `status` field to `InstalledPackageState`

Replace the overloaded `loaded: boolean` with a discriminator:

```ts

  | 'incompatible' // api_version_incompatible specifically (subset of invalid)

export interface InstalledPackageState {

  state: ExtensionState;
  diagnostics: ExtensionDiagnostic[];

}
```

The loader already knows the rejection reason at every branch. Deriving `status` is mechanical:
- Validation fails â†’ `'invalid'` (or `'incompatible'` if code is `api_version_incompatible`)
- Duplicate ID â†’ `'duplicate'`
- `state.enabled === false` â†’ `'disabled'`
- Otherwise â†’ `'loaded'`

M5 brief Locked Decisions call for this explicitly; making it first-class eliminates the fragile `diagnostics.length === 0` heuristic.
