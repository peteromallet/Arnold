## Rejected Or Overstated

**Wave1's claim that `set_params`, `set_theme`, `set_theme_overrides` bypass proposal policy is false.** These route through `executeCommand` (`loop.ts:500-533`) with `timelineMutationMode` correctly forwarded. `executePreparedTransaction` (registry.ts:464) and `handleRepeat`/`undo` subpaths all check `timelineMutationMode === "propose"` and return dry-run proposals. The claim of "shortcut bypass" for these three is incorrect — they respect policy correctly when it reaches them.

**The inlined `MANIFEST_SCHEMA` drift (Finding 3 from wave1 contract drift) is over-weighted.** It's dead code — `validateManifestSchema()` never references it. Deleting it is cleanup, not a contract integrity fix. The actual comparison should be TypeScript types ↔ JSON schema file ↔ runtime validator ↔ docs.

**Wave1's `create_task` bypass claim is overstated.** `create_task` creates generation-queue tasks, not timeline config mutations. It's debatable whether task creation falls under timeline-mutation policy. The brief's "Out" section explicitly excludes "Full extension-contributed agent tools."

## Required Plan Changes

**Resequence: Move the drift gate before M1 can be declared done.** Do not allow M1 to close until `test:extensions` includes a cross-source comparison script (`scripts/quality/check-extension-contract-drift.mjs`) that parses `contributionFamilies.ts` statuses, the JSON schema's allowed collections, `compatibility.md`'s family table, and `extension.ts` exports — failing on disagreement. This must be the blocking golden fixture.

**Resequence: M2 must deliver the Supabase DDL and `DataProvider` extension method signatures, not just factory contracts.** Without concrete `saveExtensionState`/`getExtensionStateRepository` method signatures on `DataProvider` and migration files for `extension_state`/`extension_settings`/`timeline_proposals` tables, M3 inherits undefined contracts. The blocking test: a provider conformance suite that actually writes/reads/reloads extension state through Supabase (not just capability flag assertions).

**Add concrete blocking tests for M3:** (1) An integration test that sends `proposal_policy: 'always'` from frontend through the edge function and asserts the response includes a non-null `proposal_response` and the timeline config version is unchanged. (2) A test that `create_shot` in proposal mode returns a proposal instead of inserting. (3) A test that proposal envelopes survive a reload (provider-backed persistence).

**Add `removeByExtensionId` and crash-reset to M4's done criteria with golden fixtures.** The diagnostics store must gain `removeByExtensionId`; `ExtensionRenderBoundary` must accept a `resetKey` prop tied to extension load state; extension-render source must call `replaceBySource('extension-render', [])` on disable. The blocking test: crash → disable → re-enable → fresh render succeeds (not just config/settings correctness).

**Split M5's readiness artifact from its manager UI.** The Phase 4 readiness gate (render planner contract, ToolRegistry stub, Lifecycle contract, per-family schema reservations) is a blocking dependency for Phase 4 families — not just a deliverable of the last foundation milestone. It should be extracted as a lightweight `M5a` contract artifact, with the manager UI as `M5b`. This prevents the readiness gate from fake-passing as "we wrote a doc" while the actual interfaces future families depend on remain undefined.
