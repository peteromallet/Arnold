**Grounding (read before scoping):** `interrogation/SYNTHESIS.md` (Theme C ŌĆö the policy spine, the three uncoordinated ledgers, the dispatch god-object; missing-abstraction #3); `decision/{interface-feasibility,resident-shape}.md` (the load-bearing constraint: dispatch/emit/store unify **only as interfaces-with-backends, NOT one forced mechanism**); `confidence/{a2-concurrency,a4-worker-stall,a6-prompt-assembly,d1-cost-attribution,d3-event-schema}.md`. **Guiding principle (EPIC ┬¦Discipline + interface-feasibility OVERALL CALL + Theme C):** an SDK piece is an **interface with multiple real backends**, drawn at a deliberately LOW altitude. M4 gives each of the four services planning and resident *each already have their own version of* one interface + Ōēź2 backends (each with Ōēź1 real in-tree user) ŌĆö AND it adds the **missing horizontal owner** the four services lean on: a **driver policy spine**. The spine is why the dispatch request can stay thin: liveness, cost, rate, and recovery are consulted through spine owners, **not smuggled per-request through both backends** (the "dispatch request as god-object" that interrogation Theme C flagged as a symptom of missing-abstraction #3). M4 does NOT fold one mechanism into the other (no async loop forced into subprocess shape; no `state.json` forced into the revisioned Store).

## Outcome

1. **`dispatch` is one interface with TWO real backends.** `subprocess-cli` (planning's `run_step_with_worker`, `workers/_impl.py:2478`) and `async-api` (resident's `OpenAICompatibleAgentRunner.run`, `resident/agent_loop.py:169`) both satisfy one `Dispatcher` protocol returning a typed result+cost+session ref. The shared seam is the **`prompt_override` + minimal-shim path** (a6) ŌĆö the caller supplies a rendered body + a thin shim; the worker keeps sessions/sandbox/schema/contract/retry. The request carries the **liveness/`run_id` sink** (a4) but consults cost/rate/recovery **through the spine**, not as request fields.
2. **`emit` is one verb ŌĆö `EventSink.emit(kind, payload, scope)` ŌĆö over TWO backends.** `events.ndjson` (`observability/events.py:276`, flock'd monotonic seq, plan-dir-scoped journal) and the Store progress/epic path (`store/base.py` `append_progress_event:1295` / `record_epic_event:443`, row-identity-scoped). The scope token differs per backend; the **event schema is pinned** (d3). The spine's **composition-observability contract** rides on this verb.
3. **`evidence` splits into TWO pieces, not one.** **Attestation** (a notary ŌĆö today's `validate_execution_evidence`, `orchestration/execution_evidence.py:15`: asserts a claim holds, does not branch flow) and **oracle** (a measuring instrument: `run(cmd) -> {exit, stdout, stderr}` whose result *branches control flow*). Both behind injected strategy; each with a real user.
4. **`config` is an N-layer precedence resolver a package extends.** Generalize today's 2-layer `get_effective`/`setting_is_explicit` (`_core/io.py:764,777`) into the spine's **precedence resolver** (env > args > state.config(override) > profile > robustness > DEFAULTS) with a **characterization test**; planning's args bus and resident's `ResidentConfig` re-express as bindings over it.
5. **THE policy spine ŌĆö the missing horizontal owner (interrogation #3).** A **`RecoveryPolicy`** (`classify(error) -> {retry_fresh | retry_transient | escalate | halt(kind)}` + per-class budgets) extracting auto.py's brain (the `ExitKind` taxonomy + three retry loops + `_is_retryable_external_error`, "retry" ├Ś111). **ONE live budget/cost authority** consulted before each dispatch and folding across fan-out shards, homed in the key/rate broker (rate & spend are the same shared-depletable-resource problem). A **backend-agnostic composition-observability event contract** the driver emits by default, onto which **introspect/doctor/trace/cost are RE-HOMED** (today hardwired to ndjson + planning payloads).

**Acceptance:** for each of the four interfaces, BOTH backends are exercised by a real in-tree user (planning + resident/loop), AND one deliberately non-planning caller (an oracle-`run(cmd)` consumer per acceptance test #1) uses oracle-evidence + dispatch + the spine together. A hermes step dispatched through the service writes liveness on token progress and not on silence (a4 guard). A fourth tool hitting a transient error gets a retry classification from the injected `RecoveryPolicy` **without reaching into planning's auto.py** (Theme C's load-bearing test). Two tenants over a shared key set observe one shared budget+backoff (a2). The four observability commands run off the composition contract, not `find_plan_dir`.

## Locked decisions (interfaces-with-backends, NOT forced-unify)

- **Dispatch is request/result-shape sharing only.** Subprocess stall/sandbox/session-resume and async in-process tool loop stay strictly below the line, per-backend (interface-feasibility VERDICT ┬¦1). NOT a unified tool-execution or liveness *engine*.
- **The dispatch request is thin BECAUSE the spine owns the cross-cutting concerns.** Liveness travels as request data + an injected sink (a4); cost/rate/recovery are consulted via the spine owners (#5), not re-implemented per backend. This is the explicit fix for the "dispatch request as god-object" (interrogation ┬¦over-complications).
- **Emit unifies the verb, not the event model.** Two sinks, opaque scope token, separate taxonomies; `record_epic_event`'s transactional path is NOT in the emit call.
- **Store is NOT in M4 scope as a mutating contract.** interface-feasibility VERDICT ┬¦3: LWW `state.json` and the revisioned/leased Store are irreconcilable as one mutating contract (M3's state-evolution axis). M4 touches `state.json` only as the liveness/cost *artifact* sink.
- **`RecoveryPolicy` extracts auto.py's policy, not its mechanism.** M3 owns the loop/process driver mechanism (teardown + max-iters); M4 owns the errorŌåÆconsequence classification + per-class budgets that drive it. The two are deliberately separate (Theme C: M3 has mechanism, M4 supplies the brain).
- **ONE budget authority, homed in the broker.** Rate and spend collapse into one shared-depletable-resource authority; the three counters reconcile through it. Single-process behaviour byte-identical when no shared ledger.
- **Each backend has Ōēź1 real user.** subprocess-cli ŌåÉ planning + loop; async-api ŌåÉ resident; NDJSON sink ŌåÉ planning; Store sink ŌåÉ resident; attestation ŌåÉ planning finalize; oracle ŌåÉ the non-planning acceptance caller; config base ŌåÉ both bindings; `RecoveryPolicy` ŌåÉ auto.py's three loops + the fourth-tool transient test; budget authority ŌåÉ planning `CostTracker` + the a2 two-tenant test; observability contract ŌåÉ the four re-homed commands.

- **Do not regress the blessed escape hatches:** the `prompt_override` path (a6) and the in-process pool fallback (a2).
