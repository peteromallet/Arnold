# M5-eval — Evaluand + the one Ledger: versioned attributable judgments

**Epic:** Pipeline Unification (`.megaplan/briefs/pipeline-unification-EPIC.md` §M5-eval + §"The architecture" Evaluand/one-Ledger; `committed-uu/SYNTHESIS.md` Part 2 "Evaluand + the (single) Ledger", Tier-S UU#2, Reshaper #5, Principles #6/#7). **Position (PROGRAM.md):** M4 (services + Effect Ledger + one-Ledger R5 + **Evaluand SCAFFOLD**) → **M5-eval [T2]** (completes the scaffold) → **GATES M5-cal** (non-negotiable; PROGRAM critical-path apex edge). Fans out ∥ M5a (node verbs) ∥ M5b (execute) off the M4 service base. **Tier/robustness:** premium · thorough/high. **Depends_on:** M4 (the one Ledger + the Evaluand record shape + write path), M5a (the `verify`/`judge` node verbs), M2 (a judge is a Port-typed piece + the routing-key/`Reduce[T]` types). **Code verified against `main` 2026-05-29.**

> **What M5-eval is (vs the M4 scaffold).** M4 landed the Evaluand *record shape* `{judge-version, rubric-version, input-set-hash, score}` + a no-recompute write/read into the one Ledger (`m4-services-spine.md` Outcome #7, Done #10). M5-eval makes that record **attributable and joinable**: the judge becomes a Port-typed, content-hashed, *versioned piece* (eval-as-pipeline-kind), the score stops being a bare float anywhere on the new path, the model I/O is hashed + re-injectable keyed by `(prompt_hash, model_version, params)`, and "is the new version better?" becomes a **join** over `(piece-version × rubric-version × judge-version × input-set × provenance)` — recorded into ONE content-addressed log every surface READS and nothing recomputes. This is the spine for the self-improving heart; calibrating against bare floats is the Goodhart / co-degradation failure (UU#2,#10) — so this milestone HARD-GATES M5-cal.

---

## Outcome

1. **A judge is a Port-typed, content-hashed, versioned PIECE — `judge-as-pipeline-kind`.** A judge's identity = `judge_version = hash(judge step-code + resolved rubric body + judge-model-identity + Port set + ABI version)`; it `consumes`/`produces` typed Ports like any node (M2). Today the critic is an implicit roster pick (`audits/critique_evaluator.py` `CRITIC_MODEL_ROSTER`) with no recorded version identity; M5-eval makes the judge a first-class versioned piece whose every judgment is attributable to its closure.
2. **A score is a JOIN, never a bare float.** Every judgment recorded on the new path carries `(piece-version × rubric-version × judge-version × input-set-hash × provenance/taint)`. `PipelineVerdict(score=1.0, ...)` literals (`_pipeline/pattern_joins.py:40,83`) and the `Receipt.verdict: str | None` / `metrics: dict` shape (`receipts/schema.py:34,35`) are superseded on the new path by an attributable Evaluand record; the bare float is forbidden by a grep gate in eval-emitting SDK modules (mirrors the M2 `GateRecommendation` gate).
3. **One content-addressed Ledger every surface reads and nothing recomputes (R5 completion for eval).** "Did v_n+1 beat v_n?" is answered by a JOIN over the one Ledger — not by re-running a judge, not by a substring/vibe over `cost.py`-style read-time classification. The Ledger records **hashed, re-injectable model I/O** keyed by `(prompt_hash, model_version, params)` (the M4 §7 seed made load-bearing for eval) so a past judgment stays re-evaluable when a model drifts behind a stable name (UU#7).
4. **The judge is decoupled from the optimizer (anti-Goodhart spine).** The eval ruler is recorded as an exogenous, versioned reference signal: a rubric change is a visible `rubric_version` bump, not a silent meaning-shift; the judge-version is pinned per judgment so a *measurement* regression is separable from a *quality* regression (UU#2, Principle #6/#7). M5-eval does NOT build routing/calibration (that is M5-cal, downstream of this gate).
5. **Re-injectable, replayable eval.** Given two piece-versions and a fixed input-set, the system computes "is the new version better?" as a deterministic join over recorded judgments — and can RE-JUDGE the past by re-injecting the hashed model I/O against a new judge-version, producing a NEW attributable record (never overwriting the old), so ruler upgrades are themselves versioned events.

**Acceptance (build these, don't assert):** (a) two piece-versions judged by one judge-version over one input-set → "better?" resolves as a Ledger JOIN returning the winning piece-version + the attribution tuple, with ZERO recompute (the second read hits the log, runs no judge); (b) a rubric edit produces a NEW `rubric_version` and a NEW judgment record — the OLD judgment is unchanged and still joinable (versioned, not mutated); (c) a judge-model swap behind a stable name produces a distinct `judge_version` (model-identity in the hash) so the two judgments do not collide to one record (the taint/identity-collision-launders class, UU#4/#7); (d) the eval **oracle gate** below is green; (e) the grep gate proves zero bare-float scores in eval-emitting SDK modules.

## Locked decisions

- **The judge is a versioned, attributable, content-hashed piece; the score is a join over `(piece-version × rubric-version × judge-version × input-set)`, never a bare float** (SYNTHESIS Principle #7, Part 2 Evaluand). Eval-as-pipeline-kind: a judge is discovered/typed/versioned like any node, no privileged path.
- **Recorded-into, never recomputed-from (R5).** The one Ledger (M4) is the single eval observation log; comparisons are joins, re-judgments are NEW records. No surface re-runs a judge to answer "better?"; no substring/heuristic read-time classification.
- **Model-identity is in the judge-version hash (Reshaper #7).** A model drifting behind a stable name yields a distinct `judge_version`; identical-byte judgments with different judge/model identity never collide to one record (UU#4/#7 collision-launders defense — taint/provenance ride the M2 hash, not beside the value).
- **Optimizer ≠ evaluator (anti-Goodhart, Principle #6).** M5-eval makes the ruler exogenous + versioned; it does NOT build the calibration loop that consumes it. The Evaluand→Calibration edge is one-directional and gated: M5-cal may only READ this Ledger, never define the ruler.
- **The full attribution join completes the M4 scaffold; it does not re-lay the one Ledger.** M4 owns the write path + envelope + join key; M5-eval adds the attribution tuple, the eval-piece versioning, and the no-recompute join read on top of M4's R5 substrate.
- **Each surface has ≥1 real user:** judge-piece ← planning's gate/critique (`editorial/gating.py`, `audits/critique_evaluator.py`); the join read ← a "did v_n+1 beat v_n?" comparison over two recorded plan iterations; `re_judge` ← a recorded-I/O re-injection test; the grep gate ← the eval-emitting SDK modules.
