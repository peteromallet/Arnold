# Sprint 2 - Dispatch Grants And Enforcement Spine

## Outcome

Build the enforceable Megaplan authority spine on top of Sprint 1's inventory, contracts, and shadow views:

1. introduce Megaplan `DispatchGrant` as the domain wrapper over `CapabilityGrant`;
2. require worker results and merge/reconcile inputs to identify dispatch id, plan revision, coordinator fence, task/sense-check scope, prerequisite digest, and worker identity;
3. route all merge-like paths through a single validator;
4. derive ready frontier and dependency authority from accepted task attempts rather than raw artifact status.

## Scope

In scope:

- Define and implement `DispatchGrant` data structures and persistence/projection points.
- Update execute dispatch and worker result paths so results echo grant identity, plan revision, fence, scope, prerequisite digest, and worker identity.
- Implement one unified merge/reconcile validator covering normal merge, failure-boundary reconcile, no-pending replay, review rework, and creative/doc merge.
- Reject or quarantine off-scope updates, wrong revision, wrong dispatch id, stale prerequisite digest, unsupported creative cross-task updates, and insufficient evidence.
- Implement or extend `PlanExecutionView.next_ready_wave` from accepted task attempts and dependency closure.
- Stop scheduler/frontier authority from depending on raw `finalize.json` status where the new authority path is available.
- Add migration/shadow diagnostics for any consumer that cannot yet be fully rewired.
- Add tests that cover full read-decide-mutate paths for grants, validation, quarantine, idempotency, and dependency closure.

Out of scope:

- Full watchdog, PR publication, human gate, and chain consumer rewiring unless needed to prove the enforcement spine.
- Deleting compatibility projections.
- Broad non-authority cleanup.

## Done Criteria

- Worker result acceptance requires valid dispatch/grant identity and evidence-backed scope.
- All merge-like execute paths share the same validator or a clearly documented adapter to that validator.
- Off-scope, stale, wrong-revision, wrong-dispatch, and insufficient-evidence updates are rejected or quarantined with useful diagnostics.
- Scheduler/frontier tests prove ready work comes from accepted task attempts and dependency closure.
- Existing execute/reconcile behavior remains compatible where safe, with feature/shadow gates if needed.
- The sprint leaves a clear handoff for Sprint 3 consumer migration, sibling operational views, and compatibility cleanup.
