# Progress Auditor Stage Metrics

## Outcome

Add an end-to-end deterministic `stage_metrics` or `phase_metrics` block to the 6-hour progress auditor report so operators and downstream model phases can see stalls, retries, repair attempts, meta-repair attempts, human/PR/CI waits, handoff gaps, no-op loops, and duration by stage.

The finished work should make the auditor answer the user's question directly: "Will it find the number of stalls, retries, etc. by different stages?"

## Scope

In scope:

- Define a stable report schema for stage metrics in report-only output.
- Classify available evidence into at least these stages when present:
  - `prep`
  - `plan`
  - `critique`
  - `gate`
  - `revise`
  - `finalize`
  - `execute`
  - `review`
  - `feedback`
  - `chain`
  - `repair`
  - `meta_repair`
  - `human_pr_ci`
  - `deployment_runtime`
- Count per stage:
  - `stalls`
  - `retries`
  - `repair_attempts`
  - `meta_repair_attempts`
  - `human_waits`
  - `ci_waits`
  - `handoff_gaps`
  - `no_op_loops`
  - `dead_workers`
  - `duration_seconds` when derivable
  - `unknowns` / `missing_evidence`
- Include evidence pointers for nonzero counts. Prefer file paths, timestamps, event ids, session ids, sidecar names, or compact log references.
- Add top-level `coverage`, `data_quality`, and `dispatch_summary` blocks if they are absent or incomplete.
- Keep the existing pure gather/report behavior safe: report-only mode must not dispatch model or repair processes.
- Preserve and test existing deterministic signals:
  - `repair_data_ghost_running`
  - `green_with_recent_repair_churn`
  - `watchdog_chain_health_disagreement`
- Add focused unit tests and fixtures in `tests/cloud/test_progress_auditor.py` and related cloud wrapper tests as needed.
- Run focused tests locally and run a cloud report-only audit against the Hetzner workspace to confirm the block appears on real data.
- Update the progress-auditor debug skill or nearby docs if the operator workflow changes.

Out of scope:

- Full watchdog rewrite.
- Full repair custody redesign.
- Full PR/CI dashboard.
- Changing merge policy or broad cloud deployment mechanics unless required to verify this sprint.

## Constraints

- Work in the existing Arnold repo style and use existing parser/helper patterns before adding new abstractions.
- Do not rely on model summaries for counts. The report should compute counts from deterministic artifacts.
- Treat missing evidence as `missing` or `unknown`, not zero.
- Keep shell wrapper changes minimal and covered by regression tests.
- Do not revert unrelated local dirty work.
- The final cloud run should target the existing Hetzner ssh provider and the `editible-install` branch/workspace convention.

## Done Criteria

- A report-only audit includes a deterministic top-level `stage_metrics` or `phase_metrics` block.
- Stage metrics include per-stage counters for stalls, retries, repair attempts, meta-repair attempts, external waits, no-op loops, handoff gaps, dead workers, and duration where evidence supports them.
- Every nonzero count includes at least one compact evidence pointer.
- Missing evidence is visible under `coverage`/`data_quality` and does not collapse to zero.
- Existing deterministic findings still appear when fixtures reproduce them.
- Report-only mode remains guaranteed non-agentic in tests.
- Focused tests pass:
  - `PYENV_VERSION=3.11.11 pytest tests/cloud/test_progress_auditor.py`
  - relevant wrapper tests from `tests/cloud/test_watchdog_wrappers.py`
- A Hetzner cloud report-only audit is run after deployment, and its output path plus summary counts are recorded in the sprint result.
