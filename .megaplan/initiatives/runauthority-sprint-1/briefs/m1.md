# Sprint 1 - Authority Freeze And Shadow Foundation

## Outcome

Deliver the first two-week implementation slice of the RunAuthority main plan:

1. freeze the known Megaplan split-authority incident class with focused tests and fixes;
2. add an authority input inventory that classifies current sources as observation, claim, decision, or projection;
3. define the initial generic RunAuthority kernel contracts;
4. produce read-only shadow views for Megaplan execution plus basic runner/publication diagnostics.

This sprint should make the next enforcement sprint possible. It should not attempt to complete the whole RunAuthority migration.

## Locked Decisions

- Build a narrow generic `RunAuthorityKernel`, not a generic god-view.
- Use generic kernel names such as `CapabilityGrant`, `SubjectAttempt`, `Claim`, and `Decision`.
- Megaplan wraps those as `DispatchGrant`, `TaskAttempt`, `TaskClaim`, and `TaskValidationDecision`.
- `PlanExecutionView` is Megaplan-specific.
- `RunnerView`, `PublicationView`, `HumanGateView`, and `MegaplanRecoveryView` are sibling views, not generic-kernel fields.
- `MegaplanPlanView` may exist as a facade, but internal consumers should use the smallest relevant view.
- Mutable artifacts are not authority.
- Legacy adapters are allowed in shadow mode, but imported legacy state without revision, attempt, grant, fence, and evidence identity cannot mutate current authority.

## Scope

In scope:

- Add regression tests and fixes for `no_pending_execution` replay preserving original dispatch scope.
- Add regression tests and fixes for `reconcile_latest_execution_batch` preserving original dispatch scope.
- Decide and encode the Sprint 1 creative/doc cross-task merge policy. Prefer quarantine or explicit bounded scope over implicit all-task authority.
- Add a VibeComfy/Megaplan incident-style fixture or synthetic reproducer covering stale/off-scope batch artifacts and contradictory task status.
- Implement an authority input inventory command/module that lists current run sources and classifies them as observation, claim, decision, or projection.
- Include sources such as `state.json`, `finalize.json`, execution batches, chain state, cloud markers, watchdog reports, repair sidecars, Git/PR observations, process/session observations, `run_state`, store/repository/compat, and WAL/event readers where present.
- Define initial data contracts for evidence envelope, observation envelope, `CapabilityGrant`, `SubjectAttempt`, claim envelope, decision record, quarantine record, coordinator fence, projection metadata, idempotency key, and CAS expectations.
- Add read-only shadow derivation for `RunAuthorityView` and initial `PlanExecutionView`.
- Add basic `RunnerView` and `PublicationView` diagnostics sufficient to separate execution, liveness, and publication failures in status/debug output.
- Add focused tests for view hash determinism or deterministic replay where practical in this slice.
- Document remaining enforcement work as follow-up tasks if it does not fit.

Out of scope:

- Full enforce-mode migration.
- Full scheduler, chain, watchdog, repair, and publication rewiring.
- Removing legacy readers.
- Signed warrant infrastructure.
- Generic native-pipeline adoption beyond ensuring the kernel contracts are not Megaplan-named.
- Broad cleanup unrelated to authority.

## Constraints

- Preserve existing public behavior unless a path is demonstrably unsafe and covered by a regression test.
- Do not revert unrelated dirty work.
- Keep the kernel generic and small.
- Prefer adapters over invasive rewrites for Sprint 1.
- Tests must exercise complete read-decide-mutate or observe-decide-project cycles where possible, not only isolated helpers.
- New diagnostics must identify source paths and reasons for contradictions.
- Feature gates should default to shadow/read-only behavior unless the specific regression fix is already known safe.

## Done Criteria

- Existing tests pass for the touched areas.
- New tests prove replay/reconcile/creative scope behavior cannot reproduce the known corruption class.
- A run authority inventory can be generated for a plan/run and reports sources with roles.
- Initial kernel contracts exist in code with clear names and tests for basic idempotency/projection behavior.
- Read-only shadow `RunAuthorityView` and `PlanExecutionView` can be derived from current artifacts.
- Basic runner/publication diagnostics are separate from execution authority.
- Legacy artifacts remain compatibility projections/observations, not new authority.
- The sprint leaves a clear handoff for Sprint 2 enforcement: dispatch grants, unified merge validator, scheduler frontier from accepted task attempts, and consumer rewiring.

## Recommended Megaplan Settings

Overall plan difficulty: 5/5; selected profile: `partnered-5`; because a bad abstraction here can pass local tests while locking in the wrong authority boundary across scheduler, recovery, status, and publication.

Planning complexity: `full`; this is a real cross-cutting engineering sprint, but the scope is deliberately bounded to the first two-week slice.

Depth: `high`; the planner needs to reason about existing split authority surfaces, migration boundaries, and contract design.

Prep: enabled; the planner should inspect current authority readers and incident-adjacent paths before committing to the task split.
