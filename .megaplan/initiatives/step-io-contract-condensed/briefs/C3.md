# C3: Suspend / Resume + Suspension-Aware Fan-Out

Condenses old m4. Bases off `arnold-generalized-pipeline @ 944d10cc`. The migration already BUILT
the suspension TYPE, the generic status-lattice reduce, the generic fan-out reduce, and the subloop
suspension lift. C3 closes the genuine remainder: composite-cursor cross-restart persistence, the
simulated targeted+batch resume path, the legacy side-channel bridge, and the designed (un-built)
extension seams.

## C3 PLAN_DIR-COUPLING FINDING (the roadmap's gate — RESOLVED by reading the driver)

The roadmap gated C3 on whether resume is fundamentally `plan_dir`-coupled. **It is NOT, at the
generic runtime layer.** `arnold/runtime/driver.py:135` defines
`StepwiseDriver.resume(envelope: RuntimeEnvelope, cursor: ResumeCursorRef) -> RuntimeEnvelope` — the
generic resume protocol takes an OPAQUE `ResumeCursorRef` (`arnold/runtime/resume.py:51`,
`arnold/runtime/envelope.py:479`; documented in `arnold/runtime/CONTRACT.md`: "Opaque cursor; `None`
for fresh runs"). The generic suspend/resume surface is already package-level decoupled.

`plan_dir` coupling lives ONLY in the MEGAPLAN adapter:
`arnold/pipelines/megaplan/_pipeline/resume.py` (`ResumeCursor.load(plan_dir)`),
`arnold/pipelines/megaplan/_core/workflow.py:612` (`resume_plan`),
`arnold/pipelines/megaplan/_pipeline/executor.py:235` (`check_awaiting_user(plan_dir)`).

**Consequence: C3 does NOT grow to decouple the generic resume protocol** — the migration already
delivered the decoupled opaque-cursor surface. C3 stays scoped to suspension-aware composition +
generic composite-cursor persistence + the legacy-side-channel bridge, and treats `plan_dir` as a
megaplan adapter detail behind the opaque `ResumeCursorRef`, not a generic concern to refactor.

## Outcome

Make the `status = suspended` discriminant behave correctly when results compose, and make a
suspended subtree resumable autonomously (test-driven, never waiting on a real human). The migration
already wired the lattice and the subloop lift; C3 finishes the persistence and resume mechanics and
locks the designed extension seams.

## Scope

IN:

- **Composite-cursor cross-process-restart persistence (generic):** a composite suspension group's
  cursor is the source of truth and survives a process exit/reload. The generic reduce already
  composes a composite suspension (`arnold/pipeline/contract_reduce.py:157` `_compose_suspension`,
  `kind="composite_suspension"`), but the cursor's DURABLE persistence + reload-and-resume path must
  be made a generic concern (behind the opaque `ResumeCursorRef`), not left to the megaplan
  `awaiting_user.json` detail.
- **Targeted + batch composite resume, driven by simulated fixtures:** resume supports both a
  targeted (one child) and a batch (all) answer against a composite suspension group; the suspend→
  resume cycle is driven entirely by a PROGRAMMATIC fixture matching `resume_input_schema` and NEVER
  waits on a real human (autonomy fix m4#7).
- **Bridge the legacy side-channel onto the typed Suspension:** the `next="halt"` +
  `state_patch={"_pipeline_paused": True}` side-channel (`arnold/pipelines/megaplan/_pipeline/
  executor.py:1016/1026`) and `awaiting_user.json` (`check_awaiting_user(plan_dir)`, `_legacy_
  awaiting_user_contract` :223) must bridge onto the typed `Suspension` interaction envelope so a
  suspended run is represented by the contract, not the side-channel — without breaking the existing
  megaplan resume path during the transition.
- **Lock the two DESIGNED-but-UNBUILT extension seams** (ship only the v1 default):
  (i) SCOPED suspension propagation — a streaming/supervisor parent with a suspending child branch
  must NOT auto-lift the whole parent; the v1 barrier/max-wins lattice is the DEFAULT, scoping is the
  EXTENSION. Leave a scope-boundary interception point; do not implement scoping.
  (ii) Policy-driven result-reduce `{quorum | best-effort | budget | saturation}` as a PLUGGABLE
  policy OVER the default `MAX_WINS` lattice (`ReducePolicy`, `contract_reduce.py:31`). The slot
  already exists as `ReducePolicy.MAX_WINS`; C3 confirms it is a real extension point and ships ONLY
  the default policy.
- Verify the built composition behavior end-to-end against the suspension regression class.

OUT:

- The `human_review` verb itself, resume UX/CLI, and who-answers routing — features that plug INTO
  this primitive; C3 provides the hook, not the human verb.
- Producing suspended results from workers (that is the C2 model seam emitting `status`); C3 is about
  PROPAGATION + persistence + resume.
- Decoupling the GENERIC resume protocol from `plan_dir` — already decoupled (opaque
  `ResumeCursorRef`); C3 does not refactor it.
- Migration of IO sites and authoring-API enforcement (C4).
- BUILDING the two extension points (scoped propagation, alternate reduce policies) — designed, not
  built.
- Any change to the `ContractResult` type / `Suspension` shape (frozen in the migration).

## Locked Decisions (carried from the prep dossier jury)

- The reduce lattice is `completed < suspended < failed` with MAX-WINS; a `failed` parent STILL
  carries `pending_suspensions` — a suspended child's cursor is NEVER dropped because a sibling
  failed (resolved decision #4). Already realized; C3 preserves it.
- Multiple suspended children form a COMPOSITE suspension group; the v1 barrier blocks on ALL
  children; resume supports targeted + batch; the composite cursor is the source of truth and
  survives process exit.
- `SubloopStep` maps a child `suspended`/`awaiting_user` → parent `suspended` BEFORE the legacy
  `GateRecommendation` promote. Already built; C3 verifies + bridges the legacy `awaiting_user`
  representation onto the typed `Suspension`.
- The v1 lattice + barrier is the BUILT DEFAULT; two extension points are DESIGNED but NOT built
  (scoped propagation; policy-driven reduce). Ship only the default.
- `human_review`, resume UX, and routing are FEATURES that plug into this primitive; the foundation
  provides the hook, not the human verb.
- AUTONOMY (m4#7): the suspend→resume cycle is driven by simulated fixtures matching
  `resume_input_schema`, NEVER waiting on a real human — the path runs autonomously to completion.
- **NEW for C3 (the plan_dir finding):** the generic resume protocol stays decoupled (opaque
  `ResumeCursorRef`); `plan_dir` is a megaplan adapter detail behind the cursor, not a generic
  refactor target.
