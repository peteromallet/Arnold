**Key architectural constraint** (`_impl.py:86-88`, `_mock_payloads.py:20`):
```python
_EXECUTE_STEPS = {"execute", "loop_execute"}
_CROSS_CALL_PERSISTENT_STEPS = _EXECUTE_STEPS
```
The per-backend one-shot retry inside `run_step_with_worker` explicitly **skips execute** (`step in _EXECUTE_STEPS` guard at lines 4477, 4525). Execute errors propagate to `handle_execute_one_batch` where only `worker_timeout` is caught specifically for timeout recovery.

### 2. Earliest and Latest Safe Fallback Points

**Earliest safe point:** Before `run_step_with_worker` is called inside `_run_and_merge_batch` (batch.py:1028). At this juncture: no worker invoked, no payload produced, no git mutation, no merge. Only the batch template (`_write_execute_batch_template`) and prompt (`_execute_batch_prompt`) have been rendered. Tier resolution has already completed. A fallback here would mean re-resolving the tier spec to the next chain element before dispatch.

**Latest safe point:** Inside `run_step_with_worker`, but only for error codes that prove no output was produced:
- `codex_pre_first_byte_stall` ‚Äî Codex CLI hung before emitting any response bytes
- `connection_error` ‚Äî TCP/TLS handshake never completed
- Provider 429 / 5xx ‚Äî request rejected at API gateway, no model invocation
- `worker_timeout` **only if** no structured output was captured (see Signal 3 below)

**Point of no return (must not fallback after):** Once `run_step_with_worker` returns a `WorkerResult` (batch.py:1028 completes), the system immediately captures payload, runs quality checks, merges into `finalize_data`, writes `execution_batch_N.json` to disk ‚Äî all within the same function. There is no safe interleaving point between worker return and artifact persistence.

---

### 3. Signals That Prove No Accepted Output / No Mutation

1. **`codex_pre_first_byte_stall` error code** ‚Äî `_impl.py:4530`: proves Codex never emitted output bytes
2. **`connection_error` before any worker response** ‚Äî no TCP/TLS session established
3. **Provider 429 / rate-limit before first token** ‚Äî API rejected request
4. **Timeout with no checkpoint file:** `_recover_execute_timeout` (timeout.py:253) catches `FileNotFoundError` for `execution_batch_N.json` ‚Äî proves nothing was persisted
5. **Timeout with empty/invalid checkpoint:** `_merge_timeout_checkpoint` (timeout.py:173) only merges valid `task_updates` entries. If `checkpoint_data` is not a dict (line 275) or contains zero mergeable tasks (line 223), no accepted output exists
6. **Git clean tree:** `before_snapshot == after_snapshot` in `_collect_quality_deviations` (quality.py) ‚Äî no file-level mutation occurred
7. **No task status transitions:** If `_merge_batch_results` (merge.py:347) returns `merged_count == 0`, no task moved from `pending` to any terminal status
8. **Auth error for a provider with independent credentials** ‚Äî different provider/auth path may succeed

### 4. Signals That Must Permanently Block Fallback

1. **`execution_batch_N.json` exists on disk for this batch** ‚Äî accepted structured output persisted (batch.py:1198)
2. **`finalize.json` rewritten after merge** ‚Äî `_merge_batch_results` mutated task statuses (batch.py:1200)
3. **`execution_audit.json` written** ‚Äî evidence validation ran (batch.py:1199)
4. **Any task in `finalize_data` has terminal status** (`done`, `skipped`, `blocked`) **after merge** ‚Äî merge completed with accepted results
5. **Git diff shows changes attributable to worker** ‚Äî `_observe_git_changes` in quality.py detected mutation (batch.py:1140-1154)
6. **Timeout checkpoint successfully merged:** `_merge_timeout_checkpoint` merged ‚â• 1 task update or sense check acknowledgment (timeout.py:223-226)
7. **Non-infrastructure error codes:** `invalid_profile`, `unknown_agent`, `schema_validation_failed`, `model_structural_audit_error`, `blocked` result ‚Äî these are semantic failures, not availability failures
8. **Worker payload contains `task_updates` or `sense_check_acknowledgments`** ‚Äî the model produced structured output, even if partial/incomplete
9. **`blocked` result from quality gates** (batch.py:1587-1598) ‚Äî legitimate semantic outcome, not infrastructure failure
10. **Same-provider auth failure when next spec uses identical auth path** ‚Äî design doc ¬ßTrigger Policy

---

### 5. v1 Policy Recommendation

**DEFER execute fallback from v1. Ship fallback for non-mutating phases only.**

Rationale:
1. The existing execute safety architecture is already sophisticated and correct: per-backend retry is **explicitly blocked** for execute (`step in _EXECUTE_STEPS` guard, `_impl.py:4477,4525`), timeout recovery handles partial output conservatively, and the merge/evidence pipeline mutates persistent state irreversibly within a single function boundary.
2. Adding configured fallback chains to execute requires threading through **three distinct integration points** ‚Äî tier resolution (`_calibration_tier_spec` ‚Üí `_resolve_tier_spec`), the `_run_and_merge_batch` dispatch, and the `_recover_execute_timeout` recovery path ‚Äî each with different safety semantics.
3. The marginal value is low: the most common pre-output failures (`auth_error`, `connection_error`) are already handled by the ambient fallback in `run_step_with_worker` (lines 4637-4665). A configured chain would add little for execute specifically.
4. Phased approach is lower risk: ship fallback first for `plan`, `critique`, `review`, `finalize`, `gate`, `prep`, and `feedback` where the centralized `run_step_with_worker` loop applies cleanly. Gather field telemetry on actual failure modes. Design execute fallback with real data in v2.

**If execute fallback must ship in v1:** Limit to pre-`run_step_with_worker` tier re-resolution ‚Äî i.e., if the selected tier spec's model is unavailable at connection-open time, try the next spec in the tier's chain **before** any worker is invoked. This is the only safe boundary given the merge/mutation architecture.

### 6. Specific Tests Needed

**`execute` tests:**
1. **`test_execute_no_fallback_after_task_update_merge`** ‚Äî worker returns valid `task_updates`; simulate infra failure on next batch; assert fallback blocked, partial progress preserved in `finalize_data`
2. **`test_execute_fallback_pre_first_byte`** ‚Äî simulate `codex_pre_first_byte_stall`; assert fallback proceeds to next chain spec, no `execution_batch_N.json` created
3. **`test_execute_no_fallback_after_batch_artifact_persisted`** ‚Äî `execution_batch_1.json` already on disk; simulate `worker_timeout` on batch 2; assert fallback does NOT retry batch 1
4. **`test_execute_timeout_checkpoint_blocks_fallback`** ‚Äî `worker_timeout` with valid checkpoint containing `task_updates`; assert `_recover_execute_timeout` merges checkpoint, fallback permanently blocked
5. **`test_execute_git_mutation_blocks_fallback`** ‚Äî worker produced output, `git diff` shows changes; assert fallback blocked even if worker subsequently times out
6. **`test_execute_quality_blocked_not_fallback_trigger`** ‚Äî worker returns `result=blocked` from quality gates; assert NOT treated as fallback trigger
7. **`test_execute_tier_chain_exhaustion`** ‚Äî all specs in tier chain fail retryably; assert `CliError` includes all attempted specs and reasons
8. **`test_execute_fallback_respects_tier_boundaries`** ‚Äî fallback within a tier does not step down to lower tier unless explicitly listed in chain

**`loop_execute` tests:**
1. **`test_loop_execute_in_execute_steps`** ‚Äî assert `"loop_execute" in _EXECUTE_STEPS`; confirm all execute safety guards apply uniformly
2. **`test_loop_execute_fallback_blocked_by_partial_loop_progress`** ‚Äî loop completed 3 of 5 iterations, finalize_data tasks mutated; assert fallback blocked
3. **`test_loop_execute_pre_first_batch_fallback`** ‚Äî connection error before first batch in loop; assert fallback proceeds cleanly
4. **`test_loop_execute_no_fallback_mid_loop`** ‚Äî batch 2 succeeds, batch 3 gets rate-limited; assert fallback blocked because partial progress already persisted
