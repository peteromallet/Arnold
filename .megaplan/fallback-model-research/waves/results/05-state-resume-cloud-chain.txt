### 2. Encoding Strategy

**Recommendation: store encoded fallback chains at string-only persistence boundaries; keep `FallbackSpecChain` objects at routing boundaries.**

For each surface above:

- **Must store encoded chains** (canonical encode → decode at load boundary): items 1, 4, 5, 6, 7, 8, 9, 10, 11. These are the unavoidable string-only persistence surfaces (`state.json`, chain YAML, ledger JSONL, receipt JSON, active step dict, preflight dict). They should use the proposed `__fallback_json__:["spec1","spec2"]` encoding.

- **Must store `list[str]` natively** (no encoding needed, just type-widening): items 2, 3. `tier_models` and `prep_models` should accept `str | list[str]` at the TOML/profile level, validated by `normalize_fallback_specs`, then stored in state as either a scalar or the encoded form. The `dict[int, str]` type annotation in `_validate_tier_models` at line 319 is the tightest constraint — widening to `dict[int, str | list[str]]` and normalizing is the main change.

- **Must decode to `FallbackSpecChain` before calling `parse_agent_spec`**: items 12, 13, 14, 15, 16, 17, 18. These are routing-time consumers that today pass one scalar to `parse_agent_spec`. A shared decode helper must run first, then the caller iterates or selects `primary`.

- **Must pass `FallbackSpecChain` through to `run_step_with_worker`**: items 16, 18. The dispatch choke point must receive a chain, not a pre-resolved `AgentMode`. The chain iteration loop lives inside `run_step_with_worker` (or a wrapper near it).

- **Override set-model (item 11) must remain scalar in v1** per design §"CLI and Overrides." The `set-model` handler writes one scalar `new_spec`. This is correct and must not change. Future: a distinct `set-fallback-models` command.

### 3. Resume/Backward-Compat Risks

1. **`_validate_persisted_phase_models` (state.py:157-188)** — If an existing plan has `phase_model` entries where the right-hand spec is a plain scalar like `codex:gpt-5.5`, loading must not break. The validator must detect the `__fallback_json__` prefix and decode it; plain scalar strings must pass through `parse_agent_spec` as today. No migration needed for existing plans.

2. **`apply_profile_expansion` persisted merge (policy.py:962-1005)** — When a plan resumes with an already-persisted `phase_model` that contains encoded fallback entries, the merge logic (which compares `pm.split("=",1)[0]` to detect stale overrides) must not split on `=` inside the JSON payload. The encoding prefix `__fallback_json__:` is safe because `=` only appears between phase name and the encoded value. Verified: `phase=__fallback_json__:[...]` splits correctly.

3. **`tier_models` in state.json** — Existing plans have `tier_models.execute: {4: "codex:gpt-5.5"}`. If the profile is updated to use arrays, a resumed plan carries the old scalar tier map from state. The new behavior must detect that the runtime tier map is scalar and wrap it in a single-element chain rather than failing. The `_validate_tier_models` change from `isinstance(spec, str)` to `isinstance(spec, (str, list))` is a one-way widening that won't reject old valid data.

4. **AgentMode 4-tuple compat** — `run_step_with_worker` already handles both `AgentMode` and 4-tuple `resolved=` parameters (lines 4392-4398). Adding `FallbackSpecChain` must not break callers passing bare 4-tuples — but those callers by definition have no fallback chain, so they default to single-attempt behavior.

5. **Routing ledger JSONL** — Old ledger lines lack `configured_specs`/`attempted_specs`/`selected_index` fields. Readers must tolerate missing keys. This is already the ledger pattern (all fields are optional at read time).

### 4. Cloud and Chain Preflight Changes Required

1. **`cloud/preflight.py:_resolved_phase_map` (line 79-90)** — Currently returns `dict[str, str]`. Must return `dict[str, str | FallbackSpecChain]` or always encode. The downstream loop at line 168-186 iterates `resolved.values()`, calling `parse_agent_spec(spec)` for each. For fallback chains, every spec in the chain must be parsed to collect `required_agents`, `runtime_commands`, `env_hints`, and `provider_requirements`. A chain that uses `hermes:deepseek:...` as a fallback but `codex:gpt-5.5` as primary needs both Codex and DeepSeek env hints reported. **Change**: `_resolved_phase_map` must decode fallback entries and `resolve_cloud_chain_runtime_dependencies` must walk every spec in every chain.

2. **`chain/spec.py:MilestoneSpec.phase_model` (line 614-622)** — Already accepts `list[str]`. If chain YAML authors want to express a fallback chain in a milestone, the YAML syntax would be:
   ```yaml
   phase_model:
     - "execute=codex:gpt-5.5"
   ```
   The `phase_model` list carries `phase=spec` pairs. For fallback, each spec value could be the encoded form `__fallback_json__:[...]`. This is acceptable for v1 (profile-file-only fallback). No YAML syntax change needed yet.

3. **`supervisor/chain_runner.py` (lines 91-95, 455)** — Passes `phase_model` as `list[str]` to launch args. No decode needed here — the values are passed through to `apply_profile_expansion`. Safe as-is.

4. **Preflight must report credential gaps** — When a chain includes `codex:gpt-5.5` primary and `hermes:deepseek:deepseek-v4-pro` fallback, the preflight must flag missing `DEEPSEEK_API_KEY` even if `OPENAI_API_KEY` is present. Currently only the primary spec's agent is scanned.

### 5. Override Semantics That Must Remain Scalar in v1

Per the design document §"CLI and Overrides":

- **`override set-model`** (`handlers/override.py:1552-1574`): `new_spec = format_agent_spec(AgentSpec(target_agent, model=target_model, effort=target_effort))` → writes one scalar. Must remain scalar. It also strips `tier_models[phase]` (line 1570-1574), which is correct: an explicit scalar override suppresses tier routing for that phase.

- **`--phase-model execute=codex:gpt-5.4` CLI flag**: Stays scalar. Suppresses `tier_models.execute` (line 936-938 in policy.py). Must not change.

- **`_agent_requested_explicitly`** (`workers/_impl.py:4003-4014`): When a `phase_model` entry is present for a step, it signals "explicit routing." For an encoded fallback chain in `phase_model`, this flag should still return `True` — an encoded chain IS explicit routing. The legacy ambient `detect_available_agents()` fallback must be suppressed when an explicit chain (even multi-spec) is configured.

- **`_resolved_profile_phase_spec`** (`override.py:521-551`): Returns a scalar spec string to display. For fallback, return the primary spec. The fallback is internal; status/display surfaces show the primary.
