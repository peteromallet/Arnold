## 2. Contracts That Should Stay Scalar Externally, Decode Internally

- **`state.config.phase_model`** (persisted JSON): Already a `list[str]`. Must stay that way. Internally decode `__fallback_json__:` prefix â†’ `FallbackSpecChain`.
- **CLI `--phase-model execute=codex:gpt-5.5`**: Remains a single scalar string. No CLI fallback syntax in v1. Internally treated as a single-element chain.
- **`override set-model`**: Sets a single model. Destroys tier routing for that phase. Stays scalar.

---

## 3. Contracts That Can Safely Accept `FallbackSpecChain | str`

These are internal routing surfaces where callers already pass around strings that get parsed:

- **`_resolve_tier_spec(..., tier_spec: str)`** â†’ `tuple[str, str, str | None]` (execute/batch.py:408). Change signature to accept `str | FallbackSpecChain`, return `(agent, mode, model, chain)`.
- **`resolve_prep_stage_model(state, stage) -> AgentMode`** (prep_research.py:246). Return `AgentMode` with attached `fallback_chain` attribute or a paired return.
- **`WorkerUnit.resolved`** (worker_fanout.py:51): Stays `AgentMode`. The chain is consumed upstream before fanout creates units. Individual units remain single-attempt.
- **Profile policy rewrite functions** (`apply_vendor_rewrite`, `apply_depth_rewrite`, `apply_available_model_floor`, `apply_deepseek_provider_rewrite`): Accept `dict[str, str | list[str]]` for profile and tier values, normalize internally.

## 4. Surfaces Missing from the Design Doc

1. **`_agent_requested_explicitly`** (workers/_impl.py:4003-4014): Determines whether ambient `detect_available_agents()` fallback fires. Must decode encoded chains before checking step match. Missing from the design doc entirely.

2. **`_vendor_adjusted_default_spec`** (workers/_impl.py:4029-4040): Runs inside `resolve_agent_mode` default path. If the primary spec in a chain happens to be a default, this mutates it before the attempt loop sees it. Must apply vendor adjustment to the primary spec only.

3. **`apply_profile_expansion` tier-stripping logic** (policy.py:935-939): When `--phase-model execute=...` is present, it pops the phase from `tier_models`. This stripping must still work when tier values are `list[str]` â€” skip the pop for that phase regardless of whether tier values are scalars or lists.

4. **`_canonicalize_tier_models_for_json`**: Referenced in `profiles/__init__.py:30` but not in the design doc. Must handle `list[str]` tier values for serialization without flattening them.

5. **`_validate_resolved_profile_invariants` and `_validate_named_profile_invariants`** (policy.py:718-777): Both iterate over tier/phase specs calling `parse_agent_spec(spec)`. Must map over chain elements.

6. **`_resolve_with_inheritance`** (profiles/__init__.py): Inherits tier/prep values from parent profiles. Must merge fallback chains across inheritance without accidentally concatenating unrelated arrays.

7. **`resolve_prep_models`** (policy.py): Resolves canonical prep model defaults. Must accept and preserve chain values for prep stages.

## 5. Minimal Implementation Order (No Breakage)

**Step 0**: Add `model_fallback.py` with `FallbackSpecChain`, `normalize_fallback_specs()`, `encode_fallback_chain()`, `decode_fallback_chain()`. All encode/decode goes through one pair of functions. No inline `__fallback_json__:` string checks anywhere else.

**Step 1**: Change profile and tier validation (`_validate_profile_map`, `_validate_tier_models`) to accept `str | list[str]`. Existing single-string profiles pass through unmodified. Add rejection for empty lists and non-string members. This is the gate â€” nothing downstream can receive a chain until this accepts it.

**Step 2**: Change policy rewrites (`apply_vendor_rewrite`, `apply_depth_rewrite`, `apply_available_model_floor`, `apply_deepseek_provider_rewrite`, `apply_critic_rewrite`) to map over chain elements. Each function calls a helper `_map_over_chain(value, fn)` internally. Existing scalars are wrapped in single-element chains, rewritten, and unwrapped â€” zero observable change.

**Step 3**: Change `profile_to_phase_models` to emit encoded strings for chains. Change `_replace_phase_model`, `_infer_phase_agent`, `_current_phase_spec`, `_agent_requested_explicitly`, `_validate_persisted_phase_models` to decode via the shared helper before parsing. Existing scalar `phase=spec` entries pass through `decode_fallback_chain` which returns a single-element chain â€” no behavior change.

**Step 4**: Add the attempt loop below `run_step_with_worker` / `run_step_phase`. At this point, single-element chains (all existing profiles) make exactly one attempt â€” no observable change.

**Step 5**: Add observability fields to routing ledger and batch routing records. Write new fields alongside existing scalar `selected_spec`. Old readers keep working.

**Step 6**: Write the test matrix from the design doc before shipping a single multi-element profile TOML. Gate on all single-string behavior tests passing identically.

The critical invariant: every step before Step 4 must produce identical AgentMode/routing behavior for single-string profiles. Step 4 introduces the loop, but with single-element chains it's a no-op. Only after Step 6 does any profile actually get a multi-element chain.
