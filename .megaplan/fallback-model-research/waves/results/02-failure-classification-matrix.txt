## Q2: RETRY_NEXT_SPEC Classification

### Should trigger `RETRY_NEXT_SPEC`:

| Error | Rationale | Caveat |
|---|---|---|
| `worker_timeout` | Next model may finish | **Not for execute** unless timeout-recovery proves no accepted output |
| `worker_stall` | Different provider/transport likely avoids | Same as above |
| `codex_pre_first_byte_stall` | No output produced yet | Safe even for execute |
| `connection_error` | Transport failure, provider-independent | |
| `rate_limit` (HTTP 429) | Different provider = independent rate limiter | |
| `api_error` (500/502/503) | Transient provider failure | |
| `quota_exceeded` | **Only if next spec uses different provider family** (see Q4) | |
| `context_overflow` | **Only behind explicit opt-in flag** (`fallback_on_context_exhaustion = true`) | |

### Should NOT trigger `RETRY_NEXT_SPEC` (`RAISE_PERMANENT`):

| Error | Rationale |
|---|---|
| `auth_error` on same provider family | Config problem, not availability |
| `invalid_agent_spec`, `agent_deps_missing`, `agent_not_found`, `invalid_args` | Structural config errors |
| `model_error`, `permission_error` | Config/entitlement, not transient |
| `unsupported_step` | Code bug, not model-dependent |
| `unsupported_tool_call_markup` | Model behavior quality issue |

### Should trigger `RAISE_SEMANTIC` (model output failures, never retry):

| Error | Rationale |
|---|---|
| `parse_error`, `worker_parse_error` | Malformed output ‚Äî fallback is not a repair loop |
| `schema_error`, `worker_structural_audit_failed` | Structural audit failure |
| `shannon_failed` | Generic Shannon failure, likely output-related |

### Should trigger `RAISE_EXECUTE_UNSAFE`:

| Condition | Rationale |
|---|---|
| Any infrastructure error **after** accepted structured output/task updates/mutation | Dirty tree risk (see design doc L189-200) |

### New normalized codes needed:

- `provider_unavailable` ‚Äî normalize `api_error` + 5xx patterns across all backends
- `rate_limit` is already present but only for Codex/Hermes; Shannon 429s are currently `worker_error`. Needs normalization.
- `context_overflow` is only Codex; Hermes/Shannon don't emit it. Should be a normalized code for the classifier.

## Q3: Where Should the Classifier Live and What Inputs?

**Location:** `arnold_pipelines/megaplan/model_fallback.py` (the new module proposed in the design doc), or a sibling `model_fallback_classify.py`.

**Inputs required:**

```python
@dataclass
class FallbackInput:
    error: CliError                    # .code, .message, .extra
    phase: str                         # "plan", "execute", "critique", etc.
    attempted_spec: str                # e.g. "codex:gpt-5.5"
    next_spec: str                     # e.g. "hermes:deepseek:deepseek-v4-pro"
    output_accepted: bool              # Did worker produce accepted structured output?
    mutation_observed: bool            # Did worker mutate files/tree?
    context_fallback_enabled: bool     # Policy flag
    external_error: ExternalError | None  # Provider-level detail when available
```

**Call site:** Inside the new fallback loop near `run_step_with_worker` (design doc step 7). The loop catches `CliError`, constructs `FallbackInput`, calls `classify_fallback(error=..., phase=..., ...)`, and branches on the `FallbackDecision` enum.

`ExternalError.from_exception()` already exists and should be called first to enrich the input with provider-level status codes/error kinds before the classifier runs.

## Q4: Distinguishing Same-Provider vs. Independent-Provider Auth/Quota Failures

**Key insight:** The classifier must compare provider families of `attempted_spec` and `next_spec`, not just check the error code.

**Rule:**

1. Parse both specs with `parse_agent_spec()`.
2. Extract the **provider family**:
   - `codex:gpt-5.5` ‚Üí family = `codex`
   - `hermes:deepseek:deepseek-v4-pro` ‚Üí family = (`hermes`, `deepseek`)
   - `claude:claude-opus-4-7` ‚Üí family = `claude` (routes through Shannon)
3. For `auth_error`:
   - Same family ‚Üí `RAISE_PERMANENT` (if Codex auth is broken, another Codex model also fails)
   - Different family ‚Üí `RETRY_NEXT_SPEC` (DeepSeek auth is independent of Codex auth)
4. For `quota_exceeded`:
   - Same family + same sub-provider ‚Üí `RAISE_PERMANENT`
   - Different family ‚Üí `RETRY_NEXT_SPEC` (DeepSeek quota is independent of Codex quota)
   - **Edge case:** `hermes:deepseek:*` ‚Üí `hermes:fireworks:*` ‚Äî both use the Hermes adapter but different API keys. The Hermes key pool already handles per-provider keys; the classifier can treat different Hermes sub-providers as independent.

**Implementation:** Add a helper `_same_provider_family(spec_a: str, spec_b: str) -> bool` that compares the agent and, for `hermes`, the sub-provider prefix.
