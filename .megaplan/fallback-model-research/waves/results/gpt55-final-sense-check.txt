**Verdict: revise**

The design is close and not unsafe, but I would make a few doc edits before launching implementation. The main risk is not the concept; it is hidden scalar collapse during routing, fanout, preflight, and resume.

**Top Findings**

1. **V1 execute fallback block is correct, but needs sharper wording.**  
   Keep automatic `execute` and `loop_execute` fallback out of v1. The doc should say explicitly: v1 may use the primary execute spec, but if it would move to spec index `>0`, it must raise `execute_fallback_unsafe` with the attempted spec and blocked next specs. Do not silently ignore the rest of the chain.

2. **A few scalar surfaces are still missing or too generic.**  
   Add explicit inventory entries for:
   - `arnold_pipelines/megaplan/preflight.py`: `preflight_check_profile` and `_required_env_vars_for_slot` parse scalar slot specs directly.
   - `handlers/init.py`: `_build_state_config` persists `phase_model`, expands `--hermes` into scalar phase pins.
   - `calibration/ledger.py`: projected tier models use scalar `routed_tier_spec`; projection must not erase fallback chains.
   - `routing/identity.py`: cache identity must use selected/actual model only, never the whole chain.
   - All `WorkerUnit` constructors, including prep, adaptive critique, review parallel, and repair-unit cloning. The `WorkerUnit.fallback_chain` field must survive process packing/unpacking.

3. **Encode/decode strategy is sound, but needs a strict contract.**  
   Add these constraints:
   - `phase_model` remains `list[str]` only. Multi-spec values are `phase=<encoded-chain>`.
   - Encoding is canonical compact JSON: `__fallback_json__:` + `json.dumps(specs, separators=(",", ":"), ensure_ascii=True)`.
   - `decode_fallback_chain(value)` must accept plain scalar strings and return a one-spec chain.
   - Every call path into `parse_agent_spec` from persisted/configured routing must decode first.
   - Chain YAML and cloud-upload normalization must preserve encoded strings exactly; no raw arrays inside `phase_model`.

4. **Retryability rules need a provider-family helper, not prose.**  
   Add `provider_family(spec)` with rules like:
   - `codex:*` -> `codex`
   - `claude:*` / `shannon:*` -> `claude`
   - `hermes:<provider>:...` -> `hermes:<provider>`
   - bare/unknown Hermes provider -> permanent config error unless resolved first

   Then define: auth, balance, quota, and likely 429 only retry when the next spec has a different family. Transport timeout, stall, network, and 5xx may try the next spec. Generic `worker_error` must not fallback unless it carries an allowed `ExternalError`.

5. **Implementation order is mostly right, but add golden tests first.**  
   Before adding any array profile, freeze existing behavior with tests for single-string profiles, `AgentMode.__iter__`, `profile_to_phase_models`, persisted resume, execute tier routing, prep routing, and cloud preflight. Then land helper and decode plumbing before dispatch changes.

**Concrete Doc Edits To Add**

- Add a “V1 execute semantics” paragraph: primary-only attempt, no automatic second attempt, clear unsafe error on retryable failure.
- Expand scalar inventory with `preflight.py`, `handlers/init.py`, `calibration/ledger.py`, `routing/identity.py`, `review/parallel.py`, and `WorkerUnit` pack/unpack.
- Add a formal `provider_family(spec)` and retryability table.
- Add an invariant: observability arrays are additive, but legacy scalar `model`, `selected_spec`, and cache identity always represent the selected attempt, not the configured chain.

After those edits, this is ready for a Megaplan implementation run.
