### 2. Fallback Should Be Per Unit, Not Per Batch

- **Prep research fanout**: per unit. If a unit exhausts its chain, preserve sentinel/error behavior — don't kill sibling units.
- **Adaptive/parallel critique**: per check/unit, within the selected complexity tier chain. Tier selection happens once; fallback iterates within that tier's specs.
- **Parallel review**: per unit, but v1 must not change topology (no falling from parallel Hermes to single Codex/Claude).
- **Tiebreaker**: per side (researcher, challenger fall back independently).
- **Single-phase callers** (`run_step_with_worker`): per call, not batched.

The batch itself has no single model — every unit carries its own resolved mode. Fallback at batch level would be meaningless and would violate the tier-selection invariant.

### 3. Where Fallback Chains Can Be Threaded Without Changing `AgentMode.__iter__`

`AgentMode.__iter__` returns `(agent, mode, refreshed, model)`. The design doc explicitly says: do not change this. The threading strategy:

1. **Pre-resolution layer**: At every site listed above where a single spec string is resolved, resolve a `FallbackSpecChain` *first*, then use the primary spec for the initial `AgentMode`. Store the full chain alongside (not inside) the AgentMode.

2. **In WorkerUnit**: Add `fallback_chain: FallbackSpecChain | None = None` to `WorkerUnit.__init__` with default `None` (backward-compatible). The chain rides in the unit, not in `resolved`.

3. **In `scatter_worker_unit()`** (the per-unit dispatch): Wrap `run_step_with_worker()` in a loop over the chain when `unit.fallback_chain` is present. On infrastructure failure, advance to next spec, rebuild `resolved` from the next spec, retry. On success, return. On exhaustion, propagate error (or sentinel if `on_unit_error` provided).

4. **For single-phase callers** (tiebreaker, critique evaluator, etc.): Add an attempt loop near `run_step_with_worker` in `_impl.py` that iterates the chain before the existing ambient fallback loop. The ambient loop (lines 4637-4665) becomes the legacy path when no configured chain is present.

5. **For critique tier resolution** (`handlers/critique.py:248-270`): When `_tier_spec_for(cx)` returns a TOML array (future), resolve to a `FallbackSpecChain`, create the primary `AgentMode` for `_resolved_agent_mode`, and attach the chain via `_check["_fallback_chain"]`. Parallel critique's `_scatter_raw` path needs no change — fallback is inside `scatter_worker_unit`.

### 5. Missing Observability Fields for Fanout Attempts

Current `record_step_routing()` records: `phase, step_label, agent, selected_spec, resolved_model, actual_model, tier, complexity, tier_routing_active`. Missing:

- `configured_specs` — the full ordered chain (list[str])
- `attempted_specs` — which specs were actually tried (list[str])
- `selected_index` — zero-based index of the winning spec (int)
- `fallback_triggered` — boolean (true if index > 0)
- `fallback_reasons` — list of `{spec, error_code, error_message}` per failed attempt
- `attempt_costs` — optional per-failed-attempt cost tracking

`WorkerUnitResult` already carries `agent, mode, model, resolved_model, effort` — but has no fields for attempt metadata. These should be added alongside existing fields (not replacing them) for backward compatibility.

The routing ledger JSONL format should accept the new fields as optional additions. Old readers ignore unknown keys.

### 6. Tests Needed to Prevent Topology-Changing Fallback

1. **Unit isolation**: A single prep-research unit exhausting its 3-spec chain produces a sentinel, not a batch abort. Sibling units keep their primary model.
2. **Tier boundary**: Adaptive critique fallback for complexity-5 check stays within tier 5's chain — does not step down to tier 4 unless the chain explicitly lists a tier-4 model.
3. **Parallel review topology guard**: A review batch of 5 Hermes units where one unit exhausts its chain must not cause remaining units to switch to Codex/Claude. The fallback is per-unit only.
4. **Tiebreaker independence**: Researcher exhausting its chain does not affect challenger's model selection.
5. **Execute after-output guard**: Execute fallback halts after `execution_batch_*.json` accepted, even if the unit's chain has more specs.
6. **Chain not on AgentMode**: Verify `AgentMode.__iter__` still unpacks as 4-tuple after fallback threading. All existing tuple-unpack callers (tiebreaker `_build_resolved`, critique handler, etc.) continue working.
7. **Ambient fallback suppression**: When a WorkerUnit carries a `fallback_chain`, the ambient `_runtime_fallback_candidates` loop in `run_step_with_worker` is never reached.
8. **Ledger records all attempts**: A 2-attempt success records both `configured_specs: ["A", "B"]` and `attempted_specs: ["A", "B"]` with `selected_index: 1`.
