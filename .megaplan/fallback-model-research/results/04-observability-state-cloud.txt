## Adversarial Edge Review: Sequential Fallback Model Lists

### Position
Adding sequential fallback model lists is **safe to begin** but requires touching **8 subsystems beyond the obvious profile+worker code** before it can ship. The single biggest risk is `AgentMode.__iter__` — its 4-tuple unpack contract is called in at least 6 handler files. Break that, and every phase transition silently corrupts.

---

### Hidden Breakage Risks

1. **`AgentMode.__iter__` is a 4-tuple contract** (`arnold/agent/contracts.py:246-248`). Every handler unpacks `(agent, mode, refreshed, model)` from it (`handlers/shared.py:245`, `handlers/review.py`, `handlers/execute.py`, `handlers/critique.py`, `execute/batch.py:1376`, `execute/batch.py:2885`). Adding a 5th field (model_list) will silently truncate or ValueError at every call site. **Verdict: must become a 6-tuple `(agent, mode, refreshed, model, model_list, resolved_model)` with a deprecation path, or model_list must NOT enter AgentMode at all.**

2. **`ActivePhase` TypedDict single `model` field** (`types.py:132`). `set_active_step` writes one `model` string (`_core/state.py:1203-1204`). The watchdog, status view, cloud progress auditor, and `touch_active_step` all read this as a scalar. A list silently passes through JSON serialization but breaks every reader expecting a string. **Verdict: `ActivePhase` must add `model_list: NotRequired[list[str]]`; all readers must be audited.**

3. **Session key uses model** (`_core/state.py:1208`): `session_key_for(step, agent, model)`. With fallback lists, which model names the session? The first? The successful one? A hash of the list? Wrong choice = session cache misses or poisoned sessions. **Verdict: session key must be derived from the *list* deterministically (sorted, joined, hashed) or use only the *primary* model.**

4. **Chain spec milestone lacks a model_list field** (`chain/spec.py:531-559`). `MilestoneSpec` has `phase_model` (phase→agent overrides) but no per-milestone fallback chain. Without it, chain YAML can't express tiered fallback across milestones → the feature is dead on arrival for cloud chains. **Verdict: add `model_list: list[str]` to MilestoneSpec; validate in from_dict; pass through supervisor chain_runner metadata (line 455).**

5. **Cloud `CodexSpec` is single-model** (`cloud/spec.py:38-40`). The cloud deploy template bakes `model` into `codex exec -c model=...`. A list requires either `--fallback-model` flag support in the codex CLI wrapper or a cloud.yaml schema change. **Verdict: cloud.yaml must accept `models: [gpt-5.5, gpt-5.4]`; template must render `--fallback-model`; preflight must resolve.**

---

### Required Telemetry/State Fields

| Field | Location | Purpose |
|---|---|---|
| `attempted_models: list[str]` | `routing_ledger.jsonl` | Which models were tried in sequence |
| `fallback_depth: int` | `routing_ledger.jsonl` | How deep the chain went (0 = first model worked) |
| `fallback_reason: str` | `routing_ledger.jsonl` | Why fallback triggered (timeout/quota/error) |
| `model_list: list[str]` | `ActivePhase` | Currently active fallback chain |
| `model_index: int` | `ActivePhase` | Which index in the list is currently running |
| `model_list` | `CapabilityClaim` (calibration ledger) | Which models were available for routing |
| `fallback_models: list[str]` | cost event payload | For per-model cost attribution when fallback occurs |

The `HistoryEntry` TypedDict (`types.py:149-168`) currently has **no model field at all** — model info lives only in `ActivePhase`. Every history entry that involved a fallback must record `model_list` and `model_index` for auditability.

---

### Migration/Backcompat Recommendation

- **Phase 1 (non-breaking)**: Add `model_list: NotRequired[list[str]]` to `PlanConfig`, `ActivePhase`, `MilestoneSpec`; add to `AgentMode` as an optional field alongside existing `model` (do NOT change `__iter__`). All readers fall back to `[model]` when `model_list` is absent.
- **Phase 2 (deprecation)**: Add `model_list` to `AgentMode.__iter__` as a 6th element behind a feature flag; update all unpack sites.
- **Phase 3 (cleanup)**: Remove the single `model` field from `ActivePhase`/`AgentMode`; replace with `model_list` only.
- **Store migration**: `PlanState` serialization must handle old states without `model_list` — treat missing as `[model]` if `model` is present, or empty list otherwise.
- **Session migration**: Existing session keys (computed from single model) become stale. Either recompute on first fallback-aware run or add a `session_key_version` field.

---

### Cloud/Chain/Bakeoff Implications

- **Cloud chains**: `cloud/preflight.py`'s `_resolved_phase_map` returns one spec per phase. Fallback lists require `resolve_cloud_chain_runtime_dependencies` to return a `resolved_fallback_chain` per milestone. The cloud template `cloud.yaml.tmpl` must render `--fallback-model` CLI flags.
- **Bakeoff**: Profile-based bakeoffs (`bakeoff_runner.py`) fan out by profile name, not model. Fallback lists are a *profile-internal* concern — no bakeoff changes needed *unless* you want to bakeoff-compare different fallback chains, which requires a new bakeoff dimension.
- **Epic chains**: `epic_chain.py` passes milestone metadata to child plan runs. The `phase_model` list already flows through; `model_list` must join it in the metadata dict (line 455).
- **Policy settings**: `policy_settings.py` needs a new `PolicySettingSpec` key (e.g., `model_fallback`) that reports whether sequential fallback is active and the configured chain.

### Minimum Doc Updates

1. **`policy_catalogue.md`**: Add "model_fallback" setting (owner, default, valid values).
2. **`chain.yaml.example`**: Show `model_list` under a milestone.
3. **`cloud_skill.md`**: Document `models:` in cloud.yaml, `--fallback-model` in CLI wrapper.
4. **`data/observe_skill.md`**: Document new routing_ledger fields for fallback telemetry.
5. **Changelog/ADR**: Migration plan for the `AgentMode.__iter__` contract change.

---

### Bottom Line

The **telemetry surface** (routing ledger, cost events, calibration claims, receipts) and the **AgentMode unpack contract** are the two highest-risk areas. Start there. The chain spec, cloud preflight, and session key derivation follow. Everything else — status view, policy settings, bakeoff — is additive and low-risk.
