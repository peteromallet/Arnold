Here's the analysis.

---

## Where the fallback wrapper should live

Put it **inside the Hermes worker** (`hermes.py`, near the existing MiniMax→OpenRouter path at L2467–2573), extending the `_run_attempt` / fallback block. The MiniMax fallback already demonstrates the pattern: catch exception, classify, pick a fallback provider+model, re-invoke `_make_agent` + `_run_attempt`. Generalize that.

Do **not** put it in the auto-driver (`auto.py`). The auto-driver retries whole *phases* via subprocess spawn, which re-runs the prompt, re-creates sessions, and re-does idempotent work. Worker-level fallback avoids that — it swaps the model mid-call before returning to the caller.

The **Shannon worker** (`shannon.py`) shouldn't get this — Shannon launches a real `claude` process in tmux; model-swapping there means killing tmux and re-launching, which is equivalent to a phase-level retry.

---

## What failure classes/signals should trigger "try next model"

Use the **same `ExternalError` classification** the system already produces (via `phase_result_classify.classify_external_error_payload`). Specifically:

| Signal | Why it's retryable |
|---|---|
| **429 / rate-limit** (`error_kind="rate_limit"`) | Provider capacity — next provider may succeed |
| **5xx server errors** (status 500/502/503/504) | Provider transient |
| **Timeout** (`error_layer` in `EXTERNAL_RETRYABLE_LAYERS`: `stream_content_stall`, `stream_read_timeout`, `transport_timeout`, `worker_stream_stall`) | Network/infra — next model likely different infrastructure |
| **Quota exhaustion** (`error_kind="quota"` / `provider_error_code="Limit Exhausted"`) | Account quota, not structural — fallback provider has its own quota |
| **Context exhaustion** (`error_kind="context_exhausted"`) — **only if configured** | Model context limit hit — a model with larger context may succeed |
| **Provider transient** (`error_kind` not in permanent set, or exception message matches known transient patterns) | Catch-all for unclassified provider failures |

The existing `ExternalError` already captures all these via `provider`, `error_kind`, `status_code`, `provider_error_code`, `error_layer`.

---

## What should NOT trigger fallback

| Signal | Why NOT |
|---|---|
| **`malformed_model_output`** / `parse_error` / JSON decode failure | Model produced garbage — retrying on a different model is a content-quality problem, not infrastructure |
| **`blocked_by_prereq`** / `blocked_by_quality` | Legitimate task result — not a failure at all |
| **`ModelBudgetError`** (prompt too large pre-dispatch) | Structural — would fail identically on next model unless context limit differs (that case is context exhaustion, handled separately) |
| **Schema validation failure** / `ModelStructuralAuditError` | Output didn't match schema — model quality, not infra |
| **`internal_error`** that isn't reclassified to `external_error` | Something broke in our code, not the provider |
| **`auth` / `billing` / `config` / `bad_request` / `invalid_request` / `unsupported_model`** | Permanent config issues — already in `EXTERNAL_PERMANENT_ERROR_KINDS`, retrying makes no sense |

These are **already excluded** by the auto-driver's `_is_retryable_external_error()` and the `EXTERNAL_PERMANENT_ERROR_KINDS` constant. The wrapper should reuse the same classification.

---

## How to avoid rerunning non-idempotent work dangerously

Two critical rules:

1. **Only for `run_hermes_step` / read-only phases.** The Hermes worker wraps phases like `plan`, `prep`, `critique`, `revise`, `gate`, `finalize`, `review` — these are read-analyze-write operations. The execute phase uses `run_codex_step` or `run_shannon_step` — mutating workers. The fallback is **in the Hermes worker only**. Execute should never get model-list fallback at the worker level; its retries must be phase-level (which the auto-driver already handles with tier escalation).

2. **Use the existing `session_id` continuity.** The Hermes worker's `session_key_for` produces a deterministic key per plan+step+agent. If the fallback reconnects to the same SessionDB, conversation history (read-only tool calls like `read_file`, `search_files`) is preserved. The prompt is re-sent fresh (megaplan re-sends everything from disk each call), so there's no duplication of analysis — just different model answering the same question.

3. **Cap at one fallback.** The existing MiniMax path tries primary → OpenRouter, then gives up. A generalized sequential list should iterate through the configured list but **stop at the first success**. If all models fail with retryable infra errors, surface all the errors and let the auto-driver handle it as `external_error`.

| `test_hermes_fallback_skips_on_parse_error` | Model A returns unparseable JSON → raise `CliError(code="worker_parse_error")` → do NOT fallback, propagate error. |
| `test_hermes_fallback_exhausted_raises_clierror` | All models in list fail with 503 → raise `CliError` with all `ExternalError` payloads in `extra`. |
| `test_hermes_fallback_classifies_external_error_correctly` | Given a `requests.HTTPError(429)`, verify `ExternalError.from_exception` returns `error_kind="rate_limit"`, `status_code=429`. |
| `test_fallback_not_triggered_by_model_budget_error` | `ModelBudgetError` pre-dispatch → immediate raise, no fallback loop. |
| `test_fallback_not_triggered_by_structural_audit_error` | `ModelStructuralAuditError` → propagate, no fallback. |

---

## Position

**Put the sequential-fallback wrapper in `hermes.py` at the worker level, not in the auto-driver.** Gate it on the same `ExternalError` classification used by the phase-result boundary. Use the existing `EXTERNAL_PERMANENT_ERROR_KINDS` constant to exclude non-infra failures. Don't touch Shannon or Codex — those are mutating workers where model-swapping mid-execution is either impossible (Shannon/tmux) or dangerous (Codex rewrites files). The auto-driver's existing external-retry and tier-escalation loops already handle phase-level recovery for execute. Cap at one fallback per invocation and surface all errors on exhaustion so the auto-driver can escalate properly.
