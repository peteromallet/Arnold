Read the doc and inspected the dispatch paths. Core policy: fallback is an ordered availability chain for the selected routing slot. It should change model only after a retryable infrastructure failure, before any accepted output boundary. It must not become a quality repair loop.

**Dispatch Scope**
The model-dispatch phase identities are: `plan`, `prep`, `prep-triage`, `prep-research`, `prep-distill`, `execute`, `loop_execute`, `critique`, `critique_evaluator`, `revise`, `gate`, `review`, `finalize`, `feedback`, `loop_plan`, `tiebreaker_researcher`, `tiebreaker_challenger`.

Fallback should apply to:

| Path | Policy |
|---|---|
| `plan`, `revise`, `gate`, `finalize`, `loop_plan` | Yes, centrally in `run_step_with_worker`, after existing same-spec retry/recovery is exhausted. |
| `feedback` | Yes technically, but best-effort: if the chain exhausts, preserve current behavior and scaffold empty feedback rather than sink the plan. |
| `prep` legacy direct worker | Yes, if it dispatches through the central worker. |
| `prep-triage`, `prep-research`, `prep-distill` | Yes per stage. Research fanout falls back per research unit before converting the unit to a sentinel/error finding. |
| `critique` sequential/single | Yes. |
| `critique_evaluator` | Yes for infra only. No static-lens fallback for invalid evaluator output. |
| Adaptive/parallel critique | Yes per check/unit, within the selected complexity tier chain only. |
| `execute`, `loop_execute` | Yes, but only under strict pre-output/no-accepted-result rules. Treat `loop_execute` as execute. |
| `review` single/full/light/thorough | Yes. |
| Extreme parallel review | Yes per review unit only if fallback preserves compatible review topology. Cross-agent topology changes need an owner decision. |
| `tiebreaker_researcher`, `tiebreaker_challenger` | Yes separately. A successful researcher artifact should not be rerun because challenger fell back or failed. |
| Tiebreaker synthesis / human decision | No model fallback; synthesis is local, decision is policy/human. |
| Cloud, chain, bakeoff, supervisor, auto-driver | They should preserve/pass routing chains. Fallback happens inside the child phase dispatch, not as a separate outer runner feature. |
| Existing runtime fallback | Keep only as legacy behavior when no explicit fallback chain is configured. A configured chain must not silently append ambient available agents. |

**Move-Down Triggers**
Move to the next spec only when all are true:

1. The current attempt failed before an accepted phase payload/result boundary.
2. The failure is classified as availability/infrastructure.
3. Any existing local same-spec recovery has already run where appropriate.
4. The next spec is explicitly listed in the selected chain.
5. The next spec has a plausible independent route: different model capacity, provider, executable, auth path, quota pool, or transport.

Retryable trigger classes:

- `codex_pre_first_byte_stall`
- `worker_stall`
- `worker_timeout`, only if no accepted structured output/recovery exists
- `connection_error`
- provider 429 / `rate_limit`
- provider 5xx / transient `api_error`
- quota or usage-limit exhaustion only when the next route has independent quota/auth
- missing executable/dependency for one attempted route only if another explicit chain route is viable
- `context_overflow` only behind an explicit opt-in policy flag

**Never Trigger**
Do not fallback on:

- invalid profile syntax, empty arrays, malformed fallback specs
- unknown agent or clearly unsupported model
- missing dependencies/auth for every configured route
- same-provider auth failure when the next spec uses the same auth path
- HTTP 400 / schema or invalid-request errors
- generic `worker_error` unless further classified as transient infra
- malformed model output, JSON repair failure, schema validation failure
- model structural audit failure
- legitimate `blocked`, `failed`, `pending`, `needs_rework`, gate escalation, or review rejection
- execute task update/evidence failures
- test failures or dirty-tree verification failures
- human approval/preflight/destructive-action blocks
- cost cap, budget exhaustion, or auto-driver iteration caps

**Special Handling**
Execute: fallback only before first meaningful output, on connection/open/rate-limit/quota/pre-first-byte failures, or timeout where timeout recovery proves no completed batch result was accepted. No fallback after task updates, `execution_batch_*.json`, accepted timeout recovery, evidence audit, or tree mutation that cannot be proven harmless. Use fresh sessions for fallback attempts.

Adaptive critique: fallback inside evaluator dispatch for infra only; invalid evaluator verdict remains evaluator failure. Per-check fallback stays within `tier_models.critique[complexity]`, never down to another tier unless listed in the chain.

Prep research: resolve chains for `triage`, `fanout`, `distill`. Fanout fallback is per `WorkerUnit`; exhausted chain becomes the current sentinel/error finding. No fallback for “no research areas” or weak/invalid findings.

Review: normal review uses central fallback. Extreme Hermes parallel review can fallback per unit only while preserving the same multi-unit output contract. Falling from parallel Hermes to single Codex/Claude review changes artifact topology and needs a policy decision.

Tiebreakers: researcher and challenger fallback independently. Local synthesis and human tiebreaker choice never fallback.

Cloud/chain: milestone `phase_model`, profile expansion, and preflight must preserve chains rather than flattening to primary specs. Cloud preflight should report credential/dependency status for every possible chain route.

Auto-driver: its external retries, context retries, blocked execute retries, and execute tier escalation remain outer control loops. They should see a phase only after the internal fallback chain succeeds or exhausts. Do not let semantic auto retries consume fallback models.

Existing runtime fallback: configured chains take precedence. Explicit scalar `--phase-model execute=...` remains a single route and suppresses tier fallback. Ambient `detect_available_agents()` fallback should remain legacy/no-chain only.

**Telemetry Needed**
Add alongside existing scalar `model` fields:

- `configured_specs`, `attempted_specs`, `selected_spec`, `selected_index`
- `fallback_triggered`, per-attempt `fallback_reason`
- per-attempt error code, provider status, retryability decision
- resolved model vs provider-reported actual model
- session id, fresh/persistent flag per attempt
- cost/token attribution per attempt where available
- active-step current chain index/spec and failed attempts
- fanout unit id/check id/research area for per-unit fallback
- execute-specific: pre/post git snapshot, structured output seen, timeout recovery accepted, batch artifact written, task updates seen

Primary sinks: `routing_ledger.jsonl`, history entries, receipts, active step state, status/introspect views, cloud preflight/status payloads.

**Owner Decisions**
- Whether `context_overflow` fallback is opt-in only. I recommend opt-in.
- Whether execute can fallback after any possible mutation without a clean reset mechanism. I recommend no.
- Whether extreme parallel review may change topology from parallel Hermes units to single Codex/Claude review.
- Whether first-route missing credentials/deps should fail preflight or skip at runtime.
- How to distinguish transient “model unavailable” from invalid/unknown model.
- Whether fallback lists are profile-only v1 or also expressible through repeated CLI `--phase-model`.
- Whether `prep_models` officially support fallback arrays in v1.
- How auto-driver execute tier escalation should interact with a chain that already contains lower/backup models.
