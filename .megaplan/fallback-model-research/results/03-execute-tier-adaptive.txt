## Research Findings: Sequential Fallback Model Lists

### Position: Fallback lists do not exist today — they must be added.

The codebase has exactly **zero** support for sequential fallback model lists. Every tier spec (`tier_models.execute.N`) is a **single string** consumed atomically. Adding this feature requires changes at four distinct call sites.

---

### 1. Call Paths Where a Fallback List Must Be Preserved/Resolved

**Path A — Execute batch dispatch** (`batch.py:1390–1424`):
`handle_execute_one_batch` → `compute_batch_complexity` → `_calibration_tier_spec` → `_resolve_tier_spec` → `run_step_with_worker`.  
The tier spec is resolved to `(agent, mode, model)` once and passed directly. No retry loop exists — a `worker_timeout` CliError triggers `_recover_execute_timeout` (state mutation, not model swap). **A fallback list needs to be iterated here before the `_run_and_merge_batch` call at line 1450.**

**Path B — Auto-loop** (`batch.py:2454+` `handle_execute_auto_loop`): Delegates per-batch dispatch to the same `_run_and_merge_batch` with per-batch tier resolution. Same gap.

**Path C — Adaptive critique routing** (`critique.py:149–276` `_apply_adaptive_critique_routing`): Resolves `_critique_tiers.get(complexity)` → `_resolve_tier_spec` → attaches `AgentMode` to each check. The `AgentMode` then flows into `parallel_critique.py:run_parallel_critique` → `scatter_worker_units` → `scatter_worker_unit`. **No retry/fallback loop at either the routing layer or the worker-fanout layer.**

**Path D — `_core/dispatch.py` `resolve_dispatch_spec`**: Pure dictionary lookup `slot_table.get(tier_ordinal, default)`. Returns a **single string**. This is the lowest-level bottleneck — any list representation must be parsed here.

---

### 2. How tier_models Should Represent Fallback Lists

**Recommendation: comma-separated within the existing string value**, e.g.:
```toml
[profiles.partnered-5.tier_models.execute]
4 = "codex:gpt-5.4, codex:gpt-5.5"
5 = "codex:gpt-5.5, hermes:deepseek:deepseek-v4-pro"
```

This avoids breaking every profile TOML file, every validator, every serializer (`_canonicalize_tier_models_for_json`), vendor rewrite (`apply_vendor_rewrite` which iterates `tiers[tier_int] = _swap_premium_spec(...)` per-spec), and depth rewrite. A list-of-strings TOML array would require rewriting 10+ profile files and breaking the `dict[int, str]` type contract in ~8 functions across `policy.py`.

**Resolution**: `_resolve_tier_spec` (or a new wrapper) splits on `","`, strips whitespace, and iterates. `resolve_dispatch_spec` continues returning `str | None` — callers handle the iteration.

---

### 3. Interaction with phase_model Override Precedence

`apply_profile_expansion` (policy.py:935–938, 1011–1025) explicitly **strips** `tier_models.execute` when a CLI `--phase-model execute=...` or persisted `phase_model` pin is present:

```python
# Lines 935-938: CLI override strips tier table
if tier_models:
    for phase in ("execute", "critique"):
        if phase in cli_steps:
            tier_models.pop(phase, None)

# Lines 1011-1025: Persisted pin also strips (unless it's just the profile default)
for phase in ("execute", "critique"):
    if phase not in cli_steps: continue
    default_spec = profile_default_specs.get(phase)
    if default_spec is not None and effective_specs.get(phase) == default_spec:
        continue  # don't strip if persisted pin IS the profile default
    tier_models_after_persisted.pop(phase, None)
```

**Implication**: If a user passes `--phase-model execute=codex:gpt-5.5`, the entire `tier_models.execute` table is **removed**, so fallback lists within it are irrelevant. This is correct behavior — a phase_model pin is an explicit single-model choice, not a tier-routing decision. No change needed.

The critique path (`_apply_adaptive_critique_routing` at critique.py:166–205) has its own fallback chain: `args.tier_models` → `state.config.tier_models` → reload from profile. This is already robust for the "args stripped tier_models" case.

---

### 4. Batch/Idempotency Concerns

**Session key collision**: `_resolve_tier_spec` at line 1413 forces `refreshed = True` when the tier model differs from the fallback. With a fallback list, the session must be keyed by the **first** model (not the one that eventually succeeded) to avoid stale session poisoning if a retry succeeds on model #2.

**Re-entry**: `handle_execute_one_batch` can be called with `--batch N` repeatedly. If batch 4 fails on `gpt-5.4` and succeeds on `gpt-5.5`, the routing record must capture both the attempted and actual model. The existing `routing_record` structure (`_build_routing_record`) has `selected_spec` and `actual_model` fields — add `attempted_specs: list[str]` for the full fallback chain.

**Resume cursor**: `_record_execute_blocked` (execute.py:75–87) writes a `resume_cursor` that carries `retry_strategy: "fresh_session"`. This already handles the session-poisoning concern. No additional idempotency machinery needed — the retry just starts from the first model in the fallback list again.
