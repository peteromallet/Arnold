### Retroactive Detection (a–c)

**(a) Historical same-phase failure recounted on multiple polls after later successes:** **GAP.** `_deterministic_retry_evidence` (wrapper:2113) and `_deterministic_phase_history_evidence` (wrapper:2152) only count events within the 6h audit window cutoff. If a gate failure at 18:00 succeeded at 18:30 and the auditor runs at 01:00 (7h later), the failure aged out. Watchdog recounting of the failure across polls won't be surfaced unless `prior_watchdog_report_refs` matched `stopped/dead/progress_stall` — not `"failed then recovered"`. L3 would **not** detect this.

**(b) Accepted/dispatched repair requests with no claim/attempt:** **DETECTED** — see TRACKED above.

**(c) L2 misdiagnosis + nonzero ordinary retrigger:** **PARTIALLY DETECTED.** If the retrigger happened but the chain didn't advance, `incomplete_chain` + `l1_failed` + `absent_or_stale_l2` could trigger via superfixer cycle if L2 evidence is stale. But if L2 evidence exists and is fresh (even misdiagnosed), `absent_or_stale_l2 = False` → blocks `actionable`. The `false_success` drift detection requires the brief projection to first claim recovery — a bootstrapping gap.

### Source/Test Gaps

1. **`_l2_failure_fingerprint`** has no `recursion_guard_blocked` signal (identified in prior audit at superfixer-custody-20260715/results/07-l3-retroactive.txt:39).
2. **`_l2_fix_vs_resolver_drift_finding`** requires `shipped_fix == "fixed"` AND `install_freshness == "fresh"` (six_hour_auditor:669) — both placeholders must be populated, creating a chicken-and-egg with misdiagnosis detection.
3. **No `installed_freshness` comparison** between deployed wrapper SHA and L2 fix commit SHA anywhere in the auditor pipeline.
4. **Audit-window cutoff** (`_deterministic_retry_evidence` L2113) uses `cutoff` from `cutoff = dt.datetime.now() - dt.timedelta(hours=window_h)` — historical same-phase failures across windows are invisible.
