### ✅ INCLUDE in first "all structured templates" sprint

These are the boundaries every plan execution touches, are architecturally stable, and are straightforward template targets:

| Priority | Phase | Rationale |
|---|---|---|
| **P0** | `plan` | First model output; `plan.json` is small, stable, and the template is the "entry point" for structured output. |
| **P0** | `prep` / `prep-distill` | Same schema (`prep.json`); used in every plan. `prep-distill` normalizes to `prep` so one template serves both. |
| **P0** | `execute` | Core execution output. `execution.json` is the canonical schema; `execution_batch_relaxed.json` is only a capture-mode relaxation. |
| **P0** | `finalize` | Complex schema but stable; touched in every full plan cycle. |
| **P0** | `critique` | Both serial and parallel critique use `critique.json`. Template for a single check (`checks[0]`) is the key shape. |
| **P0** | `review` | Both serial and parallel review use `review.json`. Single-check template + criteria verdict template needed. |
| **P0** | `gate` | Decision point; `gate.json` is moderate complexity. |
| **P0** | `revise` | Touched on every iterate loop. Same template family as `plan`. |

| **P1** | `feedback` | End-of-run only; small schema; cheap to add. |

## 3. Template Path Naming Convention

Based on the existing patterns (`write_single_check_template`, `_write_critique_template`, `_write_review_template`, `_write_criteria_verdict_review_template`), the convention is:

```
# Canonical schema templates (input to model seam):
schemas/<schema_key>                    e.g. schemas/plan.json
                                         schemas/execution.json
                                         schemas/critique.json

# Per-step output templates (seeded before model call):
<plan_dir>/<step>.json                  e.g. plan_abc/plan.json
<plan_dir>/critique_check_<check_id>.json   (parallel critique single-check)
<plan_dir>/review_check_<check_id>.json     (parallel review single-check)
<plan_dir>/review_criteria_verdict.json     (parallel review criteria side-unit)

# Batch artifacts:
<plan_dir>/execution_batch_<N>.json

# Derived/variant schemas (runtime-constructed):
schemas/execution_doc.json
schemas/execution_batch_relaxed.json
```

**Naming rule**: `<phase_identity>.json` for the canonical template; `<phase_identity>_<qualifier>.json` for variants. Check-scoped templates use `<phase>_check_<id>.json`.

## 4. Biggest Unknowns

1. **`critique_evaluator.json` shape**: This schema is built by `_build_critique_evaluator_schema()` at the top of `schemas/runtime.py` (lines before 178). I didn't read that function's full body. Its shape is dynamically constructed and may have non-trivial nesting. Need to confirm the exact properties before templating.

2. **Single-check subset extraction**: The parallel critique/review fan-outs require a "single check" template — i.e., the `checks[0]` item shape within `critique.json` / `review.json`. The current code writes a full `critique.json` envelope with `checks: [one_item]`. The template system needs to decide: does it template the **envelope** (`{checks: [...], flags: [], ...}`) or the **inner check object**? Both parallel runners expect the envelope. This is the biggest design question for the template sprint.

3. **`prep-research` fanout lifecycle**: `prep_research_finding.json` is used per-research-unit in `run_prep_orchestration`. The orchestration fans out N units, each producing one finding. The aggregation step then assembles them. Whether each unit needs its own template file or can share a single template path needs clarification.

4. **Compact review prompt ≠ different schema**: The compact review fallback (`compact_review_prompt`) uses the same `review.json` schema. Template files don't differ — only the prompt text is compressed. This means the template system doesn't need a separate entry; it's a prompt-level concern, not a schema-level concern.

5. **`directors_notes.json` and creative-mode forms**: The creative pipeline (`forms/`) builds custom execution schemas (e.g. `execution_doc.json`). Whether these derived schemas get their own template files or are handled by the base `execution.json` template + property overrides is unclear. The `forms/__init__.py` `Form` dataclass has an `execution_schema_key` field, suggesting each form may need its own template.
