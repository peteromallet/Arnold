## Investigation: Should `critique_evaluator` Get a Template-Fill JSON Contract?

### 1. Verdict: **Exclude** (no template-fill file needed)

### Rationale

`critique_evaluator` **is** a structured-output boundary — unambiguously. It has:
- A formal JSON schema (`_build_critique_evaluator_schema()` in `schemas/runtime.py`, line 76–175)
- A `StepContract` registered in `step_contracts.py` (line 125–134)
- A dedicated `EvaluatorVerdict` TypedDict and `validate_evaluator_verdict()` semantic validator (audits/critique_evaluator.py)
- A dedicated normalizer (`_normalize_critique_evaluator_capture_payload`, model_seam.py line 367–368)

But it is **not** a template-fill boundary in the same sense as critique/review/finalize/execute/gate. Here's why, drawn directly from the code:

#### Why critique/review/execute have template-fill files

For `critique`, the prompt module writes a pre-populated JSON template to disk **before the model runs** (`_write_critique_template()`, prompts/critique.py:441–462). The model fills in the `findings` arrays on disk. The file is the model's sole output channel. The worker (`hermes.py:1319–1353`) instructs: *"This file is your ONLY output. It contains a JSON template with the structure to fill in."* The template pre-populates the `checks` array with check ids, questions, and guidance — the model just adds `findings` entries.

For `critique_evaluator`, there is **no `_write_evaluator_template()` function anywhere**. The model produces the entire verdict JSON from scratch in its response text. This is confirmed by:
- The prompt (prompts/critique_evaluator.py:441–452) describes the JSON keys inline
- No `output_path` or `template_path` references for `critique_evaluator` in any worker file
- The handler (handlers/critique.py:274–282) reads `eval_worker.payload` directly from the worker result — no file-based fallback

#### Why a template file would be counterproductive for the evaluator

1. **The selections list is inherently unpredictable.** Critique needs 5–9 checks filled in (one per lens). The evaluator must decide *which* of the 9 lenses to select vs skip — a pre-populated template would be misleading since it can't pre-populate the selection decisions. Each selection carries a model-determined `complexity` (1–5) and `complexity_justification` that can't be pre-populated.

2. **The `other` custom areas are fully model-determined.** The evaluator can add up to `MAX_OTHER_AREAS` (= 2) bespoke `other` entries (audits/critique_evaluator.py:25). These have no predetermined shape — their `area`, `why`, `complexity`, and `complexity_justification` are all created by the model.

3. **`flag_verifications` are conditional.** They only exist on iteration ≥ 2 (prompts/critique_evaluator.py:281–345). A template would need to be versioned per iteration, adding complexity without value.

4. **The evaluator has its own recovery stack.** The handler (critique.py:270–445) runs with `_MAX_EVAL_ATTEMPTS = 2` and blocks the milestone on persistent failure rather than degrading. The `validate_evaluator_verdict()` function handles consistent duplicates by deduping + warning (not hard-rejecting). This is a more robust recovery path than template-fill.

5. **The normalizer handles schema drift already.** `_normalize_critique_evaluator_capture_payload()` (model_seam.py:367–368) and the `oneOf` union in the schema (legacy/catalog/other selection schemas) handle format evolution without a template file.
