## 1. Short Verdict

**Every phase needs exactly ONE scratch template path per logical unit (check, step, or batch). No phase requires the same template twice simultaneously, and no phase requires distinct retry templates to avoid clobbering.** All retry/reprompt/repair paths safely reuse the same single template because they are serialized — either by a state lock, a sequential retry loop, or an in-memory merge.

## 3. High-Risk Clobber Cases

**None identified that are currently exploitable.** The existing design is safe because:

1. **Parallel critique per-check templates** are already correctly namespaced by `check_id`. Two concurrent workers for checks `security` and `completeness` write to `critique_check_security.json` and `critique_check_completeness.json` — no collision.

2. **Gate reprompt** is the most subtle case. The handler calls `_run_worker("gate", ...)` twice in sequence. The second call overwrites the output file, but the first result is captured in a Python variable and merged via `_merge_gate_worker_attempt`. If this ever became parallel (e.g., speculative dual-gate), it would clobber. Currently safe.

3. **Execute batch retry** reuses `execution_batch_{N}.json` for the same batch on retry. This is safe because batch execution is sequential and the retry is a fresh worker session. If batch execution ever became concurrent (parallel batches), the shared checkpoint file (`finalize.json`) would be the clobber risk — but that's already precluded by the sequential batch loop.

4. **Critique shape-repair retry loop** (lines 520-550 of `parallel_critique.py`) re-dispatches a subset of failed units. Each unit retains its original `output_path` (`critique_check_{id}.json`). The re-scatter is parallel, but since each unit's path is unique by `check_id`, there's no cross-unit collision. A single unit's retry overwrites its own prior output — which is the intended behavior (the prior output was malformed).

**Hypothetical future risk:** If the codebase ever introduces concurrent execution of the same phase for the same plan (e.g., speculative parallel gate, or parallel batch execution), the single-template-per-unit pattern would need revision to add attempt/version suffixes. Currently all phases are serialized by `load_plan_locked`.
