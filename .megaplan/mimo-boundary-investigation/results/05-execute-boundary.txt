## Execute Boundary Report for MiMo

### 1. Boundary Verdict: **Risky**

The execute pipeline is well-engineered with extensive normalization, guardrails, and recovery. However, several boundaries are specifically risky for MiMo (a smaller model) because they assume the model can reliably produce structured JSON conforming to a complex nested schema, handle multi-task batch checklists without losing track, and perform long reasoning chains.

### 2. Likely MiMo Failure Modes

**A. Structured output schema conformance (HIGH RISK)**

The execution output schema (`execution.json`) requires a deeply nested JSON structure:
- Top-level `output`, `files_changed`, `commands_run`, `deviations`, `task_updates[]`, `sense_check_acknowledgments[]`
- Each `task_update` requires `task_id`, `status` (enum-constrained), `executor_notes`, `files_changed`, `commands_run`
- Each sense check acknowledgment requires `sense_check_id` and `executor_note`

The merge layer (`merge.py` lines 27-42, 153-178) has extensive field-alias normalization (`id` → `task_id`, `completed` → `done`, etc.) which proves models frequently get field names wrong. MiMo will likely produce `id` instead of `task_id`, or `completed` instead of `done`. The normalization handles this, but edge cases remain.

**B. Multi-task batch evidence attribution (HIGH RISK)**

The prompt explicitly requires (line 107): *"Multi-task batches: treat the batch as a checklist… Emit a SEPARATE `task_updates` entry per task carrying that task's own concrete evidence… If you find yourself about to report several tasks done without distinct per-task evidence, stop."*

MiMo's weaker instruction-following on long prompts makes it likely to:
- Collapse evidence across tasks (reusing one task's `files_changed` for another)
- Omit `task_updates` entries entirely for some tasks
- Report `pending` status despite explicit prohibition (line 106)

**C. Checkpoint file read-modify-write (MEDIUM RISK)**

Checkpoint requirements (lines 271-287) ask the model to read the checkpoint JSON, update a specific task's fields, and write the full file back. MiMo is likely to either skip checkpoints entirely or produce malformed checkpoint JSON that breaks timeout recovery.

**D. finalize.json consumption fidelity (MEDIUM RISK)**

The prompt injects the entire projected `finalize.json` (line 518) as context. This can be very large for plans with many tasks + sense checks + user actions + validation metadata. MiMo's context window pressure from this payload may cause it to drop tasks or ignore `baseline_test_failures` entries.

**E. Doc-mode `sections_written` vs code-mode `files_changed` (LOW-MEDIUM RISK)**

The doc-mode schema (`execution_doc.json`, runtime.py line 1068-1086) swaps `files_changed` → `sections_written` at the schema level. The merge layer handles both, but MiMo may conflate these across modes, especially if given code-mode examples.

**F. Status enum adherence (LOW RISK, well-mitigated)**

The status alias map (`status_constants.py`) handles `completed` → `done`, `complete` → `done`, `skip` → `skipped`, `verified` → `done`. This is good. The `normalize_execute_task_status` in capture + merge is a solid two-layer defense.

### 3. Would Smaller Per-Task Templates/Checkpoints Help?

**Yes, significantly.** Specific recommendations:

**Per-task execution would reduce MiMo's biggest risk — multi-task evidence attribution.** Currently, batches of up to 5 tasks are sent in one prompt. For MiMo:
- **Single-task batches** (`max_tasks_per_batch=1`) would eliminate cross-task evidence confusion entirely. The cost is more round trips, but each round is simpler and more reliable.
- **Per-task checkpoint writes** already exist conceptually (the checkpoint path is per-batch via `batch_artifact_path`). A per-task checkpoint after each completed task would make timeout recovery granular and reduce the blast radius of mid-batch failures.
- **Template-based task_update generation**: Instead of asking MiMo to produce the full JSON shape, a lighter template like `"task_id=?; status=?; files_changed=[]; commands_run=[]; executor_notes=?"` per task would reduce schema conformance failures. The harness already normalizes aggressively, so this would slot in.

**The batch prompt is already very long** (~700 lines of prompt construction in `execute.py`). Splitting it into a per-task micro-prompt with just the single task's description, dependencies, and sense checks would dramatically reduce context pressure on MiMo.

**Summary**: The execute pipeline has robust normalization and recovery, but MiMo's structural output reliability is the key risk. The highest-leverage change would be defaulting to `max_tasks_per_batch=1` for smaller models and adding per-task checkpoint granularity. The existing merge/normalize layer is strong but should be hardened with the alias-collision and pending-status tests above.
