## Finalize Boundary Investigation Report

### 1. Boundary Verdict: **RISKY** for MiMo

The finalize boundary is structurally sound for strong models (Claude-class) but has specific fragility vectors that disproportionately affect weaker / instruction-following-challenged models like MiMo. It is not "broken" — when the model produces valid output, the handler processes it correctly. But MiMo is likely to fail at the **capture-audit** boundary before the handler's own, more lenient validation ever runs.

**Specific risks:**

- **Prompt-to-capture-schema field gap.** The prompt (`prompts/finalize.py:107-120`) tells the model to produce 9 fields per task (`id`, `description`, `depends_on`, `status`, `executor_notes`, `reviewer_verdict`, `kind`, `complexity`, `complexity_justification`). But the capture audit (`schemas/runtime.py:586-598`) validates against `SCHEMAS["finalize.json"]`, which requires **11 fields** per task — adding `files_changed`, `commands_run`, and `evidence_files`. If MiMo follows the prompt literally and omits those three fields, the capture audit rejects the output with `ModelStructuralAuditError` **before** the handler's `_validate_finalize_payload` (which uses the more lenient `_FINALIZE_INPUT_SCHEMA` and would accept the output) ever runs. Claude likely infers the missing fields from schema familiarity; MiMo will not.

- **No default-injection normalizer.** The execute step's normalizer (`_normalize_execute_capture_payload`) adds `setdefault` for `files_changed`, `commands_run`, and `auto_attributed_files`. The finalize normalizer (`_normalize_native_capture_payload` for `step == "finalize"`) only strips null optionals (`stance`, `stop_signal`). It does **not** inject defaults for the three missing fields. This means the gap between prompt and schema has no compensating safety net.

- **170-line prompt, dense structure.** The finalize prompt is the longest in the pipeline. It asks the model to simultaneously: parse a plan, emit a DAG, score complexity with a multi-paragraph rubric, shape batches, manage doc-mode vs code-mode invariants, and produce sense-checks. MiMo's effective instruction-following context is narrower than Claude's. At 170 lines, the risk of partial compliance (emitting some fields, hallucinating others, or dropping into prose) is elevated.

- **MiMo model-family budget is supported** — `ModelFamily.MIMO` exists (`model_seam.py:95`), it has `_FAMILY_BUDGET_DEFAULTS` and a DeepSeek-V3 tokenizer fallback (`model_seam.py:465`). The budget path is not a bottleneck. The risk is purely at the semantic/structural output layer.

### 2. Whether Template-Fill Can Make Finalize Reliable

**Partially. A template reduces but does not eliminate the risk.**

A concrete JSON template injected into the prompt — with all required fields pre-populated as empty/placeholder values — would solve the field-omission problem definitively. For example:

```json
{
  "tasks": [
    {
      "id": "T1",
      "description": "<fill>",
      "depends_on": [],
      "status": "pending",
      "executor_notes": "",
      "reviewer_verdict": "",
      "files_changed": [],
      "commands_run": [],
      "evidence_files": [],
      "kind": "code",
      "complexity": 1,
      "complexity_justification": "<fill>"
    }
  ],
  "watch_items": [],
  "sense_checks": [
    {"id": "SC1", "task_id": "T1", "question": "<fill>", "executor_note": "", "verdict": ""}
  ],
  "user_actions": [],
  "meta_commentary": "<fill>"
}
```

**What it solves:** The three missing fields (`files_changed`, `commands_run`, `evidence_files`) and the `user_actions`/`meta_commentary` top-level fields that the stored schema requires but the handler's input schema doesn't.

**What it does not solve:** The model still needs to reason about the plan structure, produce a valid DAG, score complexity, and produce sense-checks. Template-fill helps with "shape compliance" but not "semantic compliance." A bad DAG or wrong `depends_on` references would still break execution. The batch-shaping guidance (lines 98-104) requires judgment that template-fill cannot substitute.

**Recommendation:** Add a JSON template AND reduce the prose instructions to field-by-field guidance. The template alone is the highest-ROI change.

### 3. Whether Finalize Should Be Split into Outline → Structured Conversion

**Yes, but only for the MiMo tier. Splitting would significantly improve reliability.**

**Rationale:** The current prompt asks for a single-step cognitive leap: read a plan (potentially 5-10 steps with gate commentary), decompose it into tasks, shape the DAG, score complexity, produce sense-checks, and emit valid JSON — all in one turn. For MiMo, this is a high-cognitive-load task that benefits from decomposition.

**Proposed split:**

**Phase 1 — Outline (lightweight, free-form → structured):**
- Input: approved plan text
- Output: ordered list of `{id, description, depends_on, kind}` per plan step
- No complexity scoring, no sense-checks, no DAG optimization
- Validates: task count ≥ plan step count, depends_on references valid

**Phase 2 — Enrichment (structured → fully scored):**
- Input: Phase 1 output
- Output: full task objects with `complexity`, `complexity_justification`, `status`, `executor_notes`, `reviewer_verdict`, `files_changed`, `commands_run`, `evidence_files`
- Plus: `sense_checks`, `watch_items`, `meta_commentary`, `user_actions`
- Validates: full stored schema

**Benefits:**
- Each phase has ~50% of the cognitive load
- Phase 1 failure is cheaper (no complexity scoring wasted)
- Phase 2 is closer to template-fill (structured input → structured output)
- Phase 1 output can be cached/reused if Phase 2 fails

**Cost:**
- Extra LLM call (latency + cost)
- More handler code
- Phase 1 output needs its own schema

**Verdict:** Worth it for MiMo-tier routing. For Claude, the single-step prompt works fine. The runtime's tier-based routing could dispatch to single-step (Claude) or two-step (MiMo) based on the model family.

### 4. Schema/Prompt Contradictions Remaining

**Contradiction 1: Missing task fields in prompt vs stored schema.**
The prompt lists 9 required fields per task. The stored schema (`SCHEMAS["finalize.json"]`, lines 586-598) requires **11** — adding `files_changed`, `commands_run`, `evidence_files`. The handler's `_FINALIZE_INPUT_SCHEMA` (lines 246-285) only requires 5 (`id`, `description`, `status`, `complexity`, `complexity_justification`). Three different field sets; the strictest one (capture audit) wins at runtime.

**Contradiction 2: Doc-mode `sections_written` vs stored schema.**
The doc-mode prompt (line 33) says: *"Task objects use `sections_written` (array of section IDs) instead of `files_changed`/`commands_run`."* But the stored finalize schema has no `sections_written` property on tasks and still requires `files_changed` and `commands_run`. A doc-mode model that follows the prompt and omits `files_changed`/`commands_run` will fail the capture audit. The execute step has a `_build_execution_doc_schema()` that swaps these fields, but no equivalent exists for finalize.

**Contradiction 3: `meta_commentary` and `user_actions` not enforced by input schema.**
The stored schema requires `user_actions` and `meta_commentary` (line 673). The handler's `_FINALIZE_INPUT_SCHEMA` does not. The prompt asks for `meta_commentary` (line 163) and `user_actions` (lines 96, 162) but the model can omit them and pass the handler's validation — only to fail at write-time if downstream consumers enforce the stored schema.

**Contradiction 4: `kind` not in input schema.**
The prompt lists `kind` as a required task field (line 114). The stored schema has it with an enum (lines 571-574). The `_FINALIZE_INPUT_SCHEMA` does not mention `kind` at all — so the handler would accept tasks without it, but downstream consumers expecting the stored schema shape would be surprised.

**Contradiction 5: `depends_on` not required by input schema.**
The prompt says "Every task object must include `depends_on`" (line 110). The stored schema requires it. The `_FINALIZE_INPUT_SCHEMA` does not require it. A model that omits `depends_on` passes handler validation but fails capture audit.
