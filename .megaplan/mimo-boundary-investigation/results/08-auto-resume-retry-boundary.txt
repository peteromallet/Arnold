## Auto/Resume/Retry Boundary Report: Malformed Model Output

### 1. Boundary Verdict: **Risky** for fast models with malformed outputs

The auto driver's handling of `malformed_model_output` is structurally correct but **unbounded in practice for a specific failure mode**: a fast model that consistently produces malformed structured output for a non-terminal phase (e.g., finalize while state is `gated`).

**The loop:**
- `auto.py` L3386–3393: when `exit_kind` is `malformed_model_output` or `internal_error`, the driver does **not bail immediately** — it records a `phase_failed` lifecycle failure with a generic `resume_cursor={"phase": ..., "retry_strategy": "rerun_phase"}` and loops.
- On the next iteration, `status()` returns `state=gated, next_step=finalize` (because state was never advanced).
- The phase is re-dispatched, produces the same malformed output, and the cycle repeats.
- Termination depends **entirely on stall detection** (`DEFAULT_STALL_THRESHOLD = 5`): same state for 5 consecutive iterations with no new progress events in the journal → `stalled` exit.
- For a fast model (<1s per call), this means ~5 wasted LLM calls before the driver gives up. For a slower model, the wall-clock cost is higher but bounded.

**Why it's not broken:** stall detection does eventually fire, and the quarantine logic (`_quarantine_phase_outputs` L1844) prevents half-written output files from being consumed by the re-dispatched phase.

**Why it's risky:** The stall threshold is the *only* guard. There is no per-phase malformed-output counter, no escalation (e.g., try a stronger model), and no strategy switch. The `RecoveryPolicy.classify` method never even matches `malformed_model_output` — it falls through to the `unclassified → halt` default (L254–258 in recovery_policy.py), but the auto driver bypasses RecoveryPolicy for this exit kind entirely.

### 2. Should auto stop earlier or switch strategy?

**Yes — auto should classify repeated malformed output differently from a generic phase failure.** Specific recommendations:

| Gap | Recommendation |
|-----|---------------|
| **No malformed-output budget** | Add a `malformed_output_count` counter (per-phase, similar to `context_retry_count`). After N malformed failures on the same phase (e.g., 2), escalate or halt. The `auto_escalation.py` already maps `malformed_model_output` → `FailureCategory.internal_error` with `escalate=True, retries_first=1` — but this policy is only used for **execute** tier escalation, not for the generic phase loop. |
| **No strategy switch** | After 1 malformed failure, the re-dispatched phase should include a different prompt or temperature. Currently the exact same call is retried with a quarantined output file. A `retry_strategy: "fresh_session"` (already available in the resume_cursor enum) would be more appropriate than `rerun_phase`. |
| **Stall detection is too coarse** | For fast models, 5 iterations of wasted calls may be fine. For expensive models, it's not. A per-phase malformed-output cap of 2 would be tighter and more intentional than relying on the generic stall threshold. |

### 3. Missing state/failure metadata

The `phase_failed` lifecycle failure recorded at auto.py L3403–3413 captures:
- `exit_code`, `stderr`, `stdout` (truncated to 400 chars), `iteration`
- `resume_cursor` with generic `retry_strategy: "rerun_phase"`

**Missing for provider-specific diagnosis:**

| Missing field | Why it matters |
|---|---|
| **`exit_kind`** | The failure records `kind="phase_failed"` but does **not** persist the `exit_kind` (`malformed_model_output` vs `internal_error`). A human reading state.json cannot distinguish a parse failure from a crash. |
| **`model` / `provider`** | Not recorded in the failure metadata. For provider-specific failures (e.g., a model that truncates JSON), knowing which model was used is critical. The model is in `active_step` at call time but not copied into the failure record. |
| **`raw_output` / `raw_output_hash`** | The malformed output itself is not preserved (quarantining renames the file but the failure record doesn't link to it). A `raw_output_hash` or path to the quarantined file would help post-mortems. |
| **`consecutive_malformed_count`** | No counter tracks how many times the same phase has produced malformed output. The stall counter is a proxy but conflates malformed output with other no-progress scenarios. |
| **`session_id` / `invocation_id`** | The PhaseResult has an `invocation_id` but it's not threaded into the lifecycle failure. Hard to correlate failure records with LLM call logs. |

### 4. Tests that should be added

1. **`test_auto_stalls_on_repeated_malformed_finalize`** (characterization corpus)
   - A new `tests/characterization/auto_drive_corpus/malformed_finalize_loop.json` fixture where finalize returns `exit_kind=malformed_model_output` repeatedly while state stays `gated`. Assert that the driver exits with `status="stalled"` after exactly `stall_threshold` same-state iterations (not more).

2. **`test_malformed_output_records_exit_kind_in_lifecycle_failure`**
   - Verify that when a phase exits with `malformed_model_output`, the persisted `lifecycle_failures` entry includes `exit_kind: "malformed_model_output"` in its metadata (currently it does not — this test will fail and serve as the implementation prompt).

3. **`test_recovery_policy_classifies_malformed_model_output`**
   - `RecoveryPolicy.classify` currently falls through to `unclassified → halt` for `malformed_model_output`. Add a test that asserts the expected decision (e.g., `retry_fresh` with a new `malformed` budget kind, or explicit `halt` with `halt_kind="malformed_output_exhausted"`).

4. **`test_malformed_output_escalates_after_budget`**
   - Unit test: after N malformed failures on the same phase, the driver should either escalate (try a different model for finalize, if tier routing supports it) or halt with a distinct status — not rely solely on stall detection.

5. **`test_quarantine_does_not_quarantine_parseable_nonempty_output`**
   - The current quarantine logic at L1877–1882 explicitly **keeps** non-empty parseable dicts/lists. Add a test confirming this behavior and that a re-dispatched phase can correctly reject (rather than "recover") a valid-JSON-but-semantically-wrong output file. This is the edge case where quarantine helps least.
