## Critique Boundary Report

### 1. Boundary verdict: **RISKY** for MiMo

The defense-in-depth architecture has **two independent normalizers** that both handle `severity_hint` aliases:

| Layer | Function | Location | `.strip().lower()`? | Catch-all for unknowns? |
|-------|----------|----------|---------------------|--------------------------|
| Capture path | `_normalize_critique_flag` | `model_seam.py:628` | **No** — exact match only | **No** — unknown values pass through unchanged |
| Recovery path | `_normalize_critique_recovery_flag` | `handlers/critique.py:795` | **Yes** | **No** — unknown values pass through unchanged |
| Registry write | `normalize_flag_record` | `flags.py:51` | No (or-check) | Yes — defaults to `"uncertain"`, but runs **too late** (after schema audit) |

**What works:** The explicit alias sets cover the common MiMo output (`"significant"`, `"major"`, `"high"`, `"low"`, `"minor"`, `"medium"`, `"moderate"`). The existing tests (`test_recovered_critique_normalizes_flag_severity_aliases` in `test_critique.py:53` and `test_capture_step_output_normalizes_critique_severity_hint_aliases` in `test_model_seam.py:699`) verify these paths.

**What's broken or risky:**

1. **Casing variants bypass the capture-path normalizer.** The model_seam normalizer at line 631 compares `severity_hint in {"high", "significant", ...}` without `.lower()`. If MiMo emits `"High"`, `"SIGNIFICANT"`, or `" Likely-Significant "`, the capture-path normalizer silently passes it through. Only the recovery normalizer (which does `.strip().lower()`) would catch it — and recovery only fires if the main payload also fails schema audit.

2. **No catch-all fallback in either normalizer.** If MiMo emits an unexpected string like `"important"`, `"severe"`, `"notable"`, or `"critical-important"`, both normalizers leave it unchanged. The schema enum `["likely-significant", "likely-minor", "uncertain"]` rejects it. The handler falls into recovery, which also leaves it unchanged. The whole critique step fails.

3. **`category` has no pre-audit normalization.** The schema allows `["correctness", "security", "completeness", "performance", "maintainability", "doc-quality", "other", "verifiability"]`. If MiMo emits `"testing"`, `"reliability"`, `"robustness"`, or `"code-quality"`, schema audit fails. `normalize_flag_record` in `flags.py:53` does map unknowns to `"other"`, but that runs at registry-write time — **after** schema validation has already rejected the payload.

4. **`flagged` bool coercion is absent.** Findings require `"flagged": boolean`. If MiMo emits `"flagged": "yes"`, `"flagged": "true"`, or `"flagged": 1`, schema audit fails. No normalizer handles string-to-bool coercion.

5. **`category` value `"tests"` is not in the schema enum.** The test at `test_critique.py:68` uses `"category": "tests"` for FLAG-002, but the schema at `runtime.py:533` doesn't include `"tests"` — this would only pass because the test runs `_normalize_critique_payload_for_recovery` directly (not the full schema audit). In a live run, `"tests"` would fail schema validation.

### 2. Whether template-fill would help

**Yes, it would help directly.** The critique prompt at `prompts/critique.py:540` tells the model to:

> Use the standard format (id, concern, category, severity_hint, evidence).

But it never enumerates the valid enum values. The output template at `_write_critique_template` writes `"flags": []` — an empty array with no schema hint.

**What the template should contain** (one example entry in the `flags` array):

```json
{
  "flags": [
    {
      "id": "FLAG-NNN",
      "concern": "<describe the concrete issue you found>",
      "category": "<one of: correctness, security, completeness, performance, maintainability, other>",
      "severity_hint": "<one of: likely-significant, likely-minor, uncertain>",
      "evidence": "<specific file/line/behavior you observed>"
    }
  ]
}
```

Alternatively, or in addition, the prompt text at line 540 should be updated from:

> Use the standard format (id, concern, category, severity_hint, evidence).

to:

> Use the standard format. `severity_hint` must be exactly one of: `likely-significant`, `likely-minor`, `uncertain`. `category` must be exactly one of: `correctness`, `security`, `completeness`, `performance`, `maintainability`, `other`.

This is a prompt-only change and is the lowest-risk fix.
