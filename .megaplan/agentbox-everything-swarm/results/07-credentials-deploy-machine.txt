## AgentBox Inventory: What Exists, What's Missing

### 1. CLOUD PROVISIONING — HIGH REUSE (80%)

Three provider backends exist, all sharing a clean `Provider` protocol:

| File | What it Does | Reuse for AgentBox |
|---|---|---|
| `cloud/providers/ssh.py` | Docker lifecycle on any reachable SSH host: `docker build/run/exec/rm`. Syncs deploy dir via rsync (fallback: scp). Injects `.env` secrets. `ssh_exec()` and `upload_file()` primitives. | **Directly usable for Hetzner.** Add a Hetzner-specific wrapper that discovers the SSH host via hcloud API, then delegates to this provider. |
| `cloud/providers/railway.py` | Railway CLI wrapper: `link`, `variables --set`, `up --detach --ci`, `ssh --session`, volume management. Structured `DeployReport` with step-by-step status. | **Directly usable.** Already production-hardened for Railway. |
| `cloud/providers/local.py` | Docker Compose local iteration. | Useful for testing. |

**Gap: No Hetzner provider.** The `ssh.py` provider is 90% of it — you need a thin `HetznerProvider` that creates the cloud VM (hcloud server create), waits for SSH readiness, then wraps `SshProvider` for the rest. No volume abstraction yet for Hetzner block storage.

**Reuse verdict:** Write a ~150-line `cloud/providers/hetzner.py` inheriting from `SshProvider` or delegating to it. The SSH transport, Docker lifecycle, secrets injection, and OAuth seeding all work as-is.

### 2. CREDENTIAL SYNC — HIGH REUSE (90%)

Two layers exist:

**Layer 1: Deploy-time secrets** (`cloud/spec.py` → `secrets: [OPENAI_API_KEY, GITHUB_TOKEN, ...]`). The deploy step reads local env vars and pushes them to Railway variables or remote `.env`. Redacted from logs.

**Layer 2: OAuth seed at deploy time** (`cloud/auth.py`):
- `seed_codex_oauth()` — reads `~/.codex/auth.json` and `~/.hermes/auth.json` from the operator's machine, base64-encodes them, and `ssh_exec`s them into `/workspace/.creds/` + `/root/.codex/` on the cloud box
- Surface-level seed: deploy-time only (not runtime refresh)

**Layer 3: Boot-time restore** (`entrypoint.sh.tmpl` lines 26-35):
```bash
install -m 600 /workspace/.creds/codex-auth.json /root/.codex/auth.json
install -m 600 /workspace/.creds/hermes-auth.json /root/.hermes/auth.json
```
Persistent volume → ephemeral `/root` on every container restart.

**Layer 4: Claude refresh-token shim** (`template.py` `_claude_auth_block()`, ~100 lines of bash):
- `CLAUDE_CODE_REFRESH_TOKEN` env var → persisted to `/workspace/.claude-creds/refresh_token`
- `claude-key-helper` cron-refreshes OAuth access token, persists rotation
- `/usr/local/bin/claude` shim: refresh → export `ANTHROPIC_API_KEY` → exec real binary
- Graceful fallback to raw `ANTHROPIC_API_KEY`

**Gap: No runtime credential refresh for Codex.** The Codex OAuth seed is one-shot at deploy time. If a box lives long enough for the Codex token to expire, there's no refresh mechanism (unlike Claude's shim). This matters for long-running AgentBox instances.

**Gap: No DeepSeek credential path.** `cloud/auth.py` only handles codex/hermes OAuth seeds. DeepSeek uses an API key model — add `DEEPSEEK_API_KEY` to the standard secrets list and wire it similarly.

### 3. RUNTIME ENVIRONMENT — HIGH REUSE (85%)

`entrypoint.sh.tmpl` is a battle-tested 200+ line entrypoint that already does:
- Git clone-if-missing with branch pinning
- GitHub credential helper from `GITHUB_TOKEN`
- Codex/Hermes OAuth restore from persistent volume
- Claude refresh-token shim installation
- `arnold[agent]` pip install at pinned ref (auto-refresh on restart)
- Agent routing configuration (`arnold config set agents.*`)
- Heartbeat watchdog in its own tmux session
- Health server (keeps Railway alive)
- Mode-specific runner (auto/chain/idle) launched in tmux `agent` session

**What AgentBox needs added:**
1. DeepSeek API key → env var, exported for subagent use
2. A `discord` mode (analogous to `auto`/`chain`/`idle`) that launches the Discord resident agent
3. `mp-refresh-megaplan` script already exists for runtime re-installs

**Reuse verdict:** The entrypoint template is the single most reusable artifact. Add Discord mode, DeepSeek env, and you have an AgentBox runtime.

### CONCRETE GAPS & RECOMMENDATIONS

| Gap | Severity | Fix |
|---|---|---|
| **No Hetzner provider** | High | Write `cloud/providers/hetzner.py` (~150 lines): hcloud API to create server, wait for SSH, delegate to SshProvider |
| **No Codex OAuth refresh** | Medium | Add a `codex-key-helper` shim analogous to the Claude one, or restart boxes before token expiry |
| **No DeepSeek credential path** | Medium | Add `DEEPSEEK_API_KEY` to secrets list + export in entrypoint |
| **Discord command surface unclear** | High | Audit `resident/discord.py` — does it accept `/start ticket` commands, or is it status-only? |
| **No Discord → cloud binding** | High | The resident agent runs locally; needs a mode where it orchestrates cloud boxes (start Hetzner/Railway, inject creds, trigger chains, collect results) |
| **Guardian needs cloud awareness** | Medium | Watchdog operates on local tmux; needs to run over SSH or from a cloud control plane |
| **No multi-box orchestration** | Low | Currently one box per `cloud.yaml`; AgentBox implies multiple concurrent boxes from Discord |

### TOP 3 ACTIONS

1. **Write the Hetzner provider** — reuse SshProvider's Docker lifecycle. This unlocks 80% of the "Discord → provision box → run work" loop.

2. **Audit and extend `resident/discord.py`** — confirm it can ingest commands (start ticket, start epic, start chain) and map them to cloud provisioning + store creation. This is the Discord-first surface.

3. **Add a Discord mode to `entrypoint.sh.tmpl`** — a `discord` runner mode that launches the resident agent in tmux, wired to receive work items and self-orchestrate. Combined with the Hetzner provider, this gives you "type `/start epic` in Discord → box spins up on Hetzner → work runs → pushes PR."
