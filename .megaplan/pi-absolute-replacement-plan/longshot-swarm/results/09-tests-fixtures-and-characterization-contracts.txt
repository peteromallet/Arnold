## Implicit Contracts Found in Tests & Fixtures (Migration Must Preserve)

### 1. Agent Dispatcher Model Override Must Be Double-Declared (agent)

**Source:** `tests/test_agentic_scenarios.py` (FLAG-002)

Every agent with `dispatcher: hermes` or `dispatcher: deepseek-subagent` must specify **both** `model` and `config.model` to the identical value. The test explicitly documents that sisypy's `HermesDispatcher` has a hardcoded default and the _only_ override path is `extra_config["model"]` from `agent_spec.config`. If `config.model` is missing, the scenario silently runs on the expensive model.

**Hazard:** A migration that changes dispatcher wiring or model config propagation could silently cause all hermes/deepseek-subagent agents to regress to `deepseek-v4-pro` at ~10× cost, with no runtime error.

**→ Epic:** Agent Runtime Extraction / **Gate:** M7 pre-merge regression suite.

### 2. WorkerResult → AgentResult Telemetry Projection Must Be Lossless (codex, shannon, hermes)

**Source:** `tests/test_codex_adapter.py` (line 56–89), `tests/test_shannon_adapter.py` (line 54–72), `tests/arnold/agent/test_shannon_adapter.py`, `tests/workers/test_hermes_execute_recovery.py`

Every field crossing the `WorkerResult → AgentResult` bridge must survive intact: `raw_output`, `payload`, `session_id`, `cost.cost_usd`, `model_actual`, `tokens.*`, `duration_ms`, `rate_limit`, `shannon_plan`, and `provenance.*`. A migration that changes either side of the bridge without a matching projection test will silently drop telemetry or cost tracking.

Additional hermes-specific contract: `_reconstruct_execute_payload` must prefer the **current** batch output (`execute_batch_N_output.json`) over prior batch checkpoints (`execution_batch_M.json`) when both exist — tested explicitly in `test_reconstruct_execute_payload_prefers_current_batch_output`.

**→ Epic:** Unified Agent Dispatcher / **Gate:** M11 cross-adapter telemetry parity test.

### 3. Hermes Tool-Call Markup Normalization Supports Five Dialects (hermes)

**Source:** `tests/workers/test_hermes_tool_markup.py`

The `_normalize_response_content_tool_calls` function must recover from **five distinct tool-call markup formats** that Hermes agents emit in content (not native tool_calls):
1. Self-closing XML aliases (`<file_read path="..." />` → `read_file`)
2. Invoke-style XML (`<invoke name="read">...`)
3. Write-file-XML-wrapped JSON payloads
4. DSML-prefixed invoke (`<｜DSML｜invoke name="write_file">...`)
5. Bash heredoc (`<bash>cat <<'EOF' > file.json\n...\nEOF</bash>`)

Alias resolution maps `file_read` and `invoke name="read"` → `read_file`. Migration that replaces the hermes worker must replicate all five recovery paths and all alias mappings.

**→ Epic:** Agent Runtime Extraction / **Gate:** M7 hermes-markup-regression.

### 5. Fanout Boundaries: Phase Modules Must Not Own Raw Agent Fanout (fanout)

**Source:** `tests/characterization/test_agent_fanout_boundaries.py`

A whitelist of three modules may own raw fanout (`hermes_fanout.py`, `process_fanout.py`, `worker_fanout.py`). All other orchestrator/handler/review modules are prohibited from importing `concurrent.futures`, `multiprocessing`, `ThreadPoolExecutor`, `ProcessPoolExecutor`, `AIAgent`, or raw fanout workers. This is enforced via AST scan, not runtime.

Fixture-backed: `tests/fixtures/workflow_authoring/invalid_parallel_fanout.py` encodes that parallel fanout in authoring DSL is rejected.

**→ Epic:** Pipeline Unification / **Gate:** Fanout boundary AST scan must pass on migrated code.

### 7. Routing-Owned State Key Inventory: No Undocumented Handler Writes (agent)

**Source:** `tests/arnold_pipelines/megaplan/test_routing_source_invariants.py`

An audited inventory (`_INVENTORIED_HANDLER_WRITES`) of every handler write to routing-owned state keys (13 keys including `current_state`, `route`, `branch`, etc.) is enforced via AST scan. Any new write to a routing-owned key that isn't in the inventory causes test failure. Stale entries (inventoried but code removed) also fail. Migration that moves handler logic must maintain or update this inventory.

The companion test `test_no_native_module_embeds_megaplan_vocabulary` (line 264–306) ensures product-neutral native code contains no megaplan-specific string literals — a critical boundary for any migration that extracts native pipelines.

**→ Epic:** Pipeline Unification / **Gate:** SD1 routing-invariant scan.

### 8. Codex Writable Roots Must Reject Engine Overlap (codex)

**Source:** `tests/arnold_pipelines/megaplan/test_codex_writable_roots.py`

Codex's writable roots must be disjoint from the engine root. Three failure modes tested: equal paths, engine-contained-within-target, and user-configured `extra_writable_roots` that overlap. Self-hosted editable opt-in via `MEGAPLAN_ENGINE_ISOLATION_PROVIDER` is the only bypass. A migration that changes sandbox/writable-root computation without preserving this barrier could let a Codex agent overwrite the engine itself.

**→ Epic:** Agent Runtime Extraction / **Gate:** Codex sandbox safety gate.

### 9. State Delta CAS: Versioned Conflict Detection with No-Mutation-on-Conflict (agent)

**Source:** `tests/arnold_pipelines/megaplan/test_state_delta.py`

The state delta system implements optimistic concurrency with per-key version counters. Three critical properties tested: (a) stale-version raises `StateDeltaConflict` with `key`/`expected`/`actual` attrs, (b) state is **not mutated** on conflict (line 69–77), (c) missing `_state_meta` defaults version to 0 (line 129–135). Any migration that changes how state updates flow through handlers must preserve these CAS semantics exactly.

**→ Epic:** DBStore Foundation / **Gate:** State delta conformance suite.
