## 1. `sync-skills.sh` — Uncoordinated parallel skill installer
**File:** `sync-skills.sh` (lines 16–21, 28–92)  
**Why it's a surface:** This script independently symlinks skills from `arnold_pipelines/megaplan/skills/` into `~/.claude/skills/`, `~/.codex/skills/`, `~/.agents/skills/`, `~/.hermes/skills/` — four agent homes. It operates **outside** the `_GLOBAL_TARGETS` mechanism in `megaplan/cli/skills.py` and outside the `megaplan setup` flow. It has its own stale-link cleanup logic targeting `_codex_skills` (the retired source).  
**How to handle:** Either delete `sync-skills.sh` outright and fold its targets into `_GLOBAL_TARGETS`, or make it delegate to `megaplan setup --global`. The four agent directories it touches overlap partially but not completely with `_GLOBAL_TARGETS` (e.g., `.agents/skills/` and `.hermes/skills/` are in both, but `.claude/skills/` entries differ). In a unified facade, only one thing should write into agent skill directories.

## 2. `generate_arnold_docs.py` → `render_codex_skills()` — Generates Codex skills into retired directory
**File:** `scripts/generate_arnold_docs.py` (lines 51, 614–621)  
**Why it's a surface:** The `SKILLS_DIR` constant points to `arnold_pipelines/megaplan/data/_codex_skills/`, which is explicitly marked "retired" by `sync-skills.sh` (line 9) and is **not** in the `_GLOBAL_TARGETS` list. The generated composed rules (`render_composed_rules`, line 682–693) also write into `_codex_skills` sibling `_composed/`. These writes are commit-tracked (the files live in the repo), but the target directory is dead code from the skill-distribution perspective. The generator creates skills for *all* `discover_shipped_pipelines()` entries (not just the surviving migrated subset), potentially resurrecting deleted surfaces.  
**How to handle:** Align the generator's output directories with the live distribution targets. Either (a) make `render_codex_skills()` write to the canonical `_composed/` or package `data/` directories that `_GLOBAL_TARGETS` actually symlinks from, or (b) remove the function if the doc generator is not meant to produce installable skills.

---

## 3. `generate_arnold_docs.py` → `render_composed_skill()` — Teaches `arnold workflow` CLI as the launcher
**File:** `scripts/generate_arnold_docs.py` (lines 624–678, especially 638–646)  
**Why it's a surface:** The composed rules (for Claude, Codex, Cursor) contain a "Launcher" section that explicitly teaches agents: `arnold workflow --help`, `arnold workflow check --module <package.module>:build_pipeline`. If the unified facade changes the launch surface (e.g., moves from the CLI to an API or different entrypoint), these generated docs become stale instructions that agents will follow. They also enumerate every `discover_migrated_pipelines()` result, potentially leaking internal pipeline IDs.  
**How to handle:** Make the "Launcher" section a template parameter fed by the facade's canonical entrypoint, or replace it with a single pointer to a stable doc page that doesn't get regenerated with pipeline-specific detail.

---

## 4. `arnold_pipelines/megaplan/skills/subagent-launcher/SKILL.md` — Teaches direct `launch_hermes_agent.py` / `fan.py` / `codex exec` invocation
**File:** `arnold_pipelines/megaplan/skills/subagent-launcher/SKILL.md` (entire file)  
**Why it's a surface:** This is the **primary agent-launch surface** in the repo. It teaches three pathways (Hermes agentic via `launch_hermes_agent.py`/`fan.py`, Codex via `codex exec`, Claude via `launch_claude_agent.py`) that completely bypass any unified control facade. The skill is distributed symlink-installed by `_GLOBAL_TARGETS` (lines 206–209) into Claude, Codex, Hermes, and `.agents` directories. Agents reading this skill learn to shell out directly to subprocesses, not through a facade.  
**How to handle:** This is the hardest surface. The skill is intentionally a launch surface. In a unified facade plan, either: (a) rewrite the skill to route through the facade (e.g., `arnold agent launch --model=...` instead of raw `python launch_hermes_agent.py`), or (b) keep the skill as-is but accept that it's an explicit bypass — agents that read it will always have a direct launch path. The `launch_hermes_agent.py` and `fan.py` scripts themselves (lines 98–100 for runtime discovery) fall back to `~/Documents/Arnold` and legacy megaplan — another indirection that could survive a facade migration.

## 5. `megaplan/cli/skills.py` `_GLOBAL_TARGETS` — Codex targets use `dir_symlink` of entire skill directories, not composed files
**File:** `arnold_pipelines/megaplan/cli/skills.py` (lines 186, 189, 191, 193, etc.)  
**Why it's a surface:** Claude gets `_composed/claude_skill.md` (pre-baked), but Codex gets raw `_codex_skills/megaplan/` directories (lines 186, 189, etc.). Subagent-launcher and cleanup-loose-branches are installed as `dir_symlink` targets pointing at `skills/subagent-launcher/` (lines 206–213). This means the Codex skill directories and the Arnold-managed skills contain **all support files** (Python scripts, fan.py, launch_claude_agent.py), not just SKILL.md. A facade that only intercepts SKILL.md content won't catch the executable scripts these directories carry.  
**How to handle:** Decide whether the facade controls skill *content* only or also skill *executable payloads*. If the latter, symlinked directory installs are a hard problem — you'd need to intercept at the filesystem level or replace dir_symlink with a controlled copy mechanism.

## 6. `docs/skill-distribution.md` — Documents the old `_codex_skills` directory as canonical home
**File:** `docs/skill-distribution.md` (line 9: "megaplan/data/")  
**Why it's a surface:** Low-confidence concern. The ADR says the canonical home is `megaplan/data/`, but the actual canonical skill directories live under `arnold_pipelines/megaplan/skills/` (for subagent-launcher and cleanup-loose-branches). The ADR table (lines 31–39) doesn't include these Arnold-managed skills. If someone follows the ADR to extend skills, they'll put files in the wrong place.  
**How to handle:** Update the ADR to reflect the actual canonical home split (`megaplan/data/` for package skills, `skills/` for Arnold-managed ones) or unify them.

## 7. Generated docs (`docs/arnold/examples/*.md`) — Teach `arnold workflow` CLI directly
**Files:** `docs/arnold/examples/megaplan.md`, `docs/arnold/examples/jokes.md`, etc.  
**Why it's a surface:** These are auto-generated by `generate_arnold_docs.py` and contain "Validation" rows like `` `arnold workflow check --module ...` ``. They're not skills loaded by agents, but they are human-facing reference docs that establish the CLI as the authoritative interface. A facade that replaces the CLI must update or regenerate these.  
**How to handle:** Make the validation command a configurable template in the generator.

---

## 8. `launch_hermes_agent.py` — Runtime discovery fallback to `~/Documents/Arnold`
**File:** `arnold_pipelines/megaplan/skills/subagent-launcher/launch_hermes_agent.py` (lines 98–100)  
**Why it's a surface:** Long-shot concern. The launcher first tries a legacy `megaplan.agent` distribution, then falls back to the Arnold checkout at `~/Documents/Arnold` (or `ARNOLD_PATH`). A unified facade deployed as a different package or in a different location won't be found by this fallback, causing silent degradation to the legacy path.  
**How to handle:** Add the facade's install location to the discovery chain, or wire `ARNOLD_PATH`/a new env var.

---

**Summary ranking:** `sync-skills.sh` is the most immediate coordination gap (two parallel installers). The subagent-launcher SKILL.md and its scripts are the hardest architectural surface — they teach direct process launch, which is inherently a bypass surface that may need to be accepted rather than closed. The doc generator creates a maintenance burden by embedding CLI commands that must track any facade rename.
