I've completed a thorough survey of all pipeline/workflow surfaces outside the megaplan worker routes. Here are the ranked findings:

## Ranked Findings

### 1. **AgentStep + PanelReviewerStep — Worker Injection Surface** (HIGH)
**Paths:** `arnold/pipeline/steps/agent.py`, `arnold/pipeline/steps/panel.py`

Both steps accept a `_worker: WorkerFn` injected at compile time. The step calls `self._worker(prompt=..., step_name=..., pipeline_name=..., inputs=..., mode=...)` — a *de facto* contract between pipeline builder and agent dispatch. There is no formalized `WorkerFn` protocol beyond `Callable[..., Any]`. The megaplan bridge classes in `megaplan/_pipeline/steps/` inject concrete workers; the neutral core accepts anything.

**Recommendation:** **Epic 2 (Facade)** — Formalize `WorkerFn` as a typed protocol in `arnold.pipeline.contracts` with the exact kwarg surface, then enforce it in both AgentStep and PanelReviewerStep.

---

### 2. **Model Seam Hook Registries — Implicit Extension Contract** (HIGH)
**Path:** `arnold/pipeline/model_seam.py`

Four module-level hook tables (`_NATIVE_NORMALIZERS`, `_COMPATIBILITY_PROJECTIONS`, `_CAPTURE_SCHEMA_RESOLVERS`, `_RECOVERY_SHAPE_CHECKS`) are populated at *consumer import time* via `register_*()` functions. This is an invisible contract: if megaplan registers hooks and then migrates, the hook tables must be re-registered identically or things silently break.

**Recommendation:** **Epic 7 (Migration)** — Add hook-registration snapshot/verification to the migration checklist. The model seam's hook surface should be audited as part of consumer migration.

---

### 3. **Dynamic Fanout — `run_fanout()` Specialization Contract** (HIGH)
**Path:** `arnold/pipeline/pattern_dynamic.py`

`run_fanout()` accepts a `base_step` (duck-typed dataclass), specializes it per spec via `dataclasses.replace`, and dispatches via ThreadPoolExecutor. The `FanoutMetadata` bundle controls concurrency, specialization keys, and governor limits. `FanoutJoinContract` describes the join function. Any step type (including AgentStep, PanelReviewerStep) can be the `base_step`. No contract validation exists between the specialization spec and the base step's fields.

**Recommendation:** **Epic 2 (Facade)** — Add a `FanoutStepProtocol` or factory contract to `pattern_dynamic` that validates specialization compatibility at construction time.

---

### 4. **StepInvocationAdapter — Reserved Model Slot** (MEDIUM)
**Paths:** `arnold/execution/step_invocation.py`, `arnold/pipeline/model_seam.py` (lines 355-373)

`StepInvocationAdapterRegistry` starts with a reserved `"model"` slot (placeholder). `ModelStepInvocationAdapter.invoke()` delegates to `render_step_message()`. `install_model_step_adapter()` replaces the placeholder with the concrete adapter. If consumer migration misses this wiring, model steps fail silently.

**Recommendation:** **Epic 7 (Migration)** — Include `install_model_step_adapter` in the migration smoke test. Validate that the registry's `"model"` slot is occupied by a real (non-placeholder) adapter post-migration.

---

### 5. **Native Graph Projection → Pipeline Bridge** (MEDIUM)
**Path:** `arnold/pipeline/native/graph_projection.py`

Converts `NativeProgram` → `Pipeline`, mapping `ParallelMapInstruction` to fan-out stages, `_NativePhaseStep` to step adapters. Contains `FanoutMetadata` wiring and `_join_validators`. The projection is the runtime bridge between native-authored programs and the graph executor. Any change to the fan-out or step contracts breaks this bridge.

**Recommendation:** **Epic 1 (Inventory)** — Catalog graph_projection as a migration-critical bridge surface. Include its `FanoutMetadata` → `ParallelStage` mapping in the compatibility matrix.

---

### 6. **Pipeline Authoring Contract (`validate_package_module`)** (MEDIUM)
**Path:** `arnold/pipelines/_authoring.py`

Enforces that pipeline packages export `driver` (starting with `"native"`), `supported_modes` (containing `"native"`), `build_pipeline()` returning a `Pipeline` with non-null `native_program`. Also checks routing purity and rejects shim/fallback patterns. This is the *gate* for native-first packages — but it doesn't cover agent dispatch surfaces.

**Recommendation:** **Epic 1 (Inventory)** — Add `build_pipeline()` return type → agent-step usage audit to the inventory.

---

### 7. **Agent Contracts — Dispatch Protocol** (MEDIUM)
**Path:** `arnold/agent/contracts.py`

`AgentDispatcher` protocol (`dispatch(request: AgentRequest) -> AgentResult`), `scatter_agent_units()`, `FanoutUnit/FanoutResult`. These are the canonical wire-format contracts. Pipeline steps don't directly reference them; megaplan workers do. But the `AgentRequest.agent/mode/model/effort` fields are the profile surface.

**Recommendation:** **Epic 7 (Migration)** — Covered by profile migration if `AgentSpec → AgentMode → AgentRequest` resolution is migrated cleanly.

---

### 8. **Runtime CONTRACT.md — `profile_validate` Operation** (LOW)
**Path:** `arnold/runtime/CONTRACT.md` (line 27)

Declares `profile_validate` as one of six runtime operation kinds. This is the runtime contract's profile-validation surface. Currently a stub; the profile validation logic lives in megaplan (`resident/profile.py`).

**Recommendation:** **Epic 7 (Migration)** — Covered by profile migration. The `profile_validate` operation kind is already declared; the implementation moves with the consumer.

---

### 9. **Pipeline Discovery — `ShippedPipelineInfo`** (LOW)
**Path:** `arnold_pipelines/discovery.py`

`builder_contract` field tracks `workflow` / `native` / `deferred-native` per pipeline package. Used by the inventory scanner. Not an agent-launch surface per se, but tracks which pipelines use which builder family.

**Recommendation:** **Epic 1 (Inventory)** — Already part of the inventory scan.

---

### 10. **Workflow Manifest Validation — Forbidden Import Scanning** (LOW)
**Path:** `arnold/workflow/validation.py` (line 36-40), `arnold/conformance/workflow_manifest_runtime.py`

`FORBIDDEN_PRODUCT_IMPORTS` blocks megaplan imports in neutral packages. The conformance scanner enforces this at the AST level. This is a compile-time guard, not a runtime surface.

**Recommendation:** **Epic 1 (Inventory)** — Already inventoried.

---

## Answer: Separate Pipeline-Runtime Compatibility Contract?

**No new epic needed.** The surfaces are split across two existing epics:

- **Agent dispatch contracts** (WorkerFn, model seam hooks, fanout specialization, StepInvocationAdapter wiring) are *pipeline-internal* seams that need **Epic 2 facade formalization**.
- **Profile/agent resolution** (AgentSpec → AgentMode, `profile_validate`, resident profile) is **Epic 7 consumer migration**.

The one gap is the **model seam hook registries** — they're an invisible, import-time contract between the neutral `model_seam.py` and whichever consumer registers hooks. This isn't covered by either epic today. Recommend adding a **single Epic 2 task**: "Formalize `model_seam` hook registration as a versioned contract with snapshot verification."
