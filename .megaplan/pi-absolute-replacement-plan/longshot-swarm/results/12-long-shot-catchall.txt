## Potentially Missing Surfaces

### 1. Subagent Launcher Fan (`skills/subagent-launcher/fan.py`)

**Evidence:**
- A standalone script that fans out N hermes/AIAgent agents in a single process using `ThreadPoolExecutor` (line 6-8: "imports megaplan/hermes ONCE... shares a single SessionDB across ThreadPoolExecutor").
- Has its own model shortcut mapping (`_MODEL_SHORTCUTS`: `pro`, `flash`, `fast`, `mimo`, `kimi`, `glm` — lines 255-264) that is separate from the main `key_pool.resolve_model`.
- Bypasses the normal megaplan step pipeline entirely — constructs `AIAgent` instances directly with the OpenAI SDK (lines 217-243), including a timeout installation path that mutates the agent's internal OpenAI client kwargs (lines 460-493).
- Supports `--isolation=processes` for per-task SIGKILL surface (line 32-34).
- Has its own kill-script companion (`fan_kill.py`).

**Proposed Plan Treatment:** This is a distinct agent execution surface that needs its own inventory entry. The model shortcuts table, OpenAI SDK timeout mutation, and process-isolation kill surface have no coverage in existing briefs. Treat as a separate subagent-launcher execution path with its own model resolution and timeout/policy surface.

### 2. Resident Bot Agent Loop (`resident/agent_loop.py`)

**Evidence:**
- Defines a `DispatchProtocol` / `AgentRunner` protocol for resident bot turns with its own tool registry and `pydantic`-based tool call validation (lines 15, 61-67).
- Uses `ToolRegistry`, `ToolCallAuditRecord`, and `ToolRuntimeContext` — a separate tool-call audit surface from the main pipeline.
- Runs through `ModelSeam.render_step_message` for budget checks (line 18), so it interacts with the model seam but is an entirely separate dispatch path.

**Proposed Plan Treatment:** This is a second agent loop surface (beyond the main step pipeline and the subagent fan). Needs inventory as a resident-bot execution path with its own tool registry, schema validation, and turn-based dispatch protocol.

### 3. Guardian Background Worker (`agentbox/guardian/worker.py` + `handlers.py`)

**Evidence:**
- `GuardianWorker` runs asynchronously (`asyncio`), claims scheduled tasks (liveness, deep_supervision, briefing, reminders), and supervises non-terminal `megaplan_chain` operations (lines 76-123, 182-204).
- Has its own inspection timeout, heartbeat lease loop, retry caps, and resume logic (lines 46-51, 206-303).
- Can mark operations FAILED after hitting retry caps (lines 322-352).
- `MegaplanChainGuardianHandler` delegates through `load_operation_adapter` to the chain adapter, but the Guardian itself is a separate background supervision surface that runs outside the normal chain execution path.

**Proposed Plan Treatment:** This is an asynchronous supervision loop outside the main chain runner. It has its own lease/retry policy, inspection timeout, and failure-marking authority. Needs coverage as a background agent supervision surface.

### 4. Repair Runner Subprocess Surface (`watchdog/repair_runner.py`)

**Evidence:**
- `RepairRunner` executes allowlisted megaplan subcommands (`doctor`, `auto`, `resume`, `chain`) as `python -m arnold_pipelines.megaplan <subcommand> ...` via subprocess (lines 30-46).
- Has `--profile partnered-5` injection logic (lines 52-60).
- Runs system commands (`rm`, `kill`) directly alongside megaplan subcommands.

**Proposed Plan Treatment:** This is an allowlisted subprocess execution surface that launches megaplan as a child process. The allowlist (`_MEGAPLAN_SUBCOMMANDS`) and profile injection are surfaces not covered by existing briefs. Needs coverage as a repair subprocess execution path.

### 5. AgentBox Host Operation Type (`agentbox/adapters.py`)

**Evidence:**
- `AGENTBOX_HOST_OPERATION_TYPE = "agentbox_host"` is listed alongside `megaplan_chain` (line 10) but has no registered adapter in `_ADAPTERS`.
- This suggests a second operation type that AgentBox manages but whose adapter is handled elsewhere (possibly tmux-based host sessions).

**Proposed Plan Treatment:** If `agentbox_host` is a live operation type with its own execution surface (tmux sessions managed via `agentbox/tmux.py`), it needs inventory. The Guardian worker only supervises `megaplan_chain` — `agentbox_host` operations may have an unmonitored execution surface.
