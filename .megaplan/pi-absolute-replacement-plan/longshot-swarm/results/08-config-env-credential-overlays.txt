**Env alias chains** ‚Äî `KeyPool` at `arnold/agent/providers/pool.py` lines 38‚Äì46 maps multiple env names to the same key: `ZHIPU_API_KEY` ‚Üí also `GLM_API_KEY`, `KIMI_API_KEY` ‚Üí also `MOONSHOT_API_KEY`, `GEMINI_API_KEY` ‚Üí also `GOOGLE_API_KEY`. ### 3. Credential/Key Pool Gaps

**Dual-load path for `~/.hermes/.env`** ‚Äî Two different code paths read this file:
1. `KeyPool._load_hermes_env_unlocked()` ‚Äî manual line-by-line parse, reads into a dict, used for key pool lookups
2. `env_loader.load_hermes_dotenv()` ‚Äî uses `python-dotenv` to push into `os.environ`, used by Hermes CLI entrypoints

These have different precedence rules. The plan must determine which path(s) survive.

**Evidence:** `arnold/agent/providers/pool.py` lines 154‚Äì163 vs `arnold/agent/providers/env_loader.py` lines 19‚Äì46.

**Governor charging on key acquire** ‚Äî `KeyPool.acquire()` at `arnold_pipelines/megaplan/runtime/key_pool.py` line 54 calls `_charge_governor_for_current_envelope()`. This coupling between key pool and budget enforcement is not captured in the provider-router contract or Epic 3.

**Codex/Copilot token resolution** ‚Äî `auth.py` resolve_provider (line 691) handles OAuth, `gh auth token` CLI fallback, and env-var API keys in a priority chain. The plan's credential mediation design doesn't address OAuth token lifecycle.

**AgentBox separate credential registry** ‚Äî `agentbox/credentials/registry.py` has its own `KNOWN_CREDENTIALS` dict (`GITHUB_TOKEN`, `ANTHROPIC_API_KEY`, `CLAUDE_API_KEY`, `OPENAI_API_KEY`, `CODEX_API_KEY`, `DISCORD_BOT_TOKEN`) plus a push/test backend. This is a parallel credential system not connected to KeyPool ‚Äî must be unified or explicitly kept separate with a contract.

### 4. Override Flags ‚Äî Missing Pipeline Steps

**`--deepseek-provider`** ‚Äî A CLI flag at `arnold_pipelines/megaplan/cli/__init__.py` line 160 with choices `["direct"]`. Not mentioned anywhere in the plan or GAP_SWARM.

**`--phase-model` append semantics** ‚Äî `init_parser.add_argument("--phase-model", action="append")` at CLI line 199 allows multiple overrides (e.g., `--phase-model plan=codex:gpt-5.5 --phase-model critique=hermes:kimi:k2`). The plan should capture this as a list-accumulating override, not a single-value flag.

**`--adaptive-critique` / `--strict-adaptive-critique`** ‚Äî Flags on `init` (CLI line 190‚Äì191) that change critique behavior. These interact with profile resolution but are not in the override pipeline inventory.

**Full transformation order** ‚Äî `apply_profile_expansion` ‚Üí `apply_vendor_rewrite` ‚Üí `apply_critic_rewrite` ‚Üí `apply_depth_rewrite` (in `profiles/policy.py`). The GAP_SWARM notes reordering would silently change behavior. The plan should add a **rewrite-order contract test** as a gate deliverable.

### 5. Test Gaps

**No credential detection characterization tests** ‚Äî `has_usable_secret()` in `auth.py` line 329 and `_check_credential()` in `preflight.py` line 49 have no test coverage. The plan's `credential-resolution-flow.md` should include test contracts.

**Existing writable-root tests not referenced** ‚Äî `tests/arnold_pipelines/megaplan/test_codex_writable_roots.py` (105 lines) covers engine/target overlap, `self_hosted_editable` opt-in, and configured engine root denial. The plan mentions writable-root policy (Epic 2) but doesn't cite these existing test contracts as baselines.

**`resolve_model()` routing matrix untested** ‚Äî The OpenRouter blocking behavior and provider prefix routing in `key_pool.py` lines 240‚Äì329 has no characterization tests. Epic 3 requires provider parity tests ‚Äî these should include the current routing matrix.
